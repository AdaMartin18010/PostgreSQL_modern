## 📋 概述

一致性模型是分布式系统和数据库系统的核心概念，它定义了系统在并发访问和网络分区情况下，数据一致性的保证程度。

本文档详细阐述各种一致性模型的定义、特性、实现方式和PostgreSQL中的体现，为系统设计提供理论指导。

**核心观点**：

- 一致性模型存在**强度谱系**：从强一致性到最终一致性
- PostgreSQL的隔离级别对应不同的**一致性模型**
- 一致性选择需要在**正确性和性能**之间权衡
- CAP定理决定了**一致性模型的边界**

## 📊 第一部分：一致性模型谱系

### 1.1 一致性模型分类

**一致性模型分类树**：

```text
一致性模型
├── 强一致性（Strong Consistency）
│   ├── 线性一致性（Linearizability）
│   ├── 顺序一致性（Sequential Consistency）
│   └── 因果一致性（Causal Consistency）
│
├── 弱一致性（Weak Consistency）
│   ├── 最终一致性（Eventual Consistency）
│   ├── 会话一致性（Session Consistency）
│   └── 单调读一致性（Monotonic Read Consistency）
│
└── 无一致性（No Consistency）
    └── 无保证
```

### 1.2 一致性强度对比

| 一致性模型 | 强度 | 保证 | 性能 | 适用场景 |
|-----------|------|------|------|---------|
| **线性一致性** | 最强 | 全局顺序 | 最低 | 金融交易 |
| **顺序一致性** | 强 | 每个节点顺序 | 低 | 分布式锁 |
| **因果一致性** | 中等 | 因果关系 | 中 | 社交网络 |
| **最终一致性** | 弱 | 最终收敛 | 高 | 日志系统 |
| **无一致性** | 无 | 无保证 | 最高 | 缓存系统 |

### 1.3 一致性模型关系图

```text
线性一致性 (Linearizability)
    ↓ (包含)
顺序一致性 (Sequential Consistency)
    ↓ (包含)
因果一致性 (Causal Consistency)
    ↓ (包含)
最终一致性 (Eventual Consistency)
    ↓ (包含)
无一致性 (No Consistency)
```

**形式化关系**：

$$
\text{Linearizability} \subset \text{Sequential Consistency}
 \subset \text{Causal Consistency} \subset \text{Eventual Consistency}
$$

## 📊 第二部分：强一致性模型

### 2.1 线性一致性（Linearizability）

**定义**：

系统操作看起来像是原子地、按某个全局顺序执行，且每个操作都能立即看到之前所有操作的结果。

**形式化定义**：

对于操作序列$O = \{o_1, o_2, ..., o_n\}$，存在一个全局顺序$\prec$，使得：

$$
\forall o_i, o_j \in O: \quad \text{if } o_i \prec o_j \text{ then }
 \text{Response}(o_i) \text{ happens-before } \text{Invoke}(o_j)
$$

**PostgreSQL实现**：

```sql
-- SERIALIZABLE隔离级别实现线性一致性（带完整错误处理）
DO $$
BEGIN
    BEGIN
        BEGIN
            SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
            RAISE NOTICE '事务隔离级别已设置为 SERIALIZABLE（线性一致性）';
        EXCEPTION
            WHEN OTHERS THEN
                RAISE WARNING '设置事务隔离级别失败: %', SQLERRM;
                RAISE;
        END;

        RAISE NOTICE '线性一致性特征：';
        RAISE NOTICE '- ✅ 所有操作按全局顺序执行';
        RAISE NOTICE '- ✅ 读操作立即看到最新写入';
        RAISE NOTICE '- ✅ 无时间窗口不一致';
    EXCEPTION
        WHEN OTHERS THEN
            RAISE WARNING '操作失败: %', SQLERRM;
            RAISE;
    END;
END $$;
```

**性能影响**：

| 指标 | 影响 | 说明 |
|------|------|------|
| 延迟 | +100-500ms | 全局协调开销 |
| 吞吐量 | -50-80% | 串行化限制 |
| 可用性 | 降低 | 分区时可能阻塞 |

### 2.2 顺序一致性（Sequential Consistency）

**定义**：

所有节点看到相同的操作顺序，但不需要全局原子性。每个节点的操作按程序顺序执行。

**形式化定义**：

对于节点$n$的操作序列$O_n = \{o_1^n, o_2^n, ..., o_k^n\}$，存在全局顺序$\prec$，使得：

$$
\forall n, \forall i < j: \quad o_i^n \prec o_j^n
$$

**PostgreSQL实现**：

```sql
-- REPEATABLE READ隔离级别接近顺序一致性（带完整错误处理）
DO $$
BEGIN
    BEGIN
        BEGIN
            SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
            RAISE NOTICE '事务隔离级别已设置为 REPEATABLE READ（顺序一致性）';
        EXCEPTION
            WHEN OTHERS THEN
                RAISE WARNING '设置事务隔离级别失败: %', SQLERRM;
                RAISE;
        END;

        RAISE NOTICE '顺序一致性特征：';
        RAISE NOTICE '- ✅ 事务内操作顺序一致';
        RAISE NOTICE '- ✅ 不同事务可能看到不同顺序';
        RAISE NOTICE '- ✅ 无全局原子性保证';
    EXCEPTION
        WHEN OTHERS THEN
            RAISE WARNING '操作失败: %', SQLERRM;
            RAISE;
    END;
END $$;
```

**与线性一致性的区别**：

| 特性 | 线性一致性 | 顺序一致性 |
|------|-----------|-----------|
| 全局顺序 | ✅ 唯一全局顺序 | ✅ 全局顺序（但可能不同视角） |
| 原子性 | ✅ 操作原子 | ❌ 操作可能重叠 |
| 实时性 | ✅ 实时可见 | ❌ 可能延迟可见 |

### 2.3 因果一致性（Causal Consistency）

**定义**：

保证因果相关的操作在所有节点上按相同顺序执行，但无关操作可以乱序。

**形式化定义**：

对于操作$o_1$和$o_2$，如果$o_1 \rightarrow o_2$（因果相关），则：

$$
\forall n: \quad o_1 \prec_n o_2
$$

其中$\prec_n$是节点$n$的操作顺序。

**PostgreSQL实现**：

```sql
-- READ COMMITTED隔离级别提供因果一致性（带完整错误处理）
DO $$
BEGIN
    BEGIN
        BEGIN
            SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
            RAISE NOTICE '事务隔离级别已设置为 READ COMMITTED（因果一致性）';
        EXCEPTION
            WHEN OTHERS THEN
                RAISE WARNING '设置事务隔离级别失败: %', SQLERRM;
                RAISE;
        END;

        RAISE NOTICE '因果一致性特征：';
        RAISE NOTICE '- ✅ 相关操作顺序一致';
        RAISE NOTICE '- ✅ 无关操作可以乱序';
        RAISE NOTICE '- ✅ 允许读已提交数据';
    EXCEPTION
        WHEN OTHERS THEN
            RAISE WARNING '操作失败: %', SQLERRM;
            RAISE;
    END;
END $$;
```

**应用场景**：

- 社交网络：用户A关注用户B，然后看到B的更新
- 评论系统：回复必须出现在原评论之后
- 消息系统：消息的因果关系必须保持

## 📊 第三部分：最终一致性模型

### 3.1 最终一致性定义

**定义**：

如果系统不再接收新的更新，经过一段时间后，所有节点最终会收敛到相同的状态。

**形式化定义**：

$$
\lim_{t \to \infty} \forall n_i, n_j \in N: \quad \text{State}(n_i, t) = \text{State}(n_j, t)
$$

**PostgreSQL实现**：

```sql
-- 异步复制实现最终一致性（带完整错误处理）
DO $$
BEGIN
    BEGIN
        BEGIN
            ALTER SYSTEM SET synchronous_standby_names = '';
            RAISE NOTICE '同步备库名称已设置为空（最终一致性，异步复制）';
        EXCEPTION
            WHEN OTHERS THEN
                RAISE WARNING '设置同步备库名称失败: %', SQLERRM;
        END;

        BEGIN
            ALTER SYSTEM SET synchronous_commit = 'local';
            RAISE NOTICE '同步提交模式已设置为 local（最终一致性）';
        EXCEPTION
            WHEN OTHERS THEN
                RAISE WARNING '设置同步提交模式失败: %', SQLERRM;
        END;

        RAISE NOTICE '最终一致性特征：';
        RAISE NOTICE '- ✅ 主库立即提交';
        RAISE NOTICE '- ✅ 备库延迟同步';
        RAISE NOTICE '- ✅ 最终所有节点一致';
    EXCEPTION
        WHEN OTHERS THEN
            RAISE WARNING '操作失败: %', SQLERRM;
            RAISE;
    END;
END $$;
```

### 3.2 最终一致性的变体

#### 3.2.1 会话一致性（Session Consistency）

**定义**：

同一会话内的操作保证一致性，不同会话间可能不一致。

**PostgreSQL实现**：

```sql
-- 会话内一致性通过事务保证（带完整错误处理）
DO $$
BEGIN
    BEGIN
        BEGIN;
        RAISE NOTICE '开始事务（会话内一致性）';

        -- 会话内的所有操作保证一致性
        -- 这里可以执行多个操作

        COMMIT;
        RAISE NOTICE '事务提交成功（会话内一致性保证）';
    EXCEPTION
        WHEN OTHERS THEN
            ROLLBACK;
            RAISE WARNING '事务失败: %', SQLERRM;
            RAISE;
    END;
END $$;
```

#### 3.2.2 单调读一致性（Monotonic Read Consistency）

**定义**：

如果节点读取到某个值，后续读取不会看到更旧的值。

```sql

            WHEN OTHERS THEN
                RAISE WARNING '设置事务隔离级别失败: %', SQLERRM;
                RAISE;
        END;

        RAISE NOTICE '单调读特征：事务内不会看到更旧的值';
    EXCEPTION
        WHEN OTHERS THEN
            RAISE WARNING '操作失败: %', SQLERRM;
            RAISE;
    END;
END $$;
```

#### 3.2.3 单调写一致性（Monotonic Write Consistency）

**定义**：

同一节点的写入操作按顺序执行。

**PostgreSQL实现**：

### 3.3 最终一致性的收敛性

**收敛时间**：

$$

## 📊 第四部分：PostgreSQL MVCC的一致性级别

### 4.1 隔离级别与一致性模型的映射

**映射关系表**：

| PostgreSQL隔离级别 | 一致性模型 | CAP选择 | 保证 |
|-------------------|-----------|---------|------|
| **SERIALIZABLE** | 线性一致性 | CP | 全局顺序、无异常 |
| **REPEATABLE READ** | 顺序一致性 | CP | 事务内顺序、可重复读 |
| **READ COMMITTED** | 因果一致性 | AP | 读已提交、允许不可重复读 |
| **READ UNCOMMITTED** | 弱一致性 | AP | 读未提交、最低保证 |

**形式化映射**：

$$
\text{Isolation Level} \rightarrow \text{Consistency Model}
$$

$$
\begin{align}
\text{SERIALIZABLE} &\rightarrow \text{Linearizability} \\
\text{REPEATABLE READ} &\rightarrow \text{Sequential Consistency} \\
\text{READ COMMITTED} &\rightarrow \text{Causal Consistency} \\
\text{READ UNCOMMITTED} &\rightarrow \text{Weak Consistency}
\end{align}
$$

### 4.2 PostgreSQL的一致性保证

#### 4.2.1 SERIALIZABLE隔离级别（线性一致性）

**实现机制**：

```sql
-- SERIALIZABLE隔离级别（带完整错误处理）
DO $$
BEGIN
    BEGIN
        BEGIN
            SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
            RAISE NOTICE '事务隔离级别已设置为 SERIALIZABLE（线性一致性）';
        EXCEPTION
            WHEN OTHERS THEN
                RAISE WARNING '设置事务隔离级别失败: %', SQLERRM;
                RAISE;
        END;

        RAISE NOTICE '实现机制：';
        RAISE NOTICE '1. 谓词锁（Predicate Lock）';
        RAISE NOTICE '2. 串行化异常检测（SSI）';
        RAISE NOTICE '3. 冲突检测与回滚';
    EXCEPTION
        WHEN OTHERS THEN
            RAISE WARNING '操作失败: %', SQLERRM;
            RAISE;
    END;
END $$;
```

**一致性保证**：

- ✅ **线性一致性**：所有操作按全局顺序执行
- ✅ **无异常**：无脏读、不可重复读、幻读
- ❌ **性能代价**：高延迟、低吞吐量

**实现机制**：

```sql

        BEGIN

            WHEN OTHERS THEN
                RAISE WARNING '设置事务隔离级别失败: %', SQLERRM;
                RAISE;
        END;

        RAISE NOTICE '实现机制：';
        RAISE NOTICE '1. 事务级快照';
        RAISE NOTICE '2. 元组级锁';
        RAISE NOTICE '3. 快照隔离（SI）';
    EXCEPTION
        WHEN OTHERS THEN

**一致性保证**：

- ✅ **顺序一致性**：事务内操作顺序一致
- ✅ **可重复读**：同一事务内多次读取结果相同
- ⚠️ **可能幻读**：可能看到新插入的行

```sql

**一致性保证**：

- ✅ **因果一致性**：相关操作顺序一致
- ⚠️ **不可重复读**：可能看到不同版本
- ⚠️ **可能幻读**：可能看到新插入的行

**配置示例**：

```sql

-- 会话级配置

```

**配置影响**：

| 配置 | 一致性模型 | 性能影响 | 适用场景 |
|------|-----------|---------|---------|
| SERIALIZABLE | 线性一致性 | -50-80% | 金融交易 |
| REPEATABLE READ | 顺序一致性 | -20-40% | 报表查询 |
| READ COMMITTED | 因果一致性 | 正常 | 通用场景 |
| READ UNCOMMITTED | 弱一致性 | +10-20% | 只读分析 |

## 📊 第五部分：一致性模型的选择指南

### 5.1 业务场景与一致性选择

**决策树**：

```text
开始
  │
  ├─ 是否需要强一致性？
  │   ├─ 是 → 线性一致性（SERIALIZABLE）
  │   │   ├─ 金融交易
  │   │   ├─ 支付系统

## 📝 总结

### 核心结论

1. **一致性模型存在强度谱系**：从线性一致性到最终一致性
2. **PostgreSQL隔离级别对应不同一致性模型**：SERIALIZABLE → 线性一致性
3. **一致性选择需要权衡**：正确性 vs 性能
4. **CAP定理决定一致性边界**：分区时无法同时保证强一致性和高可用性

### 实践建议

1. **根据业务需求选择一致性模型**：
   - 金融场景：线性一致性（SERIALIZABLE）
   - 报表场景：顺序一致性（REPEATABLE READ）
   - 通用场景：因果一致性（READ COMMITTED）

2. **监控一致性指标**：
   - 事务隔离级别使用情况
   - 一致性冲突统计
   - 复制延迟（最终一致性）

3. **理解一致性成本**：
   - 强一致性带来性能损失
   - 最终一致性提高可用性
   - 根据场景动态调整
