## 1. 并行查询理论基础

### 1.1 为什么需要并行查询

**问题背景**: 单机多核 CPU 普及，但传统查询执行仅利用单核。

**定理 1.1 (并行必要性)**: 设查询在单核上的执行时间为 $T_1$，CPU 核心数为 $P$，理想加速比为:

$$
S(P) = \frac{T_1}{T_P}
$$

对于完全可并行的查询，$S(P) = P$。

**实际约束**:

- 串行部分无法并行化（协调开销、数据合并）
- 数据依赖限制并行度
- 资源竞争导致性能下降

### 1.2 并行计算基本定律

**定义 1.1 (加速比)**: 加速比 $S(P)$ 定义为单处理器执行时间与 $P$ 个处理器执行时间的比值:

$$
S(P) = \frac{T_1}{T_P}
$$

**定理 1.2 (Amdahl 定律)**: 设程序中可并行部分的比例为 $f$，则最大加速比为:

$$
S_{max} = \lim_{P \to \infty} \frac{1}{(1-f) + \frac{f}{P}} = \frac{1}{1-f}
$$

*证明*:

- 串行部分时间: $(1-f) \cdot T_1$
- 并行部分时间: $\frac{f \cdot T_1}{P}$
- 总时间: $T_P = (1-f) \cdot T_1 + \frac{f \cdot T_1}{P}$
- 加速比: $S(P) = \frac{T_1}{T_P} = \frac{1}{(1-f) + \frac{f}{P}}$
- 当 $P \to \infty$: $S_{max} = \frac{1}{1-f}$ ∎

**定理 1.3 (Gustafson 定律)**: 对于规模可扩展的问题，加速比为:

$$
S(P) = P - f \cdot (P - 1)
$$

**数据库查询的适用性**:

- Amdahl 定律: 适用于固定数据量（OLTP 查询）
- Gustafson 定律: 适用于数据量随并行度增长（OLAP 查询）

## 2. 并行执行架构

### 2.1 领导者-工作者模型

**定义 2.1 (并行执行架构)**: PostgreSQL 采用领导者-工作者(Leader-Worker)架构:

```
┌─────────────────────────────────────────────────────────────┐
│                     Leader Process                          │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  • 查询解析与计划生成                                   │   │
│  │  • 并行度决策                                          │   │
│  │  • 启动 Worker 进程                                    │   │
│  │  • 参与执行 + 结果汇总                                  │   │
│  └──────────────────────────────────────────────────────┘   │
└────────────────────────┬──────────────────────────────────────┘
                         │ 动态共享内存 (DSM)
        ┌────────────────┼────────────────┐
        │                │                │
        ▼                ▼                ▼
┌──────────┐    ┌──────────┐    ┌──────────┐
│ Worker 1 │    │ Worker 2 │    │ Worker N │
│┌────────┐│    │┌────────┐│    │┌────────┐│
││局部扫描││    ││局部扫描││    ││局部扫描││
│├────────┤│    │├────────┤│    │├────────┤│
││局部聚合││    ││局部聚合││    ││局部聚合││
│└────────┘│    │└────────┘│    │└────────┘│
└──────────┘    └──────────┘    └──────────┘
```

**定义 2.2 (并行度)**: 并行度 $DOP$ (Degree of Parallelism) 定义为参与执行的进程数:

$$
DOP = 1 + N_{workers}
$$

其中 1 是领导者进程，$N_{workers}$ 是工作者进程数。

### 2.2 并行查询计划结构

**定义 2.3 (并行计划节点)**: 并行计划包含以下节点类型:

| 节点类型 | 符号 | 描述 |
|---------|------|------|
| Gather | $\Gamma$ | 收集工作者结果 |
| Gather Merge | $\Gamma_M$ | 有序收集结果 |
| Partial SeqScan | $\sigma_P$ | 并行顺序扫描 |
| Partial IndexScan | $\iota_P$ | 并行索引扫描 |
| Partial HashAgg | $\gamma_P$ | 并行哈希聚合 |

**定理 2.1 (并行计划正确性)**: 并行计划 $P_{parallel}$ 与串行计划 $P_{serial}$ 等价，当且仅当:

$$
\forall Q: P_{parallel}(Q) \equiv P_{serial}(Q)
$$

其中 $\equiv$ 表示结果等价（可能有序差异）。

### 2.3 数据分区策略

**定义 2.4 (数据分区)**: 将关系 $R$ 划分为 $P$ 个子集:

$$
R = \bigcup_{i=1}^{P} R_i, \quad R_i \cap R_j = \emptyset \ (i \neq j)
$$

**分区策略对比**:

| 策略 | 公式 | 优点 | 缺点 |
|------|------|------|------|
| 范围分区 | $R_i = \{r \mid key \in [v_i, v_{i+1})\}$ | 有序保留 | 数据倾斜 |
| 哈希分区 | $R_i = \{r \mid hash(key) \mod P = i\}$ | 负载均衡 | 需要哈希计算 |
| 轮询分区 | $R_i = \{r_j \mid j \mod P = i\}$ | 实现简单 | 随机访问 |
| 块分区 | $R_i = \{r \mid block(r) \in B_i\}$ | I/O 局部性好 | 可能数据倾斜 |

**定理 2.2 (分区负载均衡)**: 最优分区使各分区数据量差异最小:

$$
\min \max_{i,j} |R_i| - |R_j|
$$

**PostgreSQL 实现**: 采用块分区（Block-based），基于物理页范围划分。

## 3. 并行算子实现

### 3.1 并行扫描

**定义 3.1 (并行顺序扫描)**: 将表按页范围划分为多个子集，每个 Worker 扫描一个子集:

$$
C_{parallel\_seq}(R, P) = \frac{C_{seq}(R)}{P} + C_{setup} + C_{gather}
$$

其中:

- $C_{seq}(R)$: 串行扫描代价
- $C_{setup}$: Worker 启动代价
- $C_{gather}$: 结果收集代价

**最优并行度计算**:

**定理 3.1 (最优并行度)**: 设 $C_{seq}(R) = N \cdot t_{scan}$，$C_{setup} = t_{setup}$，$C_{gather} = N \cdot t_{gather}$，则最优并行度为:

$$
P_{opt} = \sqrt{\frac{N \cdot t_{scan}}{t_{setup}}}
$$

*推导*:

- 总代价: $C(P) = \frac{N \cdot t_{scan}}{P} + P \cdot t_{setup} + N \cdot t_{gather}$
- 对 $P$ 求导: $\frac{dC}{dP} = -\frac{N \cdot t_{scan}}{P^2} + t_{setup} = 0$
- 解得: $P_{opt} = \sqrt{\frac{N \cdot t_{scan}}{t_{setup}}}$ ∎

**配置参数**:

```sql
-- 最小并行表大小 (默认 8MB)
SET min_parallel_table_scan_size = '8MB';

-- 最小并行索引扫描大小 (默认 512KB)
SET min_parallel_index_scan_size = '512kB';

-- 并行度成本阈值 (默认 1000)
SET parallel_setup_cost = 1000;

-- 并行元组处理成本 (默认 0.1)
SET parallel_tuple_cost = 0.1;
```

### 3.2 并行连接

**定义 3.2 (并行哈希连接)**: 并行构建哈希表，每个 Worker 探测部分数据:

**代价模型**:

$$
C_{parallel\_HJ}(R, S, P) = C_{build} + \frac{C_{probe}}{P} + C_{sync}
$$

其中 $C_{sync}$ 是同步开销。

**定义 3.3 (并行合并连接)**: 要求输入已按连接键排序:

$$
C_{parallel\_MJ}(R, S, P) = \frac{C_{sort}(R) + C_{sort}(S)}{P} + C_{merge}
$$

### 3.3 并行聚合

**定义 3.4 (两阶段聚合)**: PostgreSQL 采用两阶段聚合策略:

```
┌─────────┐  ┌─────────┐  ┌─────────┐
│ Worker 1│  │ Worker 2│  │ Worker P│
│┌───────┐│  │┌───────┐│  │┌───────┐│
││Partial││  ││Partial││  ││Partial││
││ Aggregate │  ││ Aggregate │  ││ Aggregate │
││ (Hash)││  ││ (Hash)││  ││ (Hash)││
│└───┬───┘│  │└───┬───┘│  │└───┬───┘│
└────┼────┘  └────┼────┘  └────┼────┘
     └────────────┼────────────┘
                  ▼
          ┌───────────────┐
          │ Final         │
          │ Aggregate     │
          │ (Leader)      │
          └───────────────┘
```

**定理 3.2 (聚合并行效率)**: 对于分组数 $G$，数据量 $N$，并行度 $P$，两阶段聚合的加速比为:

$$
S(P) = \frac{N \cdot t_{agg}}{\frac{N \cdot t_{agg}}{P} + G \cdot t_{final}}
$$

当 $N \gg G$ 时，$S(P) \approx P$。

## 4. 并行度决策

### 4.1 成本模型

**定义 4.1 (并行查询成本)**: 并行查询总成本包括:

$$
C_{total} = C_{setup} + \frac{C_{compute}}{P} + C_{gather} + C_{sync}
$$

其中:

- $C_{setup}$: Worker 启动成本
- $C_{compute}$: 计算成本（可并行）
- $C_{gather}$: 结果收集成本
- $C_{sync}$: 同步成本

**PostgreSQL 代价参数**:

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `parallel_setup_cost` | 1000 | 启动 Worker 的代价 |
| `parallel_tuple_cost` | 0.1 | 通过 Gather 传递元组的额外代价 |
| `min_parallel_table_scan_size` | 8MB | 表的最小并行扫描大小 |
| `min_parallel_index_scan_size` | 512KB | 索引的最小并行扫描大小 |

### 4.2 自适应并行度

**定义 4.2 (自适应并行度)**: 根据运行时信息动态调整并行度:

$$
P_{adaptive} = \min\left(P_{max}, P_{opt}, P_{resource}\right)
$$

其中:

- $P_{max}$: 最大允许并行度 (`max_parallel_workers_per_gather`)
- $P_{opt}$: 基于代价模型的最优并行度
- $P_{resource}$: 基于资源限制的最大并行度

**资源限制计算**:

```
P_resource = min(
    max_parallel_workers,
    max_worker_processes - 当前活跃进程数,
    CPU核心数 - 1
)
```

### 5.2 并行查询限制

**当前不支持并行的操作**:

| 操作类型 | 原因 | 计划类型 |
|---------|------|----------|
| 修改操作 (INSERT/UPDATE/DELETE) | 可见性冲突 | 数据修改 |
| 游标扫描 | 需要有序输出 | DECLARE CURSOR |
| 包含并行不安全的 UDF | 副作用 | 函数调用 |
| 嵌套 Gather | 避免过度并行 | 多层嵌套 |
| 全局限速 | 资源控制 | LIMIT/OFFSET |

**并行安全判断**:

```sql
-- 查看函数并行安全属性
SELECT
    proname,
    CASE proparallel
        WHEN 's' THEN 'SAFE'
        WHEN 'r' THEN 'RESTRICTED'
        WHEN 'u' THEN 'UNSAFE'
    END as parallel_safety
FROM pg_proc
WHERE proname = 'my_function';

-- 修改函数并行安全属性
ALTER FUNCTION my_function() PARALLEL SAFE;
```

## 7. 实例与反例

### 7.1 正例

**场景1: 大表并行扫描**

```sql
-- 创建测试表
CREATE TABLE sales (
    id BIGSERIAL PRIMARY KEY,
    amount DECIMAL(12,2),
    status VARCHAR(20),
    created_at TIMESTAMP
);

-- 插入1000万条数据
INSERT INTO sales (amount, status, created_at)
SELECT
    random() * 1000,
    CASE WHEN random() < 0.8 THEN 'completed' ELSE 'pending' END,
    NOW() - (random() * INTERVAL '365 days')
FROM generate_series(1, 10000000);

-- 更新统计信息
ANALYZE sales;

-- 并行查询
EXPLAIN (ANALYZE, BUFFERS)
SELECT status, COUNT(*), AVG(amount)
FROM sales
GROUP BY status;

-- 结果：使用 4 个 Worker，执行时间减少 65%
```

**场景2: 并行索引构建**

```sql
-- 并行创建索引
SET max_parallel_maintenance_workers = 4;

CREATE INDEX CONCURRENTLY idx_sales_status
ON sales(status)
WITH (parallel_workers = 4);

-- 监控构建进度
SELECT
    phase,
    blocks_total,
    blocks_done,
    ROUND(100.0 * blocks_done / NULLIF(blocks_total, 0), 2) as pct_complete
FROM pg_stat_progress_create_index;
```

**场景3: 并行连接**

```sql
-- 大表连接优化
EXPLAIN (ANALYZE)
SELECT
    c.customer_name,
    SUM(s.amount) as total_spent
FROM customers c
JOIN sales s ON c.id = s.customer_id
GROUP BY c.customer_name;

-- Workers Launched: 4
```

### 7.2 反例

**反例1: 小表并行**

```sql
-- 错误：小表使用并行
EXPLAIN SELECT * FROM small_config_table WHERE key = 'value';

-- 结果：虽然计划显示 Gather，但并行开销使总时间增加
-- 正确做法：保持串行执行
```

**反例2: 过度并行**

```sql
-- 错误：设置过高的并行度
SET max_parallel_workers_per_gather = 32;

-- 问题：
-- 1. 进程启动开销大
-- 2. CPU 上下文切换频繁
-- 3. 共享资源竞争

-- 正确做法：根据 CPU 核心数设置合理值
SET max_parallel_workers_per_gather = 4; -- 8核机器
```

**反例3: 并行不安全函数**

```sql
-- 创建并行不安全函数
CREATE OR REPLACE FUNCTION log_access(user_id INT)
RETURNS VOID AS $$
BEGIN
    INSERT INTO access_log VALUES (user_id, NOW());
END;
$$ LANGUAGE plpgsql;

-- 在并行查询中使用会导致:
-- 1. 查询无法并行化
-- 2. 或者产生不可预期的结果

-- 正确做法：标记函数并行安全或避免在并行查询中使用
ALTER FUNCTION log_access(INT) PARALLEL UNSAFE;
```
