## 1. 查询优化理论基础

### 1.1 为什么需要查询优化

**问题背景**: SQL是声明式语言，用户指定"要什么"而非"怎么做"。

**定理 1.1 (优化必要性)**: 对于包含n个表的查询，可能的连接计划数量为 Catalan(n-1) × n!，穷举不可行。

*证明*:

- n个表的二叉树形状数为 Catalan(n-1) = $\frac{(2n-2)!}{(n-1)!n!}$
- 每个内部节点可以是n!种连接顺序之一
- 对于n=10: Catalan(9) = 4862, 10! = 3,628,800
- 总计划数 = 4862 × 3,628,800 ≈ 17.6 billion
∎

### 1.2 查询优化问题定义

**定义 1.1 (查询优化问题)**:
给定查询Q和代价模型C，找到执行计划P使得:

$$
P^* = \arg\min_{P \in \mathcal{P}(Q)} C(P)
$$

其中$\mathcal{P}(Q)$是所有等价计划的集合。

**定理 1.2 (NP难)**: 查询优化问题是NP难的。

*证明概要*: 可规约到旅行商问题(TSP)。∎

## 2. 代价模型形式化

### 2.1 通用代价函数

**定义 2.1 (代价模型)**:
执行计划P的总代价为:

$$
C(P) = \alpha \cdot C_{IO}(P) + \beta \cdot C_{CPU}(P) + \gamma \cdot C_{Network}(P) + \delta \cdot C_{Memory}(P)
$$

其中权重参数满足: $\alpha + \beta + \gamma + \delta = 1$

### 2.2 I/O代价模型

**顺序扫描代价**:

$$
C_{seq}(R) = \left\lceil \frac{|R| \cdot t_R}{B} \right\rceil \cdot c_{seq}
$$

其中:

- $|R|$: 关系R的元组数
- $t_R$: 元组大小
- $B$: 页大小
- $c_{seq}$: 顺序页读取代价

**索引扫描代价**:

$$
C_{index}(R, A = v) = h_I + \left\lceil sel(A = v) \cdot |R| \right\rceil \cdot c_{random}
$$

其中:

- $h_I$: B+树高度
- $sel(A = v)$: 选择率
- $c_{random}$: 随机页读取代价

**定理 2.1 (索引扫描最优条件)**: 索引扫描优于顺序扫描当且仅当:

$$
h_I + sel(A = v) \cdot |R| \cdot c_{random} < \frac{|R| \cdot t_R}{B} \cdot c_{seq}
$$

### 2.3 连接代价模型

**Nested Loop Join**:

$$
C_{NL}(R \bowtie S) = C_{outer}(R) + |R'| \cdot C_{inner}(S)
$$

其中$|R'|$是应用选择条件后的R基数。

**Block Nested Loop**:

$$
C_{BNL}(R \bowtie S) = \left\lceil \frac{|R|}{M-2} \right\rceil \cdot C_{seq}(S) + C_{seq}(R)
$$

其中M是可用内存页数。

**Hash Join**:

$$
C_{HJ}(R \bowtie S) = 2 \cdot C_{seq}(R) + C_{seq}(S) \quad \text{(R是构建表)}
$$

**Merge Join** (假设已排序):

$$
C_{MJ}(R \bowtie S) = C_{seq}(R) + C_{seq}(S)
$$

**定理 2.2 (连接算法选择)**:

- 当R很小且S有索引时，Nested Loop最优
- 当R和S都很大且无索引时，Hash Join最优
- 当R和S已排序时，Merge Join最优

### 2.4 代价模型参数

**PostgreSQL代价参数**:

| 参数 | 默认值 | 说明 | 典型SSD值 |
|------|--------|------|-----------|
| seq_page_cost | 1.0 | 顺序页读取 | 1.0 |
| random_page_cost | 4.0 | 随机页读取 | 1.1 |
| cpu_tuple_cost | 0.01 | 处理每个元组 | 0.01 |
| cpu_index_tuple_cost | 0.005 | 处理索引元组 | 0.005 |
| cpu_operator_cost | 0.0025 | 执行操作符 | 0.0025 |
| parallel_tuple_cost | 0.1 | 并行处理 | 0.1 |
| parallel_setup_cost | 1000 | 并行启动 | 1000 |

## 3. 统计信息与基数估计

### 3.1 统计信息收集

**定义 3.1 (关系统计)**:

$$
Stats(R) = \langle |R|, W_R, \{Stats(A_i)\}_{A_i \in schema(R)} \rangle
$$

其中:

- $|R|$: 元组数
- $W_R$: 平均元组宽度
- $Stats(A_i)$: 属性统计

**定义 3.2 (属性统计)**:

$$
Stats(A) = \langle n_{distinct}, n_{null}, MCV, Histogram \rangle
$$

### 3.2 选择率估计

**等值选择** (无MCV):

$$
sel(A = v) = \frac{1}{n_{distinct}(A)}
$$

**等值选择** (有MCV):

$$
sel(A = v) =
\begin{cases}
freq_i & \text{if } v = MCV_i \\
\frac{1 - \sum_j freq_j}{n_{distinct} - |MCV|} & \text{otherwise}
\end{cases}
$$

**范围选择**:

$$
sel(A \in [l, r]) = \frac{r - l}{max(A) - min(A)} \quad \text{(均匀分布假设)}
$$

**使用直方图**:

$$
sel(A \le v) = \sum_{i: bucket_i^{max} \le v} freq_i + \frac{v - bucket_k^{min}}{bucket_k^{max} - bucket_k^{min}} \cdot freq_k
$$

### 3.3 连接选择率

**外键连接**:

$$
sel(R \bowtie_{R.a = S.b} S) = \frac{1}{\max(n_{distinct}(R.a), n_{distinct}(S.b))}
$$

**一般等值连接**:

$$
sel(R \bowtie_{A=B} S) = \frac{1}{\max(n_{distinct}(A), n_{distinct}(B))}
$$

**定理 3.1 (独立性假设误差)**: 当属性相关时，独立性假设导致的选择率估计误差可达指数级。

*证明*: 考虑$P(A=a \land B=b)$，若A和B完全相关，$sel = sel(A) = sel(B)$，但独立性假设估计为$sel(A) \cdot sel(B)$。∎

## 4. 连接顺序优化

### 4.1 动态规划算法 (DPsize)

**定义 4.1 (最优子计划)**:
设$BestPlan(S)$为连接关系集S的最优计划。

**递推关系**:

$$
BestPlan(S) = \arg\min_{S' \subset S} \{ C(BestPlan(S') \bowtie BestPlan(S \\ S')) \}
$$

**复杂度**: $O(3^n)$

### 4.2 左深树 vs 浓密树

**定义 4.2 (计划树形状)**:

- **左深树**: 右子树始终是基表
- **浓密树**: 允许任意子树连接
- **右深树**: 左子树始终是基表

**定理 4.1 (左深树最优性)**: 对于简单嵌套循环连接，存在最优左深树。

### 4.3 启发式算法

**贪心算法**:

```text
1. 从最小关系开始
2. 每次选择连接代价最小的关系加入
3. 直到所有关系连接完毕
```

**遗传算法** (PostgreSQL实现):

- 适用于大量表(>12)
- 种群大小、代数可配置
- 交叉和变异操作

## 5. 物理计划选择

### 5.1 访问路径选择

**决策矩阵**:

| 条件 | 顺序扫描 | 索引扫描 | 位图扫描 |
|------|----------|----------|----------|
| 选择率低(<5%) | ✗ | ✓ | ✓ |
| 选择率高(>20%) | ✓ | ✗ | ✗ |
| 需要排序输出 | ✗ | ✓(有序索引) | ✗ |
| 并发更新高 | ✓ | ✗ | ✓ |

### 5.2 连接算法选择决策树

```text
连接算法选择:
├─ 一个表很小(< 1000行)
│   └─ 另一个表有索引
│       └─ Nested Loop with Index
├─ 等值连接条件?
│   ├─ 内存足够容纳小表
│   │   └─ Hash Join
│   └─ 两个输入已排序?
│       └─ Merge Join
└─ 否则
    └─ Nested Loop
```

### 6.2 关键源码分析

**代价计算** (src/backend/optimizer/path/costsize.c):

```c
/* 顺序扫描代价 */
void
cost_seqscan(Path *path, RelOptInfo *baserel,
             ParamPathInfo *param_info)
{
    Cost      startup_cost = 0;
    Cost      run_cost = 0;
    double    spc_seq_page_cost;

    /* I/O代价 */
    run_cost += baserel->pages * spc_seq_page_cost;

    /* CPU代价 */
    run_cost += baserel->tuples * cpu_tuple_cost;

    path->startup_cost = startup_cost;
    path->total_cost = startup_cost + run_cost;
}

/* 索引扫描代价 */
void
cost_index(IndexPath *path, PlannerInfo *root,
           double loop_count, bool partial_path)
{
    Cost      startup_cost = 0;
    Cost      run_cost = 0;
    double    num_index_pages;
    double    num_index_tuples;

    /* 索引页访问代价 */
    run_cost += num_index_pages * random_page_cost;

    /* 表页访问代价 */
    run_cost += num_tuple_pages * random_page_cost;

    /* CPU代价 */
    run_cost += num_index_tuples * cpu_index_tuple_cost;
    run_cost += path->path.rows * cpu_tuple_cost;
}
```

**连接代价**:

```c
/* Hash Join代价 */
void
cost_hashjoin(HashPath *path, PlannerInfo *root,
              SpecialJoinInfo *sjinfo)
{
    Cost      startup_cost = 0;
    Cost      run_cost = 0;

    /* 构建哈希表代价 */
    startup_cost += outer_path->total_cost;
    startup_cost += (outer_path->rows * cpu_operator_cost);

    /* 探测代价 */
    run_cost += inner_path->total_cost;
    run_cost += (inner_path->rows * cpu_operator_cost);
}
```

## 8. 实例与反例

### 8.1 正例: 正确优化

**场景1: 索引选择优化**

```sql
-- 表结构
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    user_id INT NOT NULL,
    status VARCHAR(20),
    created_at TIMESTAMP,
    amount DECIMAL(10,2)
);

CREATE INDEX idx_user_id ON orders(user_id);
CREATE INDEX idx_status ON orders(status);

-- 查询1: 高选择率 - 使用索引
EXPLAIN ANALYZE
SELECT * FROM orders WHERE user_id = 123;
-- 计划: Index Scan (cost=0.29..8.31)

-- 查询2: 低选择率 - 顺序扫描
EXPLAIN ANALYZE
SELECT * FROM orders WHERE status = 'pending';
-- 计划: Seq Scan (cost=0.00..15432.00)
```

**场景2: 连接顺序优化**

```sql
-- 小表驱动大表
SELECT * FROM small_table s
JOIN large_table l ON s.id = l.small_id;
-- 优化器自动选择small_table作为outer
```

**场景3: 统计信息更新**

```sql
-- 批量插入后更新统计
COPY orders FROM 'orders.csv';
ANALYZE orders;  -- 关键!
```

### 8.2 反例: 常见错误

**反例1: 过时的统计信息**

```sql
-- 问题: 表增长10倍后未ANALYZE
-- 优化器仍使用旧基数估计
-- 结果: 选择了错误的索引扫描

-- 解决方案
ANALYZE tablename;
-- 或设置自动分析
ALTER TABLE tablename SET (autovacuum_analyze_threshold = 1000);
```

**反例2: 不匹配的代价参数**

```ini
# SSD上仍使用HDD参数
random_page_cost = 4.0  # 错误!

# 导致: 优化器避免使用索引
# 解决方案
random_page_cost = 1.1  # SSD正确值
```

**反例3: 强制错误的连接类型**

```sql
-- 错误地禁用hash join
SET enable_hashjoin = off;

-- 大表等值连接被迫使用nested loop
-- 结果: 性能下降1000倍

-- 解决方案: 让优化器选择
SET enable_hashjoin = on;
```

**反例4: 忽略数据相关性**

```sql
-- 城市和邮编高度相关
SELECT * FROM users
WHERE city = 'Beijing' AND zip_code LIKE '100%';

-- 优化器假设独立: sel = 0.01 * 0.1 = 0.001
-- 实际相关: sel = 0.01
-- 结果: 严重低估基数

-- 解决方案: 创建多列统计
CREATE STATISTICS city_zip_corr ON city, zip_code FROM users;
ANALYZE users;
```
