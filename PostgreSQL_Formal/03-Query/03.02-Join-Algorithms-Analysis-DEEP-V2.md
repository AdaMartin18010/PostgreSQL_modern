## 1. 连接操作理论基础

### 1.1 连接定义

**定义 1.1 (关系连接)**:
给定关系 $R$ 和 $S$，在属性 $A$ 和 $B$ 上的连接定义为:

$$
R \bowtie_{A=B} S := \{(r, s) \mid r \in R \land s \in S \land r.A = s.B\}
$$

**连接基数估计**:

$$
|R \bowtie S| = \frac{|R| \cdot |S|}{\max(|\pi_A(R)|, |\pi_B(S)|)}
$$

### 1.2 连接类型

| 连接类型 | 数学定义 | SQL示例 |
|----------|----------|---------|
| **内连接** | $R \bowtie_\theta S$ | `INNER JOIN` |
| **左外连接** | $R \bowtie^L_\theta S$ | `LEFT JOIN` |
| **右外连接** | $R \bowtie^R_\theta S$ | `RIGHT JOIN` |
| **全外连接** | $R \bowtie^F_\theta S$ | `FULL JOIN` |
| **半连接** | $R \ltimes_\theta S$ | `WHERE EXISTS` |
| **反连接** | $R \rhd_\theta S$ | `WHERE NOT EXISTS` |

## 2. Nested Loop Join (嵌套循环连接)

### 2.1 基本算法

**算法 2.1 (Nested Loop Join)**:

```
NestedLoopJoin(R, S, θ):
    result ← ∅
    for each r ∈ R:           -- 外层循环
        for each s ∈ S:       -- 内层循环
            if θ(r, s) then   -- 满足连接条件
                result ← result ∪ {(r, s)}
    return result
```

**复杂度分析**:

$$
T_{NL} = |R| \cdot |S| \cdot C_{\theta}
$$

其中 $C_{\theta}$ 是单次条件评估代价。

**定理 2.1 (NL复杂度)**: Nested Loop Join的时间复杂度为 $O(|R| \cdot |S|)$。

### 2.2 Block Nested Loop Join

**定义 2.1 (块嵌套循环)**:
一次读取外层关系的多个页到内存，减少I/O。

```
BlockNestedLoopJoin(R, S, θ, M):
    result ← ∅
    for each block B_R of R:           -- 每次读取M-2页
        load B_R into memory
        for each block B_S of S:
            for each r ∈ B_R:
                for each s ∈ B_S:
                    if θ(r, s) then
                        result ← result ∪ {(r, s)}
    return result
```

**定理 2.2 (Block NL复杂度)**: Block Nested Loop的时间复杂度为:

$$
T_{BNL} = \left\lceil \frac{|R|}{M-2} \right\rceil \cdot |S| + |R|
$$

其中 $M$ 是可用内存页数。

### 2.3 Index Nested Loop Join

**定义 2.2 (索引嵌套循环)**:
当内层关系有索引时使用。

```
IndexNestedLoopJoin(R, S, A, idx):
    result ← ∅
    for each r ∈ R:
        matches ← IndexLookup(idx, r.A)  -- O(log|S|)
        for each s ∈ matches:
            result ← result ∪ {(r, s)}
    return result
```

**定理 2.3 (Index NL复杂度)**: Index Nested Loop的时间复杂度为:

$$
T_{INL} = |R| \cdot (C_{idx} + |S_{match}|)
$$

其中 $C_{idx} = O(\log |S|)$ 是索引查找代价。

## 3. Hash Join (哈希连接)

### 3.1 基本算法

**算法 3.1 (Hash Join)**:

```
HashJoin(R, S, A, B):
    -- 构建阶段 (Build Phase)
    H ← new HashTable
    for each r ∈ R:                    -- R是构建表(小表)
        key ← hash(r.A)
        H[key].append(r)

    -- 探测阶段 (Probe Phase)
    result ← ∅
    for each s ∈ S:                    -- S是探测表
        key ← hash(s.B)
        for each r ∈ H[key]:           -- 检查哈希冲突
            if r.A = s.B then          -- 验证匹配
                result ← result ∪ {(r, s)}

    return result
```

**复杂度分析**:

$$
T_{HJ} = |R| + |S| + C_{hash}
$$

**定理 3.1 (Hash Join复杂度)**: 在均匀哈希假设下，Hash Join的时间复杂度为 $O(|R| + |S|)$。

### 3.2 分区哈希连接

**定义 3.1 (分区哈希)**:
当构建表无法全部装入内存时，使用分区策略。

**算法 3.2 (Grace Hash Join)**:

```
GraceHashJoin(R, S, A, B, M):
    -- 分区阶段
    n_partitions ← M - 1
    for each r ∈ R:
        p ← hash_partition(r.A, n_partitions)
        write r to partition R_p

    for each s ∈ S:
        p ← hash_partition(s.B, n_partitions)
        write s to partition S_p

    -- 连接阶段
    result ← ∅
    for p ← 1 to n_partitions:
        if |R_p| < M then              -- 内存能容纳
            result ← result ∪ HashJoin(R_p, S_p, A, B)
        else:
            -- 递归分区
            result ← result ∪ GraceHashJoin(R_p, S_p, A, B, M)

    return result
```

**定理 3.2 (Grace Hash复杂度)**: Grace Hash Join的时间复杂度为:

$$
T_{GHJ} = 3(|R| + |S|) + (|R| + |S|) \cdot \frac{1}{M} \cdot passes
$$

其中 $passes$ 是递归分区次数。

### 3.3 布隆过滤器优化

**定义 3.2 (布隆过滤器)**:
概率数据结构，快速判断元素是否可能在集合中。

```
HashJoinWithBloomFilter(R, S, A, B):
    -- 构建阶段
    BF ← new BloomFilter
    H ← new HashTable

    for each r ∈ R:
        BF.add(r.A)
        H[hash(r.A)].append(r)

    -- 探测阶段
    result ← ∅
    for each s ∈ S:
        if not BF.contains(s.B):
            continue                    -- 快速跳过

        for each r ∈ H[hash(s.B)]:
            if r.A = s.B then
                result ← result ∪ {(r, s)}

    return result
```

## 4. Merge Join (归并连接)

### 4.1 基本算法

**前提条件**: 输入关系必须按连接键排序。

**算法 4.1 (Merge Join)**:

```
MergeJoin(R, S, A, B):
    result ← ∅
    r ← R.first()
    s ← S.first()

    while r ≠ null and s ≠ null:
        if r.A < s.B:
            r ← R.next()
        else if r.A > s.B:
            s ← S.next()
        else:
            -- 找到匹配，处理相等的键组
            key ← r.A

            -- 收集R中所有等于key的元组
            R_group ← ∅
            while r ≠ null and r.A = key:
                R_group ← R_group ∪ {r}
                r ← R.next()

            -- 收集S中所有等于key的元组
            S_group ← ∅
            while s ≠ null and s.B = key:
                S_group ← S_group ∪ {s}
                s ← S.next()

            -- 输出笛卡尔积
            for each r' ∈ R_group:
                for each s' ∈ S_group:
                    result ← result ∪ {(r', s')}

    return result
```

**复杂度分析**:

$$
T_{MJ} = |R| + |S| + |R \bowtie S|
$$

**定理 4.1 (Merge Join复杂度)**: Merge Join的时间复杂度为 $O(|R| + |S|)$。

### 4.2 带排序的Merge Join

**定义 4.1 (排序代价)**:

$$
T_{sort}(R) = |R| \cdot \log |R|
$$

**总代价**:

$$
T_{MJ+sort} = |R| \cdot \log |R| + |S| \cdot \log |S| + |R| + |S|
$$

## 5. 算法选择

### 5.1 决策矩阵

| 条件 | Nested Loop | Hash Join | Merge Join |
|------|-------------|-----------|------------|
| 小表(≤1000) | ✓✓✓ | ✓ | ✗ |
| 中表(1K-100K) | ✗ | ✓✓✓ | ✓✓ |
| 大表(>100K) | ✗✗✗ | ✓✓✓ | ✓✓ |
| 等值连接 | ✓ | ✓✓✓ | ✓✓ |
| 范围连接 | ✓✓ | ✗ | ✓✓✓ |
| 内存充足 | ✓ | ✓✓✓ | ✓✓ |
| 输入已排序 | ✓ | ✓ | ✓✓✓ |
| 需要排序输出 | ✓ | ✗ | ✓✓✓ |

### 5.2 代价模型

**选择条件**:

$$
\text{BestJoin} = \arg\min \{C_{NL}, C_{HJ}, C_{MJ}\}
$$

其中:

- $C_{NL} = |R| \cdot C_{outer} + |R| \cdot |S| \cdot sel \cdot C_{inner}$
- $C_{HJ} = 2|R| + |S|$ (若R能装入内存)
- $C_{MJ} = sort(R) + sort(S) + |R| + |S|$

## 6. PostgreSQL实现

### 6.1 连接路径生成

```c
/*
 * create_join_paths
 * 为给定的连接关系创建所有可能的连接路径
 */
void
create_join_paths(PlannerInfo *root, RelOptInfo *joinrel)
{
    /* 1. Nested Loop Join路径 */
    create_nestloop_paths(root, joinrel);

    /* 2. Hash Join路径 (仅等值连接) */
    if (is_equijoin(joinclauses))
        create_hashjoin_paths(root, joinrel);

    /* 3. Merge Join路径 */
    if (has_sort_keys(joinclauses))
        create_mergejoin_paths(root, joinrel);
}
```

### 6.2 Hash Join实现

```c
/*
 * ExecHashJoin
 * Hash Join执行节点
 */
static TupleTableSlot *
ExecHashJoin(HashJoinState *node)
{
    /* 构建阶段 */
    if (!node->hashBuilt)
    {
        /* 执行构建子计划 */
        ExecuteBuildSide(node);

        /* 构建哈希表 */
        BuildHashTable(node);

        node->hashBuilt = true;
    }

    /* 探测阶段 */
    for (;;)
    {
        /* 从探测侧获取元组 */
        outerTuple = ExecProcNode(outerPlanState(node));

        /* 计算哈希值 */
        hashvalue = ExecHashGetHashValue(node, outerTuple);

        /* 在哈希表中查找匹配 */
        match = HashTableLookup(node->hashtable, hashvalue, outerTuple);

        while (match != NULL)
        {
            /* 验证连接条件 */
            if (ExecQual(node->js.ps.qual, econtext))
            {
                return result_slot;
            }
            match = match->next;
        }
    }
}
```

### 6.3 并行Hash Join

```c
/*
 * ExecParallelHashJoin
 * 并行Hash Join实现
 */
static TupleTableSlot *
ExecParallelHashJoin(HashJoinState *node)
{
    /* 并行构建: 每个worker构建一个分区 */
    if (BarrierPhase(&node->phs->barrier) == PHJ_BUILD)
    {
        /* 扫描构建表的一部分 */
        ExecParallelScanBuildSide(node);

        /* 将元组插入共享哈希表 */
        ExecParallelHashTableInsert(node);

        /* 同步所有worker */
        BarrierArriveAndWait(&node->phs->barrier);
    }

    /* 并行探测 */
    return ExecParallelHashJoinProbe(node);
}
```

## 7. 思维表征

### 7.1 算法执行过程对比

```
Nested Loop Join:
R: [1, 2, 3]
S: [a, b, c]

外层  内层      比较
 1  × [a,b,c]   1=a? 1=b? 1=c?
 2  × [a,b,c]   2=a? 2=b? 2=c?
 3  × [a,b,c]   3=a? 3=b? 3=c?

总比较次数: |R| × |S|

Hash Join:
构建表R → 哈希表
1 → bucket[hash(1)]
2 → bucket[hash(2)]
3 → bucket[hash(3)]

探测表S:
a → hash(a) → bucket[hash(a)] → 检查a=?
b → hash(b) → bucket[hash(b)] → 检查b=?
c → hash(c) → bucket[hash(c)] → 检查c=?

总比较次数: |R| + |S| (加上哈希冲突)

Merge Join:
R(已排序): [1, 2, 3, 4]
S(已排序): [2, 3, 4, 5]

指针: r→1, s→2
1 < 2? 是, r++ → r→2
2 = 2? 是, 匹配! r++, s++
3 = 3? 是, 匹配! r++, s++
4 = 4? 是, 匹配! r++, s++
null vs 5, 结束

总比较次数: |R| + |S|
```

### 7.2 内存使用对比

| 算法 | 内存使用 | I/O模式 |
|------|----------|---------|
| **Nested Loop** | $O(1)$ | 随机读取 |
| **Block NL** | $O(M)$ | 顺序读取 |
| **Hash Join** | $O(|R|)$ | 顺序读取 |
| **Merge Join** | $O(1)$ | 顺序读取 |

## 8. 实例与反例

### 8.1 正例

**场景1: 小表驱动大表**

```sql
-- 小表users(1000行)，大表orders(1000000行)
-- 有索引: orders(user_id)

EXPLAIN SELECT * FROM users u
JOIN orders o ON u.id = o.user_id;

-- 计划: Nested Loop
--       -> Seq Scan on users
--       -> Index Scan on orders (user_id)
-- 最优选择!
```

**场景2: 大表等值连接**

```sql
-- 两个大表(各100万行)等值连接
EXPLAIN SELECT * FROM orders o
JOIN order_items oi ON o.id = oi.order_id;

-- 计划: Hash Join
-- 构建阶段: 扫描较小的表到哈希表
-- 探测阶段: 扫描另一个表进行匹配
-- 最优选择!
```

**场景3: 已排序输入**

```sql
-- 两个表都按id排序
EXPLAIN SELECT * FROM sorted_table1 t1
JOIN sorted_table2 t2 ON t1.id = t2.id;

-- 计划: Merge Join
-- 无需额外排序，线性扫描即可
-- 最优选择!
```

### 8.2 反例

**反例1: 大数据集NLJ**

```sql
-- 问题: 两张大表使用Nested Loop
SET enable_hashjoin = off;
SET enable_mergejoin = off;

SELECT * FROM big_table b1
JOIN big_table b2 ON b1.x = b2.y;
-- 10亿次比较!

-- 解决方案: 允许优化器选择Hash Join
SET enable_hashjoin = on;
```

**反例2: work_mem不足**

```sql
-- Hash Join需要内存超过work_mem
SET work_mem = '1MB';

SELECT * FROM big_table b1
JOIN big_table b2 ON b1.x = b2.y;

-- 问题: 哈希表无法装入内存，需要分批处理
-- 磁盘I/O激增，性能下降10倍

-- 解决方案: 增加work_mem
SET work_mem = '256MB';
```

**反例3: 范围连接错误选择**

```sql
-- 范围连接不能使用Hash Join
SELECT * FROM table1 t1
JOIN table2 t2 ON t1.x BETWEEN t2.y AND t2.z;

-- Hash Join需要等值条件
-- 优化器应选择Nested Loop或Merge Join
```
