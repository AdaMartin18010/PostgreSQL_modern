## 1. 分片理论基础

### 1.1 分片问题的形式化定义

分片（Sharding）是将数据集$D$划分为若干子集$D_1, D_2, \ldots, D_n$的过程，使得：

$$D = \bigcup_{i=1}^{n} D_i, \quad D_i \cap D_j = \emptyset \ (i \neq j)$$

对于关系表$R$，分片函数$\mathcal{S}$定义为：

$$\mathcal{S}: R \times \mathcal{K} \rightarrow \{1, 2, \ldots, n\}$$

其中$\mathcal{K}$为分片键集合，$n$为分片数量。分片质量由三个核心指标度量：

**均匀性度量**（Uniformity）：

$$U(\mathcal{S}) = 1 - \frac{\sigma(|D_i|)}{\mu(|D_i|)} = 1 - \frac{\sqrt{\frac{1}{n}\sum_{i=1}^{n}(|D_i| - \bar{D})^2}}{\frac{1}{n}\sum_{i=1}^{n}|D_i|}$$

其中$\bar{D} = \frac{|D|}{n}$为理想均匀分布时的分片大小。

**负载均衡度**（Load Balance Factor）：

$$\mathcal{L}(\mathcal{S}, t) = \frac{\max_{i}(Q_i(t))}{\frac{1}{n}\sum_{i=1}^{n}Q_i(t)}$$

其中$Q_i(t)$表示时间$t$内分片$i$接收的查询数量。

**路由精确度**（Routing Precision）：

$$\mathcal{P}(\mathcal{S}, q) = \frac{|\{i : q \text{ 需要访问 } D_i\}|}{n}$$

理想情况下，点查询应满足$\mathcal{P}(\mathcal{S}, q_{point}) = \frac{1}{n}$。

### 1.2 分片键选择的数学分析

选择分片键$k$时，需考虑其基数（Cardinality）和访问模式：

**基数熵**（Cardinality Entropy）：

$$H(k) = -\sum_{v \in \text{Dom}(k)} p(v) \log_2 p(v)$$

其中$p(v) = \frac{|\{r \in R : r.k = v\}|}{|R|}$，基数熵越高，分片均匀性潜力越大。

**访问熵**（Access Entropy）：

$$H_{access}(k) = -\sum_{v \in \text{Dom}(k)} f(v) \log_2 f(v)$$

其中$f(v)$为键值$v$的访问频率。理想的分布键应最大化$H(k)$同时最小化$H_{access}(k)$的偏差。

### 1.3 分片数量优化

分片数量$n$的优化目标函数：

$$\min_n \mathcal{C}(n) = \alpha \cdot \mathcal{L}(n) + \beta \cdot \mathcal{M}(n) + \gamma \cdot \mathcal{O}(n)$$

其中：

- $\mathcal{L}(n)$：负载不均衡代价，$\mathcal{L}(n) \propto \frac{1}{\sqrt{n}}$
- $\mathcal{M}(n)$：元数据管理开销，$\mathcal{M}(n) \propto n$
- $\mathcal{O}(n)$：跨分片操作开销，$\mathcal{O}(n) \propto n^2$（最坏情况）

对于Citus，推荐的分片数量计算公式：

$$n_{optimal} = \left\lceil c \cdot \frac{|R|}{T_{target}} \right\rceil$$

其中$T_{target}$为目标分片大小（通常100MB-1GB），$c$为增长系数（通常1.5-2.0）。

## 2. 分片策略形式化

### 2.1 哈希分片（Hash Sharding）

哈希分片将分片键$k$映射到分片编号：

$$\mathcal{S}_{hash}(k) = (h(k) \bmod n) + 1$$

其中$h$为哈希函数。Citus使用PostgreSQL内置的`hash_any`函数：

$$h_{citus}(k) = \text{hash\_any}(\text{DatumGetPointer}(k), \text{VARSIZE}(k))$$

**均匀性定理**：若$h$是均匀哈希函数，$k$的取值分布独立同分布，则：

$$\lim_{|R| \to \infty} P(|D_i| = m) = \binom{|R|}{m} \left(\frac{1}{n}\right)^m \left(1-\frac{1}{n}\right)^{|R|-m}$$

期望均匀度：$E[U(\mathcal{S}_{hash})] = 1 - O(\frac{1}{\sqrt{|R|}})$

**数据倾斜度量**：

$$\text{Skew} = \frac{\max_i |D_i|}{\min_i |D_i|}$$

当$\text{Skew} > 3$时，需要考虑重新分片或使用复合分片键。

### 2.2 范围分片（Range Sharding）

范围分片基于键值的有序区间划分：

$$\mathcal{S}_{range}(k) = i \iff k \in [b_{i-1}, b_i)$$

其中边界$b_0 < b_1 < \cdots < b_n$满足：

$$\bigcup_{i=1}^{n} [b_{i-1}, b_i) = \text{Dom}(k)$$

**最优边界选择**：最小化各分片大小方差

$$\min_{b_1,\ldots,b_{n-1}} \sum_{i=1}^{n} \left(|D_i| - \frac{|R|}{n}\right)^2$$

动态范围调整算法（在线场景）：

```python
def adaptive_range_split(shard, threshold):
    """
    自适应范围分片分裂算法

    当分片大小超过阈值时触发分裂
    """
    if shard.size > threshold:
        # 计算中位数作为分裂点
        median = shard.calculate_median_key()

        # 创建两个新分片
        left_shard = Shard(shard.range_start, median)
        right_shard = Shard(median, shard.range_end)

        # 数据迁移
        for record in shard.records:
            if record.key < median:
                left_shard.insert(record)
            else:
                right_shard.insert(record)

        return [left_shard, right_shard]
    return [shard]
```

### 2.3 列表分片（List Sharding）

列表分片基于离散值的显式映射：

$$\mathcal{S}_{list}(k) = M(k)$$

其中$M: \text{Dom}(k) \rightarrow \{1, \ldots, n\}$为显式映射函数。

**映射优化问题**：给定访问频率$f(v)$和分片容量$C_i$，求解：

$$\min_{M} \max_{i} \sum_{v: M(v)=i} f(v) \cdot s(v)$$

约束条件：$\sum_{v: M(v)=i} s(v) \leq C_i, \forall i$

其中$s(v)$为键值$v$对应的数据大小。

### 2.4 分片策略对比分析

| 维度 | 哈希分片 | 范围分片 | 列表分片 |
|------|----------|----------|----------|
| 数据均匀性 | ★★★ | ★★☆ | ★☆☆ |
| 范围查询效率 | ★☆☆ | ★★★ | ★☆☆ |
| 热点处理能力 | ★★★ | ★☆☆ | ★★☆ |
| 动态扩展性 | ★★★ | ★★☆ | ★★☆ |
| 跨分片事务 | 随机分布 | 可预测 | 可控制 |

## 3. 一致性哈希深度分析

### 3.1 传统哈希的问题

传统取模哈希在节点变化时需要重新映射大量数据：

$$\text{Remap Ratio} = \frac{n-1}{n} \approx 1 \text{ (当n较大时)}$$

即几乎所有数据都需要迁移。

### 3.2 一致性哈希的数学模型

一致性哈希将节点和数据映射到同一个哈希环$\mathcal{H} = [0, 2^{32}-1]$：

$$\phi_{node}: \text{Nodes} \rightarrow 2^{\mathcal{H}}$$
$$\phi_{data}: \text{Data} \rightarrow \mathcal{H}$$

数据定位规则：

$$\mathcal{S}_{consistent}(k) = \arg\min_{n} \{h_n \in \phi_{node}(n) : h_n \geq h(k)\}$$

若不存在满足条件的$h_n$，则绕回到环的起点。

### 3.3 虚拟节点与负载均衡

引入虚拟节点（Virtual Nodes）机制，每个物理节点对应$v$个虚拟节点：

$$\phi_{node}^{virtual}(n) = \{h(n, 1), h(n, 2), \ldots, h(n, v)\}$$

**负载方差分析**：

设数据均匀分布，物理节点$n_i$的期望负载：

$$E[\text{Load}(n_i)] = \frac{v}{V} \cdot |D| = \frac{|D|}{N}$$

其中$V = N \cdot v$为总虚拟节点数，$N$为物理节点数。

负载方差上界：

$$\text{Var}(\text{Load}(n_i)) \leq \frac{|D|^2}{V} + O(\frac{|D|}{V^{3/2}})$$

当$v = O(\log N)$时，以高概率保证：

$$\frac{\max_i \text{Load}(n_i)}{\min_i \text{Load}(n_i)} \leq 1 + \epsilon$$

### 3.4 一致性哈希的数据迁移分析

当节点增删时，数据迁移量计算：

**节点加入**：新节点$n_{new}$接管区间$[h_{prev}, h_{new})$的数据

$$\text{Migration Volume}_{add} = |D| \cdot \frac{v}{V} = \frac{|D|}{N}$$

**节点离开**：数据均匀分布到剩余节点

$$\text{Migration Volume}_{remove} = |D| \cdot \frac{v}{V-v} = \frac{|D|}{N-1}$$

**迁移比例**：

$$\text{Migration Ratio} = \frac{1}{N} \text{ (相比传统哈希的 } \frac{N-1}{N} \text{)}$$

### 3.5 Citus中的一致性哈希实现

Citus的shard ID到worker的映射使用一致性哈希：

```c
/* Citus源码：shard_placement.c */

{

/* 虚拟节点查找算法 */
static int
FindNodeOnHashRing(uint32 hashValue, List *workerNodes)
{
    int virtualNodeCount = VirtualNodeCountPerWorker;
    int totalVirtualNodes = list_length(workerNodes) * virtualNodeCount;

    /* 计算目标虚拟节点位置 */
    int targetVirtualNode = hashValue % totalVirtualNodes;

    /* 映射回物理节点 */
    int workerIndex = targetVirtualNode / virtualNodeCount;

    return workerIndex;
}
```

## 4. 查询路由理论

### 4.1 查询路由的形式化模型

查询$q$的路由决策函数：

$$\mathcal{R}(q) = \{s_1, s_2, \ldots, s_m\} \subseteq \{1, \ldots, n\}$$

对于带有谓词$P$的查询，路由可优化为：

$$\mathcal{R}(q, P) = \{i : D_i \cap \sigma_P(D) \neq \emptyset\}$$

其中$\sigma_P$为选择操作。

### 4.2 谓词下推与分片裁剪

**谓词蕴含分析**：

对于范围谓词$P: k \in [a, b]$和分片$D_i$对应范围$[b_{i-1}, b_i)$：

$$D_i \cap \sigma_P(D) = \emptyset \iff b_i \leq a \lor b_{i-1} \geq b$$

即当查询范围与分片范围不相交时，该分片可被裁剪。

**分片裁剪率**：

$$\eta = \frac{n - |\mathcal{R}(q, P)|}{n} = 1 - \frac{|\mathcal{R}(q, P)|}{n}$$

对于点查询$P: k = v$，理想情况下$\eta = 1 - \frac{1}{n}$。

### 4.3 分布式查询计划的成本模型

查询执行总成本：

$$\mathcal{C}_{total}(q) = \mathcal{C}_{local} + \mathcal{C}_{network} + \mathcal{C}_{coordination}$$

其中：

**本地执行成本**：

$$\mathcal{C}_{local} = \sum_{i \in \mathcal{R}(q)} c_{cpu} \cdot |D_i'| \cdot \log(|D_i'|)$$

$D_i'$为分片$i$的筛选后数据。

**网络传输成本**：

$$\mathcal{C}_{network} = c_{latency} \cdot |\mathcal{R}(q)| + c_{bw} \cdot |R_{intermediate}|$$

**协调成本**：

$$\mathcal{C}_{coordination} = c_{sync} \cdot \log(|\mathcal{R}(q)|)$$

### 4.5 自适应路由优化

基于负载的自适应路由：

$$\mathcal{R}_{adaptive}(q) = \arg\min_{p \in \text{Placements}(s)} \mathcal{L}(p)$$

其中$\mathcal{L}(p)$为placement$p$的当前负载。

```python
class AdaptiveRouter:
    """
    自适应查询路由器

    根据worker负载动态选择最佳路由
    """

    def __init__(self, shard_manager):
        self.shard_manager = shard_manager
        self.load_metrics = LoadMetricsCollector()

    def route_query(self, query, distribution_key):
        """
        根据负载选择最优路由
        """
        # 获取候选分片
        shard_id = self.compute_shard_id(distribution_key)
        placements = self.shard_manager.get_placements(shard_id)

        # 获取各placement的实时负载
        load_scores = {
            p: self.load_metrics.get_load_score(p.worker)
            for p in placements
        }

        # 选择负载最低的placement
        best_placement = min(placements,
                           key=lambda p: load_scores[p])

        return RoutingDecision(
            shard_id=shard_id,
            placement=best_placement,
            estimated_latency=self.estimate_latency(best_placement)
        )

    def estimate_latency(self, placement):
        """
        估计查询延迟
        """
        base_latency = placement.network_latency
        queue_delay = placement.pending_queries * placement.avg_query_time
        return base_latency + queue_delay
```

## 5. 分布式连接理论

### 5.1 连接类型的形式化分类

分布式连接的复杂度分类：

**共置连接（Co-located Join）**：

$$\mathcal{J}_{colocated}(R, S) = \{(r, s) : r \bowtie_{R.k=S.k} s \land \mathcal{S}_R(r.k) = \mathcal{S}_S(s.k)\}$$

当$R$和$S$使用相同的分片函数和键时，连接可完全在本地执行。

**单分片连接（Single-Shard Join）**：

$$\mathcal{J}_{single}(R, S, v) = \{(r, s) : r \in D_R^i \land s \in D_S^j \land i = \mathcal{S}_R(v)\}$$

通过分布键过滤，限制在单分片内执行。

**跨分片连接（Cross-Shard Join）**：

$$\mathcal{J}_{cross}(R, S) = \bigcup_{i=1}^{n} \bigcup_{j=1}^{m} \{(r, s) : r \in D_R^i \land s \in D_S^j \land r.k = s.k\}$$

需要数据重分布或广播。

### 5.2 连接策略选择算法

连接策略选择的决策树：

```
if 表R和S共置:
    选择 Co-located Join
elif 查询包含分布键等值条件:
    选择 Single-Shard Join
elif min(|R|, |S|) < broadcast_threshold:
    选择 Broadcast Join
else:
    选择 Repartition Join
```

**成本比较模型**：

| 连接类型 | 网络成本 | 内存成本 | 适用场景 |
|----------|----------|----------|----------|
| Co-located | $O(1)$ | $O(|R_i| + |S_i|)$ | 同分布键 |
| Broadcast | $O(n \cdot \min(|R|, |S|))$ | $O(\min(|R|, |S|))$ | 小表join大表 |
| Repartition | $O(|R| + |S|)$ | $O(\max|D_i|)$ | 大表join大表 |

### 5.3 共置连接的数学保证

**共置定理**：若两表$R$和$S$满足：

1. 使用相同的分片函数$\mathcal{S}$
2. 使用兼容的分片键（$R.k_1$和$S.k_2$满足$k_1 = f(k_2)$或$k_2 = f(k_1)$）

则对于任意连接键值$v$，匹配的记录必然位于同一分片：

$$\forall r \in R, s \in S: r.k_R = s.k_S = v \Rightarrow \mathcal{S}(r.k_R) = \mathcal{S}(s.k_S)$$

**共置连接执行计划**：

```sql
-- 共置连接示例：用户表和订单表使用相同的user_id分片
SELECT u.name, COUNT(o.order_id)
FROM users u
JOIN orders o ON u.user_id = o.user_id
GROUP BY u.name;

/*
 * 执行计划：每个worker本地执行
 * Worker 1: SELECT ... WHERE hash(user_id) % 32 = 0
 * Worker 2: SELECT ... WHERE hash(user_id) % 32 = 1
 * ...
 * Coordinator: 汇总各worker结果
 */
```

### 5.4 跨分片重分区算法

重分区连接（Repartition Join）的数据shuffle成本：

$$\text{Shuffle Cost} = c_{network} \cdot (|R| + |S|)$$

优化后的两阶段聚合：

```python
class RepartitionJoinExecutor:
    """
    重分区连接执行器

    阶段1: 按连接键重新分布数据
    阶段2: 各节点执行本地连接
    """

    def execute(self, table_r, table_s, join_key):
        """
        执行重分区连接
        """
        # 阶段1: 重分区
        repartitioned_r = self.repartition(table_r, join_key)
        repartitioned_s = self.repartition(table_s, join_key)

        # 阶段2: 本地连接
        results = []
        for shard_id in range(self.num_shards):
            local_result = self.local_join(
                repartitioned_r.get_shard(shard_id),
                repartitioned_s.get_shard(shard_id),
                join_key
            )
            results.append(local_result)

        # 合并结果
        return self.merge_results(results)

    def repartition(self, table, join_key):
        """
        按连接键重新分布数据

        使用与目标表相同的哈希函数
        """
        partitions = defaultdict(list)

        for record in table.scan():
            key_value = record[join_key]
            # 使用连接键的哈希决定目标分片
            target_shard = self.hash_function(key_value) % self.num_shards
            partitions[target_shard].append(record)

        # 并行发送到各worker
        self.parallel_send(partitions)

        return PartitionedTable(partitions)
```

### 5.5 广播连接优化

广播连接适用于小表场景：

**广播阈值决策**：

$$\text{Broadcast}(T) \iff |T| < \frac{C_{network}}{C_{cpu} \cdot n} \cdot \min_{i}|D_i|$$

其中$C_{network}$为单位数据传输成本，$C_{cpu}$为本地连接处理成本。

```c
/* Citus源码：multi_physical_planner.c */
static bool
ShouldUseBroadcastJoin(MultiNode *joinNode, MultiTable *smallerTable)
{
    /* 获取小表的大小估计 */
    double smallerTableSize = smallerTable->relationSize;

    /* 获取worker数量 */
    int workerCount = list_length(workerNodeList);

    /* 计算广播成本 vs 重分区成本 */
    double broadcastCost = smallerTableSize * workerCount;
    double repartitionCost =
        joinNode->leftRelationSize + joinNode->rightRelationSize;

    /* 考虑网络带宽和CPU成本权重 */
    double adjustedBroadcastCost =
        broadcastCost * NetworkCostFactor;
    double adjustedRepartitionCost =
        repartitionCost * (NetworkCostFactor + CPUCostFactor);

    return adjustedBroadcastCost < adjustedRepartitionCost;
}
```

## 6. 再平衡理论

### 6.1 数据再平衡的形式化定义

再平衡是将数据从过载分片迁移到欠载分片的过程：

给定当前分布$\{D_1, D_2, \ldots, D_n\}$，目标分布$\{D_1', D_2', \ldots, D_m'\}$，求解：

$$\min_{\mathcal{T}} \sum_{i,j} \text{cost}(\mathcal{T}_{i,j})$$

约束条件：

- $\forall j: |D_j'| \approx \frac{|D|}{m}$（均匀性约束）
- $\forall i: D_i = \bigcup_{j} \mathcal{T}_{i,j}$（完整性约束）

其中$\mathcal{T}_{i,j}$表示从分片$i$迁移到分片$j$的数据集合。

### 6.2 再平衡算法

**基于阈值的再平衡触发**：

$$\text{Trigger Rebalance} \iff \max_i \frac{|D_i|}{\bar{D}} > \theta_{upper} \lor \min_i \frac{|D_i|}{\bar{D}} < \theta_{lower}$$

其中$\theta_{upper} = 1.3, \theta_{lower} = 0.7$为经验阈值。

**最小迁移再平衡算法**：

```python
class RebalancePlanner:
    """
    最小数据迁移再平衡规划器

    基于网络流算法计算最优迁移方案
    """

    def plan_rebalance(self, current_shards, target_count):
        """
        规划再平衡操作

        目标：最小化总数据迁移量
        """
        target_size = sum(s.size for s in current_shards) / target_count

        # 计算每个分片的盈余/ deficit
        shard_balance = []
        for shard in current_shards:
            delta = shard.size - target_size
            shard_balance.append((shard, delta))

        # 分离源和目标
        sources = [(s, d) for s, d in shard_balance if d > 0]  # 需要迁出
        targets = [(s, d) for s, d in shard_balance if d < 0]  # 需要迁入

        # 贪心匹配
        migrations = []
        sources.sort(key=lambda x: x[1], reverse=True)
        targets.sort(key=lambda x: x[1])

        for src_shard, src_delta in sources:
            remaining = src_delta
            for tgt_shard, tgt_delta in targets:
                if remaining <= 0:
                    break
                if tgt_delta >= 0:
                    continue

                # 计算可迁移量
                transfer = min(remaining, -tgt_delta)

                # 计算迁移范围
                migrate_range = self.calculate_migration_range(
                    src_shard, transfer
                )

                migrations.append(MigrationPlan(
                    source=src_shard,
                    target=tgt_shard,
                    range=migrate_range,
                    size=transfer
                ))

                remaining -= transfer

        return migrations

    def calculate_migration_range(self, shard, size_to_migrate):
        """
        计算需要迁移的数据范围

        优先选择冷数据或连续范围
        """
        # 获取分片内数据分布
        distribution = shard.get_key_distribution()

        # 选择能覆盖目标大小的连续范围
        cumulative = 0
        start_key = None
        end_key = None

        for key, count in distribution.sorted_items():
            if start_key is None:
                start_key = key
            cumulative += count
            end_key = key

            if cumulative >= size_to_migrate:
                break

        return KeyRange(start_key, end_key)
```

### 6.3 在线再平衡的成本分析

在线再平衡对查询性能的影响：

$$\text{Performance Degradation} = \frac{\mathcal{C}_{query}^{rebalancing}}{\mathcal{C}_{query}^{normal}} - 1$$

对于正在迁移的分片，查询需要访问源和目标两个位置：

$$\mathcal{C}_{query}^{rebalancing} = p_{migrating} \cdot (\mathcal{C}_{src} + \mathcal{C}_{tgt}) + (1 - p_{migrating}) \cdot \mathcal{C}_{normal}$$

其中$p_{migrating}$为查询涉及正在迁移数据的概率。

**渐进式再平衡策略**：

```c
/* Citus源码：shard_rebalancer.c */
static void
RebalanceShardPlacement(ShardPlacement *source, ShardPlacement *target)
{
    /* 阶段1: 创建目标分片 */
    CreateShardPlacementOnWorker(target->workerNode, source->shardId);

    /* 阶段2: 双写阶段（渐进式） */
    EnableDualWrites(source, target);

    /* 阶段3: 批量数据迁移 */
    CopyShardDataInBatches(source, target,
                          batchSize = 10000,  /* 每批10k行 */
                          sleepBetweenBatches = 100ms);

    /* 阶段4: 追赶同步 */
    SyncIncrementalChanges(source, target);

    /* 阶段5: 切换读流量 */
    RedirectReadsToTarget(target);

    /* 阶段6: 禁用双写，清理源分片 */
    DisableDualWrites(source);
    DropShardPlacement(source);
}
```

### 9.3 正确实例：一致性哈希扩容

```python
# ✅ 正确的扩容实践
class CitusClusterExpansion:
    """
    Citus集群扩容最佳实践
    """

    def expand_cluster(self, new_workers):
        """
        添加新worker节点并重新平衡数据
        """
        current_workers = self.get_current_workers()
        total_workers = len(current_workers) + len(new_workers)

        print(f"扩容: {len(current_workers)} → {total_workers} workers")

        # 步骤1: 添加新worker到集群
        for worker in new_workers:
            self.add_worker_to_cluster(worker)

        # 步骤2: 使用citus_rebalance_start进行在线再平衡
        # 基于一致性哈希，仅迁移约 1/N 的数据
        rebalance_plan = self.generate_rebalance_plan()

        for migration in rebalance_plan:
            print(f"迁移分片 {migration.shard_id}: "
                  f"{migration.source} → {migration.target} "
                  f"({migration.size_mb}MB)")

        # 步骤3: 执行渐进式再平衡
        self.execute_rebalance(rebalance_plan)

        # 验证均匀性
        uniformity = self.check_uniformity()
        print(f"再平衡后均匀度: {uniformity:.2%}")

    def generate_rebalance_plan(self):
        """
        生成最小迁移计划

        利用一致性哈希，数据移动量 ≈ 1/N
        """
        shards = self.get_all_shards()
        workers = self.get_all_workers()

        # 基于一致性哈希环计算新位置
        plan = []
        for shard in shards:
            current_worker = shard.current_worker
            optimal_worker = self.find_optimal_worker_consistent_hash(
                shard.id, workers
            )

            if current_worker != optimal_worker:
                plan.append(Migration(
                    shard_id=shard.id,
                    source=current_worker,
                    target=optimal_worker,
                    size_mb=shard.size
                ))

        # 预期迁移比例: (N_new - N_old) / N_new
        total_size = sum(s.size for s in shards)
        migrate_size = sum(m.size for m in plan)
        ratio = migrate_size / total_size

        print(f"预期迁移比例: ~{ratio:.1%} "
              f"(理论值: ~{1/len(workers):.1%})")

        return plan
```
