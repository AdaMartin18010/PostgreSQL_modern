## 1. 复制理论基础

### 1.1 复制模型分类

**定义 1.1 (复制模型)**:
复制是将数据从一个节点传播到其他节点的过程：

$$
\mathcal{R} := \langle Primary, Replicas, Stream, Consistency \rangle
$$

**分类维度**:

| 维度 | 类型 | 说明 |
|------|------|------|
| **复制粒度** | 物理/逻辑 | 块级vs行级 |
| **同步方式** | 同步/异步 | 等待确认vs立即返回 |
| **拓扑结构** | 主从/多主/链式 | 单主vs多主 |
| **一致性模型** | 强/最终 | 实时一致vs延迟一致 |

### 1.2 逻辑复制vs物理复制

**定义 1.2 (物理复制)**:
物理复制传输WAL记录，实现块级复制：

$$
\text{Physical} := \{wal\_record \mid record \in WAL\}
$$

**定义 1.3 (逻辑复制)**:
逻辑复制传输逻辑变更（INSERT/UPDATE/DELETE），实现行级复制：

$$
\text{Logical} := \{change \mid change = f(row\_old, row\_new)\}
$$

**对比矩阵**:

| 特性 | 物理复制 | 逻辑复制 |
|------|----------|----------|
| **复制粒度** | 块级 (8KB) | 行级 |
| **DDL复制** | 是 | 否 (需手动) |
| **版本兼容** | 严格相同 | 跨版本 |
| **选择性复制** | 否 | 是 (表级) |
| **冲突处理** | 无 | 应用层处理 |
| **延迟** | 低 (<1ms) | 中 (1-100ms) |
| **适用场景** | HA failover | 数据分发、ETL |

## 2. 形式化模型

### 2.1 发布-订阅模型

**定义 2.1 (发布者)**:
发布者是一个元组：

$$
P := \langle Tables, Publications, Slots, Decoder \rangle
$$

**定义 2.2 (订阅者)**:
订阅者是一个元组：

$$
S := \langle Connection, Worker, Apply, Tables \rangle
$$

**复制流**:

$$
Stream := \{msg_1, msg_2, ..., msg_n\}
$$

其中$msg_i$是逻辑变更消息。

### 2.2 复制槽机制

**定义 2.3 (复制槽)**:
复制槽跟踪订阅者的消费位置：

$$
Slot := \langle name, plugin, lsn, xmin \rangle
$$

其中：

- $lsn$: 已发送到的日志序列号
- $xmin$: 最早需要保留的事务ID

**定理 2.1 (复制槽保证)**:
复制槽保证即使订阅者断开，发布者也保留所需的WAL。

*证明*: 复制槽的xmin阻止VACUUM清理旧版本，lsn阻止WAL回收。∎

### 2.3 一致性模型

**定义 2.4 (最终一致性)**:
系统保证如果没有新的更新，最终所有副本将达到一致状态：

$$
\forall t > T: \forall r_1, r_2 \in Replicas: State(r_1, t) = State(r_2, t)
$$

**复制延迟**:

$$
Lag := T_{commit}^{primary} - T_{apply}^{replica}
$$

## 3. 冲突处理理论

### 3.1 冲突类型

**定义 3.1 (主键冲突)**:
两个并发插入具有相同主键：

$$
Insert_1(pk) \parallel Insert_2(pk) \Rightarrow Conflict(pk)
$$

**定义 3.2 (更新丢失)**:
两个并发更新同一行：

$$
Update_1(row) \parallel Update_2(row) \Rightarrow LostUpdate
$$

### 3.2 解决策略

| 策略 | 算法 | 适用场景 |
|------|------|----------|
| **Last-Write-Wins** | 时间戳比较 | 简单覆盖 |
| **First-Write-Wins** | 拒绝后续 | 防止覆盖 |
| **Merge** | 自定义合并 | 复杂数据 |
| **Abort** | 回滚事务 | 严格一致 |

### 4.2 关键源码

```c
// src/backend/replication/logical/worker.c

/* 逻辑复制工作进程 */
void
ApplyWorkerMain(Datum main_arg)
{
    /* 1. 连接到发布者 */
    conn = libpqrcv_connect(conninfo);

    /* 2. 启动复制 */
    libpqrcv_startstreaming(conn, start_lsn);

    while (!got_SIGTERM) {
        /* 3. 接收变更 */
        char *msg = libpqrcv_receive(conn, &len, &lsn);

        /* 4. 解码变更 */
        LogicalRepMsgType action = get_message_type(msg);

        switch (action) {
            case LOGICAL_REP_MSG_INSERT:
                apply_insert(msg);
                break;
            case LOGICAL_REP_MSG_UPDATE:
                apply_update(msg);
                break;
            case LOGICAL_REP_MSG_DELETE:
                apply_delete(msg);
                break;
            case LOGICAL_REP_MSG_COMMIT:
                apply_commit(msg);
                break;
        }
    }
}

/* 应用INSERT */
static void
apply_insert(LogicalRepMsg *msg)
{
    /* 准备插入语句 */
    PGresult *res = PQexecPrepared(conn, "INSERT_STMT",
                                   n_params, params, ...);

    if (PQresultStatus(res) != PGRES_COMMAND_OK) {
        /* 处理冲突 */
        handle_conflict(msg, res);
    }
}
```

## 6. 实例与反例

### 6.1 正例: 正确使用逻辑复制

**场景1: 数据仓库ETL**

```sql
-- 发布者: 生产库
CREATE PUBLICATION prod_tables
FOR TABLE orders, order_items, customers;

-- 订阅者: 数据仓库
CREATE SUBSCRIPTION data_warehouse
CONNECTION 'host=prod dbname=postgres'
PUBLICATION prod_tables;
```

**场景2: 跨版本升级**

```sql
-- PG16 发布
CREATE PUBLICATION upgrade_pub FOR ALL TABLES;

-- PG18 订阅
CREATE SUBSCRIPTION upgrade_sub
CONNECTION 'host=oldpg dbname=postgres'
PUBLICATION upgrade_pub;

-- 零停机升级
```

### 6.2 反例: 常见错误

**反例1: DDL不复制**

```sql
-- 发布者执行
ALTER TABLE orders ADD COLUMN discount NUMERIC;

-- 订阅者不会自动添加列!
-- 导致复制失败

-- 解决方案: 手动在订阅者执行DDL
```

**反例2: 大事务延迟**

```sql
-- 大事务导致复制延迟
BEGIN;
INSERT INTO logs SELECT * FROM huge_table;
COMMIT;

-- 订阅者需要等待整个事务完成才能应用
-- 解决方案: 分批提交
```
