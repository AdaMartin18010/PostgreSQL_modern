## 1. 理论基础与动机

### 1.1 为什么需要B+Tree

**问题背景**: 数据库需要支持高效的**范围查询**和**等值查询**。

**哈希索引的局限**:

- 仅支持等值查询: $O(1)$
- 不支持范围查询: $O(n)$

**二叉搜索树的局限**:

- 磁盘I/O次数多: $O(\log_2 n)$
- 树高不平衡

**B+Tree的优势**:

- 磁盘友好: 节点大小 = 页大小
- 范围查询高效: $O(\log_d n + k)$
- 始终平衡

### 1.2 形式化定义

**定义 1.1 (B+Tree)**:
一个阶数为$d$的B+Tree是满足以下性质的树:

$$
\mathcal{B}_d := \langle r, N, L, K, P \rangle
$$

其中:

- $r$: 根节点
- $N$: 内部节点集合
- $L$: 叶节点集合
- $K$: 键集合
- $P$: 指针集合

**结构约束**:

1. **阶数约束**: 每个节点最多$2d-1$个键，最少$d-1$个键（根节点除外）
2. **叶节点链接**: 所有叶节点通过双向链表连接
3. **平衡性**: 所有叶节点在同一层

**键约束**:

对于内部节点$n$，其键$k_1 < k_2 < ... < k_m$和指针$p_0, p_1, ..., p_m$满足:

$$
\forall x \in \text{subtree}(p_i):
\begin{cases}
k_i < x < k_{i+1} & \text{if } 0 < i < m \\
x < k_1 & \text{if } i = 0 \\
k_m < x & \text{if } i = m
\end{cases}
$$

## 2. 算法形式化

### 2.1 搜索算法

**算法 2.1 (B+Tree搜索)**:

```
Search(key, node):
    IF node is leaf:
        IF key in node.keys:
            RETURN node.record[key]
        ELSE:
            RETURN NOT_FOUND

    // 内部节点: 找到合适的子节点
    i = 0
    WHILE i < node.key_count AND key >= node.keys[i]:
        i = i + 1

    RETURN Search(key, node.pointers[i])
```

**定理 2.1 (搜索复杂度)**:
B+Tree搜索的时间复杂度为 $O(\log_d n)$。

*证明*:

- 树高$h \leq \log_d n$
- 每层最多1次磁盘I/O
- 叶节点内二分查找: $O(\log d)$
- 总复杂度: $O(\log_d n + \log d) = O(\log_d n)$∎

### 2.2 插入算法

**算法 2.2 (B+Tree插入)**:

```
Insert(key, value):
    leaf = FindLeaf(key)

    IF leaf has space:
        InsertIntoLeaf(leaf, key, value)
    ELSE:
        // 叶节点分裂
        new_leaf = SplitLeaf(leaf)
        InsertIntoLeaf(new_leaf, key, value)

        // 提升键到父节点
        parent_key = new_leaf.keys[0]
        InsertIntoParent(leaf, parent_key, new_leaf)

SplitLeaf(leaf):
    new_leaf = CreateLeaf()
    mid = leaf.key_count / 2

    Move keys[mid:] from leaf to new_leaf
    Link new_leaf after leaf

    RETURN new_leaf
```

**定理 2.2 (插入复杂度)**:
B+Tree插入的平均时间复杂度为 $O(\log_d n)$，最坏情况下为 $O(\log_d n)$（需要分裂到根）。

### 2.3 分裂传播分析

**定理 2.3 (分裂概率)**:
节点分裂的概率与填充因子$\rho$相关:

$$
P_{split} = 1 - \rho
$$

其中$\rho$是平均填充因子，通常$\rho \approx 0.67$。

*证明*: 当节点已满时需要分裂，填充因子越低，分裂概率越高。∎

## 3. 并发控制

### 3.1 B-Link Tree协议

**Lehman & Yao (1981)** 提出的B-Link Tree允许高并发:

**核心思想**: 在节点分裂期间，允许其他操作继续进行。

**协议规则**:

1. **搜索操作**:
   - 不需要锁
   - 如果到达的节点已被分裂，通过"右兄弟指针"继续搜索

2. **插入操作**:
   - 乐观锁: 先获取乐观锁
   - 如果冲突，升级为先悲锁

**形式化定义**:

```
节点结构:
┌─────────────────────────────────────┐
│ keys[0..m-1]                        │
│ pointers[0..m]                      │
│ right_sibling →                     │  // 新增: 右兄弟指针
│ high_key                            │  // 新增: 最大键值
└─────────────────────────────────────┘
```

### 3.2 锁类型

| 锁类型 | 模式 | 兼容性 | 使用场景 |
|--------|------|--------|----------|
| **读锁** | S | 与S兼容 | 搜索 |
| **写锁** | X | 不兼容 | 插入、删除 |
| **意向读锁** | IS | 与IS、IX兼容 | 准备读子节点 |
| **意向写锁** | IX | 与IS、IX兼容 | 准备写子节点 |

**锁协议**:

$$
\text{Lock}(node, mode) \Rightarrow \forall l \in \text{parent}(node): \text{IntentionLock}(l, mode)
$$

### 4.2 搜索实现

```c
// src/backend/access/nbtree/nbtsearch.c

BTStack
_bt_search(Relation rel, int keysz, ScanKey scankey,
           bool nextkey, Buffer *bufP, int access)
{
    // 从根节点开始
    buf = _bt_getroot(rel, access);

    while (!P_ISLEAF(opaque)) {
        // 在内部节点中找到合适的子节点
        offnum = _bt_binsrch(rel, buf, keysz, scankey, nextkey);

        // 获取子节点指针
        itemid = PageGetItemId(pg, offnum);
        itup = (IndexTuple) PageGetItem(pg, itemid);
        child = ItemPointerGetBlockNumber(&itup->t_tid);

        // 移动到子节点
        buf = _bt_relandgetbuf(rel, buf, child, access);
        opaque = (BTPageOpaque) PageGetSpecialPointer(pg);
    }

    return buf;
}
```

### 4.3 插入实现

```c
// src/backend/access/nbtree/nbtinsert.c

bool
_bt_insertonpg(Relation rel, Buffer buf, BTStack stack,
               IndexTuple itup, OffsetNumber newitemoff)
{
    Page page = BufferGetPage(buf);

    // 检查是否有足够空间
    if (PageGetFreeSpace(page) < itemsz) {
        // 需要分裂
        ritem = _bt_split(rel, buf, stack->bts_parent,
                         newitemoff, itemsz, itup);

        // 将新键插入父节点
        _bt_insertonpg(rel, pbuf, stack->bts_parent + 1,
                      ritem.hiitem, ritem.offnum);
    } else {
        // 直接插入
        _bt_pgaddtup(page, itemsz, itup, newitemoff);
    }

    return true;
}
```

## 5. 思维表征

### 5.1 B+Tree vs B-Tree 对比矩阵

| 特性 | B-Tree | B+Tree | 说明 |
|------|--------|--------|------|
| **数据存储** | 内部节点+叶节点 | 仅叶节点 | B+Tree内部节点不存数据 |
| **叶节点链接** | 无 | 双向链表 | B+Tree支持高效范围扫描 |
| **键重复** | 是 | 是 | 都需要 |
| **空间利用率** | 低 | 高 | B+Tree内部节点更小 |
| **范围查询** | $O(\log n + k)$ | $O(\log n + k)$ | B+Tree顺序访问更快 |
| **等值查询** | $O(\log n)$ | $O(\log n)$ | 相当 |
| **插入复杂度** | $O(\log n)$ | $O(\log n)$ | 相当 |
| **删除复杂度** | $O(\log n)$ | $O(\log n)$ | 相当 |
| **适用场景** | 通用 | 数据库索引 | B+Tree更适合磁盘存储 |

```

Step 2: 检查空间
    Node full (3 keys, max=3) → Need split

Step 3: 分裂节点
    [20,22,25] ──Split──→ [20] + [22,25]
                          │      │
                          │      └─ New Leaf

Step 4: 插入新键
    Insert 21 into [22,25] → [21,22,25]

Step 5: 更新父节点
    Parent [10,20,30] add pointer to new leaf
    → Promote key 22 to parent

Step 6: 如果父节点也满，递归分裂
    (本例中父节点未满，停止)
```

## 6. 实例与反例

### 6.1 正例: 正确使用B+Tree

**场景1: 用户表主键索引**

```sql
CREATE TABLE users (
    user_id BIGINT PRIMARY KEY,  -- 自动创建B-tree索引
    username VARCHAR(50),
    created_at TIMESTAMPTZ
);

-- 等值查询: O(log n)
SELECT * FROM users WHERE user_id = 12345;

-- 范围查询: O(log n + k)
SELECT * FROM users WHERE user_id BETWEEN 10000 AND 20000;
```

**性能**:

- 1000万数据，等值查询 < 1ms
- 范围查询1000行 < 5ms

### 6.2 反例: 错误使用B+Tree

**反例1: 低选择性列索引**

```sql
-- 错误: 性别只有两种值
CREATE INDEX idx_gender ON users(gender);

-- 查询只会返回50%数据
SELECT * FROM users WHERE gender = 'M';
-- 优化器选择全表扫描，索引无用
```

**解决方案**:

```sql
-- 使用部分索引
CREATE INDEX idx_male_users ON users(user_id) WHERE gender = 'M';
```

**反例2: 索引列参与运算**

```sql
-- 错误: 函数调用导致索引失效
SELECT * FROM orders WHERE EXTRACT(YEAR FROM created_at) = 2025;
-- 全表扫描!

-- 正确做法: 范围查询
SELECT * FROM orders
WHERE created_at >= '2025-01-01'
  AND created_at < '2026-01-01';
```

### 6.3 边界条件

**边界1: 极端倾斜数据**

```sql
-- 90%数据有相同键值
INSERT INTO logs SELECT 'ERROR', ... FROM generate_series(1, 9000000);
INSERT INTO logs SELECT 'INFO', ... FROM generate_series(1, 1000000);

-- 查询 ERROR 会返回900万行
-- B-tree需要遍历大量叶子页
```
