## 1. 日志理论基础

### 1.1 为什么需要WAL

**问题背景**: 数据库需要在崩溃后保持一致性。

**直接修改的问题**:

- 数据页写入可能部分完成
- 无法回滚未提交事务
- 无法重做已提交事务

**定理 1.1 (WAL必要性)**:
没有日志机制，数据库无法在崩溃后保证ACID特性。

*证明*:

1. 原子性: 无法撤销部分完成的修改
2. 持久性: 无法确认已提交事务
3. 一致性: 无法恢复到一致状态
∎

### 1.2 WAL原则

**定义 1.1 (先写日志原则)**:
任何数据修改必须先写入日志，再写入数据页：

$$
\forall op \in Write: Log(op) \prec Apply(op)
$$

**顺序保证**:

$$
LSN_{log} > LSN_{data} \Rightarrow \text{Recoverable}
$$

## 2. 日志形式化模型

### 2.1 日志记录结构

**定义 2.1 (日志记录)**:

$$
LogRecord := \langle LSN, PrevLSN, XID, Type, Info \rangle
$$

其中：

- $LSN$: 日志序列号
- $PrevLSN$: 前一条记录的LSN
- $XID$: 事务ID
- $Type$: 记录类型
- $Info$: 记录特定信息

**记录类型**:

| 类型 | 内容 | 用途 |
|------|------|------|
| **XLOG_XACT_COMMIT** | 提交记录 | 事务提交标记 |
| **XLOG_XACT_ABORT** | 中止记录 | 事务回滚标记 |
| **XLOG_HEAP_INSERT** | 插入记录 | 重做插入 |
| **XLOG_HEAP_UPDATE** | 更新记录 | 重做/撤销更新 |
| **XLOG_HEAP_DELETE** | 删除记录 | 重做删除 |
| **XLOG_CHECKPOINT** | 检查点 | 恢复起点 |

### 2.2 LSN机制

**定义 2.2 (LSN)**:
日志序列号是一个单调递增的64位整数：

$$
LSN: \mathbb{N}_{64}, \quad LSN_{i+1} = LSN_i + length(record_i)
$$

**定理 2.1 (LSN单调性)**:
LSN严格单调递增，保证日志顺序。

*证明*: 每个记录的长度为正，因此$LSN_{i+1} > LSN_i$。∎

### 2.3 日志链

**定义 2.3 (日志链)**:
通过PrevLSN链接同一事务的所有记录：

$$
Chain(T) := \{r \mid r.XID = T \land r.PrevLSN \rightarrow r.LSN\}
$$

## 3. ARIES算法形式化

### 3.1 三阶段恢复

**ARIES** = Algorithm for Recovery and Isolation Exploiting Semantics

**阶段1: 分析 (Analysis)**

- 从最后一个检查点开始扫描日志
- 确定DirtyPageSet和ActiveTransactionSet

**阶段2: 重做 (Redo)**

- 重做所有已提交事务的修改
- 从RedoLSN开始扫描

**阶段3: 撤销 (Undo)**

- 撤销所有未提交事务的修改
- 从后向前扫描

**定理 3.1 (ARIES正确性)**:
ARIES恢复后，数据库状态等价于崩溃前已提交事务的效果。

*证明*:

1. 重做阶段: 所有已提交事务的修改被重新应用
2. 撤销阶段: 所有未提交事务的修改被撤销
3. 最终状态只包含已提交事务的效果
∎

### 3.2 模糊检查点

**定义 3.1 (模糊检查点)**:
检查点记录不需要数据页同步刷盘：

$$
Checkpoint := \langle LSN, DirtyPages, ActiveTxns \rangle
$$

**重做范围**:

$$
RedoLSN = \min_{p \in DirtyPages} PageLSN(p)
$$

## 4. PostgreSQL实现

### 4.1 WAL文件结构

```
pg_wal/
├── 000000010000000000000001  (16MB)
├── 000000010000000000000002
└── ...

WAL文件命名: TTTTLLLLLLLLLLLL
- TTTT: 时间线ID
- LLLLLLLLLLLL: 日志ID和段号
```

### 4.2 关键源码

```c
// src/backend/access/transam/xlog.c

/* XLogInsert: 插入WAL记录 */
XLogRecPtr
XLogInsert(RmgrId rmid, uint8 info, XLogRecData *rdata)
{
    XLogCtlInsert *Insert = &XLogCtl->Insert;
    XLogRecord *recptr;

    /* 1. 预留WAL空间 */
    START_CRIT_SECTION();

    /* 2. 填充记录头 */
    recptr->xl_prev = Insert->PrevRecord;
    recptr->xl_xid = GetCurrentTransactionIdIfAny();
    recptr->xl_rmid = rmid;
    recptr->xl_info = info;

    /* 3. 复制数据 */
    CopyXLogRecord(rdata, recptr);

    /* 4. 更新插入位置 */
    Insert->CurrBytePos += size;
    Insert->PrevRecord = recptr->xl_prev;

    END_CRIT_SECTION();

    return recptr->xl_prev;
}

/* XLogFlush: 强制刷盘 */
void
XLogFlush(XLogRecPtr record)
{
    /* 等待WAL写盘 */
    while (XLogCtl->Write.LogwrtResult.Flush < record) {
        XLogWrite(NULL, false);
        pg_usleep(1000L);
    }
}
```

### 4.3 检查点机制

```c
/* CreateCheckPoint */
void
CreateCheckPoint(int flags)
{
    /* 1. 获取排他锁 */
    LWLockAcquire(CheckpointLock, LW_EXCLUSIVE);

    /* 2. 记录检查点位置 */
    checkPoint.redo = Insert->PrevBytePos;

    /* 3. 刷盘所有脏页 */
    BufferSync(flags);

    /* 4. 写入检查点记录 */
    XLogBeginInsert();
    XLogRegisterData(&checkPoint, sizeof(checkPoint));
    recptr = XLogInsert(RM_XLOG_ID, XLOG_CHECKPOINT_SHUTDOWN);

    /* 5. 强制刷盘 */
    XLogFlush(recptr);

    /* 6. 更新控制文件 */
    UpdateControlFile();

    LWLockRelease(CheckpointLock);
}
```

## 5. 思维表征

### 5.1 WAL数据流向图

```
Transaction
    │
    │ 1. INSERT/UPDATE/DELETE
    ↓
┌─────────────────┐
│ XLogInsert()    │
│ 生成WAL记录      │
└────────┬────────┘
         │
         │ 2. 写入WAL Buffer
         ↓
┌─────────────────┐
│ WAL Buffer      │
│ (共享内存)       │
└────────┬────────┘
         │
         │ 3. XLogFlush (事务提交)
         ↓
┌─────────────────┐
│ pg_wal/         │
│ (磁盘文件)       │
└────────┬────────┘
         │
         │ 4. 归档 (可选)
         ↓
┌─────────────────┐
│ Archive/        │
│ (备份存储)       │
└─────────────────┘
```

### 5.2 恢复流程图

```
Crash Recovery
    │
    ├─→ 1. 读取控制文件
    │       获取LastCheckpoint
    │
    ├─→ 2. 分析阶段 (Analysis)
    │       扫描WAL从检查点开始
    │       确定DirtyPageSet
    │       确定ActiveTransactionSet
    │
    ├─→ 3. 重做阶段 (Redo)
    │       从RedoLSN开始扫描
    │       应用所有已提交事务
    │       ├─→ XLOG_HEAP_INSERT → 重做插入
    │       ├─→ XLOG_HEAP_UPDATE → 重做更新
    │       └─→ XLOG_HEAP_DELETE → 重做删除
    │
    └─→ 4. 撤销阶段 (Undo)
            从后向前扫描
            回滚所有未提交事务
            使用补偿日志记录 (CLR)
```

### 5.3 同步级别对比矩阵

| 级别 | 性能 | 数据安全 | 适用场景 |
|------|------|----------|----------|
| **off** | 最高 | 可能丢失最后事务 | 临时数据 |
| **local** | 高 | 本地持久化 | 单机可靠 |
| **remote_write** | 中 | 副本收到 | 中等可靠 |
| **remote_apply** | 低 | 副本应用 | 强一致 |
| **on** | 最低 | 双同步 | 金融级 |

## 6. 实例与反例

### 6.1 正例: 正确配置

**场景1: 金融系统配置**

```ini
# 最高可靠性
synchronous_commit = on
synchronous_standby_names = 'standby1,standby2'
wal_level = replica
wal_sync_method = fdatasync
full_page_writes = on
```

**场景2: 批量加载优化**

```sql
-- 临时降低同步级别
SET synchronous_commit = off;

-- 批量加载
COPY large_table FROM 'data.csv';

-- 恢复同步
SET synchronous_commit = on;
```

### 6.2 反例: 常见错误

**反例1: WAL磁盘满**

```bash
# 问题: WAL归档失败导致磁盘满
# 数据库崩溃!

# 解决方案:
# 1. 监控WAL大小
# 2. 配置合理的保留策略
# 3. 确保归档目录可用空间
```

**反例2: 部分页写入**

```
问题: 断电导致数据页部分写入
页头损坏，无法恢复

解决方案:
- full_page_writes = on
- 在检查点后首次修改页面前，写入完整页到WAL
```
