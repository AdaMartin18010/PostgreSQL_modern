## 1. 可观测性理论基础

### 1.1 可观测性三大支柱

可观测性(Observability)是现代分布式系统的核心能力，通过系统的外部输出推断内部状态。其理论基础建立在**三大支柱**之上：

#### 1.1.1 指标(Metrics)

指标是系统状态的数值化表示，遵循时间序列模型：

$$
M(t) = \{m_1(t), m_2(t), ..., m_n(t)\} \quad \text{其中} \quad t \in \mathbb{R}^+
$$

**指标类型分类：**

| 类型 | 数学定义 | PostgreSQL示例 |
|------|----------|----------------|
| Counter | $C(t) = \int_{0}^{t} r(\tau) d\tau$ | 事务提交总数 |
| Gauge | $G(t) \in \mathbb{R}$ | 当前连接数 |
| Histogram | $H = \{(b_i, c_i)\}_{i=1}^{n}$ | 查询延迟分布 |
| Summary | $S = (\phi_{0.5}, \phi_{0.9}, \phi_{0.99})$ | 百分位延迟 |

**PostgreSQL核心指标公式：**

事务吞吐量计算公式：
$$
\text{TPS} = \frac{\Delta \text{Commited} + \Delta \text{RolledBack}}{\Delta t}
$$

缓存命中率计算：
$$
\text{CacheHitRatio} = \frac{\text{BlksHit}}{\text{BlksHit} + \text{BlksRead}} \times 100\%
$$

#### 1.1.2 日志(Logs)

日志是离散事件的文本记录，构成事件序列：

$$
\mathcal{L} = \{l_1, l_2, ..., l_n\} \quad \text{其中} \quad l_i = (t_i, s_i, m_i, a_i)
$$

其中：

- $t_i$: 时间戳
- $s_i$: 严重级别
- $m_i$: 消息内容
- $a_i$: 属性集合

**PostgreSQL日志结构化模型：**

```json
{
  "timestamp": "2026-01-15T09:23:47.123Z",
  "severity": "ERROR",
  "message": "deadlock detected",
  "attributes": {
    "process_id": 12345,
    "session_id": "abc-123",
    "database": "production",
    "query_id": "xyz-789"
  }
}
```

#### 1.1.3 追踪(Traces)

追踪记录请求在分布式系统中的完整路径，构成有向无环图(DAG)：

$$
\mathcal{T} = (S, R) \quad \text{其中} \quad S = \{span_i\}, \quad R \subseteq S \times S
$$

**跨度(Span)数学模型：**

$$
span = (trace\_id, span\_id, parent\_id, name, start\_time, end\_time, attrs, events)
$$

跨度时间关系约束：
$$
\forall span_i, span_j \in S: \text{if } (span_i, span_j) \in R \Rightarrow span_i.start \leq span_j.start \land span_j.end \leq span_i.end
$$

**追踪拓扑结构：**

```
┌─────────────────────────────────────────────────────────┐
│  Trace: order-processing-trace                          │
│  ┌───────────────────────────────────────────────────┐  │

│  │  │ Span C: Database Query (60ms)               │  │  │
│  │  │  ┌───────────────────────────────────────┐  │  │  │
│  │  │  │ Span D: Parse (5ms)                   │  │  │  │
│  │  │  └───────────────────────────────────────┘  │  │  │
```

### 1.2 OpenTelemetry数据模型

#### 1.2.1 Resource模型

Resource描述产生遥测数据的实体：

$$
Resource = \{(k_1, v_1), (k_2, v_2), ..., (k_n, v_n)\}
$$

**PostgreSQL Resource属性规范：**

| 属性名 | 类型 | 说明 | 示例值 |
|--------|------|------|--------|
| `service.name` | string | 服务名称 | `postgresql-prod-01` |
| `service.version` | string | 版本 | `18.0.0` |
| `deployment.environment` | string | 部署环境 | `production` |
| `host.name` | string | 主机名 | `db-server-01` |
| `db.system` | string | 数据库类型 | `postgresql` |

#### 1.2.2 Semantic Conventions

OTel语义约定标准化属性命名：

**数据库操作属性：**

$$
\text{db.system} = \text{"postgresql"}
$$

$$
\text{db.statement} = \text{normalized\_sql}
$$

$$
\text{db.operation} \in \{\text{SELECT}, \text{INSERT}, \text{UPDATE}, \text{DELETE}, \text{BEGIN}, \text{COMMIT}\}
$$

$$
\text{db.postgresql.plan} = \text{query\_plan\_json}
$$

#### 1.2.3 Context传播机制

分布式追踪依赖上下文传播，基于W3C Trace Context标准：

```
┌────────────────────────────────────────────────────────────┐
│                    Trace Context Header                     │
├────────────────────────────────────────────────────────────┤
│ traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-          │
│             00f067aa0ba902b7-01                            │
│              │  │                │              │          │
│              │  │                │              └── flags   │
│              │  │                └── parent span ID         │
│              │  └── trace ID (128-bit)                      │
│              └── version                                     │
└────────────────────────────────────────────────────────────┘
```

上下文传播概率模型：

$$
P(span\_received) = \prod_{i=1}^{n} p_i
$$

其中 $p_i$ 是第 $i$ 个跃点的传播成功率。

### 1.3 采样策略理论

#### 1.3.1 采样类型对比

| 采样策略 | 数学定义 | 适用场景 | 精度 |
|----------|----------|----------|------|
| 头部采样 | $S_{head}(trace) = f(trace\_id)$ | 简单场景 | 高 |
| 尾部采样 | $S_{tail}(spans) = g(latency, error)$ | 异常检测 | 极高 |
| 概率采样 | $P(sample) = p$ | 大规模部署 | 中等 |
| 速率限制 | $rate \leq R$ | 资源受限 | 低 |

#### 1.3.2 概率采样算法

基于一致性的概率采样保证完整性：

$$
sample(trace\_id) = \begin{cases}
true & \text{if } hash(trace\_id) \mod 10000 < sampling\_rate \times 10000 \\
false & \text{otherwise}
\end{cases}
$$

采样误差分析：

$$
\sigma_{sample} = \sqrt{\frac{p(1-p)}{N}} \times 100\%
$$

其中 $p$ 为采样率，$N$ 为样本数。

#### 1.3.3 自适应采样

动态调整采样率以平衡精度与开销：

$$
p_{adaptive}(t) = \min\left(1, \frac{T_{target}}{T_{actual}(t-1)} \times p_{current}\right)
$$

#### 2.1.3 自动仪表化机制

**查询生命周期仪表化：**

```c
/* pg_tracing自动仪表化钩子实现 */
static void
tracing_planner_hook(Query *parse, const char *query_string,
                     int cursorOptions, ParamListInfo boundParams)
{
    /* 创建查询解析跨度 */
    Span parse_span = {
        .name = "postgresql.parse",
        .start_time = get_current_timestamp(),
        .attributes = {
            {"db.statement", query_string},
            {"db.system", "postgresql"},
            {"db.user", GetUserNameFromId(GetUserId(), true)}
        }
    };

    /* 调用原始planner */
    PlannedStmt *result = standard_planner(parse, query_string,
                                           cursorOptions, boundParams);

    /* 结束跨度 */
    parse_span.end_time = get_current_timestamp();
    span_buffer_push(&parse_span);

    /* 创建计划跨度 */
    Span plan_span = {
        .name = "postgresql.plan",
        .parent_id = parse_span.span_id,
        .start_time = parse_span.end_time,
        .end_time = get_current_timestamp(),
        .attributes = {
            {"plan.cost", result->planTree->total_cost},
            {"plan.rows", result->planTree->plan_rows}
        }
    };
    span_buffer_push(&plan_span);

    return result;
}
```

### 2.2 自定义跨度创建

#### 2.2.1 存储过程内追踪

```sql
-- 创建带追踪的业务存储过程
CREATE OR REPLACE FUNCTION process_order_with_tracing(
    p_order_id BIGINT,
    p_customer_id BIGINT,
    p_amount DECIMAL
) RETURNS VOID AS $$
DECLARE
    v_span_id TEXT;
    v_payment_span TEXT;
    v_inventory_span TEXT;
    v_notification_span TEXT;
BEGIN
    -- 创建主业务跨度
    SELECT pg_tracing.start_span(
        name := 'process_order',
        attributes := jsonb_build_object(
            'order.id', p_order_id,
            'customer.id', p_customer_id,
            'order.amount', p_amount,
            'service.name', 'order-service'
        )
    ) INTO v_span_id;

    BEGIN
        -- 支付处理子跨度
        SELECT pg_tracing.start_span(
            name := 'process_payment',
            parent_id := v_span_id,
            attributes := jsonb_build_object('payment.method', 'credit_card')
        ) INTO v_payment_span;

        PERFORM process_payment(p_order_id, p_amount);

        PERFORM pg_tracing.end_span(v_payment_span, 'ok');

        -- 库存扣减子跨度
        SELECT pg_tracing.start_span(
            name := 'update_inventory',
            parent_id := v_span_id
        ) INTO v_inventory_span;

        UPDATE inventory
        SET quantity = quantity - 1
        WHERE product_id = (SELECT product_id FROM order_items WHERE order_id = p_order_id);

        PERFORM pg_tracing.end_span(v_inventory_span, 'ok');

        -- 通知发送子跨度
        SELECT pg_tracing.start_span(
            name := 'send_notification',
            parent_id := v_span_id,
            attributes := jsonb_build_object('notification.type', 'email')
        ) INTO v_notification_span;

        PERFORM pg_notify('order_completed', jsonb_build_object(
            'order_id', p_order_id,
            'status', 'success'
        )::TEXT);

        PERFORM pg_tracing.end_span(v_notification_span, 'ok');

        -- 结束主跨度
        PERFORM pg_tracing.end_span(v_span_id, 'ok');

    EXCEPTION WHEN OTHERS THEN
        -- 记录错误事件
        PERFORM pg_tracing.add_event(
            span_id := v_span_id,
            name := 'exception',
            attributes := jsonb_build_object(
                'exception.type', SQLSTATE,
                'exception.message', SQLERRM
            )
        );
        PERFORM pg_tracing.end_span(v_span_id, 'error');
        RAISE;
    END;
END;
$$ LANGUAGE plpgsql;
```

```sql
RETURNS TRIGGER AS $$
DECLARE

    v_new_data JSONB;
BEGIN
    -- 创建触发器执行跨度
    SELECT pg_tracing.start_span(
        name := 'trigger.audit_log',
        attributes := jsonb_build_object(
            'trigger.table', TG_TABLE_NAME,
            'trigger.operation', TG_OP,
            'trigger.when', TG_WHEN
        )
    ) INTO v_span_id;

    -- 记录变更数据
    IF TG_OP = 'DELETE' THEN
        v_old_data := to_jsonb(OLD);
        v_new_data := null;
    ELSIF TG_OP = 'INSERT' THEN
        v_old_data := null;
        v_new_data := to_jsonb(NEW);
    ELSE
        v_old_data := to_jsonb(OLD);
        v_new_data := to_jsonb(NEW);
    END IF;

    -- 插入审计日志
    INSERT INTO audit_log (
        table_name, operation, old_data, new_data,
        changed_by, changed_at, tx_id
    ) VALUES (
        TG_TABLE_NAME, TG_OP, v_old_data, v_new_data,
        current_user, now(), txid_current()
    );

    -- 添加维度属性
    PERFORM pg_tracing.set_attribute(
        span_id := v_span_id,
        key := 'audit.rows_affected',
        value := 1::TEXT
    );

    PERFORM pg_tracing.end_span(v_span_id, 'ok');

    RETURN COALESCE(NEW, OLD);
END;
$$ LANGUAGE plpgsql;
```

```sql
-- 复杂事务的多层追踪
CREATE OR REPLACE FUNCTION complex_business_transaction()
RETURNS VOID AS $$
DECLARE
    v_main_span TEXT;
    v_savepoint_span TEXT;
    v_attempt INT := 0;
    v_max_attempts INT := 3;
BEGIN
    -- 主事务跨度
    SELECT pg_tracing.start_span(
        name := 'complex_transaction',
        attributes := jsonb_build_object(
            'transaction.type', 'multi_step',
            'transaction.isolation', current_setting('transaction_isolation')
        )
    ) INTO v_main_span;

    WHILE v_attempt < v_max_attempts LOOP
        v_attempt := v_attempt + 1;

        -- 保存点子跨度
        SELECT pg_tracing.start_span(
            name := 'transaction_attempt',
            parent_id := v_main_span,
            attributes := jsonb_build_object('attempt.number', v_attempt)
        ) INTO v_savepoint_span;

        BEGIN
            -- 步骤1: 数据验证
            PERFORM pg_tracing.add_event(
                span_id := v_savepoint_span,
                name := 'validation.start'
            );
            PERFORM validate_data();

            -- 步骤2: 核心处理
            PERFORM pg_tracing.add_event(
                span_id := v_savepoint_span,
                name := 'processing.start'
            );
            PERFORM process_core_logic();

            -- 步骤3: 数据持久化
            PERFORM pg_tracing.add_event(
                span_id := v_savepoint_span,
                name := 'persistence.start'
            );
            PERFORM persist_results();

            -- 成功，结束尝试跨度
            PERFORM pg_tracing.set_attribute(
                span_id := v_savepoint_span,
                key := 'attempt.success',
                value := 'true'
            );
            PERFORM pg_tracing.end_span(v_savepoint_span, 'ok');
            EXIT;

        EXCEPTION WHEN serialization_failure THEN
            -- 串行化失败，重试
            PERFORM pg_tracing.add_event(
                span_id := v_savepoint_span,
                name := 'retry.scheduled',
                attributes := jsonb_build_object('retry.reason', 'serialization_failure')
            );
            PERFORM pg_tracing.end_span(v_savepoint_span, 'error');

            IF v_attempt >= v_max_attempts THEN
                PERFORM pg_tracing.set_attribute(
                    span_id := v_main_span,
                    key := 'transaction.final_status',
                    value := 'max_retries_exceeded'
                );
                PERFORM pg_tracing.end_span(v_main_span, 'error');
                RAISE EXCEPTION 'Max retries exceeded for transaction';
            END IF;

            PERFORM pg_sleep_random(0.1, 0.5); -- 随机退避
        END;
    END LOOP;

    PERFORM pg_tracing.set_attribute(
        span_id := v_main_span,
        key := 'transaction.attempts',
        value := v_attempt::TEXT
    );
    PERFORM pg_tracing.end_span(v_main_span, 'ok');
END;
$$ LANGUAGE plpgsql;
```

```sql
        -- 获取当前span的traceparent用于下游传播
        v_context := pg_tracing.get_current_context(v_span_id);
```

```sql
    MAX(duration) as max_duration_ms

-- 查找错误率最高的操作
SELECT
    operation_name,
    COUNT(*) as total_spans,
    SUM(CASE WHEN status_code = 'ERROR' THEN 1 ELSE 0 END) as error_count,
    ROUND(
        100.0 * SUM(CASE WHEN status_code = 'ERROR' THEN 1 ELSE 0 END) / COUNT(*),
        2
    ) as error_rate_pct
FROM jaeger_spans
WHERE service_name = 'postgresql-prod-01'
  AND start_time >= NOW() - INTERVAL '24 hours'
GROUP BY operation_name
HAVING COUNT(*) > 100
ORDER BY error_rate_pct DESC;

-- 追踪依赖分析
WITH RECURSIVE trace_tree AS (
    SELECT
        trace_id,
        span_id,
        parent_id,
        operation_name,
        0 as depth
    FROM jaeger_spans
    WHERE parent_id IS NULL
      AND start_time >= NOW() - INTERVAL '1 hour'

    UNION ALL

    SELECT
        s.trace_id,
        s.span_id,
        s.parent_id,
        s.operation_name,
        tt.depth + 1
    FROM jaeger_spans s
    JOIN trace_tree tt ON s.parent_id = tt.span_id
)
SELECT
    operation_name,
    depth,
    COUNT(*) as occurrence_count,
    AVG(duration) as avg_duration
FROM trace_tree
GROUP BY operation_name, depth
ORDER BY depth, occurrence_count DESC;
```

## 4. 性能追踪与分析

### 4.1 慢查询追踪

#### 4.1.1 慢查询识别算法

**基于统计学的慢查询检测：**

$$
\text{SlowQuery}(q) = \begin{cases}
true & \text{if } T(q) > \mu + 3\sigma \\
false & \text{otherwise}
\end{cases}
$$

其中：

- $T(q)$: 查询执行时间
- $\mu$: 平均查询时间
- $\sigma$: 标准差

**自适应阈值计算：**

$$
\theta_{adaptive}(t) = \alpha \cdot \theta_{prev} + (1-\alpha) \cdot P_{95}(t-1)
$$

#### 4.1.2 慢查询追踪实现

```sql
-- 创建慢查询追踪视图
CREATE VIEW slow_query_traces AS
SELECT
    t.trace_id,
    s.span_id,
    s.operation_name,
    s.duration / 1000000.0 as duration_ms,
    s.start_time,
    s.attributes->>'db.statement' as query,
    s.attributes->>'db.postgresql.plan' as query_plan,
    s.attributes->>'postgresql.rows' as rows_returned,
    s.attributes->>'postgresql.buffers.shared.hit' as buffers_hit,
    s.attributes->>'postgresql.buffers.shared.read' as buffers_read
FROM jaeger_spans s
JOIN jaeger_traces t ON s.trace_id = t.trace_id
WHERE s.operation_name LIKE 'postgresql%'
  AND s.duration > 1000000  -- > 1秒
  AND s.start_time >= NOW() - INTERVAL '24 hours'
ORDER BY s.duration DESC;

-- 慢查询根因分析
CREATE OR REPLACE FUNCTION analyze_slow_query(p_trace_id TEXT)
RETURNS TABLE (
    span_name TEXT,
    duration_ms NUMERIC,
    percentage_of_total NUMERIC,
    potential_issue TEXT
) AS $$
DECLARE
    v_total_duration BIGINT;
BEGIN
    -- 获取总持续时间
    SELECT MAX(duration) INTO v_total_duration
    FROM jaeger_spans
    WHERE trace_id = p_trace_id;

    RETURN QUERY
    SELECT
        s.operation_name::TEXT,
        (s.duration / 1000000.0)::NUMERIC,
        (100.0 * s.duration / v_total_duration)::NUMERIC,
        CASE
            WHEN s.operation_name LIKE '%plan%' AND s.duration > 100000 THEN
                '复杂查询计划，考虑优化SQL或添加索引'
            WHEN s.attributes->>'postgresql.buffers.shared.read'::bigint > 10000 THEN
                '大量磁盘读取，检查缓存配置或查询效率'
            WHEN s.operation_name LIKE '%lock%' THEN
                '锁等待时间高，检查并发和事务隔离级别'
            WHEN s.duration > v_total_duration * 0.5 THEN
                '此操作消耗了大部分时间，重点优化'
            ELSE
                '正常执行'
        END::TEXT
    FROM jaeger_spans s
    WHERE s.trace_id = p_trace_id
    ORDER BY s.duration DESC;
END;
$$ LANGUAGE plpgsql;
```

#### 4.1.3 查询计划追踪

```sql
-- 扩展pg_tracing以追踪查询计划
CREATE OR REPLACE FUNCTION trace_query_execution(
    p_query TEXT,
    p_params JSONB DEFAULT NULL
) RETURNS JSONB AS $$
DECLARE
    v_span_id TEXT;
    v_plan JSONB;
    v_result JSONB;
    v_start_time TIMESTAMP;
    v_plan_time INTERVAL;
    v_exec_time INTERVAL;
BEGIN
    v_start_time := clock_timestamp();

    -- 创建查询跨度
    SELECT pg_tracing.start_span(
        name := 'postgresql.manual_query',
        attributes := jsonb_build_object(
            'db.statement', p_query,
            'query.params', p_params
        )
    ) INTO v_span_id;

    -- 获取执行计划
    EXECUTE format('EXPLAIN (FORMAT JSON, ANALYZE, BUFFERS) %s', p_query)
    INTO v_plan;

    v_plan_time := clock_timestamp() - v_start_time;

    -- 提取计划信息
    PERFORM pg_tracing.set_attribute(
        span_id := v_span_id,
        key := 'postgresql.plan',
        value := v_plan::TEXT
    );

    PERFORM pg_tracing.set_attribute(
        span_id := v_span_id,
        key := 'postgresql.planning.time_ms',
        value := (EXTRACT(EPOCH FROM v_plan_time) * 1000)::TEXT
    );

    -- 添加计划节点为子跨度
    PERFORM pg_tracing.add_plan_spans(v_span_id, v_plan);

    -- 执行实际查询(如果是SELECT)
    IF p_query ~* '^\s*SELECT' THEN
        EXECUTE p_query INTO v_result;
    END IF;

    v_exec_time := clock_timestamp() - v_start_time - v_plan_time;

    PERFORM pg_tracing.set_attribute(
        span_id := v_span_id,
        key := 'postgresql.execution.time_ms',
        value := (EXTRACT(EPOCH FROM v_exec_time) * 1000)::TEXT
    );

    PERFORM pg_tracing.end_span(v_span_id, 'ok');

    RETURN jsonb_build_object(
        'plan', v_plan,
        'plan_time_ms', EXTRACT(EPOCH FROM v_plan_time) * 1000,
        'exec_time_ms', EXTRACT(EPOCH FROM v_exec_time) * 1000,
        'result', v_result
    );
END;
$$ LANGUAGE plpgsql;
```

### 4.2 锁等待分析

#### 4.2.1 锁等待追踪模型

**锁等待图模型：**

$$
G_{lock} = (P, E) \quad \text{其中} \quad E = \{(p_i, p_j) | p_i \text{等待} p_j\text{持有的锁}\}
$$

**死锁检测：**

$$
\text{Deadlock} \Leftrightarrow \exists \text{ cycle in } G_{lock}
$$

**锁等待时间分解：**

$$
T_{total} = T_{acquire} + T_{hold} + T_{queue}
$$

其中：

- $T_{acquire}$: 获取锁的时间
- $T_{hold}$: 持有锁的时间
- $T_{queue}$: 队列等待时间

#### 4.2.2 锁等待追踪实现

```sql
-- 创建锁等待追踪表
CREATE TABLE lock_wait_traces (
    id BIGSERIAL PRIMARY KEY,
    trace_id TEXT NOT NULL,
    span_id TEXT NOT NULL,
    blocked_pid INTEGER,
    blocking_pid INTEGER,
    lock_type TEXT,
    lock_mode TEXT,
    relation TEXT,
    wait_start_time TIMESTAMP,
    wait_duration_ms NUMERIC,
    query TEXT,
    created_at TIMESTAMP DEFAULT NOW()
);

-- 锁等待追踪函数
CREATE OR REPLACE FUNCTION trace_lock_waits()
RETURNS TRIGGER AS $$
DECLARE
    v_span_id TEXT;
    v_lock_info RECORD;
BEGIN
    FOR v_lock_info IN
        SELECT
            bl.pid as blocked_pid,
            kl.pid as blocking_pid,
            bl.locktype,
            bl.mode as blocked_mode,
            kl.mode as blocking_mode,
            bl.relation::regclass as relation_name,
            bl.query as blocked_query,
            kl.query as blocking_query,
            EXTRACT(EPOCH FROM (NOW() - bl.query_start)) * 1000 as wait_ms
        FROM pg_locks bl
        JOIN pg_locks kl ON bl.locktype = kl.locktype
            AND bl.relation = kl.relation
            AND bl.pid != kl.pid
        JOIN pg_stat_activity ba ON bl.pid = ba.pid
        JOIN pg_stat_activity ka ON kl.pid = ka.pid
        WHERE NOT bl.granted AND kl.granted
          AND EXTRACT(EPOCH FROM (NOW() - bl.query_start)) > 0.1
    LOOP
        -- 创建锁等待跨度
        SELECT pg_tracing.start_span(
            name := 'postgresql.lock_wait',
            attributes := jsonb_build_object(
                'lock.type', v_lock_info.locktype,
                'lock.blocked_mode', v_lock_info.blocked_mode,
                'lock.blocking_mode', v_lock_info.blocking_mode,
                'lock.relation', v_lock_info.relation_name,
                'lock.blocked_pid', v_lock_info.blocked_pid,
                'lock.blocking_pid', v_lock_info.blocking_pid,
                'query.blocked', LEFT(v_lock_info.blocked_query, 500),
                'query.blocking', LEFT(v_lock_info.blocking_query, 500),
                'wait.duration_ms', v_lock_info.wait_ms
            )
        ) INTO v_span_id;

        -- 存储到本地表
        INSERT INTO lock_wait_traces (
            trace_id, span_id, blocked_pid, blocking_pid,
            lock_type, lock_mode, relation, wait_start_time,
            wait_duration_ms, query
        ) VALUES (
            pg_tracing.get_trace_id(v_span_id),
            v_span_id,
            v_lock_info.blocked_pid,
            v_lock_info.blocking_pid,
            v_lock_info.locktype,
            v_lock_info.blocked_mode,
            v_lock_info.relation_name,
            NOW() - (v_lock_info.wait_ms || 'ms')::INTERVAL,
            v_lock_info.wait_ms,
            v_lock_info.blocked_query
        );

        -- 如果等待时间过长，标记为错误
        IF v_lock_info.wait_ms > 5000 THEN
            PERFORM pg_tracing.add_event(
                span_id := v_span_id,
                name := 'lock_wait.warning',
                attributes := jsonb_build_object(
                    'warning.type', 'long_lock_wait',
                    'threshold_ms', 5000
                )
            );
        END IF;

        PERFORM pg_tracing.end_span(v_span_id,
            CASE WHEN v_lock_info.wait_ms > 10000 THEN 'error' ELSE 'ok' END
        );
    END LOOP;

    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- 创建定时触发器(每分钟检查一次)
SELECT cron.schedule('lock-wait-monitor', '* * * * *',
    'SELECT trace_lock_waits()');
```

#### 4.2.3 死锁检测追踪

```sql
-- 死锁追踪函数
CREATE OR REPLACE FUNCTION trace_deadlock_info()
RETURNS EVENT_TRIGGER AS $$
DECLARE
    v_span_id TEXT;
    v_deadlock_info RECORD;
BEGIN
    -- 从pg_stat_database获取死锁计数
    FOR v_deadlock_info IN
        SELECT
            datname,
            deadlocks - LAG(deadlocks) OVER (PARTITION BY datname ORDER BY stats_reset) as new_deadlocks
        FROM pg_stat_database
        WHERE deadlocks > 0
    LOOP
        IF v_deadlock_info.new_deadlocks > 0 THEN
            SELECT pg_tracing.start_span(
                name := 'postgresql.deadlock',
                attributes := jsonb_build_object(
                    'deadlock.database', v_deadlock_info.datname,
                    'deadlock.count', v_deadlock_info.new_deadlocks,
                    'deadlock.severity', 'critical'
                )
            ) INTO v_span_id;

            PERFORM pg_tracing.add_event(
                span_id := v_span_id,
                name := 'deadlock.occurred',
                attributes := jsonb_build_object(
                    'resolution', 'transaction_rolled_back'
                )
            );

            PERFORM pg_tracing.end_span(v_span_id, 'error');
        END IF;
    END LOOP;
END;
$$ LANGUAGE plpgsql;

-- 创建事件触发器
CREATE EVENT TRIGGER trace_deadlock
    ON sql_drop
    EXECUTE FUNCTION trace_deadlock_info();
```

### 4.3 连接池监控

#### 4.3.1 连接池性能模型

**连接池利用率计算：**

$$
U_{pool} = \frac{N_{active}}{N_{max}} \times 100\%
$$

**连接等待时间模型(基于M/M/c队列)：**

$$
W_q = \frac{P_0 \cdot (\lambda/\mu)^c \cdot \rho}{c! \cdot (1-\rho)^2 \cdot \lambda}
$$

其中：

- $\lambda$: 连接请求到达率
- $\mu$: 服务率
- $c$: 连接池大小
- $\rho = \lambda/(c\mu)$: 系统利用率

**最优连接池大小估算：**

$$
N_{optimal} = \frac{N_{cores} \times (1 + W)}{C}
$$

其中：

- $N_{cores}$: CPU核心数
- $W$: 等待时间与计算时间比
- $C$: 并发度

#### 4.3.2 连接池追踪实现

```sql
-- 连接池监控视图
CREATE VIEW connection_pool_metrics AS
SELECT
    datname as database,
    state,
    COUNT(*) as connection_count,
    AVG(EXTRACT(EPOCH FROM (NOW() - state_change))) as avg_state_duration,
    MAX(EXTRACT(EPOCH FROM (NOW() - state_change))) as max_state_duration
FROM pg_stat_activity
WHERE backend_type = 'client backend'
GROUP BY datname, state;

-- 连接池追踪函数
CREATE OR REPLACE FUNCTION trace_connection_pool()
RETURNS VOID AS $$
DECLARE
    v_span_id TEXT;
    v_metrics RECORD;
    v_total_connections INTEGER;
    v_max_connections INTEGER;
BEGIN
    -- 获取最大连接数
    SHOW max_connections INTO v_max_connections;

    SELECT COUNT(*) INTO v_total_connections
    FROM pg_stat_activity
    WHERE backend_type = 'client backend';

    -- 创建连接池监控跨度
    SELECT pg_tracing.start_span(
        name := 'postgresql.connection_pool.metrics',
        attributes := jsonb_build_object(
            'pool.total_connections', v_total_connections,
            'pool.max_connections', v_max_connections,
            'pool.utilization_pct', ROUND(100.0 * v_total_connections / v_max_connections, 2),
            'pool.available', v_max_connections - v_total_connections
        )
    ) INTO v_span_id;

    -- 按状态追踪
    FOR v_metrics IN
        SELECT state, COUNT(*) as cnt,
               AVG(EXTRACT(EPOCH FROM (NOW() - query_start))) as avg_query_time
        FROM pg_stat_activity
        WHERE backend_type = 'client backend'
          AND state IS NOT NULL
        GROUP BY state
    LOOP
        PERFORM pg_tracing.add_event(
            span_id := v_span_id,
            name := 'connection.state.stats',
            attributes := jsonb_build_object(
                'state', v_metrics.state,
                'count', v_metrics.cnt,
                'avg_query_time_sec', COALESCE(v_metrics.avg_query_time, 0)
            )
        );
    END LOOP;

    -- 检查长连接
    FOR v_metrics IN
        SELECT pid, usename, application_name,
               EXTRACT(EPOCH FROM (NOW() - backend_start)) as connection_age,
               state, query
        FROM pg_stat_activity
        WHERE backend_type = 'client backend'
          AND EXTRACT(EPOCH FROM (NOW() - backend_start)) > 3600  -- 超过1小时
        ORDER BY backend_start
        LIMIT 10
    LOOP
        PERFORM pg_tracing.add_event(
            span_id := v_span_id,
            name := 'connection.long_lived',
            attributes := jsonb_build_object(
                'pid', v_metrics.pid,
                'user', v_metrics.usename,
                'app', v_metrics.application_name,
                'age_seconds', v_metrics.connection_age,
                'query', LEFT(v_metrics.query, 200)
            )
        );
    END LOOP;

    -- 检查空闲连接
    IF EXISTS (
        SELECT 1 FROM pg_stat_activity
        WHERE state = 'idle in transaction'
          AND EXTRACT(EPOCH FROM (NOW() - state_change)) > 300
    ) THEN
        PERFORM pg_tracing.add_event(
            span_id := v_span_id,
            name := 'connection.idle_in_transaction.warning',
            attributes := jsonb_build_object(
                'warning', 'long_idle_transactions_detected'
            )
        );
    END IF;

    PERFORM pg_tracing.end_span(v_span_id, 'ok');
END;
$$ LANGUAGE plpgsql;

-- 连接等待追踪
CREATE OR REPLACE FUNCTION trace_connection_wait()
RETURNS TRIGGER AS $$
DECLARE
    v_span_id TEXT;
    v_wait_start TIMESTAMP;
BEGIN
    -- 记录连接请求开始时间
    v_wait_start := clock_timestamp();

    -- 在连接成功后创建追踪跨度
    IF NEW.backend_type = 'client backend' THEN
        SELECT pg_tracing.start_span(
            name := 'postgresql.connection_lifecycle',
            start_time := v_wait_start,
            attributes := jsonb_build_object(
                'connection.pid', NEW.pid,
                'connection.user', NEW.usename,
                'connection.database', NEW.datname,
                'connection.app', NEW.application_name,
                'connection.client_addr', NEW.client_addr::TEXT
            )
        ) INTO v_span_id;

        -- 存储span_id以便在断开时结束
        INSERT INTO active_connection_traces (pid, span_id, connect_time)
        VALUES (NEW.pid, v_span_id, v_wait_start);
    END IF;

    RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

### 5.1 分布式事务追踪

```sql
-- 两阶段提交追踪
CREATE OR REPLACE FUNCTION trace_two_phase_commit(
    p_transaction_id TEXT,
    p_participants TEXT[]
) RETURNS VOID AS $$
DECLARE
    v_main_span TEXT;
    v_prepare_span TEXT;
    v_commit_span TEXT;
    v_participant TEXT;
BEGIN
    -- 主事务跨度
    SELECT pg_tracing.start_span(
        name := 'distributed_transaction.2pc',
        attributes := jsonb_build_object(
            'transaction.id', p_transaction_id,
            'transaction.type', 'two_phase_commit',
            'transaction.participants', array_to_json(p_participants)
        )
    ) INTO v_main_span;

    -- 准备阶段
    SELECT pg_tracing.start_span(
        name := 'distributed_transaction.prepare',
        parent_id := v_main_span
    ) INTO v_prepare_span;

    FOREACH v_participant IN ARRAY p_participants
    LOOP
        PERFORM pg_tracing.add_event(
            span_id := v_prepare_span,
            name := 'prepare.sent',
            attributes := jsonb_build_object('participant', v_participant)
        );
    END LOOP;

    PERFORM pg_tracing.end_span(v_prepare_span, 'ok');

    -- 提交阶段
    SELECT pg_tracing.start_span(
        name := 'distributed_transaction.commit',
        parent_id := v_main_span
    ) INTO v_commit_span;

    FOREACH v_participant IN ARRAY p_participants
    LOOP
        PERFORM pg_tracing.add_event(
            span_id := v_commit_span,
            name := 'commit.sent',
            attributes := jsonb_build_object('participant', v_participant)
        );
    END LOOP;

    PERFORM pg_tracing.end_span(v_commit_span, 'ok');
    PERFORM pg_tracing.end_span(v_main_span, 'ok');
END;
$$ LANGUAGE plpgsql;
```

## 6. 性能开销评估

### 6.1 开销计算模型

**追踪总开销公式：**

$$
T_{overhead} = T_{capture} + T_{serialization} + T_{export} + T_{network}
$$

**采样对精度的影响：**

$$
\text{Error}_{metric} = \frac{Z_{\alpha/2} \cdot \sigma}{\sqrt{N \cdot p}}
$$

其中：

- $Z_{\alpha/2}$: 置信区间Z值
- $\sigma$: 总体标准差
- $N$: 总请求数
- $p$: 采样率

### 6.2 推荐配置
