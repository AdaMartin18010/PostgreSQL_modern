# PG17 MERGE 命令增强详解

> **版本**: PostgreSQL 17 (2024-09-26 发布)
> **类型**: SQL/DML
> **影响等级**: 中
> **升级建议**: 推荐
> **文档长度**: ~5500字
> **创建日期**: 2026-04-07

---

## 摘要

PostgreSQL 17 对 MERGE 命令进行了重大增强，使其从 PostgreSQL 15 的基础实现演进为功能完备的 SQL 标准 DML 工具。
本文深入解析 PG17 引入的三大核心增强：**WHEN NOT MATCHED BY SOURCE** 子句、**视图作为目标表** 支持，以及 **RETURNING 子句与 merge_action()** 函数。

这些增强使 MERGE 能够处理更复杂的 CDC（变更数据捕获）、数据仓库 SCD Type 2 维度更新等场景，大幅提升数据同步效率和代码简洁性。

## 1. 特性概述

### 1.1 MERGE 命令演进历史

MERGE 命令（又称 "Upsert"）是 SQL:2003 标准定义的DML语句，用于根据源数据对目标表执行条件性插入、更新或删除操作。

| 版本 | 特性 | 状态 | 关键限制 |
|------|------|------|----------|
| **PG 15** | MERGE 初始实现 | ✅ 稳定 | 仅支持 `WHEN MATCHED` 和 `WHEN NOT MATCHED [BY TARGET]`，无 RETURNING |
| **PG 16** | 优化器改进 | 🔧 内部 | 更好的 join 选择和代价估计 |
| **PG 17** | 完整功能增强 | 🚀 推荐 | 支持 `BY SOURCE`、视图目标、RETURNING |

### 1.2 PG17 三大核心增强

**图 1.1：PG17 MERGE 增强架构**

```

│  │ WHEN NOT MATCHED    │  │  视图目标支持     │  │ RETURNING│   │
│  │      BY SOURCE      │  │                  │  │ 子句增强 │   │
```

### 1.3 语法演进对比

```sql
-- PostgreSQL 15/16 语法
MERGE INTO target_table t
USING source_table s ON t.id = s.id
WHEN MATCHED THEN UPDATE SET ...
WHEN NOT MATCHED THEN INSERT ...;

-- PostgreSQL 17 完整语法
MERGE INTO target_table t
USING source_table s ON t.id = s.id
WHEN MATCHED AND condition THEN UPDATE SET ...
WHEN MATCHED THEN DELETE
WHEN NOT MATCHED BY TARGET THEN INSERT ...
WHEN NOT MATCHED BY SOURCE THEN DELETE
RETURNING t.id, s.column, merge_action();
```

## 2. WHEN NOT MATCHED BY SOURCE 详解

### 2.1 语法和语义

**定义 2.1 (WHEN NOT MATCHED BY SOURCE)**:
当目标表中的行在源表中找不到匹配时（即目标表存在但源表不存在的行），执行指定的操作。

**形式化描述**:

$$
\text{NotMatchedBySource}(t, S) := \nexists s \in S: t.key = s.key
$$

其中 $t$ 是目标表元组，$S$ 是源表。

**图 2.1：MERGE 匹配语义矩阵**

```
                源表 (Source)
              ┌──────────┬──────────┐
              │  存在    │  不存在  │
         ┌────┼──────────┼──────────┤
目  存在  │    │ MATCHED  │ NOT      │
标       │    │          │ MATCHED  │
表       │    │          │ BY SOURCE│
(BY     ├────┼──────────┼──────────┤
TARGET) 不存在│    │ NOT      │   N/A    │
         │    │ MATCHED  │          │
         │    │ BY TARGET│          │
         └────┴──────────┴──────────┘
```

### 2.2 基本语法

```sql
MERGE INTO target_table AS t
USING source_table AS s ON t.id = s.id

-- 目标表有但源表没有的行 → 删除
WHEN NOT MATCHED BY SOURCE THEN DELETE

-- 或更新标记
WHEN NOT MATCHED BY SOURCE THEN UPDATE SET is_deleted = true;
```

### 2.3 与 WHEN NOT MATCHED BY TARGET 的区别

| 特性 | WHEN NOT MATCHED BY TARGET | WHEN NOT MATCHED BY SOURCE |
|------|---------------------------|---------------------------|
| **匹配条件** | 目标表无，源表有 | 目标表有，源表无 |
| **典型操作** | INSERT | DELETE / UPDATE |
| **别名** | `WHEN NOT MATCHED`（默认）| 必须显式指定 |
| **使用场景** | 新增数据同步 | 删除/标记已移除数据 |
| **数据流向** | Source → Target | Target ← (处理孤例行) |

**图 2.2：数据同步流程对比**

```
BY TARGET (PG15 支持):
┌─────────┐      ┌─────────┐
│ Source  │ ──▶  │ Target  │
│ (新增)  │      │ (插入)  │
└─────────┘      └─────────┘

BY SOURCE (PG17 新增):
┌─────────┐      ┌─────────┐
│ Source  │ ◀──  │ Target  │
│ (无数据)│      │ (删除)  │
└─────────┘      └─────────┘

双向同步 (PG17 完整):
┌─────────┐      ┌─────────┐
│ Source  │ ◀──▶ │ Target  │
│         │ MERGE│         │
└─────────┘      └─────────┘
```

### 2.4 实际应用场景

**场景1: 全量数据同步（CDC 同步）**

```sql
-- 源表: 上游系统的最新数据快照
-- 目标表: 本地数据仓库

MERGE INTO warehouse.customers AS target
USING staging.customers_snapshot AS source
ON target.customer_id = source.customer_id

-- 1. 匹配的 → 更新
WHEN MATCHED THEN
    UPDATE SET
        name = source.name,
        email = source.email,
        updated_at = CURRENT_TIMESTAMP,
        sync_status = 'UPDATED'

-- 2. 源表有目标表无 → 插入
WHEN NOT MATCHED BY TARGET THEN
    INSERT (customer_id, name, email, created_at, sync_status)
    VALUES (source.customer_id, source.name, source.email, CURRENT_TIMESTAMP, 'INSERTED')

-- 3. 目标表有源表无 → 删除 (PG17新增)
WHEN NOT MATCHED BY SOURCE THEN
    DELETE;
```

**场景2: 软删除标记**

```sql
-- 更安全的方式：标记删除而不是物理删除
MERGE INTO warehouse.customers AS target
USING staging.customers_snapshot AS source
ON target.customer_id = source.customer_id

WHEN MATCHED THEN
    UPDATE SET
        name = source.name,
        email = source.email,
        is_deleted = false,
        updated_at = CURRENT_TIMESTAMP

WHEN NOT MATCHED BY TARGET THEN
    INSERT (customer_id, name, email, is_deleted, created_at)
    VALUES (source.customer_id, source.name, source.email, false, CURRENT_TIMESTAMP)

WHEN NOT MATCHED BY SOURCE THEN
    UPDATE SET
        is_deleted = true,
        deleted_at = CURRENT_TIMESTAMP;
```

## 3. 视图支持

### 3.1 可更新视图要求

**定义 3.1 (可更新视图)**: 满足以下条件的视图可以作为 MERGE 的目标：

1. 视图基于单一基表（无 JOIN）
2. 包含基表的所有 NOT NULL 列（INSERT 场景）
3. 无 DISTINCT、GROUP BY、HAVING、LIMIT 等聚合操作
4. 无窗口函数或集合返回函数

**检查视图可更新性**:

```sql
-- PostgreSQL 系统视图检查
SELECT
    viewname,
    is_insertable_into,
    is_updatable,
    is_trigger_updatable
FROM information_schema.views
WHERE viewname = 'your_view';
```

### 3.2 INSTEAD OF 触发器配合

对于复杂视图（多表 JOIN），通过 `INSTEAD OF` 触发器支持 MERGE。

**图 3.1：视图 MERGE 触发器架构**

```
┌────────────────────────────────────────────────────────────┐
│                    MERGE 语句                               │
│  MERGE INTO customer_view v USING source s ON v.id = s.id  │
└────────────────────────────────────────────────────────────┘
                          │
                          ▼
┌────────────────────────────────────────────────────────────┐
│                 INSTEAD OF 触发器                           │
├────────────────────────────────────────────────────────────┤
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐     │
│  │ INSERT 规则   │  │ UPDATE 规则   │  │ DELETE 规则   │     │
│  │ (拆分到多表)  │  │ (维护一致性)  │  │ (级联处理)    │     │
│  └──────────────┘  └──────────────┘  └──────────────┘     │
└────────────────────────────────────────────────────────────┘
                          │
          ┌───────────────┼───────────────┐
          ▼               ▼               ▼
    ┌──────────┐   ┌──────────┐   ┌──────────┐
    │ customers│   │ addresses│   │  orders  │
    └──────────┘   └──────────┘   └──────────┘
```

**完整示例：客户视图 MERGE**

```sql
-- 基表定义
CREATE TABLE customers (
    customer_id INT PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    email VARCHAR(100),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE customer_addresses (
    address_id SERIAL PRIMARY KEY,
    customer_id INT REFERENCES customers(customer_id),
    address TEXT,
    city VARCHAR(50),
    is_primary BOOLEAN DEFAULT false
);

-- 创建视图
CREATE VIEW customer_with_address AS
SELECT
    c.customer_id,
    c.name,
    c.email,
    ca.address,
    ca.city,
    ca.is_primary
FROM customers c
LEFT JOIN customer_addresses ca ON c.customer_id = ca.customer_id AND ca.is_primary = true;

-- INSTEAD OF INSERT 触发器
CREATE OR REPLACE FUNCTION customer_view_insert()
RETURNS TRIGGER AS $$
DECLARE
    v_customer_id INT;
BEGIN
    -- 插入客户主表
    INSERT INTO customers (customer_id, name, email)
    VALUES (NEW.customer_id, NEW.name, NEW.email)
    RETURNING customer_id INTO v_customer_id;

    -- 如果有地址，插入地址表
    IF NEW.address IS NOT NULL THEN
        INSERT INTO customer_addresses (customer_id, address, city, is_primary)
        VALUES (v_customer_id, NEW.address, NEW.city, true);
    END IF;

    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_customer_view_insert
    INSTEAD OF INSERT ON customer_with_address
    FOR EACH ROW EXECUTE FUNCTION customer_view_insert();

-- INSTEAD OF UPDATE 触发器
CREATE OR REPLACE FUNCTION customer_view_update()
RETURNS TRIGGER AS $$
BEGIN
    -- 更新客户主表
    UPDATE customers
    SET name = NEW.name, email = NEW.email
    WHERE customer_id = OLD.customer_id;

    -- 更新或插入地址
    IF EXISTS (SELECT 1 FROM customer_addresses
               WHERE customer_id = OLD.customer_id AND is_primary = true) THEN
        UPDATE customer_addresses
        SET address = NEW.address, city = NEW.city
        WHERE customer_id = OLD.customer_id AND is_primary = true;
    ELSIF NEW.address IS NOT NULL THEN
        INSERT INTO customer_addresses (customer_id, address, city, is_primary)
        VALUES (OLD.customer_id, NEW.address, NEW.city, true);
    END IF;

    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_customer_view_update
    INSTEAD OF UPDATE ON customer_with_address
    FOR EACH ROW EXECUTE FUNCTION customer_view_update();

-- INSTEAD OF DELETE 触发器
CREATE OR REPLACE FUNCTION customer_view_delete()
RETURNS TRIGGER AS $$
BEGIN
    -- 删除地址（级联）
    DELETE FROM customer_addresses WHERE customer_id = OLD.customer_id;
    -- 删除客户
    DELETE FROM customers WHERE customer_id = OLD.customer_id;
    RETURN OLD;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_customer_view_delete
    INSTEAD OF DELETE ON customer_with_address
    FOR EACH ROW EXECUTE FUNCTION customer_view_delete();
```

**使用 MERGE 操作视图**:

```sql
-- 现在可以对视图使用 MERGE
MERGE INTO customer_with_address AS target
USING staging.new_customers AS source
ON target.customer_id = source.customer_id

WHEN MATCHED THEN
    UPDATE SET
        name = source.name,
        email = source.email,
        address = source.address,
        city = source.city

WHEN NOT MATCHED BY TARGET THEN
    INSERT (customer_id, name, email, address, city)
    VALUES (source.customer_id, source.name, source.email, source.address, source.city)

WHEN NOT MATCHED BY SOURCE THEN
    DELETE;
```

### 3.3 使用场景

| 场景 | 说明 | 优势 |
|------|------|------|
| **数据抽象层** | 通过视图暴露规范化数据 | 应用无需了解底层表结构 |
| **安全控制** | 视图过滤敏感列 | MERGE 只能操作授权列 |
| **遗留系统集成** | 旧表结构通过视图适配 | 不修改应用代码 |
| **多表逻辑实体** | 主从表通过视图统一 | 简化应用逻辑 |

## 4. RETURNING 子句增强

### 4.1 merge_action() 函数

**定义 4.1 (merge_action())**: 系统函数，返回当前 MERGE 操作对行执行的动作类型。

**返回值**:

| 返回值 | 说明 |
|--------|------|
| `INSERT` | 行被插入 |
| `UPDATE` | 行被更新 |
| `DELETE` | 行被删除 |

**图 4.1：merge_action() 工作流程**

```
MERGE 执行流程:

源表行 ──▶ 匹配检查 ──┬──▶ MATCHED ─────────────┬──▶ UPDATE ──▶ merge_action() = 'UPDATE'
                     │    (条件满足)            │
                     │                         └──▶ DELETE ──▶ merge_action() = 'DELETE'
                     │
                     ├──▶ NOT MATCHED BY TARGET ──▶ INSERT ──▶ merge_action() = 'INSERT'
                     │    (源表有，目标表无)
                     │
                     └──▶ NOT MATCHED BY SOURCE ──▶ DELETE ──▶ merge_action() = 'DELETE'
                          (目标表有，源表无)
```

### 4.2 引用 source 和 target 列

RETURNING 子句可以引用：

- **目标表列**: 操作后的值（UPDATE/INSERT 后的新值）
- **源表列**: 用于审计追踪
- **merge_action()**: 操作类型标识

```sql
MERGE INTO products AS target
USING product_updates AS source
ON target.product_id = source.product_id

WHEN MATCHED THEN
    UPDATE SET
        price = source.price,
        stock = target.stock + source.stock_delta,
        updated_at = CURRENT_TIMESTAMP

WHEN NOT MATCHED BY TARGET THEN
    INSERT (product_id, price, stock, created_at)
    VALUES (source.product_id, source.price, source.stock_delta, CURRENT_TIMESTAMP)

RETURNING
    target.product_id,
    target.price AS new_price,
    target.stock AS new_stock,
    source.stock_delta AS delta,
    merge_action() AS action,
    CURRENT_TIMESTAMP AS processed_at;
```

### 4.3 实际应用：审计日志记录

**完整 CDC 审计系统**:

```sql
-- 审计日志表
CREATE TABLE merge_audit_log (
    log_id BIGSERIAL PRIMARY KEY,
    operation_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    table_name VARCHAR(100),
    record_id VARCHAR(100),
    action VARCHAR(10),  -- INSERT, UPDATE, DELETE
    old_data JSONB,
    new_data JSONB,
    source_info JSONB,
    processed_by VARCHAR(100) DEFAULT CURRENT_USER
);

-- 使用 CTE 记录审计日志
WITH merge_result AS (
    MERGE INTO inventory AS target
    USING inventory_staging AS source
    ON target.sku = source.sku

    WHEN MATCHED AND target.quantity != source.quantity THEN
        UPDATE SET
            quantity = source.quantity,
            last_updated = CURRENT_TIMESTAMP

    WHEN NOT MATCHED BY TARGET THEN
        INSERT (sku, product_name, quantity, last_updated)
        VALUES (source.sku, source.product_name, source.quantity, CURRENT_TIMESTAMP)

    WHEN NOT MATCHED BY SOURCE THEN
        DELETE

    RETURNING
        target.sku,
        target.product_name,
        target.quantity AS new_quantity,
        merge_action() AS action
)
INSERT INTO merge_audit_log (table_name, record_id, action, new_data)
SELECT
    'inventory',
    sku,
    action,
    jsonb_build_object(
        'sku', sku,
        'product_name', product_name,
        'quantity', new_quantity
    )
FROM merge_result;

-- 查看审计日志
SELECT * FROM merge_audit_log ORDER BY operation_time DESC LIMIT 10;
```

**更精细的审计（包含旧值）**:

```sql
-- 需要旧值的审计
WITH
-- 首先获取将要更新的行的旧值
target_snapshot AS (
    SELECT sku, quantity AS old_quantity, product_name
    FROM inventory
    WHERE sku IN (SELECT sku FROM inventory_staging)
),
-- 执行 MERGE
merge_result AS (
    MERGE INTO inventory AS target
    USING inventory_staging AS source
    ON target.sku = source.sku

    WHEN MATCHED THEN
        UPDATE SET quantity = source.quantity, last_updated = CURRENT_TIMESTAMP

    WHEN NOT MATCHED BY TARGET THEN
        INSERT (sku, product_name, quantity)
        VALUES (source.sku, source.product_name, source.quantity)

    RETURNING
        target.sku,
        target.quantity AS new_quantity,
        target.product_name,
        merge_action() AS action
)
-- 合并旧值并记录审计
INSERT INTO merge_audit_log (table_name, record_id, action, old_data, new_data)
SELECT
    'inventory',
    m.sku,
    m.action,
    CASE
        WHEN m.action = 'UPDATE' THEN
            jsonb_build_object('quantity', t.old_quantity)
        ELSE NULL
    END,
    jsonb_build_object(
        'sku', m.sku,
        'product_name', m.product_name,
        'quantity', m.new_quantity
    )
FROM merge_result m
LEFT JOIN target_snapshot t ON m.sku = t.sku;
```

## 5. 实际案例

### 5.1 CDC 场景：源表同步到目标表

**变更数据捕获**是 MERGE 的核心应用场景，PG17 的增强使其能够处理完整的双向同步。

**图 5.1：CDC 同步架构**

```
┌─────────────────────────────────────────────────────────────────────┐
│                        CDC 同步流程                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  ┌──────────┐     变更事件      ┌──────────────┐                    │
│  │ 源数据库 │  ──────────────▶  │  消息队列    │                    │
│  │ (MySQL)  │   (Debezium)     │  (Kafka)     │                    │
│  └──────────┘                   └──────────────┘                    │
│                                        │                            │
│                                        ▼                            │
│                              ┌──────────────────┐                   │
│                              │  staging 表      │                   │
│                              │  (临时存储)       │                   │
│                              └──────────────────┘                   │
│                                        │                            │
│                                        ▼                            │
│  ┌──────────────────────────────────────────────────────────────┐   │
│  │                    MERGE 同步任务                             │   │
│  │  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐       │   │
│  │  │   INSERT     │  │   UPDATE     │  │   DELETE     │       │   │
│  │  │ (新数据)     │  │ (修改数据)   │  │ (已删数据)   │       │   │
│  │  └──────────────┘  └──────────────┘  └──────────────┘       │   │
│  └──────────────────────────────────────────────────────────────┘   │
│                                        │                            │
│                                        ▼                            │
│                              ┌──────────────────┐                   │
│                              │  目标表          │                   │
│                              │  (PostgreSQL)    │                   │
│                              └──────────────────┘                   │
└─────────────────────────────────────────────────────────────────────┘
```

**完整 CDC 同步实现**:

```sql
-- 源表变更 staging (由 Debezium/Kafka Connect 填充)
CREATE TABLE cdc_customer_changes (
    change_id BIGSERIAL PRIMARY KEY,
    source_id VARCHAR(50) NOT NULL,  -- 源系统ID
    operation VARCHAR(10) NOT NULL,   -- INSERT/UPDATE/DELETE
    change_time TIMESTAMP NOT NULL,
    processed BOOLEAN DEFAULT false,

    -- 业务数据
    customer_id VARCHAR(50),
    name VARCHAR(100),
    email VARCHAR(100),
    phone VARCHAR(20),
    address TEXT
);

-- 创建索引加速查询
CREATE INDEX idx_cdc_unprocessed ON cdc_customer_changes(processed, change_time)
    WHERE processed = false;

-- 目标表
CREATE TABLE dim_customers (
    customer_id VARCHAR(50) PRIMARY KEY,
    name VARCHAR(100),
    email VARCHAR(100),
    phone VARCHAR(20),
    address TEXT,

    -- CDC 元数据
    source_system VARCHAR(50),
    first_seen_at TIMESTAMP,
    last_updated_at TIMESTAMP,
    last_change_id BIGINT,
    is_deleted BOOLEAN DEFAULT false,
    version INT DEFAULT 1
);

-- CDC 同步 MERGE (定时任务执行)
WITH changes_to_process AS (
    SELECT DISTINCT ON (customer_id)
        change_id,
        source_id,
        operation,
        customer_id,
        name,
        email,
        phone,
        address
    FROM cdc_customer_changes
    WHERE processed = false
    ORDER BY customer_id, change_time DESC
),
merge_result AS (
    MERGE INTO dim_customers AS target
    USING changes_to_process AS source
    ON target.customer_id = source.customer_id

    -- 匹配且源是 DELETE → 标记删除
    WHEN MATCHED AND source.operation = 'DELETE' THEN
        UPDATE SET
            is_deleted = true,
            last_updated_at = CURRENT_TIMESTAMP,
            last_change_id = source.change_id,
            version = target.version + 1

    -- 匹配且源是 UPDATE → 更新
    WHEN MATCHED AND source.operation = 'UPDATE' THEN
        UPDATE SET
            name = COALESCE(source.name, target.name),
            email = COALESCE(source.email, target.email),
            phone = COALESCE(source.phone, target.phone),
            address = COALESCE(source.address, target.address),
            is_deleted = false,
            last_updated_at = CURRENT_TIMESTAMP,
            last_change_id = source.change_id,
            version = target.version + 1

    -- 不匹配且源是 INSERT → 插入
    WHEN NOT MATCHED BY TARGET AND source.operation = 'INSERT' THEN
        INSERT (customer_id, name, email, phone, address,
                source_system, first_seen_at, last_updated_at, last_change_id)
        VALUES (source.customer_id, source.name, source.email,
                source.phone, source.address, source.source_id,
                CURRENT_TIMESTAMP, CURRENT_TIMESTAMP, source.change_id)

    -- 目标有但 staging 本次无 (可能是延迟) → 不处理
    -- 如需严格同步，可添加 WHEN NOT MATCHED BY SOURCE

    RETURNING
        target.customer_id,
        merge_action() AS action,
        target.last_change_id
)
-- 标记已处理
UPDATE cdc_customer_changes cdc
SET processed = true
FROM merge_result mr
WHERE cdc.change_id = mr.last_change_id;

-- 检查同步状态
SELECT
    action,
    COUNT(*)
FROM (
    SELECT customer_id, merge_action() AS action
    FROM dim_customers
    WHERE last_updated_at > CURRENT_DATE
) t
GROUP BY action;
```

### 5.2 数据仓库维度表更新（SCD Type 2）

**缓慢变化维度 Type 2** 需要保存历史记录，MERGE 可以简化此过程。

**图 5.2：SCD Type 2 时间线**

```
时间轴 ───────────────────────────────────────────▶

产品A记录1: [2024-01-01 ────────── 2024-06-15)  价格: $100
                              ↓ 价格变更
产品A记录2:                      [2024-06-15 ───────────▶)  价格: $120
                              ↑ 生效日期                 ↑ 当前

查询历史价格:
- 2024-03-01 → 产品A记录1, 价格 $100
- 2024-07-01 → 产品A记录2, 价格 $120
```

**SCD Type 2 MERGE 实现**:

```sql
-- 源表：最新产品数据
CREATE TABLE staging_products (
    product_code VARCHAR(50) PRIMARY KEY,
    product_name VARCHAR(200),
    category VARCHAR(50),
    price DECIMAL(10,2),
    cost DECIMAL(10,2)
);

-- 目标表：SCD Type 2 维度表
CREATE TABLE dim_products_scd2 (
    product_key SERIAL PRIMARY KEY,
    product_code VARCHAR(50) NOT NULL,
    product_name VARCHAR(200),
    category VARCHAR(50),
    price DECIMAL(10,2),
    cost DECIMAL(10,2),
    effective_date DATE NOT NULL,
    expiration_date DATE,  -- NULL 表示当前有效
    is_current BOOLEAN DEFAULT true,

    UNIQUE (product_code, effective_date)
);

-- 索引加速查询
CREATE INDEX idx_scd2_lookup ON dim_products_scd2(product_code, is_current);

-- SCD Type 2 MERGE 过程
WITH
-- 步骤1: 结束已变更记录的当前版本
expired_records AS (
    UPDATE dim_products_scd2 target
    SET
        expiration_date = CURRENT_DATE,
        is_current = false
    FROM staging_products source
    WHERE target.product_code = source.product_code
      AND target.is_current = true
      AND (target.price != source.price OR target.category != source.category)
    RETURNING target.product_code
),
-- 步骤2: 插入新版本和全新记录
merge_result AS (
    MERGE INTO dim_products_scd2 AS target
    USING staging_products AS source
    ON target.product_code = source.product_code
       AND target.is_current = true
       AND (target.price = source.price AND target.category = source.category)

    -- 完全匹配 → 仅更新非关键属性
    WHEN MATCHED THEN
        UPDATE SET
            product_name = source.product_name,
            cost = source.cost

    -- 新记录或已变更 → 插入新版本
    WHEN NOT MATCHED BY TARGET THEN
        INSERT (product_code, product_name, category, price, cost,
                effective_date, expiration_date, is_current)
        VALUES (source.product_code, source.product_name, source.category,
                source.price, source.cost, CURRENT_DATE, NULL, true)

    RETURNING
        target.product_key,
        target.product_code,
        merge_action() AS action
)
-- 统计结果
SELECT action, COUNT(*) FROM merge_result GROUP BY action
UNION ALL
SELECT 'EXPIRED' AS action, COUNT(*) FROM expired_records;

-- 查询当前有效的产品
SELECT * FROM dim_products_scd2 WHERE is_current = true;

-- 查询某历史时间点的产品价格
SELECT * FROM dim_products_scd2
WHERE product_code = 'PROD001'
  AND effective_date <= '2024-03-01'
  AND (expiration_date IS NULL OR expiration_date > '2024-03-01');
```

### 5.3 批量数据导入与更新

**ETL 场景优化**:

```sql
-- 大规模数据导入优化配置
SET work_mem = '256MB';
SET maintenance_work_mem = '512MB';

-- 批量导入 staging
COPY staging_orders FROM '/data/orders_2024.csv' WITH (FORMAT csv, HEADER true);

-- 分析 staging 表
ANALYZE staging_orders;

-- 分区表 MERGE (假设 orders 是分区表)
MERGE INTO orders AS target
USING staging_orders AS source
ON target.order_id = source.order_id

WHEN MATCHED AND target.status != 'COMPLETED' THEN
    UPDATE SET
        status = source.status,
        total_amount = source.total_amount,
        updated_at = CURRENT_TIMESTAMP

WHEN NOT MATCHED BY TARGET THEN
    INSERT (order_id, customer_id, order_date, status, total_amount)
    VALUES (source.order_id, source.customer_id, source.order_date,
            source.status, source.total_amount)

WHEN NOT MATCHED BY SOURCE AND target.status = 'PENDING' THEN
    UPDATE SET status = 'CANCELLED', updated_at = CURRENT_TIMESTAMP;

-- 重置配置
RESET work_mem;
RESET maintenance_work_mem;
```

```
│  └──────────────────────────────────────────────────────┘     │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

**执行计划示例**:

```sql
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
MERGE INTO inventory t
USING inventory_staging s ON t.sku = s.sku
WHEN MATCHED THEN UPDATE SET quantity = s.quantity
WHEN NOT MATCHED BY TARGET THEN INSERT (sku, quantity) VALUES (s.sku, s.quantity)
WHEN NOT MATCHED BY SOURCE THEN DELETE;
```

```
               Sort Key: s.sku
```

### 6.2 索引建议

| 场景 | 推荐索引 | 说明 |
|------|---------|------|
| **匹配列** | `CREATE UNIQUE INDEX ON target(join_column)` | 加速 JOIN 匹配 |
| **源表排序** | 源表按 JOIN 列排序或使用索引 | 启用 Merge Join |
| **更新列** | 被 UPDATE 的列如果用于 WHERE | 考虑覆盖索引 |
| **大表** | 分区表按 JOIN 列分区 | 减少扫描范围 |

**优化示例**:

```sql

-- 2. 为源表创建临时索引（如果源表大）
CREATE INDEX idx_staging_sku ON inventory_staging(sku);

-- 3. 分析统计信息
ANALYZE inventory;
ANALYZE inventory_staging;

-- 4. 调整工作内存
SET work_mem = '128MB';
```

### 6.3 与 INSERT ON CONFLICT 的对比

| 特性 | MERGE | INSERT ON CONFLICT (UPSERT) |
|------|-------|---------------------------|
| **SQL标准** | ✅ SQL:2003 | ❌ PostgreSQL 扩展 |
| **INSERT + UPDATE** | ✅ 支持 | ✅ 支持 |
| **DELETE** | ✅ 支持 | ❌ 不支持 |
| **多条件分支** | ✅ 支持多个 WHEN | ⚠️ 仅 DO UPDATE/DO NOTHING |
| **源表支持** | ✅ 任意查询 | ⚠️ 仅 VALUES 或子查询 |
| **NOT MATCHED BY SOURCE** | ✅ PG17 支持 | ❌ 不支持 |
| **RETURNING** | ✅ PG17 支持 | ✅ 支持 |
| **性能** | 稍慢（复杂）| 更快（简单）|

**选择建议**:

```sql
-- 场景1: 简单 UPSERT（推荐 INSERT ON CONFLICT）
INSERT INTO users (id, name, email)
VALUES (1, 'John', 'john@example.com')
ON CONFLICT (id) DO UPDATE SET
    name = EXCLUDED.name,
    email = EXCLUDED.email;

-- 场景2: 复杂条件逻辑（推荐 MERGE）
MERGE INTO inventory AS target
USING stock_update AS source ON target.sku = source.sku
WHEN MATCHED AND source.quantity = 0 THEN DELETE
WHEN MATCHED AND target.quantity != source.quantity THEN
    UPDATE SET quantity = source.quantity, updated_at = NOW()
WHEN NOT MATCHED BY TARGET THEN
    INSERT (sku, quantity) VALUES (source.sku, source.quantity)
WHEN NOT MATCHED BY SOURCE AND target.last_sale < NOW() - INTERVAL '1 year' THEN
    DELETE;
```

## 8. 最佳实践

### 8.1 设计原则

1. **明确主键**: MERGE 的 ON 条件应当使用唯一键或主键
2. **批量处理**: 大数据量时分批处理，避免长事务
3. **索引优化**: 确保 JOIN 列有适当的索引
4. **测试验证**: 使用 RETURNING 验证操作结果

### 8.2 注意事项

```sql
-- 1. 避免在 MERGE 中修改 JOIN 列
MERGE INTO t USING s ON t.id = s.id
WHEN MATCHED THEN UPDATE SET id = s.new_id;  -- ⚠️ 危险！

-- 2. 注意并发控制
-- MERGE 会获取行级锁，高并发场景可能产生死锁
-- 建议按相同顺序处理数据

-- 3. 触发器行为
-- MERGE 的 UPDATE/INSERT/DELETE 会触发相应触发器
-- 注意触发器中的递归问题
```

### 8.3 监控与诊断

```sql
-- 查看 MERGE 语句的执行统计
SELECT
    query,
    calls,
    mean_exec_time,
    rows
FROM pg_stat_statements
WHERE query LIKE '%MERGE%'
ORDER BY mean_exec_time DESC;

-- 锁等待监控
SELECT
    blocked_locks.pid AS blocked_pid,
    blocked_activity.usename AS blocked_user,
    blocking_locks.pid AS blocking_pid,
    blocking_activity.usename AS blocking_user,
    blocked_activity.query AS blocked_statement
FROM pg_catalog.pg_locks blocked_locks
JOIN pg_catalog.pg_stat_activity blocked_activity ON blocked_activity.pid = blocked_locks.pid
JOIN pg_catalog.pg_locks blocking_locks ON blocking_locks.locktype = blocked_locks.locktype
JOIN pg_catalog.pg_stat_activity blocking_activity ON blocking_activity.pid = blocking_locks.pid
WHERE blocking_locks.relation = blocked_locks.relation
  AND blocked_locks.pid != blocking_locks.pid;
```

## 9. 总结

PostgreSQL 17 的 MERGE 增强使其从基础实现进化为生产级的数据同步工具：

| 增强特性 | 核心价值 |
|---------|---------|
| **WHEN NOT MATCHED BY SOURCE** | 支持完整双向同步，CDC 场景必备 |
| **视图目标支持** | 提供数据抽象层，简化应用逻辑 |
| **RETURNING + merge_action()** | 实现细粒度审计，提升可观测性 |

**适用场景总结**:

- ✅ 数据仓库 CDC 同步
- ✅ SCD Type 2 维度更新
- ✅ 跨系统数据一致性维护
- ✅ 批量 ETL 处理
- ✅ 需要审计追踪的数据变更
