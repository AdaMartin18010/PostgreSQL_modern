## 1. 概述

时态约束是PostgreSQL 18引入的革命性功能，它通过声明式约束机制解决了数据库中长期存在的时态数据管理难题。
传统的时态数据管理依赖应用层逻辑或复杂触发器实现，存在性能低下、容易出错、维护困难等问题。
时态约束提供了原生的数据库级解决方案，确保时态数据完整性。

> **定义 1.1 (时态约束)**：时态约束是数据库完整性约束的扩展，专门用于确保时间维度数据的正确性，包括有效时间有效性、有效期不重叠、时态引用完整性等。

## 2. 时态数据模型理论

### 2.1 时间维度基础

时态数据库理论建立在双时间维度框架之上，区分数据的两种时间语义：

#### 2.1.1 有效时间 (Valid Time, VT)

有效时间描述数据在**现实世界**中真实的有效期间。

> **定义 2.1 (有效时间)**：设实体 $e$ 具有属性 $a$，其有效时间是一个时间区间 $[v_s, v_e)$，表示属性 $a$ 在现实世界中从时刻 $v_s$ 开始有效，到时刻 $v_e$ 结束有效。

$$VT(e) = [v_s, v_e) \subseteq \mathbb{T}$$

其中 $\mathbb{T}$ 是时间域，$v_s \in \mathbb{T}$ 是有效开始时间，$v_e \in \mathbb{T} \cup \{\infty\}$ 是有效结束时间。

**特性**：

- 用户可感知和修改
- 反映业务语义
- 支持未来日期的规划
- 可回溯历史状态

#### 2.1.2 事务时间 (Transaction Time, TT)

事务时间描述数据在**数据库中**存在的期间。

> **定义 2.2 (事务时间)**：设关系 $R$ 中的元组 $t$，其事务时间是一个时间区间 $[t_s, t_e)$，表示该元组从时刻 $t_s$ 开始被存储于数据库，到时刻 $t_e$ 被逻辑删除。

$$TT(t) = [t_s, t_e) \subseteq \mathbb{T}_{db}$$

其中 $\mathbb{T}_{db}$ 是数据库系统时间域，满足 $t_s \leq NOW$，$t_e \in (NOW, \infty]$。

**特性**：

- 系统自动生成和维护
- 单调递增，不可修改
- 支持审计和追溯
- 确保数据血缘追踪

#### 2.1.3 双时态对比表

| 特性 | 有效时间 (Valid Time) | 事务时间 (Transaction Time) |
|------|----------------------|----------------------------|
| **时间语义** | 现实世界时间 | 数据库时间 |
| **修改权限** | 用户可修改 | 系统管理 |
| **时间范围** | 过去、现在、未来 | 过去到现在 |
| **主要用途** | 业务逻辑 | 审计追踪 |
| **表示符号** | $[v_s, v_e)$ | $[t_s, t_e)$ |
| **典型应用** | 合同有效期、价格历史 | 数据变更日志 |

**图 2.1：双时态模型示意**

```text
                    数据库时间轴 →
                    t₁      t₂      t₃      t₄      t₅
                    |       |       |       |       |
    现              ├───────┼───────┼───────┼───────┤
    实     VT₁ [vₛ₁───────vₑ₁)
    世              │       │       │       │       │
    界     VT₂      │  [vₛ₂──────────vₑ₂)   │       │
    时              │       │       │       │       │
    间     VT₃      │       │   [vₛ₃──────────vₑ₃)  │
    轴              │       │       │       │       │
              ↓     │       │       │       │       │
                    │       │       │       │       │
    [数据库状态]    │  S₁   │  S₂   │  S₃   │  S₄   │
```

### 2.2 双时态模型 (Bitemporal Model)

双时态模型将有效时间和事务时间结合，构成二维时间空间。

> **定义 2.3 (双时态数据)**：一个双时态数据项是一个四元组 $(v_s, v_e, t_s, t_e, D)$，其中 $D$ 是非时态数据，$(v_s, v_e)$ 是有效时间区间，$(t_s, t_e)$ 是事务时间区间。

$$Bitemporal\_Data = \{(v_s, v_e, t_s, t_e, D) \mid v_s < v_e \land t_s < t_e\}$$

**图 2.2：双时态笛卡尔平面**

```text
    有效时间轴 (Valid Time) →

    vₑ₄ ↑
        │    ┌─────────┐
    vₑ₃ │    │  D₁(t₁) │ ← 事务时间 t₁ 时的数据状态
        │    │  (D的VT)│
    vₛ₃ │    └─────────┘
        │         ┌─────────┐
    vₑ₂ │         │  D₂(t₂) │ ← 事务时间 t₂ 时修改后的数据
        │         │  (D'的VT)│
    vₛ₂ │         └─────────┘
        │              ┌─────────┐
    vₑ₁ │              │  D₃(t₃) │ ← 事务时间 t₃ 时的最新状态
    vₛ₁ │              └─────────┘
        └────────────────────────────────→
            tₛ₁      tₛ₂      tₛ₃    事务时间轴 (Transaction Time)
```

双时态查询的时态谓词：

$$
Valid_{t_v}(D) \land Transaction_{t_t}(D) = \begin{cases}
true & \text{if } v_s \leq t_v < v_e \land t_s \leq t_t < t_e \\
false & \text{otherwise}
\end{cases}
$$

### 2.3 SQL:2011 时态标准

SQL:2011标准正式引入了时态数据支持，定义了标准化语法。

#### 2.3.1 标准语法

```sql
-- 应用周期表 (Application-time period table) - 有效时间
CREATE TABLE employees (
    emp_id INTEGER PRIMARY KEY,
    name VARCHAR(100),
    dept_id INTEGER,
    salary DECIMAL(10,2),
    valid_start DATE,
    valid_end DATE,
    PERIOD FOR valid_period (valid_start, valid_end)
);

-- 系统版本表 (System-versioned table) - 事务时间
CREATE TABLE employees_history (
    emp_id INTEGER,
    name VARCHAR(100),
    sys_start TIMESTAMP GENERATED ALWAYS AS ROW START,
    sys_end TIMESTAMP GENERATED ALWAYS AS ROW END,
    PERIOD FOR SYSTEM_TIME (sys_start, sys_end)
) WITH SYSTEM VERSIONING;
```

#### 2.3.2 标准运算符

| 运算符 | 含义 | PostgreSQL 18等效实现 |
|--------|------|----------------------|
| `OVERLAPS` | 区间重叠检测 | `&&` (范围类型) |
| `IMMEDIATELY PRECEDES` | 紧邻前置 | 自定义函数 |
| `IMMEDIATELY SUCCEEDS` | 紧邻后继 | 自定义函数 |
| `CONTAINS` | 包含关系 | `@>` (范围类型) |
| `EQUALS` | 相等 | `=` (范围类型) |

## 3. 时态约束类型

### 3.1 WITHOUT OVERLAPS 约束

这是PostgreSQL 18引入的核心时态约束。

> **定义 3.1 (WITHOUT OVERLAPS)**：对于关系 $R$ 的属性集 $A$ 和时态属性 $T$，约束 WITHOUT OVERLAPS 确保任意两个在 $A$ 上相等的元组，其时态区间 $T$ 不重叠。

$$\forall t_1, t_2 \in R: (t_1.A = t_2.A \land t_1 \neq t_2) \Rightarrow T(t_1) \cap T(t_2) = \emptyset$$

#### 3.1.1 语法定义

```sql
-- 基础语法
CREATE TABLE employees (
    emp_id INTEGER,
    department_id INTEGER,
    valid_period DATERANGE,
    PRIMARY KEY (emp_id, valid_period WITHOUT OVERLAPS)
);

-- 多列键 + 时态约束
CREATE TABLE position_history (
    emp_id INTEGER,
    position_code VARCHAR(20),
    valid_period TSTZRANGE,
    PRIMARY KEY (emp_id, position_code, valid_period WITHOUT OVERLAPS)
);

-- 独立约束形式
CREATE TABLE price_history (
    product_id INTEGER,
    valid_period DATERANGE,
    price DECIMAL(10,2),
    CONSTRAINT unique_valid_period
        EXCLUDE USING gist (product_id WITH =, valid_period WITH &&)
);
```

#### 3.1.2 数学语义

**图 3.1：WITHOUT OVERLAPS 约束原理**

```
    时间轴 →

    允许的情况:
    emp_id=100:  [===========)
                             [===========)

    emp_id=200:  [============================)

    禁止的情况 (违反约束):
    emp_id=100:  [===========)
                       [===========)  ← 重叠! 错误!
                       ↑
                    交集 ≠ ∅
```

设 $K$ 是键属性集，$P$ 是时态区间属性，则约束的集合表示为：

$$R' = \{(k, p, d) \in R \mid \nexists (k', p', d') \in R: k = k' \land p \cap p' \neq \emptyset \land (p, d) \neq (p', d')\}$$

#### 3.1.3 复杂度分析

**时间复杂度**：

- 插入检查：$O(\log n + m)$，其中 $n$ 是B-tree索引大小，$m$ 是重叠区间数
- 使用 GiST 索引：平均 $O(\log n)$

**空间复杂度**：$O(n)$ 用于索引存储

### 3.2 有效期不重叠检查

这是 WITHOUT OVERLAPS 的具体实现，支持多种检查级别。

#### 3.2.1 完全重叠禁止

```sql
-- 禁止任何时间上的重叠
CREATE TABLE employee_dept_history (
    emp_id INTEGER NOT NULL,
    dept_id INTEGER NOT NULL,
    valid_period TSTZRANGE NOT NULL,

    CONSTRAINT no_overlap_any
        EXCLUDE USING gist (
            emp_id WITH =,
            valid_period WITH &&
        )
);
```

> **定义 3.2 (完全重叠禁止)**：禁止任何两个相同键值的区间存在非空交集。

$$\forall t_1, t_2: (t_1.K = t_2.K \land t_1 \neq t_2) \Rightarrow t_1.P \cap t_2.P = \emptyset$$

#### 3.2.2 相邻区间合并

```sql
-- 禁止重叠但允许相邻
CREATE TABLE salary_history (
    emp_id INTEGER,
    salary DECIMAL(12,2),
    valid_period DATERANGE,

    CONSTRAINT no_overlap_mergeable
        EXCLUDE USING gist (
            emp_id WITH =,
            valid_period WITH &&
        )
);

-- 相邻合并触发器 (可选优化)
CREATE OR REPLACE FUNCTION merge_adjacent_periods()
RETURNS TRIGGER AS $$
BEGIN
    -- 检查并合并相邻区间
    UPDATE salary_history
    SET valid_period = daterange(
        lower(valid_period),
        upper(NEW.valid_period),
        '[)'
    )
    WHERE emp_id = NEW.emp_id
      AND upper(valid_period) = lower(NEW.valid_period);

    RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

**图 3.2：重叠检查类型对比**

```
    时间轴 →

    类型A: 完全禁止重叠
    ─────────────────────────────────
    [========)        [========)
            ↑
        不允许 [====) 插入

    类型B: 允许相邻 (可合并)
    ─────────────────────────────────
    [========)[========)[========)
            ↑
        相邻可自动合并为 [================)
```

#### 3.2.3 部分键重叠控制

```sql
-- 仅当部门相同时禁止重叠
CREATE TABLE dept_assignments (
    emp_id INTEGER,
    dept_id INTEGER,
    valid_period DATERANGE,

    CONSTRAINT unique_dept_assignment
        EXCLUDE USING gist (
            emp_id WITH =,
            dept_id WITH =,
            valid_period WITH &&
        )
);

-- 同一员工可在不同部门有重叠时段
-- emp_id=1, dept_id=10, [2024-01-01, 2024-06-01)  ✓
-- emp_id=1, dept_id=20, [2024-03-01, 2024-09-01)  ✓ (不同部门)
-- emp_id=1, dept_id=10, [2024-05-01, 2024-12-01)  ✗ (同部门重叠)
```

### 3.3 外键时态引用完整性

时态外键确保引用关系在时间上的一致性。

> **定义 3.3 (时态引用完整性)**：设关系 $R$ 引用关系 $S$ 的时态键 $K$，时态外键约束要求对于 $R$ 中的每个元组 $r$，存在 $S$ 中的元组 $s$ 满足：
>
> 1. 键值匹配：$r.FK = s.K$
> 2. 时间包含：$r.P \subseteq s.P$

$$\forall r \in R: \exists s \in S: r.FK = s.K \land r.P \subseteq s.P$$

#### 3.3.1 基础时态外键

```sql
-- 被引用表: 部门有效性历史
CREATE TABLE departments_history (
    dept_id INTEGER,
    dept_name VARCHAR(100),
    valid_period DATERANGE,
    PRIMARY KEY (dept_id, valid_period WITHOUT OVERLAPS)
);

-- 引用表: 员工部门分配 (必须在部门有效期间内)
CREATE TABLE emp_dept_assignments (
    emp_id INTEGER PRIMARY KEY,
    dept_id INTEGER,
    assignment_period DATERANGE,

    -- 时态外键约束
    CONSTRAINT fk_temporal_dept
        FOREIGN KEY (dept_id, assignment_period)
        REFERENCES departments_history(dept_id, valid_period)
        MATCH TEMPORAL  -- PostgreSQL 18新语法
);
```

#### 3.3.2 区间包含语义

```sql
-- 更精确的区间包含约束
CREATE OR REPLACE FUNCTION check_temporal_fk()
RETURNS TRIGGER AS $$
BEGIN
    IF NOT EXISTS (
        SELECT 1 FROM departments_history dh
        WHERE dh.dept_id = NEW.dept_id
          AND dh.valid_period @> NEW.assignment_period
    ) THEN
        RAISE EXCEPTION 'Temporal FK violation: no matching department for period %',
            NEW.assignment_period;
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_check_temporal_fk
BEFORE INSERT OR UPDATE ON emp_dept_assignments
FOR EACH ROW EXECUTE FUNCTION check_temporal_fk();
```

**图 3.3：时态外键约束语义**

```
    时间轴 →

    部门有效性:    [========================================)
```

### 4.2 范围类型应用

PostgreSQL的范围类型是时态约束的技术基础。

#### 4.2.1 范围类型对比

| 范围类型 | 元素类型 | 时态精度 | 典型应用 |
|----------|----------|----------|----------|
| `INT4RANGE` | INTEGER | 离散 | 版本号范围 |
| `INT8RANGE` | BIGINT | 离散 | 序列号范围 |
| `NUMRANGE` | NUMERIC | 连续 | 数值区间 |
| `TSRANGE` | TIMESTAMP | 微秒 | 无时区时间 |
| `TSTZRANGE` | TIMESTAMPTZ | 微秒 | 全局时间 |
| `DATERANGE` | DATE | 日 | 日期范围 |

#### 4.2.2 范围操作优化

```sql
-- 范围边界计算函数
CREATE OR REPLACE FUNCTION range_intersection(
    r1 TSTZRANGE,
    r2 TSTZRANGE
) RETURNS TSTZRANGE AS $$
BEGIN
    -- 交集公式: [max(l1,l2), min(u1,u2))
    IF r1 && r2 THEN
        RETURN TSTZRANGE(
            GREATEST(lower(r1), lower(r2)),
            LEAST(upper(r1), upper(r2)),
            '[)'
        );
    END IF;
    RETURN NULL;
END;
$$ LANGUAGE plpgsql IMMUTABLE;

-- 范围并集 (仅对重叠或相邻区间)
CREATE OR REPLACE FUNCTION range_union(
    r1 TSTZRANGE,
    r2 TSTZRANGE
) RETURNS TSTZRANGE AS $$
BEGIN
    IF r1 && r2 OR upper(r1) = lower(r2) OR upper(r2) = lower(r1) THEN
        RETURN TSTZRANGE(
            LEAST(lower(r1), lower(r2)),
            GREATEST(upper(r1), upper(r2)),
            '[)'
        );
    END IF;
    RAISE EXCEPTION 'Ranges cannot be unioned: % and %', r1, r2;
END;
$$ LANGUAGE plpgsql IMMUTABLE;

-- 范围差集
CREATE OR REPLACE FUNCTION range_minus(
    r1 TSTZRANGE,
    r2 TSTZRANGE
) RETURNS TSTZRANGE[] AS $$
DECLARE
    result TSTZRANGE[];
BEGIN
    IF NOT (r1 && r2) THEN
        RETURN ARRAY[r1];
    END IF;

    IF lower(r1) < lower(r2) THEN
        result := array_append(result,
            TSTZRANGE(lower(r1), lower(r2), '[)'));
    END IF;

    IF upper(r2) < upper(r1) THEN
        result := array_append(result,
            TSTZRANGE(upper(r2), upper(r1), '[)'));
    END IF;

    RETURN result;
END;
$$ LANGUAGE plpgsql IMMUTABLE;
```

**图 4.1：范围集合运算**

```

    A:           [===========)

    A && B:          [===)

    并集运算 (+
    A:           [===========)
    B:               [===========)
    A + B:       [================)

    差集运算 (-):
    A:           [================)
```

### 4.3 排除约束优化

EXCLUDE约束是实现WITHOUT OVERLAPS的核心机制。

#### 4.3.1 约束语法详解

```sql
-- 基础排除约束
EXCLUDE USING gist (
    column1 WITH operator1,
    column2 WITH operator2,
    ...
)

-- 等价逻辑: 对于任意两行r1, r2，不存在以下情况同时成立:
-- NOT (r1.col1 operator1 r2.col1) AND
-- NOT (r1.col2 operator2 r2.col2) AND
-- ...
```

> **定理 4.1 (排除约束正确性)**：EXCLUDE约束确保对于指定的列组合，不存在两行同时满足所有指定的操作符条件。

#### 4.3.2 操作符选择

| 操作符 | 含义 | 应用场景 |
|--------|------|----------|
| `=` | 相等 | 键值匹配 |
| `&&` | 重叠 | 时态区间检测 |
| `@>` | 包含 | 范围包含检查 |
| `<>` | 不相等 | 差异检测 |
| `=` + `&&` | 相等且重叠 | WITHOUT OVERLAPS |

```sql
-- 复杂排除约束示例
CREATE TABLE room_bookings (
    room_id INTEGER,
    booking_period TSTZRANGE,
    status VARCHAR(20),  -- 'confirmed', 'tentative'

    CONSTRAINT no_double_booking
        EXCLUDE USING gist (
            room_id WITH =,
            booking_period WITH &&
        ) WHERE (status = 'confirmed'),

    CONSTRAINT limited_tentative
        EXCLUDE USING gist (
            room_id WITH =,
            booking_period WITH &&
        ) WITH (max_tentative = 3)  -- 最多3个待定预订
);
```

### 5.2 价格历史追踪

#### 5.2.1 数据模型设计

```sql
-- 产品信息
CREATE TABLE products (
    product_id SERIAL PRIMARY KEY,
    sku VARCHAR(50) UNIQUE NOT NULL,
    name VARCHAR(200) NOT NULL,
    category_id INTEGER,
    base_cost DECIMAL(10,2),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 产品价格历史 (支持多币种)
CREATE TABLE product_price_history (
    price_id SERIAL,
    product_id INTEGER NOT NULL REFERENCES products(product_id),
    currency_code CHAR(3) NOT NULL DEFAULT 'CNY',
    price DECIMAL(10,2) NOT NULL CHECK (price >= 0),
    price_period TSTZRANGE NOT NULL,
    price_type VARCHAR(20) DEFAULT 'standard',  -- standard, promotional, vip

    -- 核心约束: 同一产品、同币种、同价格类型，时段不能重叠
    CONSTRAINT no_overlap_price
        EXCLUDE USING gist (
            product_id WITH =,
            currency_code WITH =,
            price_type WITH =,
            price_period WITH &&
        ),
    CONSTRAINT positive_price CHECK (price > 0)
);

-- 促销价格 (特殊逻辑)
CREATE TABLE promotional_prices (
    promo_id SERIAL,
    product_id INTEGER NOT NULL REFERENCES products(product_id),
    promo_code VARCHAR(50),
    discount_rate DECIMAL(4,2) CHECK (discount_rate BETWEEN 0 AND 1),
    fixed_price DECIMAL(10,2),
    promo_period TSTZRANGE NOT NULL,

    CONSTRAINT no_overlap_promo
        EXCLUDE USING gist (
            product_id WITH =,
            promo_period WITH &&
        ),
    CONSTRAINT valid_promo CHECK (
        (discount_rate IS NOT NULL AND fixed_price IS NULL) OR
        (discount_rate IS NULL AND fixed_price IS NOT NULL)
    )
);
```

#### 5.2.2 业务操作示例

```sql
    (100, 'CNY', 219.00, '[2024-06-01 00:00:00+08, infinity)', 'standard');
COMMIT;

-- 3. 查询特定时间点的价格
CREATE OR REPLACE FUNCTION get_price_at_time(
    p_product_id INTEGER,
    p_currency CHAR(3),
    p_timestamp TIMESTAMPTZ
) RETURNS DECIMAL(10,2) AS $$
DECLARE
    v_price DECIMAL(10,2);
    v_promo_price DECIMAL(10,2);
BEGIN
    -- 先检查促销价格
    SELECT fixed_price * (1 - discount_rate)
    INTO v_promo_price
    FROM promotional_prices
    WHERE product_id = p_product_id
      AND promo_period @> p_timestamp
    ORDER BY lower(promo_period) DESC
    LIMIT 1;

    IF v_promo_price IS NOT NULL THEN
        RETURN v_promo_price;
    END IF;

    -- 返回标准价格
    SELECT price INTO v_price
    FROM product_price_history
    WHERE product_id = p_product_id
      AND currency_code = p_currency
      AND price_type = 'standard'
      AND price_period @> p_timestamp;

    RETURN v_price;
END;
$$ LANGUAGE plpgsql;

-- 使用

    CASE
        WHEN upper(pph.price_period) = 'infinity' THEN 'Current'
        ELSE 'Historical'
    END AS status
FROM products p
JOIN product_price_history pph ON p.product_id = pph.product_id
WHERE p.sku = 'PROD-001'
ORDER BY pph.currency_code, lower(pph.price_period);

-- 5. 价格重叠检测 (审计)
SELECT
    p.sku,
    pph1.price_period AS period_1,
    pph2.price_period AS period_2,
    pph1.price_period * pph2.price_period AS overlap
FROM product_price_history pph1
JOIN product_price_history pph2 ON pph1.product_id = pph2.product_id
JOIN products p ON p.product_id = pph1.product_id
WHERE pph1.price_id < pph2.price_id
  AND pph1.price_period && pph2.price_period
  AND pph1.currency_code = pph2.currency_code
  AND pph1.price_type = pph2.price_type;
```

```
    约束保证: 同一币种标准价格不重叠
             促销价格可独立管理
```

### 5.3 地址变更历史

#### 5.3.1 数据模型设计

```sql
-- 地址类型枚举
CREATE TYPE address_type AS ENUM ('home', 'work', 'billing', 'shipping');

-- 地址历史表
CREATE TABLE address_history (
    address_id SERIAL,
    entity_type VARCHAR(20) NOT NULL,  -- 'employee', 'customer', 'supplier'
    entity_id INTEGER NOT NULL,
    address_type address_type NOT NULL,

    -- 地址字段
    street_address VARCHAR(200),
    city VARCHAR(100),
    state_province VARCHAR(100),
    postal_code VARCHAR(20),
    country_code CHAR(2),

    -- 时态字段
    valid_period DATERANGE NOT NULL,
    is_primary BOOLEAN DEFAULT false,

    -- 约束: 同一实体、同类型地址，时段不重叠
    CONSTRAINT no_overlap_address
        EXCLUDE USING gist (
            entity_type WITH =,
            entity_id WITH =,
            address_type WITH =,
            valid_period WITH &&
        ),
    -- 约束: 同一实体、同时段只能有一个主地址
    CONSTRAINT unique_primary_address
        EXCLUDE USING gist (
            entity_type WITH =,
            entity_id WITH =,
            valid_period WITH &&
        ) WHERE (is_primary = true)
);

-- 地址索引
CREATE INDEX idx_address_lookup ON address_history
    (entity_type, entity_id, address_type, valid_period);
```

### 5.4 合同有效期管理

#### 5.4.1 数据模型设计

```sql
-- 合同类型
CREATE TYPE contract_status AS ENUM (
    'draft', 'negotiating', 'active', 'suspended', 'expired', 'terminated'
);

-- 合同主表
CREATE TABLE contracts (
    contract_id SERIAL PRIMARY KEY,
    contract_no VARCHAR(50) UNIQUE NOT NULL,
    contract_type VARCHAR(50) NOT NULL,  -- 'service', 'purchase', 'employment'
    party_a_id INTEGER NOT NULL,  -- 甲方
    party_b_id INTEGER NOT NULL,  -- 乙方
    title VARCHAR(200) NOT NULL,
    contract_status contract_status DEFAULT 'draft',

    -- 合同价值
    contract_value DECIMAL(15,2),
    currency_code CHAR(3) DEFAULT 'CNY',

    -- 时态字段 (合同有效期)
    valid_period DATERANGE NOT NULL,

    -- 元数据
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    created_by INTEGER,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 合同版本历史 (双时态)
CREATE TABLE contract_versions (
    version_id SERIAL PRIMARY KEY,
    contract_id INTEGER NOT NULL REFERENCES contracts(contract_id),
    version_no INTEGER NOT NULL,

    -- 合同内容
    content TEXT,
    terms JSONB,
    attachments JSONB,

    -- 版本时态 (事务时间)
    tx_period TSTZRANGE NOT NULL DEFAULT
        tstzrange(CURRENT_TIMESTAMP, 'infinity', '[)'),

    -- 修改信息
    modified_by INTEGER,
    modification_reason TEXT,

    -- 约束: 同一合同的版本，事务时间不重叠
    CONSTRAINT no_overlap_version
        EXCLUDE USING gist (
            contract_id WITH =,
            tx_period WITH &&
        )
);

-- 合同履行阶段 (项目里程碑)
CREATE TABLE contract_phases (
    phase_id SERIAL,
    contract_id INTEGER NOT NULL REFERENCES contracts(contract_id),
    phase_name VARCHAR(100) NOT NULL,
    phase_order INTEGER NOT NULL,
    phase_period DATERANGE NOT NULL,
    deliverables JSONB,
    completion_status VARCHAR(20) DEFAULT 'pending',

    -- 约束: 同一合同的阶段，时间不能重叠
    CONSTRAINT no_overlap_phase
        EXCLUDE USING gist (
            contract_id WITH =,
            phase_period WITH &&
        )
);

-- 付款计划
CREATE TABLE payment_schedules (
    schedule_id SERIAL,
    contract_id INTEGER NOT NULL REFERENCES contracts(contract_id),
    payment_no INTEGER NOT NULL,
    payment_description VARCHAR(200),
    scheduled_amount DECIMAL(15,2) NOT NULL,
    scheduled_date DATE NOT NULL,
    due_period DATERANGE NOT NULL,  -- 付款期限
    payment_status VARCHAR(20) DEFAULT 'pending',

    -- 约束: 同一合同的付款，期限不重叠
    CONSTRAINT no_overlap_payment
        EXCLUDE USING gist (
            contract_id WITH =,
            due_period WITH &&
        )
);
```

#### 5.4.2 业务操作示例

```sql
-- 1. 创建新合同
INSERT INTO contracts
    (contract_no, contract_type, party_a_id, party_b_id, title,
     contract_value, currency_code, valid_period, contract_status)
VALUES
    ('CTR-2024-001', 'service', 100, 200, '软件开发和维护服务合同',
     500000.00, 'CNY', '[2024-01-01, 2024-12-31)', 'active');

-- 2. 创建合同版本
INSERT INTO contract_versions
    (contract_id, version_no, content, terms, modified_by, modification_reason)
VALUES
    (1, 1, '合同正文...', '{"payment_terms": "30 days"}'::jsonb,
     1001, 'Initial contract creation');

-- 3. 修改合同 (创建新版本，旧版本结束事务时间)
BEGIN;
UPDATE contract_versions
SET tx_period = tstzrange(lower(tx_period), CURRENT_TIMESTAMP, '[)')
WHERE contract_id = 1
  AND upper(tx_period) = 'infinity';

INSERT INTO contract_versions
    (contract_id, version_no, content, terms, modified_by, modification_reason)
VALUES
    (1, 2, '修改后的合同正文...',
     '{"payment_terms": "15 days", "penalty_clause": "updated"}'::jsonb,
     1001, 'Updated payment terms per negotiation');
COMMIT;

-- 4. 合同延期
UPDATE contracts
SET valid_period = daterange(
    lower(valid_period),
    '2025-06-30',
    '[)'
)
WHERE contract_id = 1;

-- 5. 查询合同在特定日期的有效版本
SELECT
    c.contract_no,
    c.title,
    cv.version_no,
    cv.tx_period AS version_valid_period,
    cv.modification_reason
FROM contracts c
JOIN contract_versions cv ON c.contract_id = cv.contract_id
WHERE c.contract_id = 1
  AND cv.tx_period @> '2024-06-15 10:00:00+08'::timestamptz
ORDER BY cv.version_no DESC
LIMIT 1;

-- 6. 合同履行时间线
SELECT
    cp.phase_name,
    cp.phase_order,
    cp.phase_period,
    lower(cp.phase_period) AS start_date,
    upper(cp.phase_period) AS end_date,
    cp.completion_status
FROM contract_phases cp
WHERE cp.contract_id = 1
ORDER BY cp.phase_order;

-- 7. 付款计划冲突检测
SELECT
    c.contract_no,
    ps1.payment_no AS payment_1,
    ps2.payment_no AS payment_2,
    ps1.due_period * ps2.due_period AS overlap_period
FROM payment_schedules ps1
JOIN payment_schedules ps2 ON ps1.contract_id = ps2.contract_id
JOIN contracts c ON c.contract_id = ps1.contract_id
WHERE ps1.schedule_id < ps2.schedule_id
  AND ps1.due_period && ps2.due_period;

-- 8. 即将到期的合同预警
SELECT
    contract_no,
    title,
    upper(valid_period) AS expiry_date,
    (upper(valid_period) - CURRENT_DATE) AS days_remaining,
    contract_value,
    CASE
        WHEN upper(valid_period) - CURRENT_DATE <= 30 THEN 'URGENT'
        WHEN upper(valid_period) - CURRENT_DATE <= 90 THEN 'WARNING'
        ELSE 'NORMAL'
    END AS alert_level
FROM contracts
WHERE upper(valid_period) != 'infinity'
  AND upper(valid_period) > CURRENT_DATE
  AND upper(valid_period) <= CURRENT_DATE + INTERVAL '180 days'
ORDER BY upper(valid_period);
```

### 6.2 时态数据一致性保证

```sql
-- 触发器: 确保时态数据的连续性
CREATE OR REPLACE FUNCTION check_temporal_continuity()
RETURNS TRIGGER AS $$
DECLARE
    gap_count INTEGER;
BEGIN
    -- 检查是否存在时间间隙 (针对需要连续性的场景)
    SELECT COUNT(*) INTO gap_count
    FROM (
        SELECT
            upper(valid_period) AS period_end,
            LEAD(lower(valid_period)) OVER (ORDER BY lower(valid_period)) AS next_start
        FROM emp_department_history
        WHERE emp_id = NEW.emp_id
    ) t
    WHERE period_end IS NOT NULL
      AND next_start IS NOT NULL
      AND period_end < next_start;

    IF gap_count > 0 THEN
        RAISE WARNING 'Temporal gap detected for emp_id %', NEW.emp_id;
    END IF;

    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- 触发器: 自动维护更新时间戳
CREATE OR REPLACE FUNCTION update_timestamp()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = CURRENT_TIMESTAMP;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_update_contract_timestamp
BEFORE UPDATE ON contracts
FOR EACH ROW EXECUTE FUNCTION update_timestamp();
```

## 8. 总结与最佳实践

### 8.1 设计原则

1. **选择合适的时间粒度**：日期精度使用`DATERANGE`，时间精度使用`TSTZRANGE`
2. **合理设置上界**：当前有效记录使用`infinity`而非NULL
3. **左闭右开区间**：统一使用`[)`语义，便于连续区间拼接
4. **索引策略**：GiST索引用于重叠检测，B-tree用于边界查询

### 8.2 实施建议

1. **渐进式迁移**：现有系统可先用触发器模拟，再逐步迁移到原生约束
2. **分区表应用**：大数据量时态表务必分区，按时间或键值分区
3. **监控与审计**：建立时态数据质量监控，定期检查约束有效性
4. **文档化**：时态约束的业务含义必须文档化，便于维护

### 8.3 版本兼容性

| PostgreSQL版本 | 时态约束支持 | 说明 |
|---------------|-------------|------|
| 16 及以下 | ❌ 不支持 | 需使用 EXCLUDE 约束模拟 |
| 17 | ❌ 不支持 | 时态约束在开发周期中被回滚，未发布 |
| 18+ | ✅ **原生支持** | 完整的 `WITHOUT OVERLAPS` 和 `PERIOD` 语法 |

#### 历史背景

- **PG17 开发周期**: 时态约束最初实现，但在发布前被回滚
- **PG18 (2025-09-25)**: 时态约束正式发布，包含完整功能

#### 迁移建议

从 PG16 及以下版本迁移到 PG18 时，可以将现有的 `EXCLUDE` 约束改写为新的时态约束语法：

```sql
-- PG16 及以下的写法
CREATE TABLE bookings (
    room_id INT NOT NULL,
    booking_period TSTZRANGE NOT NULL,
    CONSTRAINT no_double_booking
        EXCLUDE USING gist (room_id WITH =, booking_period WITH &&)
);

-- PG18+ 的推荐写法
CREATE TABLE bookings (
    room_id INT NOT NULL,
    booking_period TSTZRANGE NOT NULL,
    PRIMARY KEY (room_id, booking_period WITHOUT OVERLAPS)
);
```
