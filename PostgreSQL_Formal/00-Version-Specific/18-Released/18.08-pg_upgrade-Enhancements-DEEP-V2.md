## 1. 升级机制理论

### 1.1 原地升级 vs 逻辑升级

PostgreSQL提供两种主要的大版本升级路径，各有其适用场景和技术特点。

**原地升级(In-place Upgrade):**

```text
┌─────────────────────────────────────────────────────────────────┐
│                    原地升级流程图                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  PostgreSQL 17 Cluster          PostgreSQL 18 Cluster          │
│  ┌──────────────────┐           ┌──────────────────┐           │
│  │ Data Directory   │           │ Data Directory   │           │
│  │ ┌──────────────┐ │           │ ┌──────────────┐ │           │
│  │ │ PG_VERSION   │─┼──┐    ┌───┼─│ PG_VERSION   │ │           │
│  │ │ (17)         │ │  │    │   │ │ (18)         │ │           │
│  │ ├──────────────┤ │  │    │   │ ├──────────────┤ │           │
│  │ │ base/        │◄─┼──┼────┼───┼─►│ base/        │ │           │
│  │ │ global/      │◄─┼──┼────┼───┼─►│ global/      │ │           │
│  │ │ pg_xact/     │◄─┼──┼────┼───┼─►│ pg_xact/     │ │           │
│  │ └──────────────┘ │  │    │   │ └──────────────┘ │           │
│  └──────────────────┘  │    │   └──────────────────┘           │
│                        │    │                                    │
│                   ┌────┘    └────┐                              │
│                   │   pg_upgrade   │                              │
│                   │  ├─ 复制数据   │                              │
│                   │  ├─ 更新目录   │                              │
│                   │  └─ 迁移配置   │                              │
│                   └───────────────┘                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**原地升级的核心公式:**

总升级时间 = 数据复制时间 + 目录升级时间 + 验证时间

$$T_{in-place} = T_{copy} + T_{catalog} + T_{verify}$$

其中数据复制时间:

$$T_{copy} = \frac{DataSize}{Bandwidth_{effective}} + Latency_{metadata}$$

**逻辑升级(Logical Upgrade):**

```text
┌─────────────────────────────────────────────────────────────────┐
│                    逻辑升级流程图                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  PostgreSQL 17              pg_dump/pg_restore              18  │
│  ┌─────────────┐           ┌──────────────┐           ┌────────┐│
│  │ Source DB   │──────────►│ SQL/Custom   │──────────►│ Target ││
│  │             │   dump    │ Format File  │  restore  │   DB   ││
│  └─────────────┘           └──────────────┘           └────────┘│
│        │                          │                      ▲      │
│        │ Schema                   │ Data                 │      │
│        └──────────────────────────┼──────────────────────┘      │
│                                   │                             │
│                              ┌────┴────┐                        │
│                              │ Storage │                        │
│                              │  (Disk/ │                        │
│                              │  S3)    │                        │
│                              └─────────┘                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**逻辑升级的时间模型:**

$$T_{logical} = T_{schema\_dump} + T_{data\_dump} + T_{transfer} + T_{restore}$$

各分量展开:

$$T_{schema\_dump} = N_{tables} \cdot t_{schema} + N_{objects} \cdot t_{object}$$

$$T_{data\_dump} = \frac{DataSize}{CompressionRatio \cdot DumpBandwidth}$$

$$T_{restore} = \frac{DataSize}{InsertBandwidth} + N_{indexes} \cdot T_{index\_build}$$

**两种方案对比:**

| 维度 | 原地升级 | 逻辑升级 |
|------|----------|----------|
| 停机时间 | 短(分钟级) | 长(小时级) |
| 数据复制 | 物理复制(O(1)复杂度) | 逻辑转换(O(N)复杂度) |
| 存储需求 | 2×峰值 | 1.5-3×持续 |
| 回滚难度 | 较难 | 较易 |
| 跨架构支持 | 不支持 | 支持 |
| 选择性升级 | 不支持 | 支持 |
| 数据验证 | 自动页校验 | 需额外实现 |

**选择决策模型:**

定义总拥有成本函数:

$$TCO = \alpha \cdot T_{downtime} + \beta \cdot StorageCost + \gamma \cdot Risk$$

其中 $\alpha$ 是停机成本系数，$\beta$ 是存储成本系数，$\gamma$ 是风险系数。

原地升级适合条件:

$$T_{copy} < T_{max\_downtime} \land SameArchitecture = true$$

逻辑升级适合条件:

$$CrossArchitecture = true \lor SelectiveObjects \neq \emptyset$$

### 1.2 系统目录变更

**目录版本控制机制:**

PostgreSQL使用目录版本号(Catalog Version)管理兼容性:

```c
/* pg_control.h */
#define CATALOG_VERSION_NO  202406071
```

版本号变更触发条件:

$$\Delta Catalog = \{SchemaChange, TypeChange, InternalChange\}$$

**目录结构映射:**

```text
┌─────────────────────────────────────────────────────────────────┐
│              系统目录升级映射关系                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  PostgreSQL 17        pg_upgrade转换规则      PostgreSQL 18    │
│  ┌──────────────┐                              ┌──────────────┐│
│  │ pg_class     │──┐  ┌──────────────────┐   │ pg_class     ││
│  │ (oid=1259)   │  ├──►│ 列映射表         ├──►│ (oid=1259)   ││
│  └──────────────┘  │  │ - 新增列赋值     │   │ +新列        ││
│                    │  │ - 删除列处理     │   └──────────────┘│
│  ┌──────────────┐  │  │ - 类型转换       │   ┌──────────────┐│
│  │ pg_attribute │──┼──►└──────────────────┘   │ pg_attribute ││
│  │ (oid=1249)   │  │                          │ (oid=1249)   ││
│  └──────────────┘  │  ┌──────────────────┐    └──────────────┘│
│                    └──►│ 内置对象映射     │                    │
│  ┌──────────────┐     │ - OID保留        │    ┌──────────────┐│
│  │ pg_type      │────►│ - 类型兼容检查   ├───►│ pg_type      ││
│  │ (oid=1247)   │     └──────────────────┘    │ (oid=1247)   ││
│  └──────────────┘                              └──────────────┘│
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**目录升级公式:**

目录升级时间复杂度:

$$T_{catalog} = O(N_{system\_tables} \cdot \log N_{rows})$$

具体计算:

$$T_{catalog} = \sum_{i=1}^{n} (R_i \cdot t_{row\_transform} + S_i \cdot t_{index\_rebuild})$$

其中 $R_i$ 是第 $i$ 个系统表的行数，$S_i$ 是索引大小。

**OID保留策略:**

用户对象OID必须保留，内置对象OID可映射:

$$OID_{user} \xrightarrow{preserve} OID_{new}$$

$$OID_{builtin} \xrightarrow{map} OID_{new}$$

映射表大小估计:

$$Size_{map} = N_{builtin} \cdot (Size_{oid} + Size_{oid}) = N_{builtin} \cdot 8B$$

PostgreSQL页结构保证跨版本兼容:


**兼容性矩阵:**

| 组件 | 17→18兼容 | 处理方式 |
|------|-----------|----------|
| Heap页格式 | ✓ | 直接使用 |
| 索引页格式 | ✓ | 直接使用 |
| TOAST格式 | ✓ | 直接使用 |
| WAL格式 | ✗ | 不复制 |
| 控制文件 | ✗ | 重建 |

**数据复制公式:**

需要复制的数据总量:

$$DataSize_{total} = DataSize_{heap} + DataSize_{index} + DataSize_{toast} + DataSize_{fsm}$$

排除的文件:

$$DataSize_{exclude} = DataSize_{wal} + DataSize_{stat} + DataSize_{tmp}$$

实际复制量:

$$DataSize_{copy} = DataSize_{total} - DataSize_{exclude}$$

## 2. PostgreSQL 18增强

### 2.1 并行数据复制

**并行复制架构:**

```text
┌─────────────────────────────────────────────────────────────────┐
│              PostgreSQL 18 并行复制架构                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────┐                                            │
│  │  pg_upgrade     │                                            │
│  │  (Coordinator)  │                                            │
│  │                 │                                            │
│  │  ┌───────────┐  │    Worker Pool                             │
│  │  │ Scheduler │──┼───►┌────────┬────────┬────────┐           │
│  │  │           │  │    │Worker 1│Worker 2│Worker N│           │
│  │  │ - 分片    │  │    ├──┬────┼──┬────┼──┬────┤           │
│  │  │ - 调度    │  │    │  │    │  │    │  │    │           │
│  │  │ - 合并    │  │    │ Copy │    │ Copy │    │ Copy │           │
│  │  └───────────┘  │    │  │    │  │    │  │    │           │
│  └─────────────────┘    ├──┴────┼──┴────┼──┴────┤           │
│         │               │ 17  │    │ 17  │    │ 17  │           │
│         │               └──┬────┴──┬─────┴──┬────┘           │
│         │                  └───┬───┴────┬───┘                │
│         │                      └────────┘                    │
│    ┌────┴────┐                                              │
│    │ Progress │                                              │
│    │ Tracker  │                                              │
│    └─────────┘                                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**并行度计算公式:**

最优复制Worker数:

$$N_{workers} = \min\left( N_{cpu}, \left\lceil \frac{DataSize}{ChunkSize_{opt}} \right\rceil, N_{max\_workers} \right)$$

最优分片大小考虑I/O模式:

$$ChunkSize_{opt} = \arg\max_{s} \left( \frac{s}{T_{seq}(s) + T_{setup}} \right)$$

对于SSD存储，经验值约为256MB-1GB。

**并行复制时间模型:**

$$T_{parallel\_copy} = T_{setup} + \frac{DataSize}{N_{workers} \cdot Bandwidth} + T_{merge} + T_{sync}$$

与串行复制对比:

$$Speedup = \frac{T_{sequential}}{T_{parallel}} = \frac{DataSize/Bandwidth}{T_{setup} + DataSize/(N_{workers} \cdot Bandwidth) + T_{sync}}$$

当 $DataSize \gg N_{workers} \cdot Bandwidth \cdot (T_{setup} + T_{sync})$ 时:

$$Speedup \approx N_{workers}$$

**动态负载均衡:**

文件分配算法:

```text
算法: File Assignment for Parallel Copy
────────────────────────────────────────
输入: 文件列表F, Worker数N
输出: 分配方案A

1. Sort F by size descending
2. Initialize worker_load[1..N] = 0
3. FOR each file f in F DO
4.     w = argmin(worker_load)
5.     A[f] = w
6.     worker_load[w] += size(f)
7. END FOR
8. RETURN A

时间复杂度: O(|F| log |F| + |F| log N)
```

负载不均衡度:

$$Imbalance = \frac{\max(worker\_load) - \min(worker\_load)}{\sum worker\_load / N}$$

### 2.2 增量升级支持

**增量升级原理:**

```text
┌─────────────────────────────────────────────────────────────────┐
│              增量升级时间线                                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  T0          T1          T2          T3                         │
│  │           │           │           │                          │
│  ▼           ▼           ▼           ▼                          │
│  ┌───┐   ┌───────┐   ┌───────┐   ┌───────┐                     │
│  │Full│   │Delta 1│   │Delta 2│   │Delta 3│                     │
│  │Copy│   │Changes│   │Changes│   │Changes│                     │
│  └───┘   └───┬───┘   └───┬───┘   └───┬───┘                     │
│      │       │           │           │                          │
│      └───────┴───────────┴───────────┘                          │
│              │                                                   │
│              ▼                                                   │
│         ┌─────────┐                                             │
│         │ Final   │                                             │
│         │ Sync    │                                             │
│         └─────────┘                                             │
│                                                                 │
│  T0-T1: 全量复制(后台运行, 17仍提供服务)                        │
│  T1-T3: 增量同步(定期或持续)                                     │
│  T3:    最终同步(短时间停服务)                                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**变更追踪机制:**

使用WAL(Write-Ahead Log)追踪变更:

$$Changes = \{INSERT, UPDATE, DELETE\} \times Tables$$

增量数据量估计:

$$DeltaSize = DataSize_{total} \cdot ChangeRate \cdot TimeWindow$$

其中 $ChangeRate$ 是每秒变更率。

**增量同步算法:**

```text
算法: Incremental Sync
──────────────────────
输入: 源集群S, 目标集群T, 同步窗口W
输出: 同步完成

1. WHILE NOT final_sync DO
2.     /* 获取变更集 */
3.     C = GetChanges(S, last_lsn, W)
4.
5.     /* 应用变更 */
6.     FOR each change c in C DO
7.         IF c.type == INSERT THEN
8.             T.apply_insert(c.data)
9.         ELSE IF c.type == UPDATE THEN
10.            T.apply_update(c.tid, c.data)
11.        ELSE IF c.type == DELETE THEN
12.            T.apply_delete(c.tid)
13.        END IF
14.    END FOR
15.
16.    last_lsn = C.max_lsn
17.    WAIT sync_interval
18. END WHILE
19.
20. /* 最终同步 */
21. T.stop_receiving_connections()
22. FinalSync(S, T)
```

**时间计算:**

全量升级停机时间:

$$T_{downtime\_full} = T_{copy} + T_{catalog} + T_{verify}$$

增量升级停机时间:

$$T_{downtime\_incremental} = T_{final\_sync} + T_{catalog} + T_{verify}$$

其中:

$$T_{final\_sync} = \frac{DeltaSize_{final}}{Bandwidth}$$

停机时间减少比例:

$$Reduction = 1 - \frac{T_{downtime\_incremental}}{T_{downtime\_full}} \approx 1 - \frac{DeltaSize_{final}}{DataSize_{total}}$$

### 2.3 回滚机制

**回滚架构:**

```text
┌─────────────────────────────────────────────────────────────────┐
│              回滚机制架构                                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                    pg_upgrade --rollback                 │   │
│  └─────────────────────────────────────────────────────────┘   │
│                              │                                  │
│           ┌──────────────────┼──────────────────┐               │
│           │                  │                  │               │
│           ▼                  ▼                  ▼               │
│  ┌──────────────┐   ┌──────────────┐   ┌──────────────┐        │
│  │ State Backup │   │ Data Restore │   │ Config Revert│        │
│  │              │   │              │   │              │        │
│  │ - 控制文件   │   │ - 页级恢复   │   │ - 参数文件   │        │
│  │ - 系统目录   │   │ - 一致性检查 │   │ - 权限设置   │        │
│  │ - 事务状态   │   │ - 索引重建   │   │ - 连接配置   │        │
│  └──────────────┘   └──────────────┘   └──────────────┘        │
│                                                                 │
│  回滚触发条件:                                                   │
│  1. 升级验证失败                                                 │
│  2. 性能退化超过阈值                                             │
│  3. 应用兼容性问题                                               │
│  4. 手动触发                                                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**回滚点管理:**

定义回滚点(Recovery Point):

$$RP = (LSN, Timestamp, BackupSet, ConfigSnapshot)$$

回滚窗口:

$$RollbackWindow = [RP_{oldest}, RP_{newest}]$$

最大回滚时间:

$$T_{max\_rollback} = \frac{Storage_{backup}}{Rate_{change}}$$

**回滚时间模型:**

$$T_{rollback} = T_{restore\_control} + T_{restore\_catalog} + T_{verify\_consistency}$$

各分量:

$$T_{restore\_control} = O(1)$$

$$T_{restore\_catalog} = O(N_{catalog\_tables} \cdot \log N_{rows})$$

$$T_{verify\_consistency} = \frac{DataSize}{VerifyBandwidth}$$

**快速回滚优化:**

使用Copy-on-Write机制:

$$T_{fast\_rollback} = T_{metadata\_restore} + T_{cow\_remap}$$

其中:

$$T_{cow\_remap} = O(N_{modified\_pages})$$

## 3. 大版本升级实践

### 3.1 17→18升级步骤

**升级前准备:**

```bash
#!/bin/bash
# pre_upgrade_check.sh - 升级前检查脚本

PG17_HOME="/usr/pgsql-17"
PG18_HOME="/usr/pgsql-18"
OLD_DATA="/var/lib/pgsql/17/data"
NEW_DATA="/var/lib/pgsql/18/data"

echo "=== PostgreSQL 17→18 升级前检查 ==="

# 1. 版本检查
echo "[1/8] 检查PostgreSQL 17安装..."
$PG17_HOME/bin/pg_ctl --version | grep "17\." || exit 1

# 2. 数据库状态检查
echo "[2/8] 检查数据库状态..."
$PG17_HOME/bin/pg_ctl -D $OLD_DATA status

# 3. 无连接检查

    -t -c "SELECT count(*) FROM pg_stat_activity WHERE datname IS NOT NULL;")
if [ "$CONN_COUNT" -gt 1 ]; then
    echo "警告: 存在 $CONN_COUNT 个活跃连接"
    exit 1
fi

    exit 1
fi

echo "[5/8] 检查扩展兼容性..."
INCOMPATIBLE=$($PG17_HOME/bin/psql -U postgres -d postgres -t -c "
    SELECT extname FROM pg_extension
    WHERE extname NOT IN (
        SELECT name FROM pg_available_extensions
        WHERE installed_version IS NOT NULL
    );")
if [ -n "$INCOMPATIBLE" ]; then
    echo "不兼容扩展: $INCOMPATIBLE"
fi

# 6. 数据完整性检查
echo "[6/8] 执行数据校验..."
$PG17_HOME/bin/pg_dumpall -U postgres --schema-only > /tmp/schema_dump.sql

# 7. 配置文件备份
echo "[7/8] 备份配置文件..."
cp $OLD_DATA/postgresql.conf $OLD_DATA/postgresql.conf.backup.$(date +%Y%m%d)
cp $OLD_DATA/pg_hba.conf $OLD_DATA/pg_hba.conf.backup.$(date +%Y%m%d)

# 8. 创建新数据目录
echo "[8/8] 初始化新数据目录..."
mkdir -p $NEW_DATA
$PG18_HOME/bin/initdb -D $NEW_DATA --encoding=UTF8 --locale=en_US.UTF-8

echo "=== 检查完成，可以执行升级 ==="
```

```sql
-- 1. 数据库对象计数验证
WITH object_counts AS (
    SELECT
        'Tables' as object_type,
        COUNT(*) as count
    FROM pg_tables
    WHERE schemaname NOT IN ('pg_catalog', 'information_schema')
    UNION ALL
    SELECT 'Indexes', COUNT(*)
    FROM pg_indexes
    WHERE schemaname NOT IN ('pg_catalog', 'information_schema')
    UNION ALL
    SELECT 'Views', COUNT(*)
    FROM pg_views
    WHERE schemaname NOT IN ('pg_catalog', 'information_schema')
    UNION ALL
    SELECT 'Functions', COUNT(*)
    FROM pg_proc p
    JOIN pg_namespace n ON p.pronamespace = n.oid
    WHERE n.nspname NOT IN ('pg_catalog', 'information_schema')
    UNION ALL
    SELECT 'Triggers', COUNT(*)
    FROM pg_trigger
    WHERE NOT tgisinternal
)
SELECT * FROM object_counts ORDER BY object_type;

-- 2. 数据行数抽样验证
CREATE OR REPLACE FUNCTION verify_row_counts(sample_percent int DEFAULT 10)
RETURNS TABLE (
    schema_name text,
    table_name text,
    estimated_rows bigint,
    exact_rows bigint,
    difference bigint
) AS $$
DECLARE
    rec record;
BEGIN
    FOR rec IN
        SELECT schemaname, tablename
        FROM pg_tables
        WHERE schemaname NOT IN ('pg_catalog', 'information_schema')
        ORDER BY pg_total_relation_size(schemaname||'.'||tablename) DESC
        LIMIT 50
    LOOP
        schema_name := rec.schemaname;
        table_name := rec.tablename;

        EXECUTE format('SELECT reltuples::bigint FROM pg_class WHERE oid = %L::regclass',
            rec.schemaname || '.' || rec.tablename)
        INTO estimated_rows;

        EXECUTE format('SELECT count(*) FROM %I.%I TABLESAMPLE SYSTEM(%s)',
            rec.schemaname, rec.tablename, sample_percent)
        INTO exact_rows;

        exact_rows := exact_rows * (100 / sample_percent);
        difference := ABS(estimated_rows - exact_rows);

        RETURN NEXT;
    END LOOP;
END;
$$ LANGUAGE plpgsql;

SELECT * FROM verify_row_counts(5) WHERE difference > estimated_rows * 0.1;

-- 3. 索引完整性验证
DO $$
DECLARE
    r record;
    cnt bigint;
BEGIN
    FOR r IN
        SELECT indexrelname, tablename, schemaname
        FROM pg_stat_user_indexes
        WHERE indisvalid
    LOOP
        EXECUTE format('SELECT count(*) FROM %I.%I', r.schemaname, r.tablename) INTO cnt;
        RAISE NOTICE 'Index % on %.%: table has % rows',
            r.indexrelname, r.schemaname, r.tablename, cnt;
    END LOOP;
END $$;

-- 4. 扩展版本验证
SELECT
    extname,
    extversion as current_version,

    CASE WHEN extversion = (SELECT default_version FROM pg_available_extensions WHERE name = extname)
         THEN 'UP TO DATE'
         ELSE 'UPDATE AVAILABLE'
    END as status
FROM pg_extension
ORDER BY extname;
```

```bash
echo "执行升级前检查..."
pg_upgrade \
    --old-bindir=/usr/lib/postgresql/17/bin \
    --new-bindir=/usr/lib/postgresql/18/bin \
    --old-datadir=$OLD_DATA \
    --new-datadir=$NEW_DATA \
    --check

# 执行升级
echo "开始升级..."
pg_upgrade \
    --old-bindir=/usr/lib/postgresql/17/bin \
    --new-bindir=/usr/lib/postgresql/18/bin \
    --old-datadir=$OLD_DATA \
    --new-datadir=$NEW_DATA \
    --jobs=${PARALLEL_WORKERS:-4} \
    --link

echo "升级完成"
```

## 5. 总结与展望

### 5.1 PostgreSQL 18升级特性总结

| 特性 | 改进点 | 适用场景 |
|------|--------|----------|
| 并行数据复制 | 多Worker并行复制 | 大型数据库(>100GB) |
| 增量升级 | 支持持续同步 | 7×24业务系统 |
| 回滚机制 | 快速回滚到旧版本 | 风险规避 |
| 增强CLI | 更多参数和进度监控 | 自动化运维 |
| 容器化支持 | K8s Operator | 云原生环境 |

### 5.2 升级策略决策矩阵

```
                    数据量
           ┌─────────────────────────┐
           │  <100GB  │  100GB-1TB  │  >1TB
    ───────┼──────────┼─────────────┼─────────
    停机容 │ 原地升级 │  原地+并行  │ 增量升级
    忍时间 │ (分钟级) │  (分钟级)   │ (秒级)
    ───────┼──────────┼─────────────┼─────────
    高可用 │ 原地升级 │  逻辑复制   │ 增量升级
    要求   │ +预热    │  +切换      │ +蓝绿部署
    ───────┼──────────┼─────────────┼─────────
    跨架构 │ 逻辑升级 │  逻辑升级   │ 逻辑升级
           │ (pg_dump)│  (pg_dump)  │ (流式)
    ───────┴──────────┴─────────────┴─────────
```

### 5.3 未来发展方向

1. **零停机升级**: 基于逻辑复制的在线热升级
2. **智能回滚**: AI辅助的升级风险评估和自动回滚
3. **多云支持**: 跨云平台的混合升级方案
4. **细粒度升级**: 按表/按库的增量升级

## 附录A: 公式汇总

| 编号 | 公式 | 描述 |
|------|------|------|
| 1 | $T_{in-place} = T_{copy} + T_{catalog} + T_{verify}$ | 原地升级总时间 |
| 2 | $T_{copy} = \frac{DataSize}{Bandwidth_{effective}} + Latency_{metadata}$ | 数据复制时间 |
| 3 | $T_{logical} = T_{schema\_dump} + T_{data\_dump} + T_{transfer} + T_{restore}$ | 逻辑升级时间 |
| 4 | $T_{catalog} = O(N_{system\_tables} \cdot \log N_{rows})$ | 目录升级复杂度 |
| 5 | $N_{workers} = \min(N_{cpu}, \lceil \frac{DataSize}{ChunkSize_{opt}} \rceil, N_{max\_workers})$ | 最优Worker数 |
| 6 | $T_{parallel\_copy} = T_{setup} + \frac{DataSize}{N_{workers} \cdot Bandwidth} + T_{merge} + T_{sync}$ | 并行复制时间 |
| 7 | $Speedup = \frac{T_{sequential}}{T_{parallel}}$ | 加速比 |
| 8 | $T_{downtime\_incremental} = T_{final\_sync} + T_{catalog} + T_{verify}$ | 增量升级停机时间 |
| 9 | $Reduction = 1 - \frac{DeltaSize_{final}}{DataSize_{total}}$ | 停机时间减少比例 |
| 10 | $T_{rollback} = T_{restore\_control} + T_{restore\_catalog} + T_{verify\_consistency}$ | 回滚时间 |
| 11 | $TCO = \alpha \cdot T_{downtime} + \beta \cdot StorageCost + \gamma \cdot Risk$ | 总拥有成本 |
| 12 | $DataSize_{copy} = DataSize_{total} - DataSize_{exclude}$ | 实际复制量 |
