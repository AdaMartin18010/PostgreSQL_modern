## 1. Skip Scan理论基础

### 1.1 问题背景

**传统索引限制**:
给定复合索引 $I = (A_1, A_2, ..., A_n)$，传统优化器只能在前导列有等值条件时使用索引:

$$
\text{Indexable}_{trad} := \exists k \ge 1: \bigwedge_{i=1}^{k} (A_i \text{ has equality condition})
$$

**实际问题**: 生产环境中大量查询只涉及非前导列:

```sql
-- 索引: (user_id, status, created_at)
-- 查询: 查询所有pending状态的订单
SELECT * FROM orders WHERE status = 'pending';
-- 传统: 被迫使用Seq Scan
```

### 1.2 Skip Scan的核心思想

**定义 1.1 (Skip Scan)**:
Skip Scan是一种索引扫描策略，允许在跳过前导列不同值的同时，在后续列上执行条件搜索。

**形式化描述**:

$$
\text{SkipScan}(Q, I) := \exists k > 1: (A_1 \text{ unconstrained}) \land \bigwedge_{i=k}^{n} (A_i \text{ has condition})
$$

**定理 1.1 (Skip Scan存在性)**: 对于复合索引和跳过前导列的查询，Skip Scan的时间复杂度严格优于顺序扫描。

*证明*:

- 顺序扫描: $O(N)$，N为表大小
- Skip Scan: $O(M \cdot (\log N + K))$，M为不同前缀数，K为每前缀匹配数
- 当 $M \cdot K \ll N$ 时，Skip Scan显著更优 ∎

## 2. 算法形式化

### 2.1 算法定义

**定义 2.1 (Skip Scan算法)**:

输入:

- 复合索引 $I$ 在属性 $(A_1, A_2, ..., A_n)$ 上
- 查询条件 $\sigma_{\theta}$，其中 $\theta$ 约束 $A_k, ..., A_n$ ($k > 1$)

输出:

- 满足条件的元组集合 $R$

**算法步骤**:

```text
Algorithm SkipScan(Index I, Condition θ):
    R ← ∅
    prefix ← NULL

    // 扫描索引的B+树叶子节点
    for each leaf page P in I:
        for each entry e in P:
            // 检测前缀变化
            current_prefix ← e.A₁

            if current_prefix ≠ prefix:
                prefix ← current_prefix

                // 在当前前缀内搜索满足条件的项
                // 使用B+树搜索定位
                matches ← SearchInPrefix(I, prefix, θ)

                for each match m in matches:
                    R ← R ∪ {m.tuple}

                // 跳跃到下一个前缀
                next_pos ← FindNextPrefix(I, prefix)
                if next_pos exists:
                    SeekTo(I, next_pos)

    return R
```

### 2.2 关键操作形式化

**跳跃操作**:

$$
\text{SkipToNext}(I, p) := \min \{ e \in I \mid e.A_1 > p \}
$$

**前缀内搜索**:

$$
\text{SearchInPrefix}(I, p, \theta) := \{ e \in I \mid e.A_1 = p \land \theta(e.A_k, ..., e.A_n) \}
$$

### 2.3 复杂度分析

**定理 2.1 (Skip Scan复杂度)**:

设:

- $N$: 表中总记录数
- $M$: 不同前导列值的数量
- $H$: B+树高度
- $K$: 每个前缀内满足条件的平均记录数

则Skip Scan的时间复杂度为:

$$
T_{skip} = O(M \cdot (H + \log(N/M) + K))
$$

*证明*:

1. 每个不同前缀需要一次跳跃: $M$ 次
2. 每次跳跃代价: $O(H)$ (从根到叶)
3. 前缀内搜索: $O(\log(N/M))$ (二分查找)
4. 返回匹配记录: $O(K)$
5. 总计: $M \cdot (H + \log(N/M) + K)$ ∎

**定理 2.2 (最优条件)**: 当 $M \cdot K \le N \cdot \alpha$ (其中 $\alpha < 1$) 时，Skip Scan优于顺序扫描。

### 2.4 与传统扫描对比

| 扫描方式 | 时间复杂度 | 空间复杂度 | 适用条件 |
|----------|-----------|-----------|----------|
| **顺序扫描** | $O(N)$ | $O(1)$ | 无索引或低选择性 |
| **索引扫描** | $O(H + N_{match})$ | $O(1)$ | 前导列有约束 |
| **位图扫描** | $O(N_{index} + N_{heap})$ | $O(N_{pages})$ | 中等选择性 |
| **Skip Scan** | $O(M \cdot (H + K))$ | $O(1)$ | 前导列无约束，后续列高选择性 |

## 3. 代价模型

### 3.1 代价组成

**定义 3.1 (Skip Scan代价)**:

$$
C_{skip} = C_{seek} + C_{scan} + C_{filter} + C_{fetch}
$$

其中:

- $C_{seek}$: 跳跃定位代价
- $C_{scan}$: 前缀内扫描代价
- $C_{filter}$: 条件评估代价
- $C_{fetch}$: 元组获取代价

### 3.2 各项代价计算

**跳跃定位代价**:

$$
C_{seek} = M \cdot H \cdot c_{random}
$$

其中 $c_{random}$ 是随机页读取代价。

**前缀内扫描代价**:

$$
C_{scan} = M \cdot N_{prefix} \cdot sel \cdot c_{seq}
$$

其中 $N_{prefix} = N/M$ 是每个前缀的平均记录数。

**总代价公式**:

$$
C_{skip} = M \cdot H \cdot c_{random} + M \cdot sel \cdot \frac{N}{M} \cdot c_{seq} + N_{match} \cdot c_{cpu}
$$

简化:

$$
C_{skip} = M \cdot H \cdot c_{random} + sel \cdot N \cdot c_{seq} + N_{match} \cdot c_{cpu}
$$

### 3.3 选择性估计

**定义 3.2 (Skip Scan选择性)**:

对于条件 $\theta$ 作用于 $A_k, ..., A_n$:

$$
sel_{skip}(\theta) = \prod_{i=k}^{n} sel(A_i)
$$

在独立性假设下。

**定理 3.1 (选择性估计误差)**: 当后续列相关时，独立性假设的选择性估计误差可达 $\prod_{i=k}^{n} \frac{1}{sel(A_i)}$ 倍。

### 4.1 架构集成

**优化器路径生成** (src/backend/optimizer/path/indxpath.c):

```c
/*
 * create_index_paths
 * 为给定索引创建所有可能的访问路径
 */
void
create_index_paths(PlannerInfo *root, RelOptInfo *rel)
{
    ListCell   *lc;

    foreach(lc, rel->indexlist)
    {
        IndexOptInfo *index = (IndexOptInfo *) lfirst(lc);

        /* 1. 尝试普通索引扫描 */
        create_index_scan_path(root, rel, index);

        /* 2. 尝试位图索引扫描 */
        create_bitmap_paths(root, rel, index);

        /* 3. 尝试Skip Scan (PG 18新增) */
        if (enable_skip_scan && index->ncolumns > 1)
        {
            create_skip_scan_path(root, rel, index);
        }
    }
}
```

**Skip Scan路径创建**:

```c
/*
 * create_skip_scan_path
 * 创建Skip Scan访问路径
 */
static void
create_skip_scan_path(PlannerInfo *root, RelOptInfo *rel,
                      IndexOptInfo *index)
{
    SkipScanPath *spath = makeNode(SkipScanPath);

    /* 确定可跳过的前导列 */
    int skip_columns = 0;
    for (int i = 0; i < index->ncolumns; i++)
    {
        if (!index->indexkeys[i] has condition)
            skip_columns++;
        else
            break;
    }

    if (skip_columns == 0 || skip_columns >= index->ncolumns)
        return;  /* 不需要或无法Skip Scan */

    /* 计算不同前缀数 */
    spath->ndistinct_prefixes = estimate_distinct_prefixes(
        index, skip_columns);

    /* 计算Skip Scan代价 */
    cost_skip_scan(spath, root, rel, index);

    /* 如果代价最优，添加到路径列表 */
    add_path(rel, (Path *) spath);
}
```

### 4.2 执行引擎实现

**Skip Scan状态** (src/backend/executor/nodeIndexskipscan.c):

```c
typedef struct SkipScanState
{
    IndexScanState ss;          /* 基础索引扫描状态 */

    /* Skip Scan特有 */
    int         skipNkeys;      /* 跳过的列数 */
    ScanKey     skipKeys;       /* 跳过列的键 */

    Datum       current_prefix; /* 当前前缀值 */
    bool        prefix_valid;   /* 前缀是否有效 */

    /* 跳跃相关 */
    int         nskips;         /* 已执行跳跃次数 */
    int         max_skips;      /* 最大跳跃次数(限制) */

    /* 统计 */
    double      nprefixes;      /* 遇到的不同前缀数 */
    double      ntuples_per_prefix; /* 每前缀平均元组数 */
} SkipScanState;

/*
 * ExecSkipScan
 * Skip Scan主执行函数
 */
static TupleTableSlot *
ExecSkipScan(IndexScanState *node)
{
    SkipScanState *sstate = (SkipScanState *) node;

    if (!sstate->prefix_valid)
    {
        /* 定位到下一个前缀 */
        if (!SkipToNextPrefix(sstate))
            return NULL;  /* 扫描完成 */

        /* 在当前前缀内搜索 */
        BeginPrefixSearch(sstate);
    }

    /* 获取当前前缀内的匹配项 */
    TupleTableSlot *slot = GetNextMatchInPrefix(sstate);

    if (TupIsNull(slot))
    {
        /* 当前前缀搜索完成，标记需要跳跃 */
        sstate->prefix_valid = false;

        /* 递归调用以处理下一个前缀 */
        return ExecSkipScan(node);
    }

    return slot;
}

/*
 * SkipToNextPrefix
 * 跳跃到下一个不同的前导列值
 */
static bool
SkipToNextPrefix(SkipScanState *sstate)
{
    IndexScanDesc scandesc = sstate->ss.ss_ScanDesc;

    /* 如果已有前缀值，在其后搜索 */
    if (sstate->prefix_valid)
    {
        /* 设置扫描键 > 当前前缀 */
        ScanKey sk = &sstate->skipKeys[0];
        sk->sk_argument = sstate->current_prefix;
        sk->sk_strategy = BTGreaterStrategyNumber;

        /* 重新定位索引扫描 */
        index_rescan(scandesc, sstate->skipKeys, 1, NULL, 0);
    }

    /* 获取下一个不同的前缀值 */
    Datum next_prefix;
    if (!index_getnext_prefix(scandesc, &next_prefix))
        return false;  /* 无更多前缀 */

    sstate->current_prefix = next_prefix;
    sstate->prefix_valid = true;
    sstate->nskips++;

    return true;
}
```

### 4.3 代价计算

**代价函数** (src/backend/optimizer/path/costsize.c):

```c
/*
 * cost_skip_scan
 * 计算Skip Scan代价
 */
void
cost_skip_scan(SkipScanPath *path, PlannerInfo *root,
               RelOptInfo *baserel, IndexOptInfo *index)
{
    Cost        startup_cost = 0;
    Cost        run_cost = 0;
    double      num_prefixes;
    double      tuples_per_prefix;
    Selectivity selectivity;

    /* 获取不同前缀数估计 */
    num_prefixes = path->ndistinct_prefixes;

    /* 计算每前缀元组数 */
    tuples_per_prefix = index->rel->tuples / num_prefixes;

    /* 跳跃代价: 每个前缀需要一次索引定位 */
    run_cost += num_prefixes * index->tree_height * random_page_cost;

    /* 前缀内扫描代价 */
    selectivity = clauselist_selectivity(root, path->indexclauses,
                                         baserel->relid);
    run_cost += num_prefixes * tuples_per_prefix * selectivity * seq_page_cost;

    /* CPU代价 */
    run_cost += path->path.rows * cpu_tuple_cost;

    path->path.startup_cost = startup_cost;
    path->path.total_cost = startup_cost + run_cost;
}
```

## 5. 适用场景与优化

### 5.1 决策矩阵

**定理 5.1 (最优场景)**:
Skip Scan在以下条件下性能最优:

1. **高基数前导列**: $M > 100$
2. **高选择性后续列**: $sel < 0.01$
3. **均匀分布**: $\sigma(N_{prefix}) < \mu(N_{prefix}) \cdot 0.5$
4. **大表**: $N > 100000$

### 5.2 场景分析

## 6. 思维表征

### 6.1 算法执行流程

```
复合索引: (user_id, status, created_at)
查询: WHERE status = 'pending'

索引数据布局:
user_id=1, status=active, created_at=...
user_id=1, status=completed, created_at=...
user_id=1, status=pending, created_at=...    ←┐ 第一次跳跃
                                              │ 前缀=user_id=1
user_id=2, status=active, created_at=...      │ 匹配1条
user_id=2, status=pending, created_at=...    ←┤
                                              │ 第二次跳跃
user_id=3, status=pending, created_at=...    ←┤ 前缀=user_id=2
                                              │ 匹配1条
...                                           │
                                              │ 第三次跳跃
user_id=N, status=pending, created_at=...    ←┘ 前缀=user_id=3
                                                 匹配1条

执行步骤:
1. 跳跃到user_id=1，搜索status='pending' → 找到1条
2. 跳跃到user_id=2，搜索status='pending' → 找到1条
3. 跳跃到user_id=3，搜索status='pending' → 找到1条
...
N. 直到遍历所有user_id值
```

### 6.2 B+树跳跃可视化

```
B+树结构 (user_id, status):

                    [10 | 20 | 30]
                   /    |    |    \
                  /     |     |     \
             [1|5|9] [11|15|19] [21|25|29] [31|35|39]
             / | | \   / | | \   / | | \    / | | \
            1  5  9   11 15 19  21 25 29   31 35 39
            ↓  ↓  ↓   ↓  ↓  ↓   ↓  ↓  ↓    ↓  ↓  ↓
           A1  C  P   A  P  C   P  A  C    P  A  C

A=active, C=completed, P=pending

查询: WHERE status='pending'

执行路径:
1. 根节点 → 找到第一个叶子 [1|5|9]
2. 在 [1|5|9] 中找到 status='pending' → 9号叶子的P
3. 跳跃: 从9直接到11 (通过叶子节点指针)
4. 在 [11|15|19] 中找到 status='pending' → 15号叶子的P
5. 跳跃到 [21|25|29] → 找到21号叶子的P
6. 跳跃到 [31|35|39] → 找到31号叶子的P
7. 结束
```

```
       │               │

       └── 评估代价模型
```


**反例2: 极高基数导致过度跳跃**

```sql
-- 表: user_sessions
-- 索引: (session_id, event_type)
-- session_id几乎唯一(100M不同值)
-- 查询: WHERE event_type = 'click'

-- 问题: 需要跳跃100M次!
-- 每次跳跃O(log N) = 4次I/O
-- 总I/O: 400M次页面访问
-- 结果: 比Seq Scan慢100倍

-- 解决方案: 使用BRIN索引或分区
CREATE INDEX idx_event_type ON user_sessions(event_type);
-- 或按event_type分区
```

**反例3: 忽略统计信息**

```sql
-- 表数据分布发生大变化后
INSERT INTO orders SELECT * FROM orders_backup;  -- 翻倍数据
-- 未执行ANALYZE

-- 查询使用Skip Scan但性能极差
EXPLAIN (ANALYZE) SELECT * FROM orders WHERE status = 'pending';
-- 估计: 跳跃10K次，匹配5K条
-- 实际: 跳跃100K次，匹配500K条
-- 原因: 统计信息过时

-- 解决方案
ANALYZE orders;
-- 或设置自动分析
ALTER TABLE orders SET (autovacuum_analyze_threshold = 10000);
```

**反例4: 前缀内大量匹配**

```sql
-- 表: notifications
-- 索引: (user_id, is_read)
-- 查询: 获取所有未读通知 WHERE is_read = false

-- 问题: 每个用户平均有50条未读通知
-- 10M用户 × 50条 = 500M条匹配
-- Skip Scan需要访问几乎所有索引页

-- 优化器正确选择: Seq Scan
-- 人工强制Skip Scan会导致灾难
```

## 8. 形式化验证

### 8.1 正确性证明

**定理 8.1 (Skip Scan完备性)**: Skip Scan返回所有满足查询条件的元组。

*证明*:

1. 设索引I按$(A_1, A_2, ..., A_n)$排序
2. 具有相同$A_1$值的记录在索引中连续存储
3. Skip Scan遍历所有不同的$A_1$值（通过跳跃）
4. 对于每个$A_1$值，算法搜索满足$\theta(A_k, ..., A_n)$的记录
5. 由于扫描覆盖所有$A_1$值，且每个前缀内搜索完整，无遗漏 ∎

**定理 8.2 (Skip Scan正确性)**: Skip Scan只返回满足查询条件的元组。

*证明*:

1. 算法仅在确认记录满足$\theta$时才返回
2. 前缀匹配确保$A_1$值正确
3. 条件检查确保$\theta$满足
4. 因此返回的每条记录都满足完整条件 ∎

### 8.2 复杂度下界

**定理 8.3 (下界)**: 对于跳过k个前导列的Skip Scan，时间复杂度下界为 $\Omega(M \cdot H)$。

*证明*:

- 必须访问至少$M$个不同的前缀
- 每个前缀定位至少需要$H$次I/O（B+树高度）
- 因此 $\Omega(M \cdot H)$ ∎

## 9. 监控与诊断

### 9.1 监控视图

```sql
-- Skip Scan使用统计
SELECT
    schemaname,
    tablename,
    indexrelname,
    idx_scan as total_scans,
    idx_tup_read as tuples_read,
    idx_tup_fetch as tuples_fetched
FROM pg_stat_user_indexes
WHERE indexrelname LIKE '%skip%'
ORDER BY idx_scan DESC;

EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON)
```
