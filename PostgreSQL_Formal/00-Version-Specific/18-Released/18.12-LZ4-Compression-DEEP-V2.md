## 1. 压缩算法理论基础

### 1.1 LZ4算法原理

#### 1.1.1 核心算法机制

LZ4属于**LZ77系列**字典压缩算法，其核心基于**滑动窗口匹配**机制：

**滑动窗口模型：**

$$
W = [W_{-L}, W_{-L+1}, ..., W_{-1}] \quad \text{窗口大小} \quad |W| = L
$$

压缩过程将输入数据流 $S = [s_1, s_2, ..., s_n]$ 转换为压缩流 $C$：

$$
C = \{(l_1, d_1, c_1), (l_2, d_2, c_2), ..., (l_m, d_m, c_m)\}
$$

其中每个三元组表示：

- $l_i$: 匹配长度 (Length)
- $d_i$: 偏移距离 (Distance)
- $c_i$: 后续字面量 (Literal)

**LZ4压缩格式结构：**

```text
┌─────────────────────────────────────────────────────────────┐
│                    LZ4 Block Format                         │
├─────────────────────────────────────────────────────────────┤
│ Token (1B) │ [Literal Length] │ Literals │ [Match Length] │
│            │   (0-4B varint)  │   (0-N)  │  (0-4B varint) │
├────────────┴──────────────────┴──────────┴────────────────┤
│                                                             │
│  Token格式:                                                 │
│  ┌─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┐        │
│  │ 0-3 │  4  │  5  │  6  │  7  │ 0-3 │  4  │  5  │        │
│  │ 字面量长度高位 │  字面量长度低位  │ 匹配长度高位 │        │
│  └────────────────┴──────────────────┴────────────────┘        │
│                                                             │
│  匹配距离: 2字节小端序                                       │
└─────────────────────────────────────────────────────────────┘
```

#### 1.1.2 压缩过程详解

**算法伪代码表示：**

```text
函数 LZ4_Compress(input, output):
    ip = input起始位置
    anchor = ip

    当 ip < input末尾 - MFLIMIT 时:
        // 1. 哈希计算与查找
        forwardIp = ip
        循环:
            seq = forwardIp读取4字节
            hash = Hash(seq)
            ref = hashTable[hash]  // 前向引用
            hashTable[hash] = forwardIp

            如果 forwardIp - ref <= MAX_DISTANCE 且 4字节匹配:
                跳出循环

            forwardIp += 步进值
            如果 forwardIp > ipLimit: 返回

        // 2. 编码匹配
        matchLength = 4 + 扩展匹配长度(forwardIp, ref)
        literalLength = forwardIp - anchor

        // 3. 写入Token和字面量
        写入Token(literalLength, matchLength - 4)
        写入字面量(anchor, literalLength)
        写入偏移量(forwardIp - ref)
        写入匹配长度扩展(matchLength)

        // 4. 更新状态
        ip = forwardIp + matchLength
        anchor = ip
```

**哈希函数：**

$$
hash(x) = (x \times 2654435761) \gg (32 - HASH\_BITS) \mod HASH\_SIZE
$$

标准配置：

- $HASH\_BITS = 12$ (4KB哈希表)
- $HASH\_SIZE = 2^{12} = 4096$

**压缩比估算公式：**

$$
CR = \frac{InputSize}{OutputSize} = \frac{N}{\sum_{i=1}^{k}(1 + ll_i + ml_i + 2 + L_i)}
$$

其中：

- $ll_i$: 第$i$个块的字面量长度
- $ml_i$: 第$i$个块的匹配长度扩展字节
- $L_i$: 第$i$个块的字面量数据

#### 1.1.3 解压过程

LZ4解压是**确定性的、无条件的**，这是其高性能的关键：

$$
Decompress(C) = \bigoplus_{i=1}^{m} Expand(l_i, d_i, c_i)
$$

其中 $\oplus$ 表示字节拼接操作。

**解压速度优势来源：**

1. **无分支预测**：解压流程固定，无需条件判断
2. **内存顺序访问**：完全线性读取
3. **向量化操作**：可使用SIMD加速

解压速度公式：

$$
V_{decompress} = \frac{MemoryBandwidth \times CompressionRatio}{CompressionRatio - 1 + \alpha}
$$

其中 $\alpha$ 是解压指令开销系数，通常接近0。

### 1.2 压缩比vs速度权衡

#### 1.2.1 理论权衡模型

**时间-压缩比权衡曲线：**

$$
CR(t) = CR_{max} \cdot \left(1 - e^{-\lambda t}\right)
$$

其中：

- $CR_{max}$: 最大可能压缩比
- $\lambda$: 压缩效率系数
- $t$: 压缩时间

**Ziv-Lempel压缩理论极限：**

$$
H(X) \leq \frac{CompressedSize}{OriginalSize} \leq H(X) + \epsilon
$$

其中 $H(X)$ 是信源熵，$\epsilon$ 是冗余度。

#### 1.2.2 LZ4压缩级别分析

PostgreSQL 18支持LZ4多种加速级别：

| 级别 | 哈希链深度 | 搜索范围 | 压缩比 | 速度(MB/s) | 适用场景 |
|------|-----------|----------|--------|------------|----------|
| 1 | 1 | 4KB | 1.5-2.0x | 780 | 实时写入 |
| 3 | 4 | 16KB | 1.8-2.3x | 450 | 常规OLTP |
| 9 | 256 | 64KB | 2.2-2.8x | 120 | 归档存储 |
| 12 | 4096 | 256KB | 2.5-3.2x | 35 | 冷数据 |

**最优级别选择公式：**

$$
L_{opt} = \arg\min_{l} \left( \frac{DataSize}{Speed(l)} + IOWait(CompressionRatio(l)) \right)
$$

#### 1.2.3 数据特征对压缩的影响

**可压缩性熵模型：**

$$C(d) = \frac{H_{max} - H(d)}{H_{max}} \times 100\%$$

其中：

- $H(d) = -\sum_{i} p_i \log_2 p_i$: 数据熵
- $H_{max} = \log_2 N$: 最大熵

**不同数据类型的LZ4压缩比：**

```text
┌────────────────────────────────────────────────────────────┐
│  数据类型          │ 典型熵值   │ LZ4压缩比  │ 备注        │
├────────────────────────────────────────────────────────────┤
│  JSON文本          │ 4.2 bits  │ 2.0-2.5x  │ 高冗余      │
│  CSV数据           │ 3.8 bits  │ 2.2-2.8x  │ 结构化      │
│  日志文本          │ 3.5 bits  │ 2.5-3.0x  │ 重复模式    │
│  XML数据           │ 3.2 bits  │ 2.8-3.5x  │ 标签重复    │
│  重复字符串        │ 1.0 bits  │ 5.0-10x   │ 最佳情况    │
│  随机二进制        │ 7.9 bits  │ 1.0-1.01x │ 接近不可压  │
│  加密数据          │ 7.99 bits │ 1.0x      │ 完全不可压  │
│  浮点数组          │ 6.5 bits  │ 1.2-1.5x  │ 低位变化    │
└────────────────────────────────────────────────────────────┘
```

### 1.3 与PGLZ/ZSTD对比

#### 1.3.1 算法特性对比

| 特性 | PGLZ | LZ4 | ZSTD |
|------|------|-----|------|
| 算法类型 | LZ77变种 | LZ4 | LZ77+FSE |

| 解压速度 | 100-300 MB/s | 1.5-4 GB/s | 300-800 MB/s |
| 压缩比 | 2.0-3.5x | 1.5-2.5x | 2.5-5.0x |
| 内存使用 | 高 | 极低 | 中等 |
| 字典大小 | 1MB | 64KB | 可变 |
| 流式支持 | 是 | 是 | 是 |
| 线程安全 | 否 | 是 | 是 |

#### 1.3.2 性能权衡公式

**综合性能评分：**

$$
Score = w_1 \cdot \frac{V_{compress}}{V_{ref}} + w_2 \cdot \frac{V_{decompress}}{V_{ref}} + w_3 \cdot CR
$$

其中 $w_1 + w_2 + w_3 = 1$。

**典型权重场景：**

| 场景 | $w_1$ | $w_2$ | $w_3$ | 推荐算法 |
|------|-------|-------|-------|----------|
| 高频写入OLTP | 0.5 | 0.3 | 0.2 | LZ4 |
| 分析型查询 | 0.1 | 0.4 | 0.5 | ZSTD |
| 归档存储 | 0.1 | 0.2 | 0.7 | ZSTD |
| 实时复制 | 0.6 | 0.2 | 0.2 | LZ4 |

#### 2.1.2 LZ4 TOAST压缩流程

**压缩触发条件：**

$$
\text{Compress}(d) = \begin{cases}
true & \text{if } |d| > TOAST\_TUPLE\_THRESHOLD \\
false & \text{otherwise}
\end{cases}
$$

默认 $TOAST\_TUPLE\_THRESHOLD = 2048$ 字节

**压缩阈值决策：**

```c
/* PostgreSQL LZ4 TOAST压缩决策逻辑 */
bool
should_compress_lz4(Datum value, int32 typemod)
{
    int32 raw_size = VARSIZE_ANY_EXHDR(value);

    /* 小数据不压缩 */
    if (raw_size < PGLZ_strategy_default.min_input_size)
        return false;

    /* 大数据强制压缩 */
    if (raw_size > TOAST_TUPLE_THRESHOLD)
        return true;

    /* 中等数据根据类型决策 */
    if (typemod > 0 && raw_size > typemod / 2)
        return true;

    return false;
}
```

**LZ4压缩级别配置：**

```sql
-- 设置默认TOAST压缩算法为LZ4
ALTER SYSTEM SET default_toast_compression = 'lz4';

-- 设置LZ4压缩级别(1-12)
ALTER SYSTEM SET lz4.default_compression_level = 3;

-- 表级覆盖
ALTER TABLE large_documents
    ALTER COLUMN content SET COMPRESSION lz4;

-- 列级压缩级别
ALTER TABLE logs
    ALTER COLUMN message SET (compression = lz4,
                              compression_level = 1);
```

#### 2.1.3 压缩数据格式

**LZ4 TOAST数据头格式：**

```text
┌─────────────────────────────────────────────────────────────┐
│              LZ4 Compressed TOAST Chunk Header              │
├─────────────────────────────────────────────────────────────┤
│  偏移   │  长度   │  字段                  │  说明          │
├─────────┼─────────┼────────────────────────┼────────────────┤
│  0      │  4字节  │  raw_size              │  原始数据大小   │
│  4      │  4字节  │  compressed_size       │  压缩后大小     │
│  8      │  1字节  │  compression_method    │  压缩方法标识   │
│  9      │  1字节  │  compression_level     │  压缩级别       │
│  10     │  2字节  │  checksum              │  CRC32校验和    │
│  12     │  N字节  │  compressed_payload    │  LZ4压缩数据    │
└─────────────────────────────────────────────────────────────┘
```

**压缩有效性验证公式：**

$$
Valid = CRC32(compressed\_payload) == checksum \land
        lz4\_decompress(compressed) == raw\_size
$$

#### 2.3.3 WAL压缩性能公式

**WAL写入性能模型：**

$$
T_{wal\_write} = T_{generate} + T_{compress} + T_{fsync}
$$

**压缩带来的I/O节省：**

$$
I\!O_{saved} = (1 - \frac{1}{CR}) \times WAL\_Volume
$$

其中 $CR$ 是压缩比。

**WAL压缩开销评估：**

$$
Overhead_{compression} = \frac{T_{compress}}{T_{wal\_write\_uncompressed}} \times 100\%
$$

典型值：LZ4压缩开销 < 5%

## 3. 性能分析与评估

### 3.1 CPU使用率分析

#### 3.1.1 压缩CPU开销模型

**LZ4压缩CPU周期估算：**

$$
Cycles_{compress} = \alpha \cdot N + \beta \cdot Matches + \gamma
$$

其中：

- $N$: 输入数据字节数
- $Matches$: 找到的匹配数
- $\alpha, \beta, \gamma$: 架构相关常数

**每字节CPU周期对比：**

| 算法 | 压缩 cycles/byte | 解压 cycles/byte | 比例 |
|------|-----------------|-----------------|------|
| 无压缩 | 0 | 0 | - |
| PGLZ | 15-25 | 3-5 | 5:1 |
| LZ4 | 2-4 | 0.3-0.5 | 8:1 |
| ZSTD(3) | 8-15 | 1-2 | 10:1 |
| ZSTD(9) | 50-100 | 1-2 | 50:1 |

**多线程压缩扩展：**

$$
Speedup_{parallel} = \frac{N}{\frac{N}{P} + O_{sync}}
$$

其中：

- $P$: 线程数
- $O_{sync}$: 同步开销

#### 3.1.2 CPU基准测试

```sql
-- LZ4压缩性能基准测试
CREATE OR REPLACE FUNCTION benchmark_lz4_compression()
RETURNS TABLE (
    data_size_kb INT,
    compression_level INT,
    compress_time_ms NUMERIC,
    decompress_time_ms NUMERIC,
    compressed_size_kb INT,
    compression_ratio NUMERIC,
    throughput_mbps NUMERIC
) AS $$
DECLARE
    v_data TEXT;
    v_compressed BYTEA;
    v_decompressed TEXT;
    v_start TIMESTAMP;
    v_compress_time INTERVAL;
    v_decompress_time INTERVAL;
    v_size INT;
    v_level INT;
BEGIN
    FOREACH v_size IN ARRAY ARRAY[1, 10, 100, 1000, 10000]
    LOOP
        -- 生成测试数据
        v_data := (SELECT string_agg(
            CASE (random() * 4)::INT
                WHEN 0 THEN 'user:' || (random() * 100000)::INT || ','
                WHEN 1 THEN 'action:click,'
                WHEN 2 THEN 'timestamp:' || EXTRACT(EPOCH FROM NOW()) || ','
                ELSE 'data:' || md5(random()::TEXT) || ','
            END, ''
        ) FROM generate_series(1, v_size * 100));

        FOREACH v_level IN ARRAY ARRAY[1, 3, 9, 12]
        LOOP
            -- 压缩测试
            v_start := clock_timestamp();
            v_compressed := lz4_compress(v_data::BYTEA, v_level);
            v_compress_time := clock_timestamp() - v_start;

            -- 解压测试
            v_start := clock_timestamp();
            v_decompressed := convert_from(lz4_decompress(v_compressed), 'UTF8');
            v_decompress_time := clock_timestamp() - v_start;

            -- 返回结果
            data_size_kb := v_size;
            compression_level := v_level;
            compress_time_ms := EXTRACT(EPOCH FROM v_compress_time) * 1000;
            decompress_time_ms := EXTRACT(EPOCH FROM v_decompress_time) * 1000;
            compressed_size_kb := LENGTH(v_compressed) / 1024;
            compression_ratio := ROUND(LENGTH(v_data)::NUMERIC / LENGTH(v_compressed), 2);
            throughput_mbps := ROUND(
                (v_size::NUMERIC / 1024) / NULLIF(EXTRACT(EPOCH FROM v_compress_time), 0),
                2
            );

            RETURN NEXT;
        END LOOP;
    END LOOP;
END;
$$ LANGUAGE plpgsql;
```

### 3.2 I/O节省计算

#### 3.2.1 I/O节省模型

**存储空间节省：**

$$
Savings_{storage} = OriginalSize - CompressedSize = OriginalSize \times \left(1 - \frac{1}{CR}\right)
$$

**I/O带宽节省：**

$$
Bandwidth_{saved} = \lambda \times OriginalSize \times \left(1 - \frac{1}{CR}\right)
$$

其中 $\lambda$ 是I/O操作频率。

**I/O操作数对比：**

```text
┌─────────────────────────────────────────────────────────────────┐
│ 场景: 读取100MB数据                                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│ 无压缩:                                                          │
│   磁盘读取: 100MB                                               │
│   内存传输: 100MB                                               │
│   总I/O: 200MB                                                  │
│                                                                 │
│ LZ4压缩(CR=2.0):                                                 │
│   磁盘读取: 50MB (压缩后)                                        │
│   解压CPU: ~10ms (100MB @ 10GB/s)                                │
│   内存传输: 100MB (解压后)                                       │
│   总I/O: 150MB + CPU开销                                         │
│                                                                 │
│ I/O节省: 25%                                                    │
│ 响应时间变化: 取决于磁盘vs CPU瓶颈                               │
│   - HDD (100MB/s): 1.0s → 0.5s + 0.01s = 0.51s (49%更快)        │
│   - SSD (500MB/s): 0.2s → 0.1s + 0.01s = 0.11s (45%更快)        │
│   - NVMe (3GB/s): 0.033s → 0.017s + 0.01s = 0.027s (18%更快)    │
└─────────────────────────────────────────────────────────────────┘
```

#### 3.2.2 I/O性能预测

**端到端延迟模型：**

$$
T_{total} = T_{seek} + \frac{CompressedSize}{Bandwidth_{disk}} + T_{decompress} + T_{process}
$$

**LZ4解压时间：**

$$
T_{decompress} = \frac{OriginalSize}{V_{lz4}} \approx \frac{OriginalSize}{2GB/s}
$$

**盈亏平衡压缩比：**

$$
CR_{breakeven} = \frac{Bandwidth_{disk}}{Bandwidth_{cpu\_decompress}}
$$

对于典型SSD (500MB/s) 和 LZ4 (2GB/s):

$$
CR_{breakeven} = \frac{500}{2000} = 0.25
$$

由于LZ4压缩比通常 >1.5，始终有益。

### 3.3 内存影响分析

#### 3.3.1 内存使用模型

**LZ4压缩内存需求：**

$$
Memory_{lz4} = InputSize + OutputSize + HashTable_{size}
$$

其中：

- $HashTable_{size} = 2^{HASH\_BITS} \times PointerSize = 4KB \times 8B = 32KB$ (快速模式)
- 高级别压缩需要额外的匹配缓冲区

**各级别内存对比：**

| 级别 | 哈希表 | 搜索缓冲区 | 总额外内存 | 适用数据量 |
|------|--------|-----------|-----------|-----------|
| 1 | 16KB | - | 16KB | 任意 |
| 3 | 64KB | 16KB | 80KB | 任意 |
| 9 | 256KB | 64KB | 320KB | 任意 |
| 12 | 4MB | 256KB | 4.25MB | >10MB |

**并发压缩内存：**

$$
Memory_{total} = N_{connections} \times Memory_{lz4\_per\_compress}
$$

## 4. 使用建议与配置

### 4.1 压缩算法选择决策树

```text
数据特征分析
    │
    ├─> 数据是否已加密?
    │   ├─> YES → 不压缩 (压缩比≈1.0)
    │   └─> NO → 继续分析
    │
    ├─> 数据熵评估
    │   ├─> 高熵 (>6 bits/byte) → 不压缩或LZ4(1)
    │   ├─> 中熵 (3-6 bits/byte) → LZ4(3)
    │   └─> 低熵 (<3 bits/byte) → LZ4(9) 或 ZSTD
    │
    ├─> 访问模式
    │   ├─> 写入密集 → LZ4(1) 最低开销
    │   ├─> 读写均衡 → LZ4(3) 平衡选择
    │   └─> 读取密集 → LZ4(9) 更高压缩比
    │
    └─> 延迟要求
        ├─> 严格低延迟 (<1ms) → 不压缩
        ├─> 可接受少量延迟 → LZ4(1-3)
        └─> 批处理/离线 → ZSTD(9-19)
```

**选择矩阵：**

| 场景 | 推荐算法 | 级别 | 预期CR | 预期延迟增加 |
|------|---------|------|--------|-------------|
| 实时交易日志 | LZ4 | 1 | 1.5x | <0.1ms |
| 用户会话数据 | LZ4 | 3 | 2.0x | <0.5ms |
| JSON文档存储 | LZ4 | 9 | 2.5x | <1ms |
| 历史归档 | ZSTD | 9 | 4.0x | <5ms |
| 冷数据备份 | ZSTD | 19 | 5.0x | 离线 |

#### 4.2.3 分区表压缩策略

```sql
-- 基于时间的分区表，不同分区不同压缩策略
CREATE TABLE events (
    event_id BIGSERIAL,
    event_time TIMESTAMPTZ NOT NULL,
    user_id BIGINT,
    event_data JSONB
) PARTITION BY RANGE (event_time);

-- 热数据: 最新7天，不压缩或LZ4(1)

    FOR VALUES FROM (NOW()) TO (NOW() + INTERVAL '7 days')
    WITH (compression = 'lz4');

ALTER TABLE events_current
    ALTER COLUMN event_data SET (compression = lz4, compression_level = 1);

-- 温数据: 7-30天，LZ4(3)
CREATE TABLE events_recent PARTITION OF events
    FOR VALUES FROM (NOW() - INTERVAL '30 days') TO (NOW() - INTERVAL '7 days')
    WITH (compression = 'lz4');

ALTER TABLE events_recent
    ALTER COLUMN event_data SET (compression = lz4, compression_level = 3);

-- 冷数据: 30-90天，LZ4(9)
CREATE TABLE events_archive PARTITION OF events
    FOR VALUES FROM (NOW() - INTERVAL '90 days') TO (NOW() - INTERVAL '30 days')
    WITH (compression = 'lz4');

ALTER TABLE events_archive
    ALTER COLUMN event_data SET (compression = lz4, compression_level = 9);

-- 自动化压缩降级
CREATE OR REPLACE FUNCTION compress_old_partitions()
RETURNS VOID AS $$
DECLARE
    r RECORD;
BEGIN
    FOR r IN
        SELECT schemaname, tablename
        FROM pg_tables
        WHERE tablename LIKE 'events_%'
          AND tablename < 'events_' || to_char(NOW() - INTERVAL '30 days', 'YYYY_MM_DD')
    LOOP
        EXECUTE format(
            'ALTER TABLE %I.%I ALTER COLUMN event_data SET (compression_level = 9)',
            r.schemaname, r.tablename
        );
    END LOOP;
END;
$$ LANGUAGE plpgsql;

-- 每周执行一次
SELECT cron.schedule('compress-old-partitions', '0 2 * * 0',
    'SELECT compress_old_partitions()');
```

```sql
-- 压缩健康检查
CREATE OR REPLACE FUNCTION check_compression_health()
RETURNS TABLE (
    check_name TEXT,
    status TEXT,
    details TEXT,
    recommendation TEXT
) AS $$
BEGIN
    -- 检查未压缩的大字段
    RETURN QUERY
    SELECT
        'Uncompressed Large Columns'::TEXT,
        'WARNING'::TEXT,
        format('%s.%s: %s bytes', c.relname, a.attname,
               pg_column_size(a.attrelid, a.attnum))::TEXT,
        'Consider enabling LZ4 compression for this column'::TEXT
    FROM pg_attribute a
    JOIN pg_class c ON a.attrelid = c.oid
    JOIN pg_namespace n ON c.relnamespace = n.oid
    WHERE n.nspname NOT IN ('pg_catalog', 'information_schema')
      AND attstorage = 'x'
      AND pg_column_size(a.attrelid, a.attnum) > 8192
      AND attcompression = 'n'
    ORDER BY pg_column_size(a.attrelid, a.attnum) DESC
    LIMIT 10;

    -- 检查低效率压缩
    RETURN QUERY
    SELECT
        'Low Efficiency Compression'::TEXT,
        'WARNING'::TEXT,
        format('%s.%s: ratio=%s', relname, attname,
               ROUND(total_bytes::NUMERIC / compressed_bytes, 2))::TEXT,
        'Compression ratio < 1.2 may not be worth the CPU overhead'::TEXT
    FROM pg_compression_stats()
    WHERE total_bytes::NUMERIC / compressed_bytes < 1.2
      AND compressed_bytes > 1048576;

    -- 检查高压缩CPU消耗
    RETURN QUERY
    SELECT
        'High Compression CPU Usage'::TEXT,
        'INFO'::TEXT,
        format('Level %s: avg %s ms/MB', compression_level,
               ROUND(AVG(compress_time_ms) / (SUM(bytes_processed)/1024/1024), 2))::TEXT,
        'Consider reducing compression level if CPU is bottleneck'::TEXT
    FROM pg_stat_compression
    WHERE timestamp > NOW() - INTERVAL '1 hour'
      AND compression_level > 6
    GROUP BY compression_level
    HAVING AVG(compress_time_ms) > 100;
END;
$$ LANGUAGE plpgsql;
```

## 5. 生产环境最佳实践

### 5.1 迁移策略

**从PGLZ迁移到LZ4的步骤：**

```sql
-- 1. 评估当前压缩情况
CREATE TABLE compression_analysis AS
SELECT
    c.relname as table_name,
    a.attname as column_name,
    pg_column_compression(a.attrelid, a.attnum) as current_compression,
    pg_column_size(a.attrelid, a.attnum) as column_size,
    pg_relation_size(c.oid) as table_size
FROM pg_attribute a
JOIN pg_class c ON a.attrelid = c.oid
WHERE a.attstorage = 'x';

-- 2. 分批迁移(避免锁表时间过长)
DO $$
DECLARE
    batch_size INT := 1000;
    r RECORD;
BEGIN
    FOR r IN
        SELECT table_name, column_name
        FROM compression_analysis
        WHERE current_compression = 'pglz'
    LOOP
        RAISE NOTICE 'Migrating %.% to LZ4', r.table_name, r.column_name;

        -- 使用ALTER TABLE ... SET COMPRESSION
        -- 该操作会重写数据，可能需要长时间锁
        EXECUTE format(
            'ALTER TABLE %I ALTER COLUMN %I SET COMPRESSION lz4',
            r.table_name, r.column_name
        );

        -- 小批量提交避免长时间事务
        COMMIT;
    END LOOP;
END $$;

-- 3. 使用pg_repack在线迁移(不停机)
-- 安装pg_repack扩展后
SELECT pg_repack.repack_apply_indexes(
    'large_table',
    'ALTER TABLE large_table ALTER COLUMN data SET COMPRESSION lz4'
);
```

### 5.2 性能调优

```sql
-- 根据工作负载自动调整压缩级别
CREATE OR REPLACE FUNCTION auto_adjust_compression()
RETURNS VOID AS $$
DECLARE
    v_avg_cpu_usage NUMERIC;
    v_avg_io_wait NUMERIC;
BEGIN
    -- 获取最近1小时的系统指标
    SELECT
        AVG(cpu_usage),
        AVG(io_wait_percent)
    INTO v_avg_cpu_usage, v_avg_io_wait
    FROM pg_stat_system_usage
    WHERE timestamp > NOW() - INTERVAL '1 hour';

    -- CPU高、IO低: 降低压缩级别
    IF v_avg_cpu_usage > 80 AND v_avg_io_wait < 20 THEN
        ALTER SYSTEM SET lz4.default_compression_level = 1;
        RAISE NOTICE 'CPU high, reducing compression level to 1';

    -- CPU低、IO高: 提高压缩级别
    ELSIF v_avg_cpu_usage < 40 AND v_avg_io_wait > 50 THEN
        ALTER SYSTEM SET lz4.default_compression_level = 9;
        RAISE NOTICE 'IO bottleneck, increasing compression level to 9';

    -- 平衡状态: 使用默认
    ELSE
        ALTER SYSTEM SET lz4.default_compression_level = 3;
        RAISE NOTICE 'System balanced, using default compression level 3';
    END IF;

    SELECT pg_reload_conf();
END;
$$ LANGUAGE plpgsql;
```

```sql
    FOR VALUES FROM ('2026-01-08') TO ('2026-01-15');

-- 效果:
```

## 7. 总结

LZ4压缩在PostgreSQL 18中提供了**速度优先**的压缩方案，核心优势：

| 指标 | LZ4 | 对比PGLZ | 对比ZSTD |
|------|-----|----------|----------|
| 压缩速度 | 400-800 MB/s | **40x** 更快 | 5-10x 更快 |
| 解压速度 | 1.5-4 GB/s | **10x** 更快 | 3-5x 更快 |
| 压缩比 | 1.5-2.5x | 略低 | 较低 |
| CPU开销 | 极低 | 低 | 中等 |

**推荐配置**: OLTP场景使用 LZ4 level 3，平衡压缩比与性能。
