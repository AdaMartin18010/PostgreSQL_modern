## 1. 升级前准备

### 1.1 版本兼容性检查

**定理 1.1 (pg_upgrade 版本兼容性)**:

pg_upgrade 工具支持以下升级路径：

$$
\text{SupportedUpgrade}(V_{\text{source}}, V_{\text{target}}) =
\begin{cases}
\text{true}, & \text{if } V_{\text{target}} - V_{\text{source}} \leq 5 \\
\text{false}, & \text{otherwise}
\end{cases}
$$

**PG17 支持的源版本**: PG12, PG13, PG14, PG15, PG16

#### 版本检查命令

```bash
# 检查当前 PostgreSQL 版本
$ psql -c "SELECT version();"
                                                    version
--------------------------------------------------------------------------------------------------------------
 PostgreSQL 16.4 on x86_64-pc-linux-gnu, compiled by gcc (GCC) 12.3.0, 64-bit
(1 row)

# 检查系统架构兼容性
$ pg_config --version
PostgreSQL 16.4

$ pg_config --configure | grep "with-"
  --with-icu
  --with-llvm
  --with-libxml
  --with-lz4

# 检查 PG17 编译选项
$ /usr/pgsql-17/bin/pg_config --configure
```

#### 升级路径决策矩阵

| 源版本 | 推荐路径 | 预计停机时间 | 风险等级 |
|--------|----------|--------------|----------|
| PG16 | 直接 pg_upgrade | 5-30 分钟 | 低 |
| PG15 | 直接 pg_upgrade | 10-45 分钟 | 低 |
| PG14 | 直接 pg_upgrade | 15-60 分钟 | 中 |
| PG13 | 直接 pg_upgrade 或分步升级 | 20-90 分钟 | 中 |
| PG12 | 建议先升级至 PG14 再至 PG17 | 40-120 分钟 | 高 |

### 1.2 已废弃和已移除功能清单

**定义 1.1 (废弃与移除)**:

- **废弃 (Deprecated)**: 功能仍可用但会在未来版本移除，产生警告
- **移除 (Removed)**: 功能已不可用，会导致错误

#### PG17 移除的功能

| 功能 | 影响范围 | 迁移方案 |
|------|----------|----------|
| `recovery.conf` 文件 | 流复制配置 | 移至 `postgresql.conf` |
| `default_with_oids` 参数 | 表创建 | 显式使用 `WITH OIDS` |
| `operator` | 全文检索 | 使用 `tsquery_phrase()` |
| `timetravel` 扩展 | 时间旅行 | 使用 `temporal_tables` |
| `chkpass` 扩展 | 密码加密 | 使用 `pgcrypto` |

#### PG17 废弃警告的功能

```sql
-- 检查废弃功能使用情况
SELECT
    name,
    setting,
    pending_restart
FROM pg_settings
WHERE name IN (
    'escape_string_warning',
    'lo_compat_privileges',
    'operator_precedence_warning',
    'quote_all_identifiers',
    'standard_conforming_strings'
);
```

**关键配置变更检查清单**:

```bash
# 1. 检查 recovery.conf 残留（PG12-11 使用）
$ ls -la $PGDATA/recovery.conf 2>/dev/null && echo "⚠️ 需要迁移 recovery.conf"

# 2. 检查废弃参数
$ grep -E "^(default_with_oids|checkpoint_segments)" $PGDATA/postgresql.conf

# 3. 检查移除的扩展
$ psql -c "SELECT extname FROM pg_extension WHERE extname IN ('timetravel', 'chkpass');"
```

### 1.3 配置参数变更检查

**定理 1.2 (配置参数变更影响评估)**:

设 $P_{\text{removed}}$ 为移除参数集合，$P_{\text{changed}}$ 为行为变更参数集合，则升级风险度量为：

$$
\text{Risk}_{\text{config}} = |P_{\text{removed}} \cap C_{\text{current}}| \times 3 + |P_{\text{changed}} \cap C_{\text{current}}| \times 1
$$

其中 $C_{\text{current}}$ 为当前配置文件中的参数集合。

#### PG17 新增关键参数

| 参数名 | 默认值 | 说明 | 建议值 |
|--------|--------|------|--------|
| `vacuum_buffer_usage_limit` | 256MB | VACUUM 缓冲区限制 | 根据内存调整 |
| `transaction_timeout` | 0 (禁用) | 事务超时 | 3600s (生产) |
| `ssl_crl_dir` | '' | CRL 目录 | 按需配置 |
| `ssl_passphrase_command` | '' | SSL 密码命令 | 按需配置 |
| `wal_summarize` | on | 增量备份支持 | 保持开启 |

#### PG17 行为变更参数

```sql
-- 生成当前配置与 PG17 默认配置对比
WITH current_settings AS (
    SELECT name, setting, unit, context
    FROM pg_settings
    WHERE name IN (
        'shared_buffers',
        'effective_cache_size',
        'work_mem',
        'maintenance_work_mem',
        'max_connections',
        'wal_level',
        'max_wal_size',
        'min_wal_size',
        'checkpoint_completion_target',
        'random_page_cost',
        'effective_io_concurrency'
    )
),
pg17_defaults AS (
    SELECT * FROM (VALUES
        ('shared_buffers', '128MB'),
        ('effective_cache_size', '4GB'),
        ('work_mem', '4MB'),
        ('maintenance_work_mem', '64MB'),
        ('max_connections', '100'),
        ('wal_level', 'replica'),
        ('max_wal_size', '1GB'),
        ('min_wal_size', '80MB'),
        ('checkpoint_completion_target', '0.9'),
        ('random_page_cost', '4.0'),
        ('effective_io_concurrency', '1')
    ) AS t(name, default_val)
)
SELECT
    c.name,
    c.setting || COALESCE(NULLIF(c.unit, ''), '') AS current_value,
    p.default_val AS pg17_default,
    CASE
        WHEN c.setting != split_part(p.default_val, 'MB', 1)
             AND c.setting != split_part(p.default_val, 'GB', 1)
        THEN '⚠️ 差异'
        ELSE '✓ 一致'
    END AS status
FROM current_settings c
JOIN pg17_defaults p ON c.name = p.name
ORDER BY c.name;
```

```bash
$ psql -c "
```

**扩展升级前检查脚本**:

```bash
#!/bin/bash
# pre_upgrade_extension_check.sh

DB_NAME="your_database"
OUTPUT_FILE="extension_check_report.txt"

echo "=== PostgreSQL 扩展兼容性检查报告 ===" > $OUTPUT_FILE
echo "生成时间: $(date)" >> $OUTPUT_FILE
echo "" >> $OUTPUT_FILE

echo "-- 已安装扩展列表 --" >> $OUTPUT_FILE
psql -d $DB_NAME -c "
SELECT
    extname AS extension,
    extversion AS current_version,
    CASE
        WHEN extname = 'postgis' AND extversion < '3.5.0' THEN '需升级至 3.5+'
        WHEN extname = 'pgvector' AND extversion < '0.6.0' THEN '建议升级至 0.6+'
        WHEN extname = 'timescaledb' AND extversion < '2.15.0' THEN '需升级至 2.15+'
        ELSE '兼容性良好'
    END AS upgrade_note
FROM pg_extension
ORDER BY extname;
" >> $OUTPUT_FILE

echo "" >> $OUTPUT_FILE
echo "-- 自定义函数检查 --" >> $OUTPUT_FILE
psql -d $DB_NAME -c "
SELECT
    n.nspname AS schema,
    p.proname AS function_name,
    pg_get_function_result(p.oid) AS return_type,
    CASE
        WHEN p.prosrc LIKE '%timetravel%' THEN '⚠️ 依赖 timetravel'
        WHEN p.prosrc LIKE '%chkpass%' THEN '⚠️ 依赖 chkpass'
        ELSE '✓ 无问题'
    END AS check_result
FROM pg_proc p
JOIN pg_namespace n ON p.pronamespace = n.oid
WHERE n.nspname NOT IN ('pg_catalog', 'information_schema')
ORDER BY n.nspname, p.proname
LIMIT 50;
" >> $OUTPUT_FILE

cat $OUTPUT_FILE
```

## 2. pg_upgrade 升级流程

### 2.1 标准升级步骤

**定义 2.1 (pg_upgrade 操作流程)**:

pg_upgrade 升级过程可形式化为状态转换系统：

$$
\mathcal{M} = (S, S_0, \Sigma, \delta, F)
$$

其中：

- $S = \{S_{\text{init}}, S_{\text{check}}, S_{\text{dump}}, S_{\text{upgrade}}, S_{\text{verify}}, S_{\text{final}}\}$
- $\Sigma = \{\text{--check}, \text{--dump}, \text{--upgrade}, \text{--verify}\}$
- $\delta: S \times \Sigma \rightarrow S$

#### 升级前完整检查清单

```bash
# ========== 步骤 1: 环境准备 ==========
# 1.1 安装 PG17
sudo dnf install -y postgresql17-server postgresql17-contrib

# 1.2 初始化新集群
sudo /usr/pgsql-17/bin/initdb -D /var/lib/pgsql/17/data --locale=en_US.UTF-8

# 1.3 停止应用写入
$ psql -c "SELECT pg_terminate_backend(pid) FROM pg_stat_activity WHERE usename = 'app_user';"

# 1.4 创建完整备份
$ pg_dumpall -U postgres > /backup/pre_upgrade_backup_$(date +%Y%m%d).sql

# 1.5 停止旧集群
$ sudo systemctl stop postgresql-16

# ========== 步骤 2: 执行 pg_upgrade 检查 ==========
$ sudo -u postgres /usr/pgsql-17/bin/pg_upgrade \
    --old-bindir=/usr/pgsql-16/bin \
    --new-bindir=/usr/pgsql-17/bin \
    --old-datadir=/var/lib/pgsql/16/data \
    --new-datadir=/var/lib/pgsql/17/data \
    --check

# 预期输出
# Performing Consistency Checks
# -----------------------------
# Checking cluster versions                                   ok
# Checking database user is the install user                  ok
# Checking database connection settings                       ok
# Checking for prepared transactions                        ok
# Checking for system-defined composite types in user tables  ok
# ...
# *Clusters are compatible*

# ========== 步骤 3: 执行实际升级 ==========
$ sudo -u postgres /usr/pgsql-17/bin/pg_upgrade \
    --old-bindir=/usr/pgsql-16/bin \
    --new-bindir=/usr/pgsql-17/bin \
    --old-datadir=/var/lib/pgsql/16/data \
    --new-datadir=/var/lib/pgsql/17/data \
    --jobs=4 \
    --link  # 或省略使用复制模式
    --verbose

# 输出示例
# Performing Consistency Checks
# -----------------------------
# Checking cluster versions                                   ok
# Checking database user is the install user                  ok
# ...
#
# If you want to use the new cluster, you will need to modify
# the configuration files in /var/lib/pgsql/17/data to match
# the old cluster's configuration.
#
# Upgrade Complete
# ---------------
# Optimizer statistics are not transferred by pg_upgrade.
# Once you start the new server, consider running:
#    ./analyze_new_cluster.sh
```

#### 升级后配置同步

```bash
# 同步配置文件
sudo cp /var/lib/pgsql/16/data/postgresql.conf /var/lib/pgsql/17/data/postgresql.conf.upgrade
sudo cp /var/lib/pgsql/16/data/pg_hba.conf /var/lib/pgsql/17/data/pg_hba.conf

# 添加 PG17 新参数
sudo cat >> /var/lib/pgsql/17/data/postgresql.conf << 'EOF'
# ===== PostgreSQL 17 新增配置 =====
vacuum_buffer_usage_limit = 512MB
transaction_timeout = 3600s
wal_summarize = on
EOF

# 启动新集群
sudo systemctl start postgresql-17
sudo systemctl enable postgresql-17

# 执行统计信息更新
sudo -u postgres /var/lib/pgsql/analyze_new_cluster.sh
```

### 2.2 --link 模式 vs 复制模式

**定理 2.1 (升级模式时间复杂度)**:

设数据库大小为 $D$ (GB)，表数量为 $N$，则：

$$
T_{\text{copy}} = O(D) + O(N \log N)
$$

$$
T_{\text{link}} = O(N \log N)
$$

**--link 模式**:

```bash
# 使用硬链接，升级时间仅取决于元数据操作
$ pg_upgrade --link --old-datadir=/old/data --new-datadir=/new/data

# 优点:
# - 升级速度快 (TB 级数据库可在 5 分钟内完成)
# - 磁盘空间需求低
#
# 缺点:
# - 无法回滚到旧版本
# - 旧集群不可再用
```

**复制模式 (默认)**:

```bash
# 复制所有数据文件
$ pg_upgrade --old-datadir=/old/data --new-datadir=/new/data

# 优点:
# - 保留旧集群完整备份
# - 可安全回滚
#
# 缺点:
# - 升级时间与数据量成正比
# - 需要 2 倍磁盘空间
```

| 模式 | 升级时间 | 磁盘需求 | 回滚能力 | 适用场景 |
|------|----------|----------|----------|----------|
| `--link` | 分钟级 | 1.1x | ❌ | 生产环境、大数据量 |
| 复制 | 小时级 | 2x | ✓ | 测试环境、小数据量 |

### 2.3 升级时间估算模型

**定理 2.2 (升级时间预测)**:

$$
T_{\text{upgrade}} = T_{\text{metadata}} + T_{\text{data}} + T_{\text{verify}}
$$

其中：

- $T_{\text{metadata}} \approx 0.1 \times N_{\text{tables}}$ (秒)
- $T_{\text{data}} = \begin{cases} 0 & \text{(link mode)} \\ \frac{D}{R_{\text{disk}}} & \text{(copy mode)} \end{cases}$
- $T_{\text{verify}} \approx 0.05 \times D$ (秒)

```bash
# 升级时间估算脚本
#!/bin/bash
# estimate_upgrade_time.sh

PGDATA_OLD="/var/lib/pgsql/16/data"
LINK_MODE=${1:-true}

# 获取数据库大小
DB_SIZE_GB=$(du -sBG $PGDATA_OLD | cut -dG -f1)
TABLE_COUNT=$(find $PGDATA_OLD/base -name "*.frm" 2>/dev/null | wc -l)
DISK_SPEED_MBPS=200  # SSD 典型值

echo "=== PostgreSQL 升级时间估算 ==="
echo "数据库大小: ${DB_SIZE_GB} GB"
echo "表数量: $TABLE_COUNT"
echo "磁盘速度: ${DISK_SPEED_MBPS} MB/s"
echo ""

# 元数据时间 (0.1s per 1000 tables)
METADATA_TIME=$((TABLE_COUNT / 10000 + 1))

if [ "$LINK_MODE" = true ]; then
    DATA_TIME=1
    echo "模式: --link (硬链接)"
else
    # 复制时间 = 大小 / 速度
    DATA_TIME=$((DB_SIZE_GB * 1024 / DISK_SPEED_MBPS / 60 + 1))
    echo "模式: 复制模式"
fi

# 验证时间
VERIFY_TIME=$((DB_SIZE_GB / 20 + 1))

TOTAL_TIME=$((METADATA_TIME + DATA_TIME + VERIFY_TIME))

echo "预计升级时间:"
echo "  - 元数据迁移: ${METADATA_TIME} 分钟"
echo "  - 数据迁移: ${DATA_TIME} 分钟"
echo "  - 验证: ${VERIFY_TIME} 分钟"
echo "  - 总计: ${TOTAL_TIME} 分钟"
echo ""
echo "建议停机窗口: $((TOTAL_TIME * 2)) 分钟 (含缓冲)"
```

## 3. 逻辑复制升级 (PG17 新方式)

### 3.1 PG17 复制槽保留特性

**定义 3.1 (复制槽保留)**:

PG17 引入了复制槽在升级后的保留机制，允许：

$$
\text{Slot}_{\text{retained}} = \{ s \in \text{Slots} \mid \text{version}(s_{\text{subscriber}}) > \text{version}(s_{\text{publisher}}) \}
$$

这是 PG17 的关键突破，支持跨版本逻辑复制而不需要重建复制槽。

#### 复制槽状态保留原理

```
┌─────────────────────────────────────────────────────────────┐
│                    PG17 复制槽保留机制                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  PG16 Publisher              升级过程               PG17    │
│  ┌───────────┐                                    Publisher │
│  │ slot_001  │ ─────────── 升级后保留 ───────────▶ │ slot_001 │
│  │ slot_002  │ ─────────── 升级后保留 ───────────▶ │ slot_002 │
│  │ confirmed │                                    │ confirmed│
│  │ lsn: XXX  │                                    │ lsn: XXX │
│  └───────────┘                                    └──────────┘
│                                                             │
│  传统方式 (PG16及以前):                                      │
│  - pg_upgrade 后所有复制槽丢失                                │
│  - 需要完全重新初始化订阅                                     │
│                                                             │
│  PG17 新方式:                                                │
│  - pg_upgrade 保留复制槽状态                                  │
│  - 订阅者自动继续复制                                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 零停机升级方案

**定理 3.1 (零停机升级可行性)**:

在满足以下条件时，零停机升级可行：

$$
\forall t \in \text{Tables}, \text{replica\_identity}(t) \neq \text{nothing}
$$

且订阅者延迟满足：

$$
\text{lag} < \text{max\_lag\_tolerance}
$$

#### 零停机升级架构


#### 零停机升级步骤

```sql
-- ========== 步骤 1: 在 PG16 主库创建发布 ==========
-- 创建所有表的发布
CREATE PUBLICATION upgrade_pub FOR ALL TABLES;

-- 验证发布
SELECT pubname, puballtables, pubinsert, pubupdate, pubdelete, pubtruncate
FROM pg_publication;

-- ========== 步骤 2: 在 PG17 从库创建订阅 ==========
-- 注意: PG17 从库需要能够连接到 PG16 主库
CREATE SUBSCRIPTION upgrade_sub
    CONNECTION 'host=pg16-primary port=5432 dbname=mydb user=replicator password=xxx'
    PUBLICATION upgrade_pub
    WITH (
        copy_data = true,
        create_slot = true,
        slot_name = 'upgrade_slot'
    );

-- 监控复制状态
SELECT
    subname,
    pid,
    received_lsn,
    latest_end_lsn,
    latest_end_time
FROM pg_stat_subscription;

-- 等待复制追平
SELECT
    client_addr,
    state,
    sent_lsn,
    write_lsn,
    flush_lsn,
    replay_lsn,
    write_lag,
    flush_lag,
    replay_lag
FROM pg_stat_replication;

-- ========== 步骤 3: 升级 PG16 主库到 PG17 ==========
-- (按照第 2 章节的 pg_upgrade 流程执行)
-- pg_upgrade 会自动保留复制槽状态

-- ========== 步骤 4: 切换应用到 PG17 ==========
-- 1. 暂停应用写入
-- 2. 等待最后的事务复制完成
SELECT pg_current_wal_lsn();
-- 在从库确认已应用
SELECT latest_end_lsn FROM pg_stat_subscription;

-- 3. 断开应用连接并切换到新主库
-- 4. 在 PG17 上移除订阅（如果需要双向复制）
DROP SUBSCRIPTION IF EXISTS upgrade_sub;

-- ========== 步骤 5: 验证 ==========
-- 检查数据一致性
SELECT
    schemaname,
    tablename,
    pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename)) as size
FROM pg_tables
WHERE schemaname = 'public'
ORDER BY pg_total_relation_size(schemaname||'.'||tablename) DESC
LIMIT 10;
```

### 3.3 跨版本逻辑复制设置

**定义 3.2 (跨版本复制约束)**:

跨版本逻辑复制的约束集合：

$$
\mathcal{C} = \{c_1, c_2, c_3, c_4\}
$$

其中：

- $c_1$: 发布者版本 ≥ PG10
- $c_2$: 订阅者版本 ≥ 发布者版本
- $c_3$: 复制标识必须配置
- $c_4$: 数据类型必须兼容

#### 跨版本复制配置

```bash
#!/bin/bash
# setup_cross_version_replication.sh

# 配置 PG16 -> PG17 跨版本复制

PG16_HOST="pg16-primary.example.com"
PG17_HOST="pg17-standby.example.com"
DB_NAME="production_db"
REPL_USER="replicator"

echo "=== 跨版本逻辑复制配置 ==="

# 1. 在 PG16 主库配置
psql -h $PG16_HOST -U postgres -c "
-- 创建复制用户
CREATE USER $REPL_USER WITH REPLICATION LOGIN PASSWORD 'secure_password';
GRANT CONNECT ON DATABASE $DB_NAME TO $REPL_USER;
GRANT USAGE ON SCHEMA public TO $REPL_USER;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO $REPL_USER;
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT SELECT ON TABLES TO $REPL_USER;

-- 为所有表添加复制标识
DO \$\$
DECLARE
    rec RECORD;
BEGIN
    FOR rec IN
        SELECT schemaname, tablename
        FROM pg_tables
        WHERE schemaname = 'public'
    LOOP
        EXECUTE format('ALTER TABLE %I.%I REPLICA IDENTITY FULL',
                      rec.schemaname, rec.tablename);
    END LOOP;
END \$\$;

-- 创建发布
DROP PUBLICATION IF EXISTS cross_version_pub;
CREATE PUBLICATION cross_version_pub FOR ALL TABLES;
"

# 2. 在 PG17 订阅库配置
psql -h $PG17_HOST -U postgres -d $DB_NAME -c "
-- 创建订阅
DROP SUBSCRIPTION IF EXISTS cross_version_sub;
CREATE SUBSCRIPTION cross_version_sub
    CONNECTION 'host=$PG16_HOST port=5432 dbname=$DB_NAME user=$REPL_USER password=secure_password'
    PUBLICATION cross_version_pub
    WITH (copy_data = true, create_slot = true, enabled = true);
"

echo "跨版本复制配置完成"
echo "监控命令: psql -h $PG17_HOST -c 'SELECT * FROM pg_stat_subscription;'"
```

#### 复制延迟监控

```sql
-- 实时监控复制延迟
WITH publisher_lsn AS (
    SELECT pg_current_wal_lsn() AS current_lsn
),
subscriber_lsn AS (
    SELECT
        subname,
        latest_end_lsn,
        latest_end_time
    FROM pg_stat_subscription
)
SELECT
    s.subname,
    p.current_lsn,
    s.latest_end_lsn,
    pg_wal_lsn_diff(p.current_lsn, s.latest_end_lsn) AS lag_bytes,
    pg_size_pretty(pg_wal_lsn_diff(p.current_lsn, s.latest_end_lsn)) AS lag_size,
    NOW() - s.latest_end_time AS lag_time
FROM subscriber_lsn s
CROSS JOIN publisher_lsn p;
```

## 4. 升级后验证

### 4.1 数据完整性检查

**定理 4.1 (数据完整性验证)**:

升级后数据完整性可通过以下不变式验证：

$$
\forall t \in \text{Tables}, H_{\text{old}}(t) = H_{\text{new}}(t)
$$

其中 $H$ 为行级哈希聚合：

$$
H(t) = \bigoplus_{r \in t} \text{MD5}(r)
$$

#### 完整性验证脚本

```bash
#!/bin/bash
# verify_data_integrity.sh

DB_OLD="postgresql://postgres@localhost:5432/mydb"
DB_NEW="postgresql://postgres@localhost:5433/mydb"
OUTPUT_FILE="integrity_report.txt"

echo "=== PostgreSQL 升级数据完整性验证报告 ===" > $OUTPUT_FILE
echo "验证时间: $(date)" >> $OUTPUT_FILE
echo "" >> $OUTPUT_FILE

# 1. 表数量对比
echo "-- 表数量对比 --" >> $OUTPUT_FILE
echo "旧库表数量:"
psql $DB_OLD -t -c "SELECT count(*) FROM information_schema.tables WHERE table_schema = 'public';" >> $OUTPUT_FILE
echo "新库表数量:"
psql $DB_NEW -t -c "SELECT count(*) FROM information_schema.tables WHERE table_schema = 'public';" >> $OUTPUT_FILE
echo "" >> $OUTPUT_FILE

# 2. 行数统计对比
echo "-- 行数统计对比 --" >> $OUTPUT_FILE
psql $DB_OLD -c "
SELECT
    schemaname || '.' || tablename as table_name,
    n_live_tup as row_count
FROM pg_stat_user_tables
WHERE schemaname = 'public'
ORDER BY n_live_tup DESC;
" > /tmp/old_counts.txt

psql $DB_NEW -c "
SELECT
    schemaname || '.' || tablename as table_name,
    n_live_tup as row_count
FROM pg_stat_user_tables
WHERE schemaname = 'public'
ORDER BY n_live_tup DESC;
" > /tmp/new_counts.txt

echo "行数差异:" >> $OUTPUT_FILE
diff /tmp/old_counts.txt /tmp/new_counts.txt >> $OUTPUT_FILE || true
echo "" >> $OUTPUT_FILE

# 3. 校验和验证（对关键表）
echo "-- 关键表校验和验证 --" >> $OUTPUT_FILE
CRITICAL_TABLES="users orders products transactions"

for table in $CRITICAL_TABLES; do
    echo "验证表: $table"

    # 使用 MD5 聚合验证（适用于中小表）
    OLD_HASH=$(psql $DB_OLD -t -c "SELECT COALESCE(md5(string_agg(row_hash, ',' ORDER BY row_hash)), 'empty') FROM (SELECT md5(row_to_json(t)::text) as row_hash FROM $table t) sub;" 2>/dev/null || echo "ERROR")
    NEW_HASH=$(psql $DB_NEW -t -c "SELECT COALESCE(md5(string_agg(row_hash, ',' ORDER BY row_hash)), 'empty') FROM (SELECT md5(row_to_json(t)::text) as row_hash FROM $table t) sub;" 2>/dev/null || echo "ERROR")

    if [ "$OLD_HASH" = "$NEW_HASH" ]; then
        echo "✓ $table: 校验和匹配" >> $OUTPUT_FILE
    else
        echo "✗ $table: 校验和不匹配!" >> $OUTPUT_FILE
        echo "  旧库: $OLD_HASH" >> $OUTPUT_FILE
        echo "  新库: $NEW_HASH" >> $OUTPUT_FILE
    fi
done

cat $OUTPUT_FILE
```

### 4.2 性能基准对比

\rho = \frac{P_{\text{new}} - P_{\text{old}}}{P_{\text{old}}} \times 100\%

### 4.4 监控指标基线建立

**定义 4.2 (监控基线)**:

监控基线 $\mathcal{B}$ 为升级后稳定运行期间各项指标的统计分布：

$$
\mathcal{B} = \{(m, \mu, \sigma, p_{95}, p_{99}) \mid m \in \text{Metrics}\}
$$

#### 基线建立脚本

```bash
#!/bin/bash
# establish_monitoring_baseline.sh

DB_URL="postgresql://postgres@localhost:5432/mydb"
BASELINE_FILE="pg17_monitoring_baseline.json"

echo "=== PostgreSQL 17 监控基线建立 ==="
echo "此脚本收集升级后的性能基线指标"
echo ""

# 收集基线数据
psql $DB_URL -c "
WITH baseline_metrics AS (
    SELECT json_build_object(
        'timestamp', now(),
        'version', version(),
        'database_stats', (
            SELECT json_build_object(
                'database_name', datname,
                'connections', numbackends,
                'commits_per_sec', xact_commit::float / NULLIF(extract(epoch from now() - stats_reset), 0),
                'rollbacks_per_sec', xact_rollback::float / NULLIF(extract(epoch from now() - stats_reset), 0),
                'blocks_read_per_sec', blks_read::float / NULLIF(extract(epoch from now() - stats_reset), 0),
                'blocks_hit_per_sec', blks_hit::float / NULLIF(extract(epoch from now() - stats_reset), 0),
                'cache_hit_ratio', CASE WHEN blks_hit + blks_read > 0
                    THEN blks_hit::float / (blks_hit + blks_read) * 100
                    ELSE 0 END
            )
            FROM pg_stat_database
            WHERE datname = current_database()
        ),
        'table_stats', (
            SELECT json_agg(json_build_object(
                'table', schemaname || '.' || relname,
                'seq_scan_per_sec', seq_scan::float / NULLIF(extract(epoch from now() - last_vacuum), 0),
                'idx_scan_per_sec', idx_scan::float / NULLIF(extract(epoch from now() - last_vacuum), 0),
                'n_tup_ins_per_sec', n_tup_ins::float / NULLIF(extract(epoch from now() - last_vacuum), 0),
                'n_tup_upd_per_sec', n_tup_upd::float / NULLIF(extract(epoch from now() - last_vacuum), 0),
                'n_tup_del_per_sec', n_tup_del::float / NULLIF(extract(epoch from now() - last_vacuum), 0),
                'dead_tup_ratio', CASE WHEN n_live_tup + n_dead_tup > 0
                    THEN n_dead_tup::float / (n_live_tup + n_dead_tup) * 100
                    ELSE 0 END
            ) ORDER BY pg_total_relation_size(schemaname||'.'||relname) DESC)
            FROM pg_stat_user_tables
            LIMIT 50
        ),
        'index_stats', (
            SELECT json_agg(json_build_object(
                'index', schemaname || '.' || indexrelname,
                'table', relname,
                'idx_scan_per_sec', idx_scan::float / NULLIF(extract(epoch from now()), 0),
                'idx_tup_read_per_sec', idx_tup_read::float / NULLIF(extract(epoch from now()), 0)
            ) ORDER BY pg_relation_size(indexrelid) DESC)
            FROM pg_stat_user_indexes
            LIMIT 50
        ),
        'settings_baseline', (
            SELECT json_agg(json_build_object(
                'name', name,
                'value', setting,
                'unit', unit
            ))
            FROM pg_settings
            WHERE name IN (
                'shared_buffers',
                'effective_cache_size',
                'work_mem',
                'maintenance_work_mem',
                'vacuum_buffer_usage_limit',
                'max_connections',
                'wal_level',
                'max_wal_size',
                'random_page_cost',
                'effective_io_concurrency'
            )
        )
    ) AS baseline_data
)
SELECT baseline_data FROM baseline_metrics;
" -t -A > $BASELINE_FILE

echo "基线数据已保存至: $BASELINE_FILE"
echo "建议将此文件纳入版本控制，用于后续性能对比"
```

### 5.2 查询计划退化处理

**定义 5.1 (查询计划退化)**:

查询计划退化发生时满足：

$$
\text{Cost}_{\text{actual}}^{\text{new}} > 1.5 \times \text{Cost}_{\text{actual}}^{\text{old}}
$$

或执行时间增加超过 50%。

#### 查询计划退化诊断脚本

```sql
-- ========== 查询计划退化诊断 ==========

-- 1. 识别执行计划变化的查询
WITH query_plan_changes AS (
    SELECT
        queryid,
        query,
        calls,
        mean_exec_time,
        stddev_exec_time,
        shared_blks_hit,
        shared_blks_read
    FROM pg_stat_statements
    WHERE calls > 100
      AND mean_exec_time > 10  -- 只关注慢查询
    ORDER BY mean_exec_time DESC
    LIMIT 50
)
SELECT
    queryid,
    left(query, 100) as query_preview,
    calls,
    round(mean_exec_time::numeric, 2) as avg_time_ms,
    round(stddev_exec_time::numeric, 2) as stddev_ms,
    round((shared_blks_hit::numeric / NULLIF(shared_blks_hit + shared_blks_read, 0) * 100), 2) as cache_hit_pct
FROM query_plan_changes;

-- 2. 强制收集统计信息（解决因统计信息不准导致的计划退化）
ANALYZE VERBOSE;

-- 3. 对特定表进行详细分析
-- 如果某表统计信息特别旧，可能导致计划退化
SELECT
    schemaname,
    relname,
    last_vacuum,
    last_autovacuum,
    last_analyze,
    last_autoanalyze,
    n_tup_ins,
    n_tup_upd,
    n_tup_del
FROM pg_stat_user_tables
WHERE last_analyze < now() - interval '1 day'
   OR last_analyze IS NULL
ORDER BY n_tup_upd + n_tup_del DESC
LIMIT 20;

-- 4. 更新这些表的统计信息
DO $$
DECLARE
    rec RECORD;
BEGIN
    FOR rec IN
        SELECT schemaname, relname
        FROM pg_stat_user_tables
        WHERE last_analyze < now() - interval '1 day'
           OR last_analyze IS NULL
        ORDER BY n_tup_upd + n_tup_del DESC
        LIMIT 20
    LOOP
        EXECUTE format('ANALYZE %I.%I', rec.schemaname, rec.relname);
        RAISE NOTICE 'Analyzed %.%', rec.schemaname, rec.relname;
    END LOOP;
END $$;
```

```bash
#!/bin/bash
# diagnose_performance_regression.sh

DB_NAME="mydb"
OUTPUT_FILE="performance_diagnosis_$(date +%Y%m%d_%H%M%S).txt"

echo "=== PostgreSQL 17 性能回退诊断报告 ===" > $OUTPUT_FILE
echo "生成时间: $(date)" >> $OUTPUT_FILE
echo "PostgreSQL 版本: $(psql -c "SELECT version();" -t -A)" >> $OUTPUT_FILE
echo "" >> $OUTPUT_FILE

# 1. 系统资源检查
echo "-- 1. 系统资源使用情况 --" >> $OUTPUT_FILE
echo "CPU 和内存:" >> $OUTPUT_FILE
free -h >> $OUTPUT_FILE
echo "" >> $OUTPUT_FILE
cat /proc/loadavg >> $OUTPUT_FILE
echo "" >> $OUTPUT_FILE

# 2. 数据库连接状态
echo "-- 2. 数据库连接状态 --" >> $OUTPUT_FILE
psql -d $DB_NAME -c "
SELECT
    datname,
    count(*) as connection_count,
    state,
    wait_event_type,
    wait_event
FROM pg_stat_activity
GROUP BY datname, state, wait_event_type, wait_event
ORDER BY count(*) DESC;
" >> $OUTPUT_FILE
echo "" >> $OUTPUT_FILE

# 3. 锁等待检查
echo "-- 3. 锁等待情况 --" >> $OUTPUT_FILE
psql -d $DB_NAME -c "
SELECT
    blocked_locks.pid AS blocked_pid,
    blocked_activity.usename AS blocked_user,
    blocking_locks.pid AS blocking_pid,
    blocking_activity.usename AS blocking_user,
    blocked_activity.query AS blocked_statement,
    blocking_activity.query AS blocking_statement
FROM pg_catalog.pg_locks blocked_locks
JOIN pg_catalog.pg_stat_activity blocked_activity ON blocked_activity.pid = blocked_locks.pid
JOIN pg_catalog.pg_locks blocking_locks ON blocking_locks.locktype = blocked_locks.locktype
    AND blocking_locks.relation = blocked_locks.relation
    AND blocking_locks.pid != blocked_locks.pid
JOIN pg_catalog.pg_stat_activity blocking_activity ON blocking_activity.pid = blocking_locks.pid
WHERE NOT blocked_locks.granted;
" >> $OUTPUT_FILE
echo "" >> $OUTPUT_FILE

# 4. 慢查询检查
echo "-- 4. Top 10 慢查询 (pg_stat_statements) --" >> $OUTPUT_FILE
psql -d $DB_NAME -c "
SELECT
    queryid,
    left(query, 100) as query_preview,
    calls,
    round(total_exec_time::numeric, 2) as total_time_ms,
    round(mean_exec_time::numeric, 2) as mean_time_ms,
    round((100 * total_exec_time / sum(total_exec_time) OVER ())::numeric, 2) as percent_time
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 10;
" >> $OUTPUT_FILE
echo "" >> $OUTPUT_FILE

# 5. 表膨胀检查
echo "-- 5. 表膨胀情况 --" >> $OUTPUT_FILE

SELECT
    schemaname,
    relname,
    n_live_tup,
    n_dead_tup,
    round((n_dead_tup::numeric / NULLIF(n_live_tup + n_dead_tup, 0) * 100), 2) as dead_tuple_ratio,
    last_vacuum,
    last_autovacuum
FROM pg_stat_user_tables
WHERE n_dead_tup > 1000
ORDER BY n_dead_tup DESC
LIMIT 10;
" >> $OUTPUT_FILE
echo "" >> $OUTPUT_FILE

# 6. 索引使用情况

    schemaname,

    indexrelname as index_name,
    idx_scan,
    idx_tup_read,
    idx_tup_fetch,
    pg_size_pretty(pg_relation_size(indexrelid)) as index_size
FROM pg_stat_user_indexes
WHERE idx_scan < 50
  AND pg_relation_size(indexrelid) > 1000000
ORDER BY pg_relation_size(indexrelid) DESC
LIMIT 10;
" >> $OUTPUT_FILE
echo "" >> $OUTPUT_FILE

# 7. 缓存命中率
echo "-- 7. 缓存命中率 --" >> $OUTPUT_FILE

" >> $OUTPUT_FILE
```

## 6. 不同环境升级策略

### 6.1 本地/裸机环境

**定义 6.1 (裸机升级)**:

裸机环境下的升级操作空间：

$$
\mathcal{O}_{\text{baremetal}} = \{\text{pg\_upgrade}, \text{logical\_replication}, \text{dump\_restore}\}
$$

#### 裸机环境升级方案

```bash
#!/bin/bash
# baremetal_upgrade.sh

# PostgreSQL 17 裸机环境升级脚本

OLD_VERSION=16
NEW_VERSION=17
OLD_DATA="/var/lib/pgsql/${OLD_VERSION}/data"
NEW_DATA="/var/lib/pgsql/${NEW_VERSION}/data"
BACKUP_DIR="/backup/pg_upgrade_$(date +%Y%m%d)"

echo "=== PostgreSQL 裸机环境升级脚本 ==="
echo "从 PG${OLD_VERSION} 升级到 PG${NEW_VERSION}"
echo ""

# 1. 预检
pre_check() {
    echo "[1/8] 执行预检..."

    # 检查磁盘空间
    REQUIRED_SPACE=$(du -s $OLD_DATA | awk '{print $1 * 2}')
    AVAILABLE_SPACE=$(df $NEW_DATA | tail -1 | awk '{print $4}')

    if [ $AVAILABLE_SPACE -lt $REQUIRED_SPACE ]; then
        echo "错误: 磁盘空间不足"
        echo "需要: ${REQUIRED_SPACE}KB, 可用: ${AVAILABLE_SPACE}KB"
        exit 1
    fi

    # 检查版本
    CURRENT_VERSION=$(psql -V | grep -oP '\d+')
    if [ "$CURRENT_VERSION" != "$OLD_VERSION" ]; then
        echo "错误: 当前运行的是 PG${CURRENT_VERSION}, 预期 PG${OLD_VERSION}"
        exit 1
    fi

    echo "✓ 预检通过"
}

# 2. 备份
backup_data() {
    echo "[2/8] 创建备份..."
    mkdir -p $BACKUP_DIR

    # 逻辑备份
    pg_dumpall > $BACKUP_DIR/full_backup.sql

    # 配置文件备份
    cp $OLD_DATA/postgresql.conf $BACKUP_DIR/
    cp $OLD_DATA/pg_hba.conf $BACKUP_DIR/
    cp $OLD_DATA/pg_ident.conf $BACKUP_DIR/

    # 创建数据目录物理备份（可选，需要大量空间）
    # tar czf $BACKUP_DIR/data_backup.tar.gz $OLD_DATA

    echo "✓ 备份完成: $BACKUP_DIR"
}

# 3. 安装新版本
install_new_version() {
    echo "[3/8] 安装 PostgreSQL ${NEW_VERSION}..."

    # RHEL/CentOS/Rocky
    if [ -f /etc/redhat-release ]; then
        sudo dnf install -y "https://download.postgresql.org/pub/repos/yum/reporpms/EL-9-x86_64/pgdg-redhat-repo-latest.noarch.rpm"
        sudo dnf -qy module disable postgresql
        sudo dnf install -y postgresql${NEW_VERSION}-server postgresql${NEW_VERSION}-contrib
    fi

    # Ubuntu/Debian
    if [ -f /etc/debian_version ]; then
        sudo sh -c 'echo "deb http://apt.postgresql.org/pub/repos/apt $(lsb_release -cs)-pgdg main" > /etc/apt/sources.list.d/pgdg.list'
        wget --quiet -O - https://www.postgresql.org/media/keys/ACCC4CF8.asc | sudo apt-key add -
        sudo apt-get update
        sudo apt-get install -y postgresql-${NEW_VERSION} postgresql-contrib-${NEW_VERSION}
    fi

    echo "✓ PostgreSQL ${NEW_VERSION} 安装完成"
}

# 4. 初始化新集群
init_new_cluster() {
    echo "[4/8] 初始化新集群..."

    sudo mkdir -p $NEW_DATA
    sudo chown postgres:postgres $NEW_DATA
    sudo -u postgres /usr/pgsql-${NEW_VERSION}/bin/initdb -D $NEW_DATA --locale=en_US.UTF-8

    echo "✓ 新集群初始化完成"
}

# 5. 执行升级
perform_upgrade() {
    echo "[5/8] 执行 pg_upgrade..."

    sudo systemctl stop postgresql-${OLD_VERSION}

    sudo -u postgres /usr/pgsql-${NEW_VERSION}/bin/pg_upgrade \
        --old-bindir=/usr/pgsql-${OLD_VERSION}/bin \
        --new-bindir=/usr/pgsql-${NEW_VERSION}/bin \
        --old-datadir=$OLD_DATA \
        --new-datadir=$NEW_DATA \
        --link \
        --jobs=$(nproc)

    echo "✓ 升级完成"
}

# 6. 同步配置
sync_configuration() {
    echo "[6/8] 同步配置..."

    # 复制主要配置
    sudo cp $BACKUP_DIR/postgresql.conf $NEW_DATA/
    sudo cp $BACKUP_DIR/pg_hba.conf $NEW_DATA/
    sudo cp $BACKUP_DIR/pg_ident.conf $NEW_DATA/

    # 添加 PG17 新配置
    sudo cat >> $NEW_DATA/postgresql.conf << EOF

# PostgreSQL ${NEW_VERSION} 新增配置
vacuum_buffer_usage_limit = 512MB
transaction_timeout = 3600s
wal_summarize = on
EOF

    echo "✓ 配置同步完成"
}

# 7. 启动服务
start_new_service() {
    echo "[7/8] 启动新服务..."

    sudo systemctl enable postgresql-${NEW_VERSION}
    sudo systemctl start postgresql-${NEW_VERSION}

    # 执行分析脚本
    sudo -u postgres ./analyze_new_cluster.sh

    echo "✓ 新服务已启动"
}

# 8. 验证
verify_upgrade() {
    echo "[8/8] 验证升级..."

    NEW_VER=$(psql -c "SELECT version();" -t -A | grep -oP '\d+\.\d+')
    echo "新版本: $NEW_VER"

    # 检查关键表
    psql -c "SELECT count(*) FROM pg_tables WHERE schemaname = 'public';"

    echo "✓ 验证完成"
}

# 执行升级流程
pre_check
backup_data
install_new_version
init_new_cluster
perform_upgrade
sync_configuration
start_new_service
verify_upgrade

echo ""
echo "=== 升级完成 ==="
echo "旧数据目录: $OLD_DATA"
echo "备份目录: $BACKUP_DIR"
echo "如需回滚，执行: sudo systemctl stop postgresql-${NEW_VERSION} && sudo systemctl start postgresql-${OLD_VERSION}"
```

```bash
#!/bin/bash
# rds_upgrade.sh

# AWS RDS PostgreSQL 17 升级脚本

REGION="us-east-1"

aws rds wait db-snapshot-available \
    --db-snapshot-identifier "${DB_INSTANCE_ID}-pre-pg17-upgrade-$(date +%Y%m%d)" \
    --region $REGION

# 2. 检查参数组兼容性
echo "检查参数组..."
aws rds describe-db-parameter-groups \
    --region $REGION \
    --query 'DBParameterGroups[?contains(DBParameterGroupName, `postgres17`)]'

# 3. 创建 PG17 参数组（如果不存在）
aws rds create-db-parameter-group \
    --db-parameter-group-name postgres17-custom-params \
    --db-parameter-group-family postgres17 \
    --description "Custom parameters for PostgreSQL 17" \
    --region $REGION || true

# 4. 执行升级
echo "开始升级数据库..."
aws rds modify-db-instance \
    --db-instance-identifier $DB_INSTANCE_ID \
    --engine-version $TARGET_VERSION \
    --db-parameter-group-name postgres17-custom-params \
    --apply-immediately \
    --allow-major-version-upgrade \
    --region $REGION

# 5. 监控升级进度
echo "监控升级进度..."
aws rds wait db-instance-available \
    --db-instance-identifier $DB_INSTANCE_ID \
    --region $REGION

# 6. 验证
echo "验证升级..."
aws rds describe-db-instances \
    --db-instance-identifier $DB_INSTANCE_ID \
    --region $REGION \
    --query 'DBInstances[0].[DBInstanceIdentifier, EngineVersion, DBInstanceStatus]'

echo "=== RDS 升级完成 ==="
```

#### K8s 升级操作

```bash
#!/bin/bash
# k8s_upgrade.sh

NAMESPACE="database"
OLD_CLUSTER="pg16-cluster"
NEW_CLUSTER="pg17-cluster"

echo "=== Kubernetes PostgreSQL 升级 ==="

# 1. 确保旧集群有完整备份
echo "检查备份状态..."
kubectl get backups -n $NAMESPACE

# 2. 创建 PG17 集群（使用逻辑复制）
echo "创建 PostgreSQL 17 集群..."
kubectl apply -f - <<EOF
apiVersion: postgresql.cnpg.io/v1
kind: Cluster
metadata:
  name: $NEW_CLUSTER
  namespace: $NAMESPACE
spec:
  imageName: ghcr.io/cloudnative-pg/postgresql:17.0
  instances: 3
  postgresql:
    parameters:
      vacuum_buffer_usage_limit: "512MB"
      wal_summarize: "on"
  storage:
    size: 100Gi
EOF

# 3. 等待新集群就绪
echo "等待新集群就绪..."
kubectl wait --for=condition=Ready pod -l cnpg.io/cluster=$NEW_CLUSTER -n $NAMESPACE --timeout=300s

# 4. 设置逻辑复制（如果需要在线迁移）
echo "设置逻辑复制..."
# 获取旧集群主节点
OLD_PRIMARY=$(kubectl get pods -n $NAMESPACE -l "cnpg.io/cluster=$OLD_CLUSTER,role=primary" -o jsonpath='{.items[0].metadata.name}')

# 获取新集群主节点
NEW_PRIMARY=$(kubectl get pods -n $NAMESPACE -l "cnpg.io/cluster=$NEW_CLUSTER,role=primary" -o jsonpath='{.items[0].metadata.name}')

# 在新集群上创建订阅
kubectl exec -it $NEW_PRIMARY -n $NAMESPACE -- psql -c "
CREATE SUBSCRIPTION upgrade_sub
    CONNECTION 'host=$OLD_PRIMARY.db-$NAMESPACE.svc.cluster.local dbname=app user=postgres'
    PUBLICATION all_tables;
"

# 5. 监控复制延迟
echo "监控复制延迟..."
for i in {1..30}; do
    LAG=$(kubectl exec -it $NEW_PRIMARY -n $NAMESPACE -- psql -t -c "
        SELECT pg_wal_lsn_diff(pg_current_wal_lsn(), latest_end_lsn)
        FROM pg_stat_subscription;
    " 2>/dev/null | tr -d ' ')

    echo "复制延迟: $LAG bytes"

    if [ "$LAG" = "0" ] || [ -z "$LAG" ]; then
        echo "复制已追平"
        break
    fi

    sleep 10
done

# 6. 切换应用连接到新集群
echo "切换应用到新集群..."
# 更新 Service 或 Ingress 指向新集群

# 7. 清理旧集群
echo "清理旧集群..."
kubectl delete cluster $OLD_CLUSTER -n $NAMESPACE

echo "=== Kubernetes 升级完成 ==="
```
