# PostgreSQL 17 逻辑复制与高可用增强详解

> **版本**: PostgreSQL 17 (2024-09-26 发布)
> **类型**: 高可用/复制
> **影响等级**: 高
> **升级建议**: 强烈推荐
> **文档长度**: ~6500字
> **创建日期**: 2026-04-07

## 摘要

PostgreSQL 17 在逻辑复制领域实现了突破性进展，解决了长期困扰 DBA 的核心痛点：**升级时必须删除逻辑复制槽**。
本文深入解析 PG17 引入的三大核心增强：**升级时保留逻辑复制槽**、**逻辑复制故障转移支持**，以及 **pg_createsubscriber 工具**。

这些增强使 PostgreSQL 具备了真正的零停机升级能力，配合故障转移机制，为高可用架构提供了完整的生命周期管理方案。

## 1. 特性概述

### 1.1 PG17 之前逻辑复制升级的核心痛点

在 PostgreSQL 16 及更早版本中，使用 `pg_upgrade` 进行主版本升级时，逻辑复制槽无法保留，导致以下严重问题：

**图 1.1：PG16 及之前升级逻辑复制的困境**

```
┌─────────────────────────────────────────────────────────────────┐
│                    PG16 及之前版本的升级困境                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   升级前状态                        升级后状态                    │
│   ┌─────────────┐                  ┌─────────────┐              │
│   │   Primary   │◄────逻辑复制────►│  Standby A  │              │
│   │   (PG16)    │    slot_a        │  (逻辑从库)  │              │
│   │             │◄────逻辑复制────►│  Standby B  │              │
│   │  slot_a ◄───┘    slot_b        │  (逻辑从库)  │              │
│   │  slot_b     │                  │             │              │
│   └──────┬──────┘                  └─────────────┘              │
│          │                                                      │
│          │  pg_upgrade                                           │
│          ▼                                                      │
│   ┌─────────────┐                  ┌─────────────┐              │
│   │   Primary   │                  │  Standby A  │              │
│   │   (PG17)    │    ❌ 连接断开   │  (逻辑从库)  │              │
│   │             │◄───需要重建────►│             │              │
│   │  无复制槽   │                  │  Standby B  │              │
│   │  (slot丢失) │    ❌ 连接断开   │  (逻辑从库)  │              │
│   └─────────────┘                  └─────────────┘              │
│                                                                 │
│   后果：                                                          │
│   • 必须删除并重建复制槽                                           │
│   • 订阅者需要全量重新同步                                         │
│   • 升级期间数据变更可能丢失                                       │
│   • 大规模数据集 = 数小时/数天的停机时间                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**定理 1.1 (升级前的复制槽丢失问题)**:
在 PG16 及之前版本中，`pg_upgrade` 执行时逻辑复制槽不会被迁移，导致：

$$
\forall slot \in LogicalSlots_{old}: slot \notin LogicalSlots_{new}
$$

这意味着订阅者必须执行全量数据刷新，升级期间的所有变更都可能丢失。

### 1.2 PG17 的三大核心增强

**图 1.2：PG17 逻辑复制增强架构**

```
┌─────────────────────────────────────────────────────────────────┐
│              PostgreSQL 17 逻辑复制三大核心增强                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────┐  ┌──────────────────┐  ┌──────────┐   │
│  │ 升级保留复制槽       │  │ 故障转移支持      │  │pg_createsubscriber│
│  │ (Slot Preservation) │  │ (Failover Slots) │  │  工具    │
│  └─────────────────────┘  └──────────────────┘  └──────────┘   │
│         │                        │                    │         │
│         ▼                        ▼                    ▼         │
│  ┌─────────────────────┐  ┌──────────────────┐  ┌──────────┐   │
│  │ • pg_upgrade 保留槽 │  │ • failover 参数  │  │ 物理→逻辑 │   │
│  │ • 无需重新同步      │  │ • synced 列      │  │ 一键转换 │   │
│  │ • 零停机升级        │  │ • 自动故障转移   │  │ 在线迁移 │   │
│  └─────────────────────┘  └──────────────────┘  └──────────┘   │
│                                                                 │
│  核心价值：真正零停机、零数据丢失的数据库升级方案                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

| 特性 | PG16 及之前 | PG17 | 改进效果 |
|------|------------|------|----------|
| **升级时复制槽** | 必须删除 | 自动保留 | 避免全量重新同步 |
| **故障转移** | 手动重建 | 自动支持 | 高可用性提升 |
| **物理→逻辑转换** | 复杂手工操作 | pg_createsubscriber | 一键完成 |
| **升级停机时间** | 小时/天级 | 分钟级 | 业务连续性 |

### 1.3 形式化定义

**定义 1.1 (逻辑复制槽)**:
逻辑复制槽是发布者上跟踪订阅者消费位置的持久化数据结构：

$$
Slot := \langle name, plugin, slot_type, restart\_lsn, confirmed\_flush\_lsn, synced \rangle
$$

**定义 1.2 (升级槽保留)**:
PG17 引入的升级槽保留保证：

$$
\forall slot_{old} \in PG_{old}: \exists slot_{new} \in PG_{new}: slot_{new}.restart\_lsn \geq slot_{old}.restart\_lsn
$$

## 2. 逻辑复制槽保留详解

### 2.1 技术实现原理

**图 2.1：pg_upgrade 复制槽保留机制**

```
┌─────────────────────────────────────────────────────────────────┐
│                 pg_upgrade 复制槽保留技术实现                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  阶段1: 升级前准备                                                │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  1. 读取源集群复制槽信息                                   │    │
│  │     SELECT * FROM pg_replication_slots;                   │    │
│  │                                                           │    │
│  │  2. 验证槽状态 (active=true 且 restart_lsn 有效)           │    │
│  │                                                           │    │
│  │  3. 导出槽元数据到映射文件                                  │    │
│  │     pg_upgrade_slot_mapping.txt                           │    │
│  └─────────────────────────────────────────────────────────┘    │
│                              │                                  │
│                              ▼                                  │
│  阶段2: 执行 pg_upgrade                                          │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  • 停止旧集群                                            │    │
│  │  • 使用 --copy 或 --link 模式复制数据文件                   │    │
│  │  • 启动新集群 (PG17)                                      │    │
│  │  • 执行系统表升级                                          │    │
│  └─────────────────────────────────────────────────────────┘    │
│                              │                                  │
│                              ▼                                  │
│  阶段3: 复制槽重建                                               │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  1. 读取映射文件中的槽信息                                  │    │
│  │  2. 使用 restart_lsn 重新创建复制槽                         │    │
│  │     pg_create_physical_replication_slot()                 │    │
│  │     pg_create_logical_replication_slot()                  │    │
│  │  3. 验证槽的 LSN 连续性                                    │    │
│  │  4. 复制槽恢复完成，订阅者可立即连接                         │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**关键实现细节**:

1. **LSN 映射**: `pg_upgrade` 会维护旧集群到新集群的 LSN 映射关系
2. **槽状态恢复**: 新集群中的复制槽继承原有的 `restart_lsn` 位置
3. **WAL 连续性保证**: 升级过程中产生的 WAL 必须包含在基础备份中

### 2.2 pg_upgrade 如何保留复制槽

**升级流程完整步骤**:

```bash
# 步骤1: 升级前检查 - 确认复制槽状态
psql -c "SELECT slot_name, plugin, slot_type, restart_lsn, confirmed_flush_lsn
         FROM pg_replication_slots;"

# 预期输出:
#   slot_name   |  plugin   | slot_type | restart_lsn | confirmed_flush_lsn
# --------------+-----------+-----------+-------------+---------------------
#  sub1_slot    | pgoutput  | logical   | 0/12345678  | 0/12345678
#  sub2_slot    | pgoutput  | logical   | 0/1234ABCD  | 0/1234ABCD

# 步骤2: 停止旧集群
pg_ctl -D /var/lib/postgresql/16/main stop

# 步骤3: 执行 pg_upgrade (PG17 自动保留复制槽)
pg_upgrade \
    --old-datadir=/var/lib/postgresql/16/main \
    --new-datadir=/var/lib/postgresql/17/main \
    --old-bindir=/usr/lib/postgresql/16/bin \
    --new-bindir=/usr/lib/postgresql/17/bin \
    --link  # 或 --copy，视存储情况而定

# 步骤4: 验证新集群中的复制槽
psql -c "SELECT slot_name, plugin, slot_type, restart_lsn
         FROM pg_replication_slots;"

# 步骤5: 检查订阅者连接状态
psql -c "SELECT subname, pid, received_lsn, latest_end_lsn
         FROM pg_stat_subscription;"
```

**SQL 验证复制槽保留**:

```sql
-- 升级前记录复制槽状态
CREATE TABLE slot_backup_pre_upgrade AS
SELECT
    slot_name,
    plugin,
    slot_type,
    database,
    active,
    restart_lsn,
    confirmed_flush_lsn,
    pg_wal_lsn_diff(confirmed_flush_lsn, restart_lsn) as lag_bytes
FROM pg_replication_slots;

-- 升级后验证
SELECT
    s.slot_name,
    s.restart_lsn as new_restart_lsn,
    b.restart_lsn as old_restart_lsn,
    s.confirmed_flush_lsn >= b.confirmed_flush_lsn as lsn_preserved
FROM pg_replication_slots s
JOIN slot_backup_pre_upgrade b ON s.slot_name = b.slot_name;
```

### 2.3 升级流程完整操作手册

**图 2.2：零停机升级完整流程**

```
时间轴 ──────────────────────────────────────────────────────────────▶

T0: 准备阶段
    ├─ 检查复制槽状态
    ├─ 创建升级前检查点
    ├─ 备份 slot 元数据
    └─ 通知订阅者准备

T1: 只读模式切换 (可选)
    ├─ 设置 default_transaction_read_only = on
    └─ 等待活跃事务完成

T2: 停止旧集群
    ├─ pg_ctl stop -m fast
    └─ 确认进程已终止

T3: 执行 pg_upgrade
    ├─ 复制数据文件 (--link 或 --copy)
    ├─ 升级系统目录
    ├─ 重建复制槽
    └─ 验证 LSN 连续性

T4: 启动新集群
    ├─ pg_ctl start
    ├─ 验证复制槽存在
    └─ 检查订阅者重连

T5: 验证阶段
    ├─ 确认逻辑复制流恢复
    ├─ 验证数据一致性
    └─ 监控复制延迟

总停机时间: 通常 < 5 分钟 (取决于数据量和硬件性能)
```

**完整升级脚本**:

```bash
#!/bin/bash
# PG17 零停机升级脚本 - 保留逻辑复制槽

set -e

OLD_PGDATA="/var/lib/postgresql/16/main"
NEW_PGDATA="/var/lib/postgresql/17/main"
OLD_BINDIR="/usr/lib/postgresql/16/bin"
NEW_BINDIR="/usr/lib/postgresql/17/bin"

LOG_FILE="/var/log/postgresql/upgrade_$(date +%Y%m%d_%H%M%S).log"

log() {
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] $1" | tee -a "$LOG_FILE"
}

# 1. 升级前检查
log "=== 阶段1: 升级前检查 ==="

# 检查复制槽状态
log "检查复制槽状态..."
psql -c "SELECT slot_name, plugin, slot_type, active, restart_lsn
         FROM pg_replication_slots;" >> "$LOG_FILE" 2>&1

SLOT_COUNT=$(psql -t -c "SELECT count(*) FROM pg_replication_slots WHERE slot_type = 'logical';")
log "发现 $SLOT_COUNT 个逻辑复制槽"

# 检查订阅者延迟
log "检查订阅者延迟..."
psql -c "SELECT subname,
                pg_size_pretty(pg_wal_lsn_diff(received_lsn, latest_end_lsn)) as lag
         FROM pg_stat_subscription;" >> "$LOG_FILE" 2>&1

# 等待订阅者同步
log "等待所有订阅者同步..."
while true; do
    LAG=$(psql -t -c "SELECT count(*) FROM pg_stat_subscription
                       WHERE pg_wal_lsn_diff(received_lsn, latest_end_lsn) > 0;")
    if [ "$LAG" -eq 0 ]; then
        log "所有订阅者已同步"
        break
    fi
    log "仍有 $LAG 个订阅者有延迟，等待中..."
    sleep 5
done

# 2. 备份复制槽信息
log "=== 阶段2: 备份复制槽信息 ==="
psql -c "COPY (SELECT * FROM pg_replication_slots) TO '/tmp/slots_backup.csv' WITH CSV HEADER;"
log "复制槽信息已备份到 /tmp/slots_backup.csv"

# 3. 停止旧集群
log "=== 阶段3: 停止旧集群 ==="
$OLD_BINDIR/pg_ctl -D "$OLD_PGDATA" stop -m fast
log "旧集群已停止"

# 4. 执行 pg_upgrade
log "=== 阶段4: 执行 pg_upgrade ==="
$NEW_BINDIR/pg_upgrade \
    --old-datadir="$OLD_PGDATA" \
    --new-datadir="$NEW_PGDATA" \
    --old-bindir="$OLD_BINDIR" \
    --new-bindir="$NEW_BINDIR" \
    --link \
    --jobs=4 \
    2>&1 | tee -a "$LOG_FILE"

log "pg_upgrade 完成"

# 5. 启动新集群
log "=== 阶段5: 启动新集群 ==="
$NEW_BINDIR/pg_ctl -D "$NEW_PGDATA" start

# 等待数据库启动
sleep 5

# 6. 验证复制槽
log "=== 阶段6: 验证复制槽 ==="
NEW_SLOT_COUNT=$(psql -t -c "SELECT count(*) FROM pg_replication_slots WHERE slot_type = 'logical';")

if [ "$NEW_SLOT_COUNT" -eq "$SLOT_COUNT" ]; then
    log "✓ 所有 $SLOT_COUNT 个逻辑复制槽已成功保留"
else
    log "✗ 警告: 复制槽数量不匹配 (期望: $SLOT_COUNT, 实际: $NEW_SLOT_COUNT)"
fi

psql -c "SELECT slot_name, plugin, restart_lsn, confirmed_flush_lsn
         FROM pg_replication_slots;" >> "$LOG_FILE" 2>&1

# 7. 验证订阅者重连
log "=== 阶段7: 验证订阅者连接 ==="
sleep 10  # 给订阅者时间重连

psql -c "SELECT subname, pid, received_lsn, latest_end_lsn,
                pg_size_pretty(pg_wal_lsn_diff(received_lsn, latest_end_lsn)) as lag
         FROM pg_stat_subscription;" >> "$LOG_FILE" 2>&1

# 8. 运行 analyze_new_cluster.sh
log "=== 阶段8: 更新统计信息 ==="
if [ -f ./analyze_new_cluster.sh ]; then
    bash ./analyze_new_cluster.sh
    log "统计信息已更新"
fi

log "=== 升级完成 ==="
log "详细日志: $LOG_FILE"
```

## 3. 逻辑复制故障转移

### 3.1 failover 参数详解

**定义 3.1 (逻辑复制故障转移)**:
当发布者发生故障时，将逻辑复制流切换到备用服务器的机制：

$$
Failover := \langle Primary_{failed}, Standby_{promoted}, Subscription_{reconnect} \rangle
$$

**图 3.1：逻辑复制故障转移架构**

```
┌─────────────────────────────────────────────────────────────────┐
│                   逻辑复制故障转移架构                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  正常状态:                                                       │
│  ┌─────────────┐         ┌─────────────┐         ┌──────────┐  │
│  │   Primary   │◄────────│  Physical   │◄────────│ Logical  │  │
│  │   (发布者)   │ 复制槽   │  Standby    │ 复制槽  │ Replica  │  │
│  │             │  slot1   │  (故障转移   │  slot2  │ (逻辑从库)│  │
│  │  slot1:     │────────►│  目标)       │────────►│          │  │
│  │  synced=t   │         │  slot1:      │         │          │  │
│  │  slot2:     │         │  synced=t    │         │          │  │
│  │  synced=f   │         │  (来自主库)   │         │          │  │
│  └─────────────┘         └─────────────┘         └──────────┘  │
│                                                                 │
│  故障发生后:                                                     │
│  ┌─────────────┐         ┌─────────────┐         ┌──────────┐  │
│  │   Primary   │   ❌    │  Physical   │◄────────│ Logical  │  │
│  │   (宕机)     │  连接断开│  Standby    │  slot2  │ Replica  │  │
│  │             │         │  (提升为    │────────►│ (自动    │  │
│  │             │         │   新主库)    │         │  重连)   │  │
│  │             │         │             │         │          │  │
│  │             │         │  slot2:     │         │          │  │
│  │             │         │  synced=t   │         │          │  │
│  │             │         │  (继续同步)  │         │          │  │
│  └─────────────┘         └─────────────┘         └──────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**failover 参数使用**:

```sql
-- 创建支持故障转移的复制槽
SELECT pg_create_logical_replication_slot(
    slot_name := 'failover_slot',
    plugin := 'pgoutput',
    temporary := false,
    failover := true  -- PG17 新增参数
);

-- 修改现有复制槽支持故障转移
SELECT pg_alter_logical_replication_slot('existing_slot', 'pgoutput', true);

-- 查看复制槽的 failover 状态
SELECT
    slot_name,
    plugin,
    failover,  -- PG17 新增列
    synced     -- PG17 新增列
FROM pg_replication_slots
WHERE slot_type = 'logical';
```

### 3.2 pg_replication_slots 的 synced 列

**表 3.1：pg_replication_slots 新增列说明 (PG17)**

| 列名 | 类型 | 说明 |
|------|------|------|
| `failover` | boolean | 复制槽是否支持故障转移 |
| `synced` | boolean | 复制槽是否已同步到物理备库 |

**监控故障转移复制槽**:

```sql
-- 查看所有故障转移相关的复制槽
SELECT
    slot_name,
    plugin,
    database,
    active,
    failover,
    synced,
    restart_lsn,
    confirmed_flush_lsn,
    CASE
        WHEN failover AND synced THEN '✓ 可故障转移'
        WHEN failover AND NOT synced THEN '⏳ 同步中'
        ELSE '✗ 不支持故障转移'
    END as status
FROM pg_replication_slots
WHERE slot_type = 'logical'
ORDER BY failover DESC, synced DESC;

-- 监控同步延迟
SELECT
    slot_name,
    pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn) as wal_distance_bytes,
    pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn)) as wal_distance
FROM pg_replication_slots
WHERE failover = true;
```

**故障转移自动化脚本**:

```bash
#!/bin/bash
# 逻辑复制故障转移自动化脚本

NEW_PRIMARY_HOST="new-primary.example.com"
NEW_PRIMARY_PORT="5432"
SUBSCRIPTION_NAME="my_subscription"

log() {
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] $1"
}

# 1. 检查新主库上的复制槽
log "检查新主库复制槽状态..."
SYNCED_SLOTS=$(psql -h "$NEW_PRIMARY_HOST" -p "$NEW_PRIMARY_PORT" -t -c \
    "SELECT count(*) FROM pg_replication_slots WHERE slot_type = 'logical' AND synced = true;")

log "新主库上有 $SYNCED_SLOTS 个已同步的逻辑复制槽"

# 2. 更新订阅连接
log "更新订阅者连接信息..."
psql -c "ALTER SUBSCRIPTION $SUBSCRIPTION_NAME
         CONNECTION 'host=$NEW_PRIMARY_HOST port=$NEW_PRIMARY_PORT dbname=postgres';"

# 3. 刷新订阅
log "刷新订阅..."
psql -c "ALTER SUBSCRIPTION $SUBSCRIPTION_NAME REFRESH PUBLICATION;"

# 4. 等待订阅者连接
log "等待订阅者连接..."
for i in {1..30}; do
    PID=$(psql -t -c "SELECT pid FROM pg_stat_subscription WHERE subname = '$SUBSCRIPTION_NAME';")
    if [ -n "$PID" ] && [ "$PID" != "NULL" ]; then
        log "✓ 订阅者已连接 (PID: $PID)"
        break
    fi
    log "等待订阅者连接... ($i/30)"
    sleep 2
done

# 5. 验证复制延迟
log "验证复制延迟..."
psql -c "SELECT subname,
                pg_size_pretty(pg_wal_lsn_diff(received_lsn, latest_end_lsn)) as lag
         FROM pg_stat_subscription
         WHERE subname = '$SUBSCRIPTION_NAME';"

log "故障转移完成"
```

### 4.2 完整使用示例

**场景1: 将现有物理副本转换为逻辑副本**

```bash
# 步骤1: 在发布者上创建发布
psql -h primary -c "CREATE PUBLICATION mypub FOR TABLE orders, customers;"

# 步骤2: 停止物理副本
pg_ctl -D /var/lib/postgresql/17/replica stop

# 步骤3: 使用 pg_createsubscriber 转换
pg_createsubscriber \
    --database=postgres \
    --subscriber-port=5433 \

    --publication=mypub \
    --subscription=mysub

# 步骤4: 启动转换后的逻辑副本
pg_ctl -D /var/lib/postgresql/17/replica start

# 步骤5: 验证订阅状态
psql -p 5433 -c "SELECT * FROM pg_stat_subscription;"
```

**场景2: 从基础备份创建逻辑订阅者**

```bash
# 步骤1: 创建发布
psql -h primary -c "CREATE PUBLICATION full_pub FOR ALL TABLES;"

# 步骤2: 创建复制槽 (支持故障转移)
psql -h primary -c "SELECT pg_create_logical_replication_slot('full_pub_slot', 'pgoutput', false, true);"

# 步骤3: 使用 pg_basebackup 创建基础备份
pg_basebackup \
    -h primary \
    -D /var/lib/postgresql/17/logical_subscriber \
    -Fp -Xs -P -v \
    -R  # 创建 recovery.conf

# 步骤4: 配置 postgresql.conf
cat >> /var/lib/postgresql/17/logical_subscriber/postgresql.conf << EOF
# 逻辑复制相关配置
wal_level = logical
max_replication_slots = 10
max_logical_replication_workers = 4
EOF

# 步骤5: 使用 pg_createsubscriber 配置订阅
pg_createsubscriber \
    --database=postgres \
    --subscriber-port=5433 \
    --subscriber-host=localhost \
    --publisher-server='host=primary port=5432 dbname=postgres user=replicator' \
    --publication=full_pub \
    --subscription=full_sub \
    --slot-name=full_pub_slot

# 步骤6: 启动订阅者
pg_ctl -D /var/lib/postgresql/17/logical_subscriber start
```

**完整部署脚本**:

```bash
#!/bin/bash
# 使用 pg_basebackup + pg_createsubscriber 部署逻辑订阅者

PRIMARY_HOST="primary.example.com"
PRIMARY_PORT="5432"
REPLICA_DIR="/var/lib/postgresql/17/logical_replica"
REPLICA_PORT="5433"
PUBLICATION_NAME="ecommerce_pub"
SUBSCRIPTION_NAME="ecommerce_sub"

# 1. 清理并创建目录
rm -rf "$REPLICA_DIR"
mkdir -p "$REPLICA_DIR"
chown postgres:postgres "$REPLICA_DIR"

# 2. 执行基础备份
echo "执行 pg_basebackup..."
pg_basebackup \
    -h "$PRIMARY_HOST" \
    -p "$PRIMARY_PORT" \
    -D "$REPLICA_DIR" \
    -U replicator \
    -Fp -Xs -P -v \
    --wal-method=stream \
    --checkpoint=fast

# 3. 配置 postgresql.conf
cat >> "$REPLICA_DIR/postgresql.conf" << EOF

# 逻辑复制配置
port = $REPLICA_PORT
wal_level = logical
max_replication_slots = 10
max_logical_replication_workers = 4
max_worker_processes = 8

# 性能优化
shared_buffers = 2GB
work_mem = 64MB
maintenance_work_mem = 512MB
EOF

# 4. 配置 pg_hba.conf
cat >> "$REPLICA_DIR/pg_hba.conf" << EOF

# 逻辑复制连接
host    replication     all             0.0.0.0/0               scram-sha-256
host    all             all             0.0.0.0/0               scram-sha-256
EOF

# 5. 使用 pg_createsubscriber 配置订阅
echo "配置逻辑订阅..."
pg_createsubscriber \
    --database=postgres \
    --subscriber-port="$REPLICA_PORT" \
    --subscriber-host=localhost \
    --publisher-server="host=$PRIMARY_HOST port=$PRIMARY_PORT dbname=postgres user=replicator" \
    --publication="$PUBLICATION_NAME" \
    --subscription="$SUBSCRIPTION_NAME"

# 6. 启动订阅者
echo "启动逻辑订阅者..."
pg_ctl -D "$REPLICA_DIR" start -l "$REPLICA_DIR/logfile"

# 7. 等待启动并验证
sleep 5

echo "验证订阅状态..."
psql -p "$REPLICA_PORT" -c "
SELECT
    subname,
    pid,
    received_lsn,
    latest_end_lsn,
    pg_size_pretty(pg_wal_lsn_diff(received_lsn, latest_end_lsn)) as lag
FROM pg_stat_subscription;
"

echo "逻辑订阅者部署完成"
echo "连接: psql -p $REPLICA_PORT"
```

**零停机升级脚本**:

```bash
#!/bin/bash
# PG16 → PG17 零停机升级脚本

PG16_PRIMARY="pg16-primary.example.com"
PG17_NEW="pg17-new.example.com"

log() {
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] $1"
}

# 阶段1: 检查 PG17 同步状态
log "=== 阶段1: 检查 PG17 同步状态 ==="
LAG=$(psql -h "$PG17_NEW" -t -c "
    SELECT pg_wal_lsn_diff(received_lsn, latest_end_lsn)
    FROM pg_stat_subscription;
")

if [ "$LAG" -gt 0 ]; then
    log "警告: PG17 还有 $LAG bytes 延迟"
    log "等待同步完成..."
    while [ "$LAG" -gt 0 ]; do
        sleep 1
        LAG=$(psql -h "$PG17_NEW" -t -c "
            SELECT pg_wal_lsn_diff(received_lsn, latest_end_lsn)
            FROM pg_stat_subscription;
        ")
    done
fi
log "✓ PG17 已完全同步"

# 阶段2: 设置 PG16 为只读 (可选)
log "=== 阶段2: 设置 PG16 只读模式 ==="
psql -h "$PG16_PRIMARY" -c "ALTER SYSTEM SET default_transaction_read_only = on;"
psql -h "$PG16_PRIMARY" -c "SELECT pg_reload_conf();"
log "✓ PG16 已设置为只读"

# 等待活跃事务完成
log "等待活跃事务完成..."
sleep 10

# 阶段3: 最终同步检查
log "=== 阶段3: 最终同步检查 ==="
FINAL_LAG=$(psql -h "$PG17_NEW" -t -c "
    SELECT pg_wal_lsn_diff(received_lsn, latest_end_lsn)
    FROM pg_stat_subscription;
")
log "最终延迟: $FINAL_LAG bytes"

# 阶段4: 切换订阅为发布模式
log "=== 阶段4: 启用 PG17 写入 ==="
psql -h "$PG17_NEW" -c "ALTER SUBSCRIPTION mysub DISABLE;"
log "✓ PG17 订阅已禁用"

# 阶段5: 停止 PG16
log "=== 阶段5: 停止 PG16 ==="
psql -h "$PG16_PRIMARY" -c "CHECKPOINT;"
pg_ctl -h "$PG16_PRIMARY" stop -m fast
log "✓ PG16 已停止"

# 阶段6: 提升 PG17 为独立集群
log "=== 阶段6: 提升 PG17 ==="
psql -h "$PG17_NEW" -c "DROP SUBSCRIPTION mysub;"
log "✓ PG17 已提升为独立集群"

log "=== 升级完成 ==="
log "新主库: $PG17_NEW"
```

**跨版本迁移配置**:

```sql
-- ========== PG16 源集群配置 ==========

-- 1. 配置 wal_level (需要重启)
ALTER SYSTEM SET wal_level = logical;
SELECT pg_reload_conf();
-- 注意: 需要重启 PostgreSQL

-- 2. 创建迁移专用发布
CREATE PUBLICATION upgrade_pub FOR ALL TABLES;

-- 3. 创建复制槽
SELECT pg_create_logical_replication_slot('upgrade_slot', 'pgoutput');

-- 4. 创建复制用户
CREATE USER replicator WITH REPLICATION LOGIN PASSWORD 'secure_password';
GRANT ALL PRIVILEGES ON ALL TABLES IN SCHEMA public TO replicator;

-- ========== PG17 目标集群配置 ==========

-- 1. 创建订阅
CREATE SUBSCRIPTION upgrade_sub
    CONNECTION 'host=pg16-primary port=5432 dbname=postgres user=replicator password=secure_password'
    PUBLICATION upgrade_pub
    WITH (
        slot_name = 'upgrade_slot',
        copy_data = true,        -- 复制已有数据
        create_slot = false,     -- 槽已在源端创建
        enabled = true
    );

-- 2. 监控同步进度
SELECT
    subname,
    pid,
    received_lsn,
    latest_end_lsn,
    pg_size_pretty(pg_wal_lsn_diff(received_lsn, latest_end_lsn)) as lag
FROM pg_stat_subscription;

-- 3. 检查表同步状态
SELECT
    schemaname,
    tablename,
    pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename)) as size
FROM pg_tables
WHERE schemaname = 'public'
ORDER BY pg_total_relation_size(schemaname||'.'||tablename) DESC;

-- ========== 切换流程 ==========

-- 1. 停止应用写入 (短暂停机)
-- (应用程序层面操作)

-- 2. 验证最终同步
SELECT pg_wal_lsn_diff(received_lsn, latest_end_lsn) = 0 as fully_synced
FROM pg_stat_subscription;

-- 3. 禁用订阅
ALTER SUBSCRIPTION upgrade_sub DISABLE;

-- 4. 删除订阅 (成为独立集群)
DROP SUBSCRIPTION upgrade_sub;

-- 5. 在 PG17 上重新创建发布 (如果需要继续作为发布者)
CREATE PUBLICATION new_pub FOR ALL TABLES;
```

## 7. 总结

PostgreSQL 17 在逻辑复制领域实现了革命性突破，彻底解决了大版本升级的核心痛点：

| 特性 | 价值 | 适用场景 |
|------|------|----------|
| **升级保留复制槽** | 零重新同步 | 所有 PG17 升级 |
| **故障转移支持** | 高可用性 | 生产环境部署 |
| **pg_createsubscriber** | 简化运维 | 快速部署逻辑副本 |

**关键要点**:

1. **升级时复制槽保留**: 使用 `pg_upgrade` 自动保留逻辑复制槽，避免全量重新同步
2. **故障转移支持**: 通过 `failover` 参数和 `synced` 列，实现逻辑复制的自动故障转移
3. **pg_createsubscriber**: 简化物理副本到逻辑副本的转换，支持快速部署
