## 1. 事务的形式化定义

### 1.1 事务作为计算单元

**定义 1.1 (事务)**:
事务是数据库状态转换的原子单位，是一个五元组：

$$
\mathcal{T} := \langle tid, Ops, \prec, \phi, \psi \rangle
$$

其中：

- $tid \in \mathbb{N}$: 事务唯一标识符
- $Ops$: 操作序列，$Ops = \langle op_1, op_2, ..., op_n \rangle$
- $\prec$: 操作间的全序关系
- $\phi$: 前置条件，$\phi: State \rightarrow \mathbb{B}$
- $\psi$: 后置条件，$\psi: State \times State \rightarrow \mathbb{B}$

### 1.2 操作的形式化

**定义 1.2 (数据库操作)**:
操作是状态转换函数：

$$
op: State \rightarrow State \cup \{\bot\}
$$

其中$\bot$表示操作失败（违反约束）。

**基本操作集**:

| 操作 | 符号 | 语义 | 前置条件 | 后置条件 |
|------|------|------|----------|----------|
| **读** | $r_i(x)$ | 读取对象$x$ | $x \in Database$ | - |
| **写** | $w_i(x, v)$ | 写入值$v$到$x$ | $x \in Database$ | $x = v$ |
| **插入** | $i_i(x, v)$ | 插入新对象 | $x \notin Database$ | $x = v$ |
| **删除** | $d_i(x)$ | 删除对象 | $x \in Database$ | $x \notin Database$ |
| **提交** | $c_i$ | 提交事务 | $\forall op \in T: op \neq \bot$ | 效果持久化 |
| **中止** | $a_i$ | 回滚事务 | - | 效果撤销 |

### 1.3 事务调度

**定义 1.3 (调度/历史)**:
调度是多个事务操作的交错序列：

$$
S := \langle op_1, op_2, ..., op_m \rangle
$$

其中$\forall op_i, \exists T_j: op_i \in Ops(T_j)$。

**串行调度**:
$$
S_{serial} := T_1 \circ T_2 \circ ... \circ T_n
$$

即一个事务的所有操作完成后才开始下一个事务。

## 2. ACID性质形式化

### 2.1 原子性 (Atomicity)

**定义 2.1 (原子性)**:
事务的原子性要求所有操作要么全部生效，要么全部不生效：

$$
\mathcal{A}(T) := (c(T) \Rightarrow \forall op \in T: \text{Applied}(op)) \land (a(T) \Rightarrow \forall op \in T: \neg\text{Applied}(op))
$$

**定理 2.1 (原子性保证)**:
使用WAL (Write-Ahead Logging) 可以实现原子性。

*证明*:

1. **提交情况**: WAL记录了所有操作的redo信息，提交时强制刷盘，保证所有修改可重做。
2. **中止情况**: WAL记录了undo信息，中止时执行回滚，撤销所有修改。
3. **崩溃恢复**: 重启时检查事务状态，已提交的重做，未提交的撤销。
∎

### 2.2 一致性 (Consistency)

**定义 2.2 (数据库一致性)**:
数据库状态$S$是一致的，如果满足所有完整性约束：

$$
\text{Consistent}(S) := \bigwedge_{c \in \mathcal{C}} c(S) = \text{true}
$$

**定义 2.3 (事务一致性)**:
事务保持一致性：

$$
\mathcal{C}(T) := \forall S_0, S_1: \text{Consistent}(S_0) \land T(S_0) = S_1 \Rightarrow \text{Consistent}(S_1)
$$

**定理 2.2 (约束检查机制)**:
IMMEDIATE约束保证每条语句后检查，DEFERRED约束保证事务提交时检查。

### 2.3 隔离性 (Isolation)

**定义 2.4 (隔离性)**:
并发事务的执行效果等同于某个串行执行：

$$
\mathcal{I}(S) := \exists S_{serial}: \text{Equivalent}(S, S_{serial})
$$

**等价关系**: 两个调度等价如果它们产生相同的最终数据库状态，并且每个事务读取相同的值。

### 2.4 持久性 (Durability)

**定义 2.5 (持久性)**:
已提交事务的效果在系统故障后仍然保持：

$$
\mathcal{D}(T) := c(T) \Rightarrow \Box(\Diamond(\text{Committed}(T)))
$$

**定理 2.3 (WAL保证持久性)**:
WAL的fsync操作保证日志在数据页之前落盘，确保可恢复性。

## 3. 调度理论与可串行化

### 3.1 冲突关系

**定义 3.1 (操作冲突)**:
两个操作$op_i$和$op_j$冲突当且仅当：

1. 属于不同事务
2. 访问同一数据对象
3. 至少有一个是写操作

$$
\text{Conflict}(op_i, op_j) := (T(op_i) \neq T(op_j)) \land (obj(op_i) = obj(op_j)) \land (op_i \in \{w\} \lor op_j \in \{w\})
$$

**冲突类型矩阵**:

| 操作1 \ 操作2 | Read | Write |
|---------------|------|-------|
| **Read** | 无冲突 | 冲突 (RW) |
| **Write** | 冲突 (WR) | 冲突 (WW) |

### 3.2 冲突可串行化

**定义 3.2 (冲突等价)**:
两个调度$S_1$和$S_2$冲突等价：

$$
S_1 \equiv_c S_2 := \forall op_i, op_j: \text{Conflict}(op_i, op_j) \Rightarrow (op_i \prec_{S_1} op_j \Leftrightarrow op_i \prec_{S_2} op_j)
$$

**定义 3.3 (冲突可串行化)**:
调度$S$是冲突可串行化的：

$$
\text{CSR}(S) := \exists S_{serial}: S \equiv_c S_{serial}
$$

### 3.3 优先图判定法

**定义 3.4 (优先图)**:
调度$S$的优先图$PG(S) = (V, E)$：

- $V = \{T_i \mid T_i \in S\}$
- $E = \{(T_i, T_j) \mid \exists op_i \in T_i, op_j \in T_j: op_i \prec_S op_j \land \text{Conflict}(op_i, op_j)\}$

**定理 3.1 (可串行化判定)**:
调度$S$是冲突可串行化的当且仅当$PG(S)$是无环的。

*证明*:

- ($\Rightarrow$): 如果$S$冲突可串行化，存在等价的串行调度，优先图是串行顺序，无环。
- ($\Leftarrow$): 如果$PG(S)$无环，存在拓扑排序，对应的串行调度与$S$冲突等价。
∎

### 3.4 视图可串行化

**定义 3.5 (读取从关系)**:
$r_j(x)$读取自$w_i(x)$：

$$
\text{ReadFrom}(r_j(x), w_i(x)) := w_i(x) \prec r_j(x) \land \neg\exists w_k(x): w_i(x) \prec w_k(x) \prec r_j(x)
$$

**定义 3.6 (视图等价)**:
两个调度视图等价如果：

1. 每个读操作读取相同的写操作
2. 最终写操作相同

**定理 3.2 (视图可串行化包含冲突可串行化)**:

$$
\text{CSR} \subset \text{VSR}
$$

即冲突可串行化是视图可串行化的真子集。

## 4. 隔离级别形式化

### 4.1 ANSI SQL隔离级别

**定义 4.1 (异常现象)**:

| 现象 | 定义 | 禁止级别 |
|------|------|----------|
| **P1 (脏读)** | $w_1(x) \prec r_2(x) \land a_1$ | READ COMMITTED+ |
| **P2 (不可重复读)** | $r_1(x) \prec w_2(x) \prec r_1(x)$ | REPEATABLE READ+ |
| **P3 (幻读)** | $r_1(P) \prec w_2(y \in P) \prec r_1(P)$ | SERIALIZABLE |

### 4.2 Adya隔离级别

基于依赖的隔离级别：

| 级别 | 禁止的依赖 | 形式化定义 |
|------|-----------|-----------|
| **PL-1** | G0 (脏写) | 无ww冲突环 |
| **PL-2** | G1 (脏读) | 无wr依赖异常 |
| **PL-2.99** | G-SI | 快照隔离 |
| **PL-3** | G2 | 无rw反依赖环 |

## 6. PostgreSQL实现

### 6.1 事务状态管理

```c
// src/include/access/xact.h

typedef enum TransState {
    TRANS_DEFAULT,          /* 初始状态 */
    TRANS_START,            /* 已开始 */
    TRANS_INPROGRESS,       /* 执行中 */
    TRANS_COMMIT,           /* 提交中 */
    TRANS_ABORT,            /* 中止中 */
    TRANS_PREPARE           /* 准备 (2PC) */
} TransState;

// 事务状态机转换
typedef struct TransactionStateData {
    TransState state;           /* 当前状态 */
    TransactionId transactionId; /* 事务ID */

    /* 嵌套事务支持 */
    int         nestingLevel;   /* 嵌套层级 */
    int         savepointLevel; /* 保存点层级 */

    /* 锁管理 */
    LOCKMASK    heldLocks;      /* 持有的锁 */

    /* 时间戳 */
    TimestampTz xactStartTimestamp; /* 开始时间 */
    TimestampTz stmtStartTimestamp; /* 语句时间 */
} TransactionStateData;
```

### 6.2 提交处理

```c
// src/backend/access/transam/xact.c

void
CommitTransaction(void)
{
    TransactionState s = CurrentTransactionState;

    // 1. 预处理提交
    PreCommit_on_commit_actions();
    PreCommit_on_commit_imports();

    // 2. 记录WAL
    XactLogCommitRecord(...);

    // 3. 强制刷盘 (保证持久性)
    XLogFlush(XactLastRecEnd);

    // 4. 释放资源
    AtCommit_Locks();
    AtCommit_Notify();

    // 5. 清理事务状态

}
```

## 7. 实例与反例

### 7.1 正例: 正确使用事务

**场景1: 银行转账**:

```sql
BEGIN ISOLATION LEVEL SERIALIZABLE;
    -- 检查余额
    SELECT balance INTO @from_bal
    FROM accounts WHERE id = 1;

    IF @from_bal >= 100 THEN
        -- 扣款
        UPDATE accounts
        SET balance = balance - 100
        WHERE id = 1;

        -- 入账
        UPDATE accounts
        SET balance = balance + 100
        WHERE id = 2;

        COMMIT;
    ELSE
        ROLLBACK;
    END IF;
```

**正确性保证**:

- 原子性: 两个UPDATE要么都成功，要么都失败
- 一致性: 余额不会为负
- 隔离性: SERIALIZABLE防止并发修改

### 7.2 反例: 常见错误

**反例1: 丢失更新**:

```sql
-- Session 1
BEGIN;
SELECT balance INTO @bal FROM accounts WHERE id = 1;  -- @bal = 1000
-- ... 其他操作 (5秒)
UPDATE accounts SET balance = @bal - 100 WHERE id = 1;  -- 结果: 900
COMMIT;

-- Session 2 (同时执行)
BEGIN;
SELECT balance INTO @bal FROM accounts WHERE id = 1;  -- @bal = 1000
UPDATE accounts SET balance = @bal - 200 WHERE id = 1;  -- 结果: 800
COMMIT;

-- 最终结果: 800 (应该是700!)
-- 问题: READ COMMITTED不能防止丢失更新
```

**解决方案**:

```sql
-- 方案1: 使用SERIALIZABLE
BEGIN ISOLATION LEVEL SERIALIZABLE;

-- 方案2: 使用乐观锁
UPDATE accounts
SET balance = balance - 100, version = version + 1
WHERE id = 1 AND version = @current_version;

-- 方案3: 使用显式锁
SELECT balance FROM accounts WHERE id = 1 FOR UPDATE;
```

**反例2: 部分提交**:

```sql
-- 错误: 在事务中使用自动提交
SET autocommit = on;

BEGIN;  -- 此BEGIN无效!
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
-- 如果这里失败，扣款不会回滚!
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;
```

### 7.3 边界条件

**边界1: 事务ID回卷**:

```text
当XID接近2^32时:
- PostgreSQL启动autovacuum冻结
- 旧元组标记为FrozenTransactionId
- 防止回卷后的事务ID比较错误
```

**边界2: 嵌套事务溢出**:

```sql

    ...
    BEGIN;  -- 最多64层
```

## 8. TLA+形式化规范

```tla
------------------------------ MODULE Transaction ------------------------------
(*
 * 事务系统的TLA+规范
 *)

EXTENDS Integers, Sequences, FiniteSets

CONSTANTS DataObjects, Values, Transactions

VARIABLES db_state,      (* 数据库状态 *)
          txn_states,    (* 事务状态 *)
          log            (* 日志 *)

TransactionState == {"ACTIVE", "COMMITTED", "ABORTED"}

(* 类型不变式 *)
TypeInvariant ==
    /\ db_state \in [DataObjects -> Values \union {NIL}]
    /\ txn_states \in [Transactions -> TransactionState]
    /\ log \in Seq([txn: Transactions, op: STRING, obj: DataObjects, val: Values])

(* 原子性不变式 *)
AtomicityInvariant ==
    \A t \in Transactions:
        txn_states[t] = "COMMITTED" =>
            \A op \in log: op.txn = t => op \in AppliedOps

        \/

        txn_states[t] = "ABORTED" =>
            \A op \in log: op.txn = t => op \notin AppliedOps

(* 一致性不变式 *)
ConsistencyInvariant ==
    \A obj \in DataObjects: db_state[obj] \in ValidValues

(* 下一步操作 *)
BeginTransaction(t) ==
    /\ txn_states[t] = NIL
    /\ txn_states' = [txn_states EXCEPT ![t] = "ACTIVE"]
    /\ UNCHANGED <<db_state, log>>

CommitTransaction(t) ==
    /\ txn_states[t] = "ACTIVE"
    /\ txn_states' = [txn_states EXCEPT ![t] = "COMMITTED"]
    /\ log' = Append(log, [txn |-> t, op |-> "COMMIT"])
    /\ UNCHANGED <<db_state>>

================================================================================
```
