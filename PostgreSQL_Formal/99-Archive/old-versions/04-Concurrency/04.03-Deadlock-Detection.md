## 1. 概念定义

### 1.1 死锁条件

死锁产生的四个必要条件(Coffman条件):

1. **互斥**: 资源互斥访问
2. **占有等待**: 持有资源同时等待新资源
3. **不可抢占**: 资源不可强制释放
4. **循环等待**: 存在循环等待链

### 1.2 等待图 (Wait-For Graph)

$$
WFG = (V, E)
$$

其中:

- $V = \{T_1, T_2, ..., T_n\}$ (事务集合)
- $E = \{(T_i, T_j) \mid T_i \text{ waits for } T_j\}$

**定理**: 存在死锁当且仅当WFG包含环。

## 2. 死锁检测算法

### 2.1 DFS检测

```
DFS-Detect-Cycle(WFG, node, visited, rec_stack):
    visited.add(node)
    rec_stack.add(node)

    for neighbor in WFG.neighbors(node):
        if neighbor not in visited:
            if DFS-Detect-Cycle(WFG, neighbor, visited, rec_stack):
                return true
        else if neighbor in rec_stack:
            return true

    rec_stack.remove(node)
    return false
```

**复杂度**: $O(V + E)$

### 2.2 超时检测

简单方法: 如果事务等待超过`deadlock_timeout`，认为是死锁。

$$
\text{Deadlock}(T) := \text{wait_time}(T) > \text{deadlock_timeout}
$$

## 3. 死锁处理

### 3.1 受害者选择

选择牺牲者中止的标准:

1. 回滚代价最小
2. 持有锁最少
3. 等待时间最短

$$
\text{VictimScore}(T) = \alpha \cdot \text{locks}(T) + \beta \cdot \text{wait_time}(T) + \gamma \cdot \text{work}(T)
$$

### 3.2 预防策略

| 策略 | 方法 | 代价 |
|------|------|------|
| **顺序加锁** | 所有事务按相同顺序获取锁 | 限制并发 |
| **预占锁** | 一次性获取所有锁 | 降低并发 |
| **超时回滚** | 等待超时后主动回滚 | 误判风险 |

## 5. 反例分析

### 5.1 常见死锁场景

**反例 1: 反向加锁顺序**

```sql
-- Session 1
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;  -- 等待Session 2

-- Session 2
BEGIN;
UPDATE accounts SET balance = balance - 50 WHERE id = 2;
UPDATE accounts SET balance = balance + 50 WHERE id = 1;   -- 等待Session 1
-- DEADLOCK!

-- 解决方案: 统一加锁顺序
UPDATE accounts SET ... WHERE id IN (1, 2) ORDER BY id;
```

**反例 2: 间隙锁死锁**

```sql
-- 使用间隙锁的RR隔离级别容易产生死锁
BEGIN ISOLATION LEVEL REPEATABLE READ;
INSERT INTO logs VALUES (1, 'log1');  -- 获取间隙锁
INSERT INTO logs VALUES (2, 'log2');  -- 可能与并发事务冲突
```
