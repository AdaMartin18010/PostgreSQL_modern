## 1. 锁理论基础

### 1.1 为什么需要锁

**问题背景**: 并发事务访问共享数据可能导致不一致。

**定理 1.1 (锁的必要性)**: 在无锁机制下，无法保证事务的可串行化。

*证明*: 考虑两个并发事务:

- T1: R(A), W(A)
- T2: R(A), W(A)

无锁时可能的交错: R1(A), R2(A), W1(A), W2(A)

- T2的写入覆盖T1的写入
- 最终结果取决于调度顺序
- 不可串行化 ∎

### 1.2 锁的基本概念

**定义 1.1 (锁)**:
锁是一个二元组 $L = \langle object, mode \rangle$，其中:

- $object$: 被锁定的数据对象
- $mode \in \{S, X, IS, IX, SIX\}$: 锁模式

**定义 1.2 (锁相容性)**:
两个锁模式$mode_1$和$mode_2$相容当且仅当:

$$
Compatible(mode_1, mode_2) \iff LockMatrix[mode_1][mode_2] = TRUE
$$

## 2. 锁类型与粒度

### 2.1 锁模式

**共享锁 (S - Shared)**:

- 允许多个事务同时读取
- 阻止其他事务写入

**排他锁 (X - eXclusive)**:

- 只允许一个事务访问
- 阻止其他读写

**意向锁 (Intention Locks)**:

- IS: 意向共享锁
- IX: 意向排他锁
- SIX: 共享+意向排他锁

### 2.2 锁相容性矩阵

**定理 2.1 (锁相容性矩阵)**:

| 请求锁 ↓ \ 持有锁 → | S | X | IS | IX | SIX |
|:---------:|:-:|:-:|:-:|:-:|:-:|
| **S** | ✓ | ✗ | ✓ | ✗ | ✗ |
| **X** | ✗ | ✗ | ✗ | ✗ | ✗ |
| **IS** | ✓ | ✗ | ✓ | ✓ | ✓ |
| **IX** | ✗ | ✗ | ✓ | ✓ | ✗ |
| **SIX** | ✗ | ✗ | ✓ | ✗ | ✗ |

*证明*: 通过冲突分析可得。例如:

- S与X冲突: 读写冲突
- IS与IX相容: 意向锁只表示意图，不实际冲突 ∎

### 2.3 多粒度锁定

**定义 2.1 (粒度层次)**:
数据库对象形成层次结构:

$$
Hierarchy = Database \supset Tablespace \supset Table \supset Page \supset Tuple
$$

**意向锁协议**:

1. 在细粒度对象上加S锁前，必须先在所有祖先上加IS锁
2. 在细粒度对象上加X锁前，必须先在所有祖先上加IX锁
3. 相容性检查从根到叶进行

**定理 2.2 (意向锁正确性)**: 意向锁协议保证不会违反锁相容性。

*证明*: 通过层次检查确保任意两个事务不会在同一个对象上持有冲突锁。∎

### 2.4 PostgreSQL锁粒度

**表级锁** (src/include/storage/lock.h):

| 锁类型 | 模式 | 冲突锁 | 典型使用 |
|--------|------|--------|----------|
| AccessShareLock | S | ACCESS_EXCLUSIVE | SELECT |
| RowShareLock | IS | EXCLUSIVE, ACCESS_EXCLUSIVE | SELECT FOR UPDATE |
| RowExclusiveLock | IX | S, SIX, X | INSERT/UPDATE/DELETE |
| ShareUpdateExclusiveLock | SIX | SIX, X, ... | VACUUM, CREATE INDEX CONCURRENTLY |
| ShareLock | S | X, IX | CREATE INDEX |
| ShareRowExclusiveLock | SIX | ... | 排他外键检查 |
| ExclusiveLock | X | S, X, ... | REFRESH MATERIALIZED VIEW |
| AccessExclusiveLock | X | ALL | DROP, ALTER, VACUUM FULL |

**行级锁**:

| 锁模式 | 兼容性 | 命令 |
|--------|--------|------|
| FOR KEY SHARE | 高 | SELECT FOR KEY SHARE |
| FOR SHARE | 中 | SELECT FOR SHARE |
| FOR NO KEY UPDATE | 低 | SELECT FOR NO KEY UPDATE |
| FOR UPDATE | 最低 | SELECT FOR UPDATE |

## 3. 两阶段锁协议 (2PL)

### 3.1 基本2PL

**定义 3.1 (两阶段锁)**:
事务T遵循2PL如果其锁操作分为两个阶段:

1. **增长阶段 (Growing Phase)**:
   $$\forall l: lock(l) \prec unlock(l')$$
   只能获取锁，不能释放锁

2. **收缩阶段 (Shrinking Phase)**:
   $$\forall l: unlock(l) \prec lock(l')$$
   只能释放锁，不能获取锁

**定理 3.1 (2PL可串行化)**: 所有遵循2PL的事务调度都是可串行化的。

*证明*:

1. 构造等待图(Wait-For Graph)
2. 2PL保证WFG无环
3. 无环WFG对应可串行化调度 ∎

### 3.2 严格两阶段锁 (Strict 2PL)

**定义 3.2 (严格2PL)**:
事务T遵循严格2PL如果:

$$
\forall l \in Locks(T): unlock(l) \prec commit(T) \lor abort(T) \prec unlock(l)
$$

即: 所有锁保持到事务结束才释放。

**定理 3.2 (严格2PL防止级联回滚)**: 严格2PL调度中，不会发生级联回滚。

*证明*:

- 事务只能在提交/中止后释放锁
- 其他事务无法读取未提交数据
- 因此无需级联回滚 ∎

### 3.3 保守两阶段锁 (Conservative 2PL)

**定义 3.3 (保守2PL)**:
事务在开始前就获取所有需要的锁。

**定理 3.3 (保守2PL无死锁)**: 保守2PL协议不会产生死锁。

*证明*:

- 所有锁一次性获取
- 不存在等待链
- 因此无环，无死锁 ∎

### 3.4 PostgreSQL 2PL实现

**锁生命周期**:

```
事务开始
    ↓
获取锁 (根据操作类型)
    ↓
执行操作
    ↓
事务提交/中止
    ↓
释放所有锁
```

**关键源码** (src/backend/storage/lmgr/lock.c):

```c
/* 获取锁 */
LOCKTABLE *
LockAcquire(const LOCKTAG *locktag,
            LOCKMODE lockmode,
            bool sessionLock,
            bool dontWait)
{
    LWLock *partitionLock;
    LOCK *lock;
    PROCLOCK *proclock;

    /* 1. 计算哈希分区 */
    uint32 hashcode = LockTagHashCode(locktag);
    partitionLock = LockHashPartitionLock(hashcode);

    /* 2. 获取分区锁 */
    LWLockAcquire(partitionLock, LW_EXCLUSIVE);

    /* 3. 查找或创建LOCK对象 */
    lock = (LOCK *) hash_search(LockMethodLockHash,
                                locktag,
                                HASH_ENTER_NULL,
                                &found);

    /* 4. 检查锁相容性 */
    if (!LockCheckConflicts(lock, proclock, lockmode))
    {
        /* 相容: 授权锁 */
        GrantLock(lock, proclock, lockmode);
    }
    else
    {
        /* 冲突: 加入等待队列 */
        WaitOnLock(lock, proclock, lockmode);
    }

    LWLockRelease(partitionLock);
    return lock;
}
```

## 4. 死锁检测与处理

### 4.1 死锁定义

**定义 4.1 (死锁)**:
一组事务${T_1, T_2, ..., T_n}$处于死锁状态如果:

$$
\forall i: T_i \text{ waits for } T_{i+1} \land T_n \text{ waits for } T_1
$$

### 4.2 等待图 (Wait-For Graph)

**定义 4.2 (等待图)**:

$$
WFG = (V, E)
$$

其中:

- $V = \{T_i\}$: 活跃事务集合
- $E = \{(T_i, T_j) \mid T_i \text{ waits for } T_j\}$: 等待边

**定理 4.1 (死锁检测)**: 存在死锁当且仅当WFG包含有向环。

*证明*:

- 环表示循环等待
- 循环等待是死锁的必要条件 ∎

### 4.3 死锁检测算法

**DFS检测**:

```
算法 DeadlockDetection:
    对每个事务T:
        if DFS(T) 发现回边:
            检测到死锁

DFS(T):
    标记T为访问中
    对T等待的每个事务T':
        if T'标记为已访问:
            return 发现环
        if T'标记为访问中:
            return 发现环
        if DFS(T') 发现环:
            return 发现环
    标记T为已访问
    return 无环
```

**复杂度**: $O(|V| + |E|)$

### 4.4 PostgreSQL死锁检测实现

**死锁检测器** (src/backend/storage/lmgr/deadlock.c):

```c
/* 死锁检测 */
bool
DeadLockCheck(PGPROC *proc)
{
    int i;
    bool found_deadlock = false;

    /* 构建等待图 */
    nEdges = 0;
    for (i = 0; i < nProcs; i++)
    {
        if (proc->links[i].waitee != NULL)
        {
            AddWaitEdge(proc, proc->links[i].waitee);
        }
    }

    /* DFS检测环 */
    found_deadlock = FindCycle(proc);

    if (found_deadlock)
    {
        /* 选择牺牲者 */
        PGPROC *victim = ChooseDeadlockVictim();

        /* 终止牺牲者 */
        SIGTERM signal to victim;
    }

    return found_deadlock;
}

/* 选择牺牲者 */
static PGPROC *
ChooseDeadlockVictim(void)
{
    /* 启发式: 选择工作量最小的事务 */
    PGPROC *victim = NULL;
    int min_cost = INT_MAX;

    for (each proc in deadlock cycle)
    {
        int cost = proc->deadlockCost;
        if (cost < min_cost)
        {
            min_cost = cost;
            victim = proc;
        }
    }

    return victim;
}
```

**死锁检测触发**:

```c
/* 当锁等待超时后触发检测 */
#define DEADLOCK_TIMEOUT 1000  /* 1秒 */
```

### 4.5 死锁预防

**等待-死亡 (Wait-Die)**:

- 老事务等待新事务: 等待
- 新事务请求老事务持有的锁: 死亡(回滚)

**伤害-等待 (Wound-Wait)**:

- 老事务请求新事务持有的锁: 伤害(抢占)
- 新事务等待老事务: 等待

**定理 4.2 (等待-死亡无饿死)**: 等待-死亡协议保证老事务不会饿死。

*证明*:

- 老事务从不回滚
- 新事务可能回滚但会重试
- 事务年龄单调递增 ∎

## 5. 锁升级与锁表

### 5.1 锁升级

**定义 5.1 (锁升级)**:
将细粒度锁转换为粗粒度锁以减少锁数量。

**升级条件**:

$$
|RowLocks(T)| > threshold \Rightarrow UpgradeToTableLock(T)
$$

**定理 5.1 (锁升级减少开销)**: 锁升级可将锁管理开销从$O(n)$降到$O(1)$。

*证明*:

- n个行锁需要n个锁表项
- 一个表锁只需要1个锁表项 ∎

### 5.2 PostgreSQL锁表管理

**锁表结构**:

```c
typedef struct LOCK
{
    LOCKTAG     tag;            /* 锁标识 */
    LOCKMASK    grantMask;      /* 已授权锁模式 */
    LOCKMASK    waitMask;       /* 等待锁模式 */
    SHM_QUEUE   procLocks;      /* 持有者队列 */
    PROC_QUEUE  waitProcs;      /* 等待者队列 */
    int         requested[MAX_LOCKMODES];  /* 请求计数 */
    int         nRequested;     /* 总请求数 */
    int         granted[MAX_LOCKMODES];    /* 授权计数 */
    int         nGranted;       /* 总授权数 */
} LOCK;
```

**锁表分区**:

- 16个分区减少锁竞争
- 哈希函数: LockTagHashCode(tag) % NUM_LOCK_PARTITIONS

## 6. 锁与MVCC的交互

### 6.1 读不阻塞写

**PostgreSQL策略**:

- 读取使用MVCC快照，不获取行锁
- 写入获取行级排他锁
- 读写互不阻塞

### 6.2 锁与快照隔离

**定义 6.1 (快照隔离)**:
事务看到快照开始时的数据状态。

**冲突检测**:

- 第一提交者获胜
- 检测到串行化异常时回滚

## 7. 思维表征

### 7.1 锁模式决策树

```
选择锁模式:
├─ 只读操作?
│   ├─ 可能访问正在修改的数据?
│   │   ├─ 是 → AccessShareLock
│   │   └─ 否 → AccessShareLock
│   └─ 需要防止并发修改?
│       ├─ 是 → ShareLock
│       └─ 否 → AccessShareLock
├─ 修改操作?
│   ├─ INSERT/UPDATE/DELETE
│   │   └─ RowExclusiveLock
│   └─ DDL操作?
│       ├─ 需要完全排他?
│       │   ├─ 是 → AccessExclusiveLock
│       │   └─ 否 → ShareUpdateExclusiveLock
└─ 特殊操作?
    └─ 根据具体需求选择
```

### 7.2 2PL执行时序图

```
时间 →

事务T1:  [lock(A)]─────[lock(B)]────────────[unlock(B)][unlock(A)]
         │<────── 增长阶段 ──────>|<── 收缩阶段 ──>│

事务T2:       [lock(A)]────────────[lock(C)]─────[unlock(C)][unlock(A)]
              │<─────── 增长 ────────>|<─ 收缩 ─>│
              ↑
              等待T1释放A
```

## 8. 实例与反例

### 8.1 正例: 正确使用锁

**场景1: 显式行锁**

```sql
-- 金融转账: 确保原子性
BEGIN;

-- 锁定转出账户
SELECT balance FROM accounts
WHERE account_id = 123
FOR UPDATE;

-- 锁定转入账户
SELECT balance FROM accounts
WHERE account_id = 456
FOR UPDATE;

-- 执行转账
UPDATE accounts SET balance = balance - 100 WHERE account_id = 123;
UPDATE accounts SET balance = balance + 100 WHERE account_id = 456;

COMMIT;
```

**场景2: 避免死锁的加锁顺序**

```sql
-- 总是按相同顺序加锁
BEGIN;

-- 先锁定id小的账户
SELECT * FROM accounts
WHERE account_id IN (456, 123)
ORDER BY account_id  -- 确保顺序
FOR UPDATE;

-- 执行操作
UPDATE accounts SET balance = balance - 100 WHERE account_id = 123;
UPDATE accounts SET balance = balance + 100 WHERE account_id = 456;

COMMIT;
```

**场景3: 表级锁用于DDL**

```sql
-- 在线创建索引(不阻塞读写)
CREATE INDEX CONCURRENTLY idx_orders_date ON orders(created_at);
-- 使用ShareUpdateExclusiveLock，允许并发DML
```

### 8.2 反例: 锁误用

**反例1: 死锁**

```sql
-- 会话1
BEGIN;
UPDATE accounts SET balance = 100 WHERE id = 1;  -- 持有锁1
-- 等待...
UPDATE accounts SET balance = 200 WHERE id = 2;  -- 等待锁2
-- 死锁!

-- 会话2
BEGIN;
UPDATE accounts SET balance = 300 WHERE id = 2;  -- 持有锁2
-- 等待...
UPDATE accounts SET balance = 400 WHERE id = 1;  -- 等待锁1
-- 死锁!

-- 解决方案: 统一加锁顺序
```

**反例2: 锁粒度不当**

```sql
-- 问题: 长时间持有行锁
BEGIN;
SELECT * FROM orders WHERE id = 1 FOR UPDATE;
-- 执行复杂的业务逻辑(10秒)
-- 在此期间其他事务无法修改此行
COMMIT;

-- 解决方案: 尽快释放锁
BEGIN;
-- 获取数据
SELECT * FROM orders WHERE id = 1;
COMMIT;

-- 执行业务逻辑

-- 再开启事务更新
BEGIN;
UPDATE orders SET ... WHERE id = 1;
COMMIT;
```

**反例3: 锁升级导致的并发下降**

```sql
-- 问题: 批量更新导致行锁升级为表锁
BEGIN;
UPDATE large_table SET status = 'processed'
WHERE status = 'pending';  -- 影响90%的行
-- 锁升级: 表级排他锁
-- 其他会话完全无法访问此表
COMMIT;

-- 解决方案: 分批处理
DO $$
DECLARE
    batch_size INT := 1000;
BEGIN
    LOOP
        UPDATE large_table SET status = 'processed'
        WHERE ctid IN (
            SELECT ctid FROM large_table
            WHERE status = 'pending'
            LIMIT batch_size
        );

        EXIT WHEN NOT FOUND;
        COMMIT;
    END LOOP;
END $$;
```

**反例4: 忽略锁超时**

```sql
-- 问题: 无限期等待锁
BEGIN;
SELECT * FROM critical_table WHERE id = 1 FOR UPDATE;
-- 如果其他会话持有锁，无限期等待
-- 可能导致连接池耗尽

-- 解决方案: 设置锁超时
SET lock_timeout = '5s';
BEGIN;
SELECT * FROM critical_table WHERE id = 1 FOR UPDATE;
-- 5秒后超时，报错: canceling statement due to lock timeout
```

## 9. 性能优化

### 9.1 减少锁竞争

1. **缩短事务**: 尽快提交
2. **统一顺序**: 避免死锁
3. **合适粒度**: 行锁优于表锁
4. **乐观并发**: 使用MVCC版本检查

### 9.2 监控锁

```sql
-- 查看当前锁
SELECT * FROM pg_locks WHERE NOT granted;

-- 查看锁等待
SELECT
    w.pid as waiting_pid,
    w.query as waiting_query,
    b.pid as blocking_pid,
    b.query as blocking_query
FROM pg_stat_activity w
JOIN pg_locks l1 ON w.pid = l1.pid AND NOT l1.granted
JOIN pg_locks l2 ON l1.locktype = l2.locktype AND l1.relation = l2.relation AND l2.granted
JOIN pg_stat_activity b ON l2.pid = b.pid;

-- 查看死锁统计
SELECT * FROM pg_stat_database WHERE deadlocks > 0;
```
