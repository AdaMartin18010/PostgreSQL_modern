## 摘要

Serializable Snapshot Isolation (SSI) 是PostgreSQL实现可串行化隔离级别的创新技术，它结合了快照隔离的非阻塞读取优势和真正的可串行化保证。
本文从形式化角度建立SSI的完整理论框架，包括冲突检测、序列化图分析和异常预防。

## 1. SSI理论基础

### 1.1 快照隔离回顾

**定义 1.1 (快照隔离, SI)**:
每个事务读取数据库的一个一致性快照，写操作基于该快照进行冲突检测。

**First-Committer-Wins规则**:
当事务T提交时，如果存在已提交事务T'修改了T也修改的数据，T必须中止。

### 1.2 可串行化定义

**定义 1.2 (可串行化)**:
一个调度是可串行化的，如果它等价于某个串行调度。

**序列化图 (Serialization Graph, SG)**:

$$
\text{SG} = (V, E), \quad E = E_{rw} \cup E_{wr} \cup E_{ww}
$$

**定理 1.1 (可串行化判定)**: 调度是可串行化的当且仅当SG无环。

### 1.3 SI的问题

**定义 1.3 (写偏异常, Write Skew)**:
两个事务读取对方写入的数据集，各自基于读取结果写入不冲突的数据。

示例:

```text
T1: R(A), R(B), W(A)  -- 检查A+B>0
T2: R(A), R(B), W(B)  -- 检查A+B>0
```

**定理 1.2 (SI非可串行化)**: 存在SI允许但非可串行化的调度。

## 2. SSI核心机制

### 2.1 危险结构检测

**定义 2.1 (rw-依赖)**:
事务$T_i$ rw-依赖于$T_j$，如果$T_i$读取了$T_j$修改前的版本，且$T_j$在$T_i$提交前提交。

$$
T_j \xrightarrow{rw} T_i := \exists x: R_i(x_j) \land W_j(x) \land C_j \prec C_i
$$

**定义 2.2 (危险结构)**:
两个连续的rw-依赖边形成危险结构:

$$
T_1 \xrightarrow{rw} T_2 \xrightarrow{rw} T_3
$$

当$T_1 = T_3$时，形成环，存在可串行化异常。

**定理 2.1 (SSI检测条件)**: SSI通过检测危险结构$T_i \xrightarrow{rw} T_j \xrightarrow{rw} T_k$并确保至少一个事务在形成环前中止，来保证可串行化。

### 2.2 冲突检测

**定义 2.3 (入边/出边跟踪)**:
每个事务跟踪:

- **入边**: 哪些事务rw-依赖于此事务
- **出边**: 此事务rw-依赖于哪些事务

**检测算法**:

```text
当T读取数据x时:
    如果x被T'修改且T'已提交:
        添加边 T' --rw--> T

        如果T有出边到T' (即T也rw-依赖于T'):
            检测到危险结构!
            中止T或T'
```

**定理 2.2 (SSI正确性)**: SSI只允许可串行化调度。

*证明*: SSI阻止了所有可能形成环的危险结构，因此SG无环。∎

## 3. PostgreSQL实现

### 3.1 谓词锁

**定义 3.1 (谓词锁)**:
保护查询范围而非具体元组的锁。

**实现方式**:

- 页面级谓词锁: 轻量级，记录事务访问的页范围
- 索引谓词锁: 通过索引确定范围

### 3.2 冲突检测源码

```c
/*
 * CheckForSerializableConflictOut
 * 检查事务T读取的数据是否与其他事务的写入冲突
 * (出边检测)
 */
void
CheckForSerializableConflictOut(
    HeapTupleHeader tuple,  /* 读取的元组 */
    Relation relation,      /* 表 */
    Snapshot snapshot)      /* 事务的快照 */
{
    SERIALIZABLEXACT *sxact = MySerializableXact;

    /* 获取写入此元组的已提交事务 */
    SERIALIZABLEXACT *writer = GetSerializableWriter(tuple);

    if (writer != NULL && writer != sxact)
    {
        /* 添加rw-依赖边: writer --rw--> reader */
        SetSerializableConflictOut(writer, sxact);

        /* 检查是否形成危险结构 */
        if (sxact->outConflicts != NULL)
        {
            /* 检查是否有 T' 使得 sxact --rw--> T' 且 T' --rw--> writer */
            if (CheckForDangerousStructure(writer, sxact))
            {
                /* 形成危险结构，需要中止一个事务 */
                if (sxact-> SeqNo < writer->SeqNo)
                    /* 先启动的事务继续，后启动的中止 */
                    FlagForDeath(sxact);
                else
                    FlagForDeath(writer);
            }
        }
    }
}

/*
 * CheckForSerializableConflictIn
 * 检查事务T写入的数据是否被其他事务读取
 * (入边检测)
 */
void
CheckForSerializableConflictIn(
    HeapTuple tuple,
    Relation relation)
{
    SERIALIZABLEXACT *sxact = MySerializableXact;

    /* 获取读取此元组版本的活跃事务列表 */
    List *readers = GetSerializableReaders(tuple);

    foreach(reader, readers)
    {
        SERIALIZABLEXACT *reader_xact = (SERIALIZABLEXACT *) lfirst(reader);

        if (reader_xact != sxact)
        {
            /* 添加rw-依赖边: sxact --rw--> reader */
            SetSerializableConflictIn(sxact, reader_xact);
        }
    }
}
```

### 3.3 与2PL对比

| 特性 | 2PL | SSI |
|------|-----|-----|
| 读取阻塞 | 是 (S锁) | 否 (快照) |
| 死锁 | 可能 | 无 |
| 写偏异常 | 无 | 检测并中止 |
| 可串行化保证 | 是 | 是 |
| 性能 | 冲突高时下降 | 更稳定 |

## 4. 形式化证明

### 4.1 完备性

**定理 4.1 (SSI完备性)**: SSI检测到所有非可串行化调度。

*证明草图*:

1. 非可串行化调度在SG中有环
2. 环中至少有两个连续的rw-依赖边(形成危险结构)
3. SSI检测所有危险结构
4. 因此SSI能检测到所有非可串行化调度 ∎

### 4.2 安全性

**定理 4.2 (SSI安全性)**: SSI不会中止可串行化调度中的事务。

*注意*: 实际上SSI可能中止一些可串行化调度中的事务(假阳性)，但不会允许非可串行化调度通过。

## 5. 思维表征

### 5.1 危险结构可视化

```text
写偏异常 (Write Skew):

T1: R(A), R(B)        W(A)  C
        \          /
         \        /
          \      /
           \    /
            \  /
T2:         R(A), R(B)  W(B)  C

序列化图:
    T1 ----rw----> T2
    ↑              |
    |              |
    └------rw------┘

危险结构: T1 --rw--> T2 --rw--> T1 (环)

SSI检测:
当T1读取B时，检测到rw-依赖于T2
当T2读取A时，检测到rw-依赖于T1
形成危险结构，中止其中一个事务
```

### 5.2 冲突类型对比

```
读写冲突 (rw-dependency):
T1: R(x) ------> W(x) ------> C
          ↑
          │
T2:       └────── R(x) ────── C

T1读取x的旧值，T2写入x并提交
T1再写入x → 形成rw-依赖

写写冲突 (ww-dependency):
T1: W(x) ────── C
          ↓
T2:       W(x) ────── (等待或中止)

写读冲突 (wr-dependency):
T1: W(x) ────── C
          ↓
T2:       R(x) ────── C
```

## 6. 实例与反例

### 6.1 正例

**场景1: 医疗预约系统**

```sql
-- 检查医生和房间是否都可用
BEGIN ISOLATION LEVEL SERIALIZABLE;

-- 检查医生可用性
SELECT COUNT(*) INTO doc_count
FROM appointments
WHERE doctor_id = 1 AND time_slot = '10:00';

-- 检查房间可用性
SELECT COUNT(*) INTO room_count
FROM appointments
WHERE room_id = 101 AND time_slot = '10:00';

-- 如果都可用，创建预约
IF doc_count = 0 AND room_count = 0 THEN
    INSERT INTO appointments VALUES (1, 101, '10:00');
END IF;

COMMIT;

-- SSI确保在并发情况下不会超预约
-- 如果两个会话同时检查并尝试预约，
-- SSI会检测写偏异常并中止一个事务
```

**场景2: 库存管理**

```sql
-- 转移库存: 从仓库A转移到B
BEGIN ISOLATION LEVEL SERIALIZABLE;

-- 检查总库存
SELECT SUM(quantity) INTO total
FROM inventory WHERE product_id = 100;

-- 检查仓库A库存
SELECT quantity INTO qty_a
FROM inventory WHERE product_id = 100 AND warehouse = 'A';

-- 确保转移后总库存正确
IF qty_a >= 10 THEN
    UPDATE inventory SET quantity = quantity - 10
    WHERE product_id = 100 AND warehouse = 'A';

    UPDATE inventory SET quantity = quantity + 10
    WHERE product_id = 100 AND warehouse = 'B';
END IF;

COMMIT;
```

### 6.2 反例

**反例1: 错误使用SERIALIZABLE**

```sql
-- 问题: 简单的单行更新不需要SERIALIZABLE
BEGIN ISOLATION LEVEL SERIALIZABLE;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;

-- 开销: 谓词锁跟踪带来额外开销
-- 应该用: BEGIN ISOLATION LEVEL READ COMMITTED;
-- 或使用 SELECT FOR UPDATE
```

**反例2: 大量事务冲突**

```sql
-- 高并发场景下大量SSI冲突
-- 导致大量事务中止

-- 监控SSI中止率
SELECT
    xact_commit,
    xact_rollback,
    ROUND(100.0 * xact_rollback / (xact_commit + xact_rollback), 2) AS abort_rate
FROM pg_stat_database;

-- 如果abort_rate > 5%，考虑:
-- 1. 优化业务逻辑减少冲突
-- 2. 使用乐观锁替代
-- 3. 降低隔离级别
```
