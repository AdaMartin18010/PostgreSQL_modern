## 1. 死锁理论基础

### 1.1 死锁定义

**定义 1.1 (死锁)**:
一组事务 $\{T_1, T_2, ..., T_n\}$ 处于死锁状态，如果每个事务都在等待组内另一个事务持有的资源，形成循环等待。

形式化:

$$
\text{Deadlock} \iff \exists \{T_1, ..., T_n\}: \forall i \in [1,n]: T_i \text{ waits for } T_{(i \mod n) + 1}
$$

### 1.2 Coffman条件

**定理 1.1 (死锁必要条件)**: 死锁产生必须同时满足以下四个条件:

1. **互斥条件 (Mutual Exclusion)**:
   $$
   \forall r \in \text{Resources}: |\text{Holder}(r)| \le 1
   $$

2. **占有等待 (Hold and Wait)**:
   $$
   \exists T: \text{Holds}(T) \neq \emptyset \land \text{Waits}(T) \neq \emptyset
   $$

3. **不可抢占 (No Preemption)**:
   $$
   \forall r, T: \text{Holder}(r) = T \Rightarrow \neg\text{ForceRelease}(r)
   $$

4. **循环等待 (Circular Wait)**:
   $$
   \exists \text{cycle in Wait-For Graph}
   $$

*证明*: 四条件同时满足时，没有事务能继续执行，形成死锁。∎

### 1.3 等待图 (Wait-For Graph, WFG)

**定义 1.2 (等待图)**:

$$
\text{WFG} = (V, E)
$$

其中:

- $V = \{T_i \mid T_i \text{ is active}\}$: 活跃事务集合
- $E = \{(T_i, T_j) \mid T_i \text{ waits for } T_j\}$: 等待边集合

**定理 1.2 (死锁检测定理)**: 存在死锁当且仅当WFG包含有向环。

*证明*:

- ($\Rightarrow$) 死锁意味着循环等待，对应WFG中的环
- ($\Leftarrow$) WFG中的环表示循环等待，满足死锁定义 ∎

## 2. 死锁检测算法

### 2.1 基于DFS的环检测

**算法 2.1 (DFS死锁检测)**:

```text
Algorithm DetectDeadlock(WFG):
    visited ← ∅
    rec_stack ← ∅

    for each node v in WFG.V:
        if v ∉ visited:
            if DFS(v, visited, rec_stack):
                return true  // 发现死锁
    return false

Algorithm DFS(v, visited, rec_stack):
    visited.add(v)
    rec_stack.add(v)

    for each neighbor u of v in WFG:
        if u ∉ visited:
            if DFS(u, visited, rec_stack):
                return true
        else if u ∈ rec_stack:
            return true  // 发现回边，存在环

    rec_stack.remove(v)
    return false
```

**定理 2.1 (DFS算法复杂度)**: DFS死锁检测的时间复杂度为 $O(|V| + |E|)$。

*证明*: 每个节点和边最多访问一次。∎

### 2.2 增量检测

**定义 2.1 (增量检测)**:
仅在锁请求冲突时检测，而非定期全量扫描。

**复杂度优势**:

- 全量检测: $O(|V| + |E|)$ 每次
- 增量检测: $O(\text{path length})$ 平均

### 2.3 超时检测

**定义 2.2 (超时检测)**:

$$
\text{Deadlock}_{timeout}(T) := \text{wait\_time}(T) > \text{deadlock\_timeout}
$$

**特点**:

- 优点: 实现简单，无需WFG
- 缺点: 可能误判(长事务)，延迟发现

## 3. 死锁解除策略

### 3.1 受害者选择

**定义 3.1 (受害者评分)**:

$$
\text{Score}(T) = \alpha \cdot \text{Cost}_{rollback}(T) + \beta \cdot \text{Locks}(T) + \gamma \cdot \text{Priority}(T)
$$

**选择标准**:

- 回滚代价最小
- 持有锁数量少
- 优先级低(非关键事务)
- 已执行工作量小

**定理 3.1 (受害者选择最优性)**: 选择持有锁最少的事务作为受害者，解锁的等待事务数期望最大。

### 3.2 解除操作

**回滚范围**:

1. **完全回滚**: 中止整个事务
2. **部分回滚**: 回滚到保存点

**PostgreSQL策略**:

- 完全回滚事务
- 释放所有锁
- 返回错误给客户端
- 客户端决定重试

## 4. 死锁预防

### 4.1 顺序加锁

**定理 4.1 (顺序加锁无死锁)**: 如果所有事务按全局一致的顺序获取锁，则不会产生死锁。

*证明*:

- 假设存在死锁，则存在循环等待 $T_1 \to T_2 \to ... \to T_1$
- 这意味着 $lock\_order(T_1) < lock\_order(T_2) < ... < lock\_order(T_1)$
- 矛盾! ∎

**实现方式**:

```sql
-- 按主键顺序更新
UPDATE accounts SET ... WHERE id IN (3, 1, 2) ORDER BY id;
-- 实际加锁顺序: 1, 2, 3
```

### 4.2 预占锁

**定义 4.1 (预占锁)**:
事务开始前获取所有需要的锁。

**定理 4.2 (预占锁无死锁)**: 预占锁策略不会产生死锁。

*证明*: 所有锁一次性获取，不存在等待链。∎

**缺点**: 降低并发度，可能提前锁定不需要的资源。

### 4.3 等待-死亡 vs 伤害-等待

**等待-死亡 (Wait-Die)**:

- 老事务请求新事务持有的锁: 等待
- 新事务请求老事务持有的锁: 死亡(回滚)

**伤害-等待 (Wound-Wait)**:

- 老事务请求新事务持有的锁: 伤害(抢占)
- 新事务请求老事务持有的锁: 等待

**定理 4.3 (无饿死)**: 两种算法都保证老事务不会饿死。

## 5. PostgreSQL实现

### 5.1 架构集成

```
锁请求
    ↓
┌─────────────────┐
│ LockAcquire()   │
└────────┬────────┘
         ↓
    冲突?
    /      \
  是        否
  /          \
加入等待队列  授权锁
    ↓
等待超时?
    ↓
是 → DeadLockCheck()
    ↓
发现死锁?
    ↓
是 → 选择受害者 → 发送SIGTERM
```

### 5.2 核心实现

**死锁检测器** (src/backend/storage/lmgr/deadlock.c):

```c
/*
 * DeadLockCheck - 检测并解决死锁
 *
 * 在锁等待超时后调用，检查是否存在死锁，
 * 如果存在，选择一个事务中止。
 */
bool
DeadLockCheck(PGPROC *proc)
{
    int         i;
    bool        found_deadlock = false;

    /* 构建等待图 */
    nEdges = 0;
    for (i = 0; i < nProcs; i++)
    {
        PGPROC *waiter = &allProcs[i];

        /* 跳过不活跃的进程 */
        if (waiter->links[0].waitee == NULL)
            continue;

        /* 添加等待边 */
        for (j = 0; j < NUM_LOCK_PARTITIONS; j++)
        {
            if (waiter->links[j].waitee != NULL)
            {
                AddWaitEdge(waiter, waiter->links[j].waitee);
            }
        }
    }

    /* 使用DFS检测环 */
    found_deadlock = FindCycle(proc);

    if (found_deadlock)
    {
        /* 找到死锁，选择受害者 */
        PGPROC *victim = ChooseDeadlockVictim();

        /* 记录死锁信息 */
        DeadLockReport(victim);

        /* 向受害者发送SIGTERM */
        SendProcSignal(victim->pid, PROCSIG_RECOVERY_CONFLICT_DEADLOCK);

        return true;
    }

    return false;
}

/*
 * FindCycle - DFS查找环
 */
static bool
FindCycle(PGPROC *startProc)
{
    int         i;
    PGPROC     *proc = startProc;

    /* 初始化访问标记 */
    for (i = 0; i < nProcs; i++)
        allProcs[i].deadlockCheckVisited = false;

    /* 递归DFS */
    return FindCycleRecurse(proc);
}

static bool
FindCycleRecurse(PGPROC *proc)
{
    int         i;

    /* 标记为访问中 */
    proc->deadlockCheckVisited = true;

    /* 遍历等待边 */
    for (i = 0; i < NUM_LOCK_PARTITIONS; i++)
    {
        PGPROC *waitee = proc->links[i].waitee;

        if (waitee == NULL)
            continue;

        /* 发现回边 = 死锁 */
        if (waitee->deadlockCheckVisited)
        {
            /* 记录死锁环 */
            deadLockDetails[deadLockCount++] = waitee;
            return true;
        }

        /* 递归检查 */
        if (FindCycleRecurse(waitee))
        {
            deadLockDetails[deadLockCount++] = waitee;
            return true;
        }
    }

    /* 取消标记 */
    proc->deadlockCheckVisited = false;
    return false;
}

/*
 * ChooseDeadlockVictim - 选择死锁受害者
 *
 * 启发式: 选择工作量最小的事务
 */
static PGPROC *
ChooseDeadlockVictim(void)
{
    PGPROC *victim = NULL;
    int     minCost = INT_MAX;
    int     i;

    for (i = 0; i < deadLockCount; i++)
    {
        PGPROC *proc = deadLockDetails[i];
        int     cost;

        /* 计算回滚代价 */
        cost = proc->deadlockCost;  /* 基于已执行命令数估计 */

        /* 优先选择只读事务 */
        if (proc->recoveryConflictPending)
            cost += 1000;

        /* 优先选择非关键事务 */
        if (proc->isBackgroundWorker)
            cost -= 500;

        if (cost < minCost)
        {
            minCost = cost;
            victim = proc;
        }
    }

    return victim;
}
```

## 6. 思维表征

### 6.1 死锁形成示意图

```
时间 →

事务A:  [锁定R1]──────[请求R2]──[等待...]
                          ↑
                          │ 死锁!
                          ↓
事务B:          [锁定R2]──[请求R1]──[等待...]

WFG表示:
    ┌───┐      ┌───┐
    │ A │─────→│ B │
    └───┘      └───┘
       ↑          │
       └──────────┘
       循环等待 = 死锁
```

### 6.2 死锁检测流程

```
锁请求冲突
    ↓
加入等待队列
    ↓
启动死锁定时器 (deadlock_timeout)
    ↓
定时器超时?
    │
    ├── 否 → 获得锁，继续执行
    │
    └── 是 → 触发死锁检测
                ↓
         构建等待图 (WFG)
                ↓
         DFS检测环
                ↓
         发现环?
           /      \
         是        否
         /          \
    选择受害者     继续等待
         ↓
    发送SIGTERM
         ↓
    受害者回滚
         ↓
    释放锁，唤醒等待者
```

### 6.3 死锁预防策略对比

| 策略 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| **顺序加锁** | 全局一致的加锁顺序 | 简单有效 | 需要预知所有锁 | 更新操作 |
| **预占锁** | 一次性获取所有锁 | 无死锁 | 并发度低 | 短事务 |
| **等待-死亡** | 新事务回滚 | 老事务优先 | 新事务可能饿死 | 事务有优先级 |
| **伤害-等待** | 老事务抢占 | 响应快 | 回滚代价高 | 实时系统 |
| **超时检测** | 等待超时回滚 | 实现简单 | 误判，延迟 | 通用方案 |

### 6.4 常见死锁模式

```text
模式1: 反向更新顺序
┌─────────────────────────────────────────┐
│  Session A    │    Session B            │
├─────────────────────────────────────────┤
│  BEGIN;       │    BEGIN;               │
│  UPDATE R1;   │    UPDATE R2;           │
│  -- 持有R1    │    -- 持有R2            │
│               │                         │
│  UPDATE R2;   │    UPDATE R1;           │
│  -- 等待R2    │    -- 等待R1            │
│               │                         │
│  DEADLOCK!    │    DEADLOCK!            │
└─────────────────────────────────────────┘
解决方案: 统一按R1,R2顺序更新

模式2: 唯一约束冲突
┌─────────────────────────────────────────┐
│  Session A          │  Session B        │
├─────────────────────────────────────────┤
│  INSERT INTO t      │  INSERT INTO t    │
│  VALUES (1);        │  VALUES (2);      │
│                     │                   │
│  INSERT INTO t      │  INSERT INTO t    │
│  VALUES (2);        │  VALUES (1);      │
│  -- 等待B的插入      │  -- 等待A的插入    │
│                     │                   │
│  DEADLOCK!          │  DEADLOCK!        │
└─────────────────────────────────────────┘
解决方案: 统一按主键顺序插入

模式3: 间隙锁冲突 (RR隔离级别)
┌─────────────────────────────────────────┐
│  Session A              │  Session B    │
├─────────────────────────────────────────┤
│  BEGIN ISOLATION LEVEL  │  BEGIN ISOLATION
│  REPEATABLE READ;       │  LEVEL REPEATABLE
│                         │  READ;        │
│  SELECT * FROM t        │  SELECT * FROM t
│  WHERE id > 10          │  WHERE id > 20
│  FOR UPDATE;            │  FOR UPDATE;  │
│  -- 持有(10,∞)间隙锁    │  -- 持有(20,∞)间隙锁│
│                         │               │
│  INSERT INTO t          │  INSERT INTO t│
│  VALUES (25);           │  VALUES (15); │
│  -- 需要(20,∞)间隙锁    │  -- 需要(10,∞)间隙锁│
│  DEADLOCK!              │  DEADLOCK!    │
└─────────────────────────────────────────┘
解决方案: 使用RC隔离级别，或按相同顺序访问
```

## 7. 实例与反例

### 7.1 正例

**场景1: 统一加锁顺序**:

```sql
-- 转账操作: 总是按账号ID从小到大加锁
CREATE OR REPLACE FUNCTION transfer(
    from_acc INT, to_acc INT, amount NUMERIC
) RETURNS void AS $$
DECLARE
    first_id INT := LEAST(from_acc, to_acc);
    second_id INT := GREATEST(from_acc, to_acc);
BEGIN
    -- 按ID顺序加锁
    PERFORM * FROM accounts WHERE id = first_id FOR UPDATE;
    PERFORM * FROM accounts WHERE id = second_id FOR UPDATE;

    -- 执行转账
    UPDATE accounts SET balance = balance - amount WHERE id = from_acc;
    UPDATE accounts SET balance = balance + amount WHERE id = to_acc;
END;
$$ LANGUAGE plpgsql;
```

**场景2: 重试机制**:

```python
import psycopg2
from psycopg2.errorcodes import DEADLOCK_DETECTED

def execute_with_retry(conn, query, max_retries=3):
    for attempt in range(max_retries):
        try:
            with conn.cursor() as cur:
                cur.execute(query)
                conn.commit()
                return
        except psycopg2.Error as e:
            conn.rollback()
            if e.pgcode == DEADLOCK_DETECTED and attempt < max_retries - 1:
                time.sleep(0.1 * (attempt + 1))  # 指数退避
                continue
            raise
```

### 7.2 反例

**反例1: 随机加锁顺序**:

```sql
-- 问题代码: 加锁顺序取决于参数顺序
CREATE FUNCTION bad_transfer(from_acc INT, to_acc INT, ...) ...
    PERFORM * FROM accounts WHERE id = from_acc FOR UPDATE;  -- 先锁from
    PERFORM * FROM accounts WHERE id = to_acc FOR UPDATE;    -- 后锁to

-- 会话1: transfer(1, 2) -- 锁顺序: 1 → 2
-- 会话2: transfer(2, 1) -- 锁顺序: 2 → 1
-- 结果: 死锁!

-- 正确做法: 按ID排序
```

**反例2: 长时间持有锁**:

```sql
-- 问题: 在事务中执行耗时操作
BEGIN;
SELECT * FROM orders WHERE id = 1 FOR UPDATE;
-- 执行复杂的业务逻辑(5秒)
PERFORM pg_sleep(5);
UPDATE orders SET status = 'processed' WHERE id = 1;
COMMIT;

-- 风险: 锁持有时间长，增加死锁概率

-- 解决方案: 尽快释放锁
-- 1. 将业务逻辑移到事务外
-- 2. 使用乐观锁
```

**反例3: 大量行锁升级**:

```sql
-- 问题: 大量行锁升级为表锁
BEGIN;
UPDATE big_table SET status = 'x' WHERE status = 'y';  -- 影响100万行
-- 行锁数量超过max_locks_per_transaction
-- 升级为表级排他锁
-- 其他会话完全阻塞
COMMIT;

-- 解决方案: 分批处理
DO $$
DECLARE
    batch_size INT := 1000;
BEGIN
    LOOP
        UPDATE big_table SET status = 'x'
        WHERE ctid IN (
            SELECT ctid FROM big_table
            WHERE status = 'y'
            LIMIT batch_size
        );
        EXIT WHEN NOT FOUND;
        COMMIT;
    END LOOP;
END $$;
```

**反例4: 忽视间隙锁**:

```sql
-- 在RR隔离级别下
BEGIN ISOLATION LEVEL REPEATABLE READ;

-- 间隙锁范围查询
SELECT * FROM accounts WHERE balance > 1000 FOR UPDATE;
-- 锁定了balance>1000的所有行和间隙

-- 并发会话插入balance=1500的记录
-- 可能被阻塞，导致死锁

-- 解决方案: 如果不需要可重复读，使用RC隔离级别
BEGIN ISOLATION LEVEL READ COMMITTED;
```
