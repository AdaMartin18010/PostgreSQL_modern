### 2.2 大数据架构模式对比

**认知目标**：理解Lambda、Kappa、Lakehouse三种架构的设计思想与适用场景
**适用场景**：架构选型、系统设计、技术规划

| 维度 | Lambda架构 | Kappa架构 | Lakehouse架构 |
|------|-----------|-----------|--------------|
| **设计思想** | 批流分离，视图合并 | 流处理统一，日志即源 | 湖仓一体，统一存储 |
| **核心组件** | Batch Layer + Speed Layer + Serving Layer | Streaming Layer + Serving Layer | Delta Lake/Iceberg + 计算引擎 |
| **数据流** | 数据同时进入批处理和流处理两条路径 | 所有数据通过流处理，无批处理层 | 统一存储层支持批流一体查询 |
| **代码维护** | 两套代码（批处理+流处理） | 一套代码 | 一套代码 |
| **数据一致性** | 可能存在视图不一致 | 强一致性 | 强一致性（ACID） |
| **延迟性能** | 批处理：小时/天级；流处理：秒/分钟级 | 秒/分钟级 | 批：分钟级；流：秒级 |
| **存储成本** | 较高（数据冗余存储） | 中等 | 较低（统一存储） |
| **复杂度** | 高 | 中等 | 中等 |
| **容错性** | 高（批处理可重跑） | 中等（依赖日志重放） | 高（版本控制+时间旅行） |
| **适用场景** | 需要精确历史重算的场景 | 实时性要求高、逻辑统一的场景 | 需要灵活分析、成本敏感的场景 |
| **代表产品** | Hadoop + Storm/Spark | Kafka + Flink | Databricks、Starburst |
| **演进趋势** | 逐渐被Lakehouse替代 | 特定场景使用 | 主流趋势 |

### 2.3 存储技术对比

**认知目标**：理解Data Warehouse、Data Lake、Lakehouse三种存储范式的本质差异
**适用场景**：存储选型、数据平台建设、技术决策

| 维度 | 数据仓库 (Data Warehouse) | 数据湖 (Data Lake) | 湖仓一体 (Lakehouse) |
|------|------------------------|-------------------|-------------------|
| **数据类型** | 结构化数据为主 | 结构化+半结构化+非结构化 | 全类型数据 |
| **Schema** | Schema-on-Write（写时定义） | Schema-on-Read（读时定义） | 两者结合 |
| **存储格式** | 专有格式 | 开放格式（Parquet/ORC/JSON） | 开放格式+元数据层 |
| **数据质量** | 高（ETL清洗后入库） | 原始数据，质量参差不齐 | 可治理、版本控制 |
| **查询性能** | 高（优化存储+索引） | 中等（需预处理） | 高（自动优化+缓存） |
| **成本效率** | 较高（存储计算耦合） | 低（廉价存储） | 低（存算分离） |
| **灵活性** | 低（需预定义模型） | 高（灵活探索） | 高（灵活+治理） |
| **ACID支持** | 完整支持 | 不支持 | 完整支持 |
| **并发能力** | 高 | 低 | 高 |
| **用户群体** | 业务分析师、BI用户 | 数据科学家、工程师 | 全角色覆盖 |
| **典型产品** | Snowflake、Redshift、Teradata | S3 + 多种计算引擎 | Databricks、Iceberg、Hudi |
| **主要优势** | 高性能、易用性 | 低成本、灵活性 | 兼顾两者优势 |
| **主要劣势** | 成本高、灵活性差 | 数据沼泽风险、性能差 | 新兴技术、生态待完善 |

### 4.2 数据质量影响推理链

**认知目标**：论证数据质量对分析结果和业务决策的影响机制

```text

│   ├── 完整性 (Completeness): 数据字段填充率
│   ├── 一致性 (Consistency): 跨系统数据一致
│   ├── 及时性 (Timeliness): 数据更新频率
│   ├── 唯一性 (Uniqueness): 无重复记录
│   └── 有效性 (Validity): 符合业务规则

│   ├── 前提: 分析模型依赖输入数据

│   │   ├── 准确性问题 → 模型学习到错误模式
│   │   ├── 完整性问题 → 样本偏差/选择偏差
│   │   ├── 一致性问题 → 跨数据源冲突
│   │   └── 及时性问题 → 决策基于过时信息

│
├── 推理2: 分析结果偏差 → 决策失误
│   ├── 前提: 决策者依赖分析结论
│   ├── 机制:
│   │   ├── 预测偏差 → 资源错配
│   │   ├── 分类错误 → 客户流失/误拒
│   │   ├── 归因错误 → 策略方向错误
│   │   └── 趋势误判 → 错失机会/过度投资
│   └── 结论: 分析错误 → 业务损失
│
├── 推理3: 决策失误 → 业务损失
│   ├── 财务损失
│   │   ├── 直接损失: 错误投资、库存积压
│   │   └── 机会成本: 错失市场机会
│   ├── 声誉损失
│   │   ├── 客户信任下降
│   │   └── 品牌价值受损
│   └── 运营损失
│       ├── 效率降低
│       └── 员工士气影响
│
└── 数据质量投资回报论证
    │
    ├── 投入: 数据质量治理成本
    │   ├── 技术投入: 质量监控工具
    │   ├── 人力投入: 数据治理团队
    │   └── 流程投入: 质量检查流程

    ├── 产出: 质量提升收益

    │
    └── ROI计算:
        ├── 质量成本 = 预防成本 + 评估成本 + 失败成本
```
